# Game Gear 국소 사례

여기의 관측은 Game Gear 전체의 규칙이 아니다. 발동 조건이 맞을 때 조사 후보로만 사용한다.

## GG-001

- **관측 범위:** Game Gear 마도물어 1의 상점 가격 대사와 한글 렌더러.
- **사고 맥락:** 가격이 `금` 옆 자리 대신 창 하단으로 밀리고 후속 대사 순서도 함께 흐트러졌다. 가격 위치를 추적하는 커서가 틀렸다고 보았지만, 가격 채우기는 호출자가 준 고정 RAM 슬롯에 쓰고 최종 VRAM writer는 완성된 행을 복사할 뿐이었다.
- **결정 실험:** 작업 버퍼→shadow diff→VRAM 전송을 역추적하고, 한글 렌더러의 창별 타일 base 선택을 원본 초기화 테이블과 대조했다. 공유 창과 일반 창을 같은 base로 취급한 차이를 고친 뒤 상점 대사 순서와 inline 가격 표시가 함께 정상화됐다.
- **확정 결론:** 이 결함의 원인은 가격 writer가 아니라 그보다 앞선 렌더 세션의 창별 base 선택이었다. 최종 복사 지점이나 한 증상만 추적하면 공유 원인을 놓칠 수 있었다.
- **전이 한계:** 고정 가격 슬롯과 base 선택 값은 이 화면·리비전의 사실이며 다른 Game Gear UI에 이식하지 않는다.
- **관련 판단 기준:** `references/platforms/gg.md` §4, `references/strategy/debugging.md` §4.

## GG-004

- **관측 범위:** Game Gear 마도물어 3의 저장 슬롯 `日記N`과 상점 버튼 `かう`·`うる`·`でる` 압축 타일 자산.
- **선택 맥락:** 화면상 라벨은 별개였지만 저장 슬롯 네 개는 접두 타일을, 상점의 두 버튼은 오른쪽 타일 열을 물리적으로 공유했다. 각 문구를 독립 이미지처럼 다시 그리면 숫자 타일을 침범하거나 다른 버튼의 절반을 깨뜨렸다.
- **검증 근거:** 해제된 blob의 실제 타일 조립 순서를 복원했다. 저장 슬롯은 공통 `일기` 타일이 네 변형에서 byte-identical인지, 상점은 `팔기`와 `가기`가 공유하는 `기` 타일 두 개가 동일한지 빌드에서 검사하고 고유 타일만 직렬화했다.
- **확정 결과:** 번역 문구와 저장 단위가 일대일이 아닌 자산에서는 공유되는 픽셀 조각을 먼저 불변식으로 정하고, 그 조각이 같은 결과를 내는 라벨·배치만 채택해 두 UI를 제자리 교체했다.
- **전이 한계:** 특정 단어를 맞추라는 번역 규칙이 아니다. 실제 공유 타일·조립 순서가 확인되고 승인된 표현 안에서 공통 픽셀을 보존할 수 있을 때만 이 선택을 고려한다.
- **관련 판단 기준:** `references/strategy/graphics-text.md` §2·§3, `references/strategy/runtime-assets.md` §2.

## GG-005

- **관측 범위:** Game Gear 마도물어 1의 여러 포인터 테이블을 같은 spare bank로 재배치한 텍스트 빌드.
- **선택 맥락:** 저장 공간을 아끼기 위해 한 문자열의 suffix를 다른 문자열의 tail에 공유할 수 있었지만, 같은 bank 안의 서로 다른 테이블까지 하나의 공유 원장으로 묶으면 별도 라우팅되는 소비자들이 우연한 바이트 관계로 결박됐다.
- **검증 근거:** bank 공간 allocator는 함께 쓰되 suffix 후보 원장은 포인터 테이블마다 새로 만들었다. 같은 테이블 안의 공유는 유지하고 교차 테이블 후보는 차단한 빌드에서 bank 3의 세 테이블과 bank 5의 두 테이블 소비 경로를 실행해 확인했다.
- **확정 결과:** suffix 공유의 동등성 범위는 물리 bank가 아니라 base·라우팅·수명이 같은 소비자 테이블이었다.
- **전이 한계:** 테이블 간 공유를 항상 금지하는 규칙이 아니다. 참조 표현, base, 라우팅, 적재 수명과 변경 단위가 실제로 같음을 증명하면 더 넓은 공유도 가능하며, bank가 같다는 사실만으로는 충분하지 않다.
- **관련 판단 기준:** `references/strategy/reinsertion.md` §1.2·§2·§5.

## GG-006

- **관측 범위:** Game Gear 마도물어 2의 83개 번역 엔트리와 185개 포트레이트 제어 `CE`.
- **사고 맥락:** 번역 과정에서 포트레이트 선택 코드가 빠졌다. 전체 코드 수만 맞추면 같은 값도 다른 페이지나 가시 텍스트 뒤에 들어가 표정 전환 순서를 바꿀 수 있었다.
- **검증 근거:** 원문과 번역의 `wait`·`print` 경계 시퀀스가 같고 원문 제어가 모두 페이지의 가시 텍스트 앞에 있다는 모집단을 확인했다. 각 페이지 접두부에 값·순서·중복·reset을 복원하고 경계 불일치나 중간 제어는 실패시켰다. 무제어 대조군과 값 `08`·`00` 실험에서 실제 handler와 selector 상태 변화를 비교했다.
- **확정 결과:** 제어 대응의 단위는 문자열 전체의 코드 개수가 아니라 소비자가 상태를 전환하는 페이지별 접두 시퀀스였다.
- **전이 한계:** 원문과 번역의 페이지 경계가 같고 제어가 페이지 접두부에만 있다는 전수 근거가 있을 때만 성립한다. 중간 제어나 페이지 구조 변경이 있으면 대응을 추정하지 않는다.
- **관련 판단 기준:** `references/strategy/text-extraction.md` §3.1·§3.3, `references/strategy/translation-workflow.md` §4, `references/conventions/translation-artifacts.md` §3.

## GG-007

- **관측 범위:** Game Gear 마도물어 1의 JP 원문 토큰과 EN 기반 대상 엔진의 상태 문자열 테이블.
- **사고 맥락:** 번역은 JP의 `{VAR:08}`을 같은 토큰으로 보존했지만, 대상 엔진에서 인덱스 8은 `recovered`의 접미사 `vered`를 가리켜 프롤로그에 영문 조각이 표시됐다. 토큰 표기가 같다는 사실을 소비 의미가 같다는 근거로 사용한 것이 문제였다.
- **결정 실험:** 토큰 핸들러와 JP·EN 포인터 테이블을 대조해 다음 바이트가 런타임 상태가 아니라 각 엔트리에 고정된 인덱스임을 확인했다. 번역의 상태 토큰 81개를 문맥에 맞는 한글 리터럴과 조사로 바꾸고 문제 장면과 전체 토큰 잔존 검사를 통과시켰다.
- **확정 결과:** 원본 토큰의 바이트·이름을 보존하는 것과 대상 소비 의미를 보존하는 것은 다르다. 대상 엔진의 매핑이 달라졌고 값이 정적임이 확인된 이 범위에서는 리터럴 치환이 의미를 복원했다.
- **전이 한계:** 변수가 런타임 상태·이름·수치에 따라 달라지거나 대상 엔진이 원본 매핑을 유지하면 리터럴 치환할 수 없다. 토큰마다 대상 소비자와 값의 수명을 다시 확인한다.
- **관련 판단 기준:** `references/strategy/translation-workflow.md` §3.1·§4, `references/strategy/text-extraction.md` §4.4, `references/conventions/translation-artifacts.md` §3.

## GG-008

- **관측 범위:** Game Gear 마도물어 A의 region 3 문자열 테이블과 필드 money·상태 박스.
- **사고 맥락:** 추출 결과가 길이 0인 유일한 엔트리를 빈 문자열로 보고 재패킹 포인터를 `0x0000`으로 바꾸자, 게임이 부트 코드를 문자열로 읽어 정상 JP 글리프 조각을 표시했다.
- **결정 실험:** 잘못 표시되는 순간의 테이블 소비를 추적해 index 37을 특정하고 원본 테이블을 전수 대조했다. 이 엔트리의 원본 포인터만 파일 bank 밖 WRAM을 가리켰고, 런타임에 조립되는 money 문자열이 그 위치에 있었다. 원본 포인터를 보존한 두 바이트 차이와 같은 장면의 재표시로 수정을 확인했다.
- **확정 결과:** 이 범위의 길이 0은 NULL이나 빈 payload가 아니라 외부에서 생성되는 문자열의 참조였다. 재패커는 보존 가능한 원본 외부 포인터를 유지하고 새 bank에서 의미를 재현할 수 없는 빈 엔트리는 실패시켰다.
- **전이 한계:** 모든 빈 엔트리의 포인터를 보존하지 않는다. NULL, 빈 문자열, 별도 저장·런타임 생성 참조를 대상 소비자와 주소 공간에서 구분해야 한다.
- **관련 판단 기준:** `references/strategy/reinsertion.md` §1.2·§2·§6, `references/strategy/runtime-assets.md` §2, `references/conventions/data-formats.md` §4.

## GG-009

- **관측 범위:** Game Gear 마도물어 A의 두 텍스트 루프가 공유하는 글리프 확장 루틴 훅.
- **사고 맥락:** 프리픽스 인덱스를 계산하려고 `B`를 임시로 쓴 훅은 한 루프에서는 동작했지만, 다른 루프에서는 `B`가 줄당 문자 카운터였다. 복귀 뒤 카운터가 작은 프리픽스 값으로 바뀌어 루프가 언더플로우하고 화면 전송까지 진행하지 못했다.
- **결정 실험:** 공유 루틴의 호출자 두 곳을 전수 대조해 진입·복귀 레지스터 의미를 분리했다. `B`를 보존하고 각 호출자의 plane 상태를 명시적으로 정규화한 뒤 대사·필드 텍스트 경로에서 글리프 전개와 화면 전송을 확인했다.
- **확정 결과:** 공유 훅은 열거한 모든 호출자의 진입·복귀 상태를 보존한 뒤에만 두 렌더 경로를 포괄할 수 있었다.
- **전이 한계:** 공통 하위 함수를 항상 훅하거나 특정 레지스터를 항상 보존하라는 뜻이 아니다. 호출자 분모와 각 경로의 입력·출력·원본 효과가 확인된 범위에서만 공유 훅을 채택한다.
- **관련 판단 기준:** `references/strategy/reinsertion.md` §4, `references/strategy/debugging.md` §3·§5.
