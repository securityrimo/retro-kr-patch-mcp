# Saturn 국소 사례

여기의 관측은 Saturn 전체의 규칙이 아니다. 발동 조건이 맞을 때 조사 후보로만 사용한다.

## SATURN-004

- **관측 범위:** Saturn 마도물어의 SEQ 스크립트 포인터 재배치.
- **사고 맥락:** 패턴 스캐너가 `23 01 00 25 13 81`의 opcode 인자 `0x25`를 주소 prefix로 오인해 뒤의 맵 전환 코드를 `13 81`에서 `09 DD`로 바꿨고, 건물 진입이 크래시했다.
- **결정 실험:** 후보가 가리키는 주소의 범위와 워드 정렬을 검사해 홀수 타깃을 거짓양성으로 제외했다. 특정 suffix를 금지하는 우회는 실제 포인터까지 누락해 NPC 텍스트를 잃었으므로 폐기했다.
- **확정 결론:** 이 포맷에서는 prefix나 값 blacklist가 아니라 실제 타깃의 유효 범위·정렬·소비 형식으로 포인터를 판정해야 했다.
- **전이 한계:** 짝수 타깃은 이 워드 정렬 스크립트에서 확인된 조건이며 다른 포인터 포맷에 자동 적용하지 않는다.
- **관련 판단 기준:** `references/strategy/text-extraction.md` §3.5, `references/strategy/reinsertion.md` §2.

## SATURN-008

- **관측 범위:** Saturn 와쿠와쿠 뿌요뿌요 던전의 headerless `FONT16S.SPR`과 `CSMES.SPR` 그래픽 텍스트.
- **선택 맥락:** 한 파일 전체를 한 폭으로 렌더하면 일부 라벨만 정상으로 보여 잘못된 전역 geometry를 유지하기 쉬웠다. 실제 파일은 여러 폭과 padding·색상 band가 섞인 구간 구조였다.
- **검증 근거:** 폭 후보별 JP·EN·DIFF contact sheet를 만들고 diff cluster의 row 범위, 반복 패턴과 fill 경계를 함께 대조했다. `FONT16S.SPR`은 8~48px 혼합 폭 라벨로, `CSMES.SPR`은 232px·40px·248px의 세 구간으로 분리해 각 구간을 독립 패치했다.
- **확정 결과:** 한 raw 자산 전체에 하나의 폭을 가정하지 않고, 선행판 diff 후보를 반복 구조·fill 경계와 대조해 구간별 geometry를 따로 확정해야 했다.
- **전이 한계:** contact sheet와 diff는 후보 생성 수단이다. 선행판 diff는 바뀐 영역의 하한일 뿐 전체 모집단이 아니며, 최종 geometry는 반복 구조·구간 경계나 실제 소비 인자로 확인한다.
- **관련 판단 기준:** `references/strategy/graphics-text.md` §1·§2, `references/strategy/initial-survey.md` §2.

## SATURN-009

- **관측 범위:** Saturn 와쿠와쿠 뿌요뿌요 던전 `MENU1S.SPR`의 같은 배경을 공유하는 메뉴 5개와 이름 3개 4bpp 라벨.
- **선택 맥락:** 원문 없는 clean plate가 없었고 각 변형에서 글자만 달랐다. 바이트 전체의 max나 한 nibble만 보고 바이트 전체를 덮으면 두 픽셀의 선택이 결박됐고, OR는 각 픽셀의 팔레트 인덱스 비트를 합쳐 다른 색을 만들었다.
- **검증 근거:** 배경 좌표와 팔레트가 같은 변형들을 맞춘 뒤 각 픽셀 nibble을 독립적으로 비교해 어두운 글자가 사라지는 최댓값 배경을 만들었다. 새 텍스트도 비배경 nibble만 덮어쓰고 나머지는 복원 배경을 유지해 메뉴와 이름 블록에 적용했다.
- **확정 결과:** 같은 배경·좌표·팔레트를 공유하는 변형은 packed pixel을 픽셀 단위로 분리한 뒤, 각 픽셀의 색 의미에 맞는 극값을 취해 글자 없는 배경을 복원했다.
- **전이 한계:** 변형의 배경 좌표·팔레트가 같아야 하고, 모든 변형에서 글자가 겹친 픽셀은 복원되지 않는다. 팔레트 인덱스 순서가 실제 명암 순서와 다르면 max/min을 쓰지 말고 색 의미에 맞는 선택 함수를 다시 정한다.
- **관련 판단 기준:** `references/strategy/graphics-text.md` §2·§4.

## SATURN-010

- **관측 범위:** Saturn 마도물어의 MP·PT·COMMON·DIARY SEQ 텍스트 성장과 중간 구조 재배치.
- **사고 맥락:** 파일 끝을 패딩해 전체 크기를 원본과 같게 하거나 4의 배수로 맞춰도 CD 진행 멈춤과 옵션 화면 크래시가 남았다. 텍스트 엔트리 사이에는 4바이트 포인터 표·이벤트 기술자·스크립트 코드가 끼어 있었다.
- **결정 실험:** 패치 범위를 격리해 각 엔트리 뒤 누적 이동량과 다음 구조의 정렬을 비교했다. 파일 끝만 맞춘 빌드는 중간 경계 170/362개가 어긋나 멈췄고, 각 엔트리의 길이 변화량을 4의 배수로 만든 빌드는 모든 경계가 맞아 정상 진행했다.
- **확정 결과:** 필요한 지점은 최종 파일 끝이 아니라 **각 중간 소비 경계**였다. 변화량이 2바이트 어긋난 엔트리 안에 유효한 공백 타일을 넣어 이후 구조의 4바이트 정렬을 유지했다.
- **전이 한계:** Saturn이나 SEQ 전체가 항상 이 정책을 요구한다는 뜻이 아니다. 텍스트 사이의 실제 구조와 longword 소비가 확인된 범위에만 적용하고, 정렬 단위와 패딩 값은 대상 소비자에서 다시 확정한다.
- **관련 판단 기준:** `references/strategy/reinsertion.md` §1.2·§3·§6, `references/conventions/data-formats.md` §5.

## SATURN-011

- **관측 범위:** Saturn 마도물어 `MP0401.SEQ`에서 `FF02 FF3A FF3C`로 끝나는 엔트리의 길이 패딩.
- **사고 맥락:** 엔트리 정렬을 맞춘 뒤에도 해당 이벤트가 하드락했다. 알려진 종료 코드 하나의 뒤에 패딩을 붙이는 로직이 끝의 여러 `FFxx` 워드가 이루는 제어 suffix를 갈라놓고 있었다.
- **결정 실험:** 패딩 위치만 바꿔 비교하고, 바이트열 끝에서 연속된 `FFxx` 워드 전체를 역방향으로 건너뛰었다. suffix 뒤 패딩에서는 하드락이 재현됐고 첫 제어 워드 앞에 유효한 공백 타일을 넣은 빌드에서는 해소됐다.
- **확정 결과:** 마지막 종료자 하나가 아니라 **완전한 trailing control sequence 앞**을 이 엔트리의 패딩 위치로 확정했다.
- **전이 한계:** `FFxx`가 언제나 독립 2바이트 제어 토큰이라는 일반 규칙은 아니다. 인자 폭과 parser 상태가 다른 포맷에서는 같은 역방향 패턴을 쓰지 말고 완전한 token suffix를 파싱한다.
- **관련 판단 기준:** `references/strategy/reinsertion.md` §1.1·§6, `references/strategy/text-extraction.md` §3.1.

## SATURN-012

- **관측 범위:** Saturn 마도물어 COMMON·PT SEQ의 `FF00`·`FF09` 구분 기술명·결과 메시지 묶음과 하위 문자열 포인터.
- **사고 맥락:** 여러 이름을 하나의 번역 엔트리로 패치하면서 전체 길이 변화 하나만 계산하자, 앞 이름의 길이 변화가 뒤 이름을 자르거나 내부 포인터를 다른 이름 중간으로 이동시켰다.
- **검증 근거:** 원문과 번역을 같은 구분자 경계로 나누고 하위 항목 수를 대조했다. 고정 슬롯 표는 각 하위 문자열을 원래 길이에 맞춰 패딩하고, 직접 포인터가 있는 표는 각 하위 문자열의 원본 시작과 개별 이동량으로 타깃을 다시 계산했다. `FF00`과 `FF09` 사례 모두 뒤 항목 포인터의 정확한 이동을 테스트하고 실제 기술명 표시를 확인했다.
- **확정 결과:** 번역 파일의 상위 엔트리가 아니라 게임이 직접 순회·참조하는 **하위 문자열**을 길이·포인터 보정의 의미 단위로 사용해 잘림과 중간 진입을 해소했다.
- **전이 한계:** 구분자와 소비 방식은 표마다 다르며, 입증된 범위는 원문과 번역의 하위 항목 수가 같은 입력뿐이다. 수가 다르거나 경계를 가역적으로 대응할 수 없으면 이 사례로 대응 관계를 추정할 수 없으며 별도 거부·처리가 필요하다.
- **관련 판단 기준:** `references/strategy/text-extraction.md` §3.1·§3.3, `references/strategy/reinsertion.md` §1.1·§1.2·§2.

## SATURN-013

- **관측 범위:** Saturn 와쿠와쿠 뿌요뿌요 던전 `MENU.SEQ`의 일반 페이지형 고정 슬롯과 별도 suffix를 이어 그리는 기술명 엔트리.
- **사고 맥락:** 짧아진 일반 문장 뒤에 원본 tail을 남기면 제어 토큰이 중복돼 phantom page가 생겼다. 반대로 모든 짧은 문자열을 공백 타일로 채우자 게임이 별도 포인터의 `を修得` suffix를 이어 그리는 기술명 사이에 가시적 공백이 생겼다.
- **검증 근거:** 일반 페이지형은 마지막 제어 토큰 전체를 떼고 공백을 앞에 채운 뒤 제어 토큰을 다시 붙여 원본 tail을 제거했다. suffix 연결형은 표시 소비자가 `FF00`에서 멈추고 다른 문자열로 이어짐을 확인해 해당 엔트리에만 무패딩 프로필을 적용했다. MENU 전체의 비패치 영역 동일성과 인게임 표시를 대조했다.
- **확정 결과:** 같은 파일·고정 슬롯 안에서도 일반 페이지형은 **제어 앞 패딩**, 연결형은 **종료 즉시 끝나는 무패딩**으로 정책을 분리해 제어 중복과 연결 공백을 모두 제거했다.
- **전이 한계:** 공백 타일과 무패딩 어느 쪽도 파일 전체 기본값이 아니다. 종료 뒤 바이트가 정말 미소비인지, 후속 문구를 별도 참조로 합성하는지와 trailing token 전체 폭을 엔트리별로 확인한다.
- **관련 판단 기준:** `references/strategy/reinsertion.md` §1.1·§6, `references/conventions/data-formats.md` §5.
