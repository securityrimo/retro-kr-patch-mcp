# Dreamcast 국소 사례

여기의 관측은 Dreamcast 전체의 규칙이 아니다. 발동 조건이 맞을 때 조사 후보로만 사용한다.

## DC-001

- **관측 범위:** Dreamcast판 뿌요뿌요~n의 중첩 스크립트 엔트리.
- **사고 맥락:** relocation 코드가 각 씬을 `다음 씬 포인터`까지만 잘랐다. 실제로는 28개 마스터 중 18개에서 여러 진입점이 뒤쪽 대사와 포트레이트 코드를 공유했고, 앞쪽 씬은 같은 `FFFF` 종료자까지 읽어야 했다.
- **결정 실험:** 각 씬 시작점에서 실제 종료자를 찾아 다음 씬 시작보다 뒤에 놓이는지를 전수 비교했다. 이 검사로 중첩 마스터와 비중첩 마스터가 갈렸다.
- **확정 결과:** `다음 씬 포인터`는 이 데이터의 끝 경계가 아니며, 18개 마스터가 같은 종료자까지 이어지는 tail을 공유했다.
- **전이 한계:** 포인터가 단조 증가한다는 사실만으로 중첩도 비중첩도 증명되지 않는다.
- **관련 판단 기준:** `references/strategy/text-extraction.md` §1.

## DC-002

- **관측 범위:** Dreamcast판 뿌요뿌요~n의 보너스 라벨 UI.
- **사고 맥락:** 보너스 라벨은 문자열을 읽지 않고 코드에 박힌 폰트 슬롯을 순서대로 그렸다. 같은 원본 슬롯이 서로 다른 라벨에서 공유되어 텍스처만 바꾸면 한쪽 라벨을 고치는 순간 다른 쪽이 깨졌다.
- **결정 실험:** 실제 sprite 호출의 슬롯 시퀀스와 UV를 대조해 공유 슬롯 충돌을 재현하고, 새 슬롯을 고정 예약한 뒤 그 슬롯을 소비하는 라벨 참조들을 함께 바꿨다.
- **확정 결론:** 이 라벨은 텍스처와 코드의 고정 슬롯 참조를 함께 바꿔야 했다.
- **전이 한계:** 한 UI의 고정 슬롯이 다른 화면에도 공유된다고 가정하지 않는다.
- **관련 판단 기준:** `references/strategy/font-strategy.md`, `references/strategy/text-extraction.md`.

## DC-005

- **관측 범위:** Dreamcast판 뿌요뿌요~n의 음성 동반 대사창과 `F00B` 대기·`F0C9` 전환 경로.
- **사고 맥락:** 대사창의 한국어 표시 글리프 수가 줄어 후반 음성이 점점 일찍 끊겼고, 씬 블롭의 부족 바이트를 공백으로 채워 일본어 원문 크기에 가깝게 맞추는 방안이 먼저 제안됐다.
- **검증 근거:** 소비자를 디스어셈블해 `F0C9`가 다음 대사창과 음성으로 즉시 전환하고 그 뒤 바이트는 읽지 않음을 확인했다. `F00B` 대기를 50프레임에서 300프레임으로 바꾼 대조 실험과 문제 장면별 `+30`프레임 HITL에서 표시 간격과 음성 잘림이 실제로 변했다.
- **확정 결과:** 이 경로의 대사창 시간은 raw 크기가 아니라 대기 프레임과 행별 글리프 렌더 시간으로 결정됐다. 소비되지 않는 끝 패딩은 버리고, 실제로 짧아진 대사창의 `F00B` 값을 장면별로 조정했다.
- **전이 한계:** 제어코드 값, 프레임 의미와 음성 전환 방식은 이 엔진의 사실이다. 번역 길이만으로 보정량을 자동 확정하지 않으며 대사창의 모든 대기·행 구성과 실제 음성 경계를 다시 측정한다.
- **관련 판단 기준:** `references/strategy/text-extraction.md` §3.1·§3.2, `references/strategy/translation-workflow.md` §4, `references/strategy/build-and-verify.md` §4.

## DC-006

- **관측 범위:** Dreamcast판 뿌요뿌요~n의 옵션 타이틀 엔진 코드와 리터럴 풀.
- **사고 맥락:** 코드 레이아웃을 생성 쪽에서 정하도록 바꾼 뒤에도 예전 고정 주소용 포인터 쓰기가 남아, 새 레이아웃에서 같은 범위를 소유한 리터럴 풀 엔트리를 덮었다.
- **검증 근거:** 최종 바이너리의 생성 레이아웃과 수동 쓰기 범위를 대조해, 예전 포인터 주소에 이제 글자 폭 엔트리가 놓였고 뒤이은 수동 쓰기가 이를 다른 포인터 값으로 바꿈을 확인했다.
- **확정 결과:** 생성 레이아웃과 예전 고정 주소 쓰기의 중복 소유가 원인이었으며, 고정 주소 쓰기를 제거해 최종 레이아웃의 소유자를 하나로 만들었다.
- **전이 한계:** 특정 명령 집합·생성기나 리터럴 풀 배치를 기본 처방으로 삼지 않는다. 최종 산출물에서 실제 쓰기 소유권과 겹침을 다시 확인한다.
- **관련 판단 기준:** `references/strategy/build-and-verify.md` §1, `references/conventions/project-conventions.md` §5.2.

## DC-007

- **관측 범위:** Dreamcast판 뿌요뿌요~n의 데이터 트랙 파일 재배치와 오프닝 동영상.
- **사고 맥락:** 파일 사이 간격이 1,000섹터보다 크면 경계라고 본 휴리스틱이 데이터 파일 사이의 큰 간격을 잘못 분류해, 재배치 파일을 뒤쪽 대형 동영상과 겹치는 위치에 썼다.
- **검증 근거:** 루트 파일의 모든 LBA 구간과 재배치 결과를 대조해 동영상 구간 침범을 확인했다. 기존 파일의 끝을 모두 반영한 위치로 옮긴 빌드에서는 HITL로 오프닝 재생이 정상임을 확인했다.
- **확정 결과:** 이 이미지에서는 디렉터리에 기록된 모든 파일 구간을 공간 점유로 계산한 뒤에만 새 파일을 배치해 동영상 덮어쓰기를 제거했다.
- **전이 한계:** 모든 파일의 최대 끝을 보편적인 할당 전략으로 이식하지 않는다. 대상 이미지의 트랙·볼륨 경계, 숨은 점유 영역, 디렉터리 완전성과 필요한 연속 공간을 다시 확정한다.
- **관련 판단 기준:** `references/strategy/reinsertion.md` §5, `references/strategy/build-and-verify.md` §2·§4.
