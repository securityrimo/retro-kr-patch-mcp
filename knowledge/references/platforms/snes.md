# SNES 한글 패치 플랫폼 레퍼런스

SNES(Super Famicom) 게임의 한글 패치 제작에 필요한 플랫폼 사실과 선행 사례 노트를 정리한다.
수치와 패턴은 대상 ROM의 헤더·매퍼·렌더 경로를 실측한 뒤 적용한다.

---

## 1. 플랫폼 개요

| 항목 | 값 |
|------|-----|
| CPU | 65816 (16비트, 65C02 상위 호환) |
| 클럭 | SlowROM 2.68MHz / FastROM 3.58MHz |
| 주소 공간 | 24비트 (뱅크 $00-$FF × 64KB) |
| WRAM | 128KB (Bank $7E:$0000 - $7F:$FFFF) |
| VRAM | 64KB (워드 단위 주소 $0000-$7FFF) |
| OAM | 544바이트 (128 스프라이트 × 4B + HI 테이블 32B) |
| CGRAM | 512바이트 (256색 팔레트) |
| PPU | BG 모드 0-7, 모드 1(4bpp+4bpp+2bpp)이 텍스트 게임의 주류 |

### 65816 요점

- **M/X 플래그**: `REP #$20`으로 A를 16비트, `SEP #$20`으로 8비트 전환. X/Y는 `$10` 비트. `LDA #imm`의 바이트 수가 M 플래그에 따라 2B/3B로 달라지므로 훅 코드에서 진입 시점의 플래그 상태를 반드시 확인한다.
- **뱅크 레지스터**: PBR(프로그램 뱅크), DBR(데이터 뱅크), D(다이렉트 페이지). 텍스트 엔진은 `PHA/PLB`로 DBR을 텍스트 뱅크로 바꾼 뒤 `LDA $0000,Y`로 읽는 패턴이 흔하다.
- **뱅크 간 이동**: `JSL` (`$22 LL HH BB`, 4B) / `JML` (`$5C LL HH BB`, 4B) / 복귀는 `RTL` (`$6B`). 훅 설치는 원본 코드 4바이트를 JSL/JML로 교체하는 것이 기본형이다.
- **블록 전송**: `MVN sb,db` (`$54 db sb`, 3B)로 ROM→WRAM 대량 복사. 폰트 스테이징에 사용한다. 겹치는 영역을 역방향으로 복사할 때는 `MVP`(`$44`, 감소 방향)를 쓴다. 두 명령 모두 X=소스·Y=목적·A=길이−1(A+1바이트 복사)이라 길이 한 칸 off-by-one에 주의한다.
- **주의 명령**: 원본 4바이트에 상대 분기(BEQ/BNE)가 포함되면 훅 이전 시 오프셋을 재계산해야 한다.

### 주요 하드웨어 레지스터

| 레지스터 | 용도 |
|---------|------|
| $2100 INIDISP | Force Blank (bit7=1) / 밝기 |
| $2115 VMAIN | VRAM 증가 모드. 통상 `$80` ($2119 쓰기 후 +1워드) |
| $2116/17 VMADDL/H | VRAM 워드 주소 |
| $2118/19 VMDATAL/H | VRAM 데이터 쓰기 |
| $2180-$2183 | WRAM 데이터/주소 ($2183 bit0=1이면 Bank $7F) |
| $43x0-$43x6 | DMA 채널 x 설정 (모드/B버스/소스/크기) |
| $420B MDMAEN | DMA 실행 (bit=채널) |

**VRAM DMA는 Force Blank 또는 VBlank 중에만 허용된다.** 그 외 시점에 전송하면 VRAM이 깨진다. WRAM DMA는 시점 제약이 없다. VRAM 전송은 `DMAPx=$01`(A→B, $2118+$2119 2레지스터), WRAM 전송은 `DMAPx=$00`($2180 단일 레지스터)이다.

## 2. 메모리 맵과 ROM 매핑

SNES ROM은 LoROM, HiROM, ExLoROM/ExHiROM 등 매핑 모드가 다르다. 먼저 내부 헤더와 알려진 덤프 메타데이터로 대상 ROM의 매핑을 확정한다. 아래 공식은 선행 사례 게임들이 사용한 **헤더리스 일반 LoROM** 기준이며, CPU 주소의 `addr`가 $8000-$FFFF일 때만 적용한다. LoROM에서는 각 뱅크의 $8000-$FFFF 절반(32KB)에 ROM이 매핑된다.

```
# SNES 주소 → 파일 오프셋 (헤더 없는 ROM 기준)
PC = (bank & 0x7F) * 0x8000 + (addr - 0x8000)

# 파일 오프셋 → SNES 주소
bank = PC // 0x8000
addr = 0x8000 + (PC % 0x8000)

# 예: $29:$93FB → 0x29 * 0x8000 + (0x93FB - 0x8000) = 0x1493FB
```

수동 계산 시 `- 0x8000` 누락이 가장 흔한 실수다. 변환 함수를 하나 두고(단일 진실 원천, SSOT) 전 코드가 그것만 쓰게 한다.

| Bank | PC 오프셋 보정 | 예시 |
|------|---------------|------|
| $00 | addr - 0x8000 | $CCA3 → 0x4CA3 |
| $01 | addr 그대로 | $8000 → 0x8000 |
| $0F | addr + 0x70000 | $8000 → 0x78000 |
| $32 | addr + 0x188000 | $D4B1 → 0x1954B1 |

- ROM 헤더 위치는 매핑별로 다르다. LoROM은 $7FC0, HiROM은 $FFC0 후보를 확인한다. $xFD5=매핑 모드, $xFD7=ROM 크기, 체크섬은 $xFDC/$xFDE이며 합이 0xFFFF여야 유효하다.
- 벡터 테이블(Reset/NMI/IRQ)은 Bank $00 끝에 있다. NMI 핸들러가 DMA 큐 실행을 담당하는 구조가 흔하다.
- $80-$BF의 LoROM 주소는 $00-$3F와 같은 ROM 데이터가 보이는 미러일 수 있지만, 이것과 FastROM 접근 가능성은 별도 개념이다. FastROM은 메모리 속도/헤더 설정 문제이고, 주소 미러는 매핑 모드 문제다. 특히 $80-$BF와 $C0-$FF 구간의 의미는 LoROM, HiROM, ExHiROM, ExLoROM, 확장 ROM에서 달라질 수 있으므로 `& 0x7F`를 전역 규칙으로 쓰지 말고 대상 ROM의 매퍼별 변환 함수로 처리한다.
- **빈 공간 조사**: 0xFF/0x00으로 채워진 빈 뱅크나 긴 빈 영역이 있을 수 있지만, 크기와 위치는 ROM마다 다르다. 한글 폰트·훅 코드·재배치 텍스트는 실측한 빈 영역이나 확장 영역에 배치한다. 빈 공간에 쓰기 전 대상 영역이 실제로 비어 있고 참조되지 않는지 검증하도록 강제하면 충돌 사고를 빌드 타임에 잡는다.

## 3. 비디오 시스템과 폰트 제약

### 타일 포맷

| 포맷 | 크기 | 용도 |
|------|------|------|
| 2bpp 8×8 | 16B/타일 | BG3 텍스트 레이어 (4색) |
| 4bpp 8×8 | 32B/타일 | BG1/BG2, OBJ 스프라이트 (16색) |
| 16×16 글리프 | 8×8 4장 조합 | 일본어 텍스트 게임에서 흔한 본문 구성 |

- SNES 표준 2bpp는 **행 단위 인터리브**다: byte0=row0_bp0, byte1=row0_bp1, byte2=row1_bp0 …. 4bpp는 bp0/bp1 인터리브 16B 뒤에 bp2/bp3 인터리브 16B가 이어진다.
- 일부 게임은 WRAM에 **플레이너 2bpp**(bp0 8바이트 + bp1 8바이트 분리)로 폰트를 보관하고 DMA 루틴이 변환한다. 포맷을 추측하지 말고 참조 글리프를 양쪽 디코더로 렌더링해 시각 비교로 확정한다 (한 사례에서 플레이너로 잘못 주입해 글리프 전체가 깨졌다).
- 16×16 글리프의 4분할(TL/TR/BL/BR) 배치도 게임마다 다르다. 연속 64B인 게임(16×16 2bpp = 64B/글리프)과 stride=256(TL+0, TR+16, BL+256, BR+272)인 게임이 모두 실존한다. DMA 루프 디스어셈블리로 stride를 확정한다.
- 본문 폰트가 2bpp 타일의 한 비트플레인만 쓰는 구성(텍스트=bp0, 배경=bp1)도 흔하다. 이 경우 텍스트 비트플레인(짝수 바이트)만 0으로 지우면 배경을 보존한 채 글자를 클리어할 수 있다(평면 분리 클리어, §4의 VRAM 클리어 훅의 한 변형). 가변폭 영문 폰트는 글자별 폭 테이블·커닝을 별도로 운용하기도 한다.

### VRAM 운용

- VRAM 주소는 **워드 단위**다. 바이트 주소의 절반이다 (byte $A000 = word $5000).
- 텍스트 CHR 영역은 게임이 정한 위치(예: word $5400-$57D0)에 글자 단위로 DMA된다. 상단 행과 하단 행이 +$0080 워드 간격으로 분리되는 식의 고정 격자를 가진다.
- 화면 설정에 따라 빈 VRAM 영역이 생긴다(한 사례: word $5800-$5FFF, 2048워드 = 2bpp 8×8 타일 256개 = 16×16 글자 64개). 동시 표시 글자 수가 적으면 이 풀에 필요한 글리프만 동적 로드하는 전략이 가능하다.

### OBJ VRAM / 스프라이트 텍스트 패칭

메뉴 타이틀·말풍선처럼 스프라이트로 그려지는 텍스트는 BG 텍스트 엔진을 거치지 않는다. 확립된 패턴:

- OBJ VRAM이 초기화 시점에 1회성 DMA로 채워지고 이후 갱신되지 않는 화면이라면, **Force Blank 상태에서 ROM 타일을 VRAM OBJ 영역에 직접 DMA**(전용 채널 사용, 예: Channel 6)하면 영구 반영된다.
- OAM은 스프라이트 위치·타일 번호·속성 테이블이고, 글리프 픽셀 데이터는 OBJ VRAM에 있다. OAM 패치는 어떤 OBJ 타일을 어느 위치에 그릴지 바꾸는 작업이고, 글리프 교체는 OBJ VRAM 타일 데이터를 바꾸는 작업이다.
- 화면별로 훅 지점이 다르다. 장비 화면은 메뉴 공통 초기화 지점에, 상점 화면은 LZ 콜백 체인 내부에 훅을 걸어 각각 타일·코드 DMA 그룹을 주입하는 식이다.
- 범용 콜백 체인에 훅을 걸 때는 **dp 레지스터 2바이트 조합 가드**(소스 뱅크 + 주소 하위 바이트)로 발화 문맥을 유일하게 좁힌다. 1바이트 가드는 다른 화면에서 오발화한다. 가드는 디컴프레서 호출 **이전**에 배치한다(디컴프레서가 dp를 덮어쓴다).
- 글리프가 스프라이트 타일 경계에 걸치면 composite 캔버스(예: 80×16)에 먼저 그린 뒤 타일 단위로 분할해 주입한다.

### 16×16 본문 + 8×8 UI 혼합 렌더링

일본어 원문이 16×16 본문 폰트를 쓰는 게임에서는 한글도 16×16 경로를 우선 검토한다. 8×8 한글 본문은 픽셀 예산이 빡빡하므로 대표 문장 비교와 실제 화면 표시로 가독성을 증명해야 한다.
반면 옵션·스탯·세이브·월드맵 같은 UI 화면은 8×8 타일맵 기반이므로 별도 처리가 필요하다.

| 구분 | 폰트 | 처리 방식 |
|------|------|----------|
| 본문 대화/전투 | 16×16 (TTF 12px 렌더링) | 텍스트 엔진 훅 + ROM 폰트 뱅크 교체 |
| UI 화면 (옵션/스탯/월드맵) | 8×8 전용 픽셀 폰트 (8px) | LZ 인터셉트 후 WRAM 디컴프 결과에 KO 타일 MVN 덮어쓰기 |
| OBJ 스프라이트 텍스트 | 8px 또는 composite | Force Blank ROM→VRAM 직접 DMA |

LZ 인터셉트 패턴: 게임의 LZ 디컴프레서 호출(`JSL` 사이트)을 커스텀 훅으로 교체 → 원본 LZ를 그대로 호출 → 디컴프 결과 WRAM 위에 한글 CHR/타일맵을 MVN으로 덮어쓰기 → 게임이 수정된 WRAM을 VRAM으로 DMA. 게임 코드는 변화를 알아채지 못한다.

## 4. 텍스트 엔진 패턴

### 엔진 구조 파악

SNES 텍스트 엔진에서 자주 보이는 구조는 다음과 같다.

```
텍스트 포인터 로드 → 바이트(또는 워드) 페치 → 제어코드 임계값 비교
  ├─ 제어코드: 점프 테이블 → 핸들러 (줄바꿈/페이지/박스/종료)
  └─ 일반 문자: 타일 인덱스 → DMA 소스 주소 계산 → VRAM 전송
```

훅 설계 전에 전수 조사로 확정할 것:
- 텍스트 렌더러 진입점이 몇 개인가 (단일 렌더러로 모든 텍스트가 수렴하는 게임도 있고 호출자가 여러 곳으로 흩어진 게임도 있다 — 모든 진입·호출 지점을 빠짐없이 찾는다)
- 텍스트 위치 카운터를 증가시키는 지점이 몇 곳인가 (`INC` 사이트가 여러 곳에 흩어져 있으면 프리픽스 핸들러를 추가할 때 여기에 합산된다)
- 제어코드 테이블 위치와 엔트리 (예: $F8=페이지, $F9=줄바꿈, $FB=한자 프리픽스, $FC=텍스트박스, $FF=종료)

### 훅 위치 선정 기준

1. **디스패치 지점**: 문자 바이트가 제어코드 분기 직전에 모이는 단일 지점에 `JML`을 설치한다. 한글 확장 프리픽스($F0/$F1 등)를 여기서 감지해 커스텀 핸들러로 분기하고, 일반 문자는 원본 경로로 복귀시킨다.
2. **뱅크 결정 지점**: 렌더러가 DMA 소스 뱅크를 하드코딩하는 지점(예: `LDA #$0F`)에 훅을 걸어, 뱅크 플래그 변수에 따라 확장 폰트 뱅크($32 등)로 오버라이드한다. 플래그는 사용 직후 클리어한다(잔류 시 경합 버그).
3. **원본 바이트 길이**: JSL/JML은 4바이트이므로 교체 대상 원본도 4바이트 경계로 잡는다. 3바이트 패턴이면 다음 1바이트(INX 등)까지 포함해 훅 코드에서 복원한다.
4. **PPU 레지스터 경계**: 임의의 텍스트 훅 안에서 PPU 레지스터(Color Math, Window 등)를 즉흥적으로 쓰지 않는다(투명/표시 버그 유발). 단 검증된 VBlank/NMI/Force Blank/DMA 큐 컨텍스트에서는 PPU 레지스터 설정과 VRAM/OAM/CGRAM 전송 자체가 정상적인 해결책이다. 정체불명 루틴(스프라이트 로더 등)에 무분별하게 훅을 걸지 않는다.

### 텍스트 블록 VRAM 클리어 훅

한글은 글자 폭과 줄바꿈이 원문과 달라 짧은 문장 뒤에 이전 텍스트 타일이 VRAM에 잔류할 수 있다. 줄마다 공백 패딩으로 덮는 방식이 불완전하면 **엔진 레벨 클리어 훅**을 검토한다: 텍스트 블록 초기화 루틴에 훅을 설치해 텍스트 VRAM 영역을 매번 풀 클리어한다. 클리어 시작 행을 한 줄 늦게 잡으면 첫 행 잔류가 남을 수 있으므로 영역 경계를 보수적으로 잡고, 실제 BLANK 타일을 VRAM에 기록하는 공백 인코딩을 함께 검증한다.

### 사례별 훅 구성

| 사례 | 훅 구성 |
|------|---------|
| SNES 나조뿌요 1(룰루의 루) | 텍스트 엔진 진입 JSL 훅 + VMAIN 설정 훅 + 인라인 패치(렌더러 범위 체크, JP 폰트 블록 테이블 비활성). 폰트는 WRAM 로딩 훅에서 MVN으로 스테이징 |
| SNES 마도물어 | 텍스트 엔진 훅 여러 개 + 텍스트 블록 VRAM 클리어 훅. 그 외 LZ 인터셉트(세이브/옵션/월드맵), OAM DMA(장비/상점), 인라인 바이트 패치(선택지 하이라이트) |
| SNES 나조뿌요 2(룰루의 철완번성기) | 렌더러 진입점이 2개($96CC 2바이트 문자 / $96D3 1바이트 문자)라 양쪽에 한글 감지 훅이 필요. 원본 `A9 01 EB E8` 4바이트를 JSL로 교체 |

핵심 디스패치, 뱅크 오버라이드, 프리픽스 핸들러, 블록 클리어에서 시작하고 UI 화면을 한글화할수록 LZ 인터셉트·OAM 훅이 추가된다. 필요한 훅 집합은 렌더 경로 수와 UI 경로 분리에 따라 게임마다 달라진다.

### 훅 코드 배치

- 훅 본체는 빈 뱅크의 프리 스페이스에 둔다. 폰트 타일 수에 따라 주소가 밀리므로, 하드코딩 상수 대신 **파이프라인 체인**(각 단계가 end 주소를 반환 → 다음 단계의 start)으로 동적 배치한다.
- ASM은 헥스 직접 코딩 대신 명령어 단위 빌더(분기 라벨의 오프셋을 자동 계산하는 어셈블러)로 생성하면 분기 오프셋 실수를 차단한다.
- 모든 ROM 쓰기는 기록 영역을 추적하는 단일 경로를 통과시키고, 설치 지점은 원본 바이트 일치, 프리 스페이스는 영역 전체가 0xFF인지를 쓰기 전에 검증하도록 강제한다.

## 5. 한글 운용 수치

### 완성형 글리프 규모

번역 전량에 등장하는 한글 음절 수는 번역 코퍼스 집계의 출력(수요)이며 게임마다 다르다. 선행 SNES 사례들은 대략 수백 자 규모였지만, 이를 수용 상한으로 쓰지 않는다. 수용 상한은 「단일바이트 잔여 코드 수 + prefix당 최대 256자 × 확보한 빈 뱅크 용량(글리프당 바이트로 환산)」처럼 게임별 입력으로 별도 산정한다(`references/strategy/font-strategy.md` 3절). 인코딩 단위 자체도 게임이 정한다. 아래처럼 단일바이트+프리픽스를 쓰는 게임도 있고, 글자 하나를 16비트 워드 코드포인트로 저장해 종결자·제어코드의 바이트 폭까지 달라지는 게임도 있으니, 자기 게임의 인코딩 단위를 먼저 실측한다. 아래는 단일바이트+프리픽스 배치의 한 사례다:

| 인코딩 | 뱅크/주소 | 용도 |
|--------|----------|---------|
| FIXED ($00-$1F) | $0F:$8000-$87FF | 숫자·구두점·화살표·괄호·BLANK |
| 단일바이트 ($20-$EF) | $0F:$8800-$BBFF | 한글 주요 음절 |
| FB 프리픽스 | $0F:$C000-$FFFF | 확장 영역 |
| F1 프리픽스 | $32:$8000-$BFFF | 추가 확장 영역 |
| F0 프리픽스 | $32:$C000-$D4FF | 잔여 음절 |

기존 폰트 뱅크를 전부 교체하고 모자라는 분량은 빈 뱅크에 프리픽스 인코딩으로 확장하는 구조다. 인코딩 테이블에 없는 문자는 **빌드 에러**로 처리한다 — 처리 없이 넘어가면 미패치 일본어 바이트가 한글 엔진 훅을 통과하며 프리픽스 오감지로 화면이 깨진다(실측 사례).

### 폰트 크기와 TTF 렌더링

| 용도 | 글리프 크기 | TTF 권장 |
|------|------------|---------|
| 본문 16×16 | 16×16 2bpp (연속 64B인 사례 기준. 배치는 게임별 상이, §3 참조) | 픽셀 폰트 계열 TTF, **12px** (외곽선 렌더링 포함 시) 또는 16px (외곽선 없음) |
| 세이브/메뉴 16×16 | 동일 | 13px |
| UI 8×8 | 8×8 2bpp (16B/타일) | 8×8 전용 픽셀 폰트, **8px** |

- TTF 래스터라이저는 아웃라인 폰트만 지원하는 경우가 있다(사례: fontdue). 비트맵 임베드 TTF는 빈 글리프를 반환할 수 있으므로 아웃라인화된 픽셀 폰트를 쓴다.
- 글리프는 bbox 기준 센터링으로 16×16 캔버스에 배치한다.
- 본문과 UI 타이틀의 폰트를 통일하면(같은 TTF 패밀리) 화면 간 위화감이 없다.
- 4bpp 화면(BG1/OBJ)에 2bpp 폰트를 넣을 때는 bp2/bp3이 미제어 상태가 되므로 palette index 0(투명)이 보장되도록 비트플레인을 명시적으로 채운다. JP 원본이 BP1=$FF(투명 없음)로 운용하는 화면은 한글도 동일하게 맞춘다.

### 텍스트 박스 폭

모든 문자가 16px(2타일) 폭이면 계산이 단순하다: 박스 폭 20타일이면 한 줄 10글자, 30타일이면 15글자다. 페이지당 줄 수와 박스 폭은 게임별로 다르므로 레이아웃 시뮬레이터를 만들어 확정한다.

## 6. 포인터 체계와 재배치

한글 번역은 일본어 원문보다 바이트가 늘어나는 경우가 많아, 인플레이스 교체로 수용 불가한 문자열은 빈 뱅크로 재배치하고 포인터를 다시 쓴다.

### 3바이트/2바이트 하이브리드

| 타입 | 구조 | 특성 |
|------|------|------|
| 3바이트 포인터 | (addr_lo, addr_hi, bank) | 뱅크 간 참조 가능. 재배치의 주력 |
| 2바이트 포인터 테이블 | 연속 테이블, addr만 | 같은 뱅크 내부만 가리킴. 재배치 시 뱅크 내 free space 필요 |
| 코드 내장 2바이트 | `LDA #$XXXX` 등 코드에 산재 | 디스어셈블리로 개별 발굴 |

세 타입이 한 게임에 섞여 있을 수 있다. 각 타입을 빠짐없이 카탈로그화하고, 인플레이스로 수용 가능한 것과 재배치가 필요한 것을 나눠 처리한다.

### 포인터 카탈로그 원칙

- 포인터 목록은 추측 스캔이 아니라 역공학 기반 **정밀 카탈로그**로 관리한다. 누락 1개가 곧 크래시다.
- **서브포인터**: 포인터가 문자열 시작이 아니라 중간(+1~+8)을 가리킬 수 있다.
- **팬텀 포인터**: 카탈로그에는 있지만 번역 단위에 대응하지 않는, 텍스트 체인 중간으로 직접 점프하는 포인터(조건 분기 대화 등). 제어코드 인덱스 카운팅(원문에서 N번째 박스 태그 → 한글에서 같은 인덱스 위치)으로 새 위치를 계산하고, 매핑 불가면 해당 원문을 원위치에 보존한다.
- **2-pass 할당**: free space가 부족한 뱅크는 cross-bank 재배치 먼저 → 해방된 텍스트 영역을 뱅크 내 재배치에 재활용한다.
- 줄 폭 초과는 포인터 재배치와 별개의 레이아웃 문제다. 먼저 창 크기·제어코드·페이지 분할·동적 width/height 계산·VWF/spacing 가능성을 확인하고, 그래도 시스템 제약이 닫히면 마지막 단계에서 번역 축약이나 줄바꿈 병합으로 해결한다.

## 7. 패치 포맷과 빌드

- 배포 포맷은 **BPS를 1순위**로 한다. BPS는 원본·결과 CRC32를 내장해 리비전이 다른 ROM·헤더 유무가 다른 ROM에 적용하는 사고를 적용 단계에서 차단한다. SNES 씬에서 역사적으로 널리 쓰인 IPS는 차선이며, 호환성 목적으로 병행 배포할 수 있다.
- IPS 구조: `PATCH`(5B) 헤더 + 레코드([오프셋 3B big-endian][크기 2B][데이터], 크기 0이면 RLE 레코드) + `EOF`(3B). 오프셋이 24비트라 16MB까지 표현 가능하며 SNES ROM(보통 1-4MB)에는 충분하다. 단 원본 검증이 없어 잘못된 ROM에 적용해도 에러 없이 망가진다.
- **헤더 없는 ROM**(.sfc, 512B 카피어 헤더 제거본) 기준으로 생성·적용한다. 헤더 유무가 섞이면 전 오프셋이 512B 밀린다.
- 빌드 파이프라인은 "JP ROM + 폰트 + 번역 + 훅 + 재배치 → 패치 ROM → verify → 패치 파일 생성" 순서로 단방향 구성한다. 패치 파일은 부산물이고 진실 원천은 빌드 스크립트다.
- ROM 파일 자체는 저작권상 저장소에 커밋하지 않는다. 배포물은 차분 패치 파일만이다.

## 8. 검증

### 에뮬레이터 선택

프로그래밍 디버깅의 요건은 Lua 스크립팅·headless 실행·메모리 콜백이다. 조사 시점 기준으로 정확도는 bsnes 계열이 가장 높고, 디버깅 도구는 이 요건을 갖춘 Mesen2가 가장 실용적이다. Mesen2는 GDB 서버를 지원하지 않으므로 **Lua 스크립팅을 주 프로그래밍 디버깅 수단으로 둔다**(역시 조사 시점 기준).

### Mesen2 Lua API 요점

- 실행: GUI(Script Window) 또는 headless `Mesen --testrunner script.lua rom.sfc` (스크립트에서 `emu.stop(0)`=PASS / `emu.stop(1)`=FAIL → CI 통합 가능).
- **모든 `addMemoryCallback`에 `emu.cpuType.snes` + memType을 명시**한다. 생략하면 NES 기본값이 적용되어 콜백이 영원히 발화하지 않는다.

```lua
-- ROM 실행 브레이크포인트: snesPrgRom은 PC 오프셋 기준
-- $04:$AE74 → (4 & 0x7F)*0x8000 + 0xAE74 - 0x8000 = 0x22E74
emu.addMemoryCallback(function(addr, val)
    emu.log("HIT $04:AE74")
end, emu.callbackType.exec, 0x22E74, 0x22E74,
    emu.cpuType.snes, emu.memType.snesPrgRom)
```

| memType | 용도 | 주의 |
|---------|------|------|
| snesMemory | CPU 버스 (WRAM/ROM/IO) | $7FFFFE 같은 24비트 평면 주소 |
| snesRegister | PPU/APU 레지스터 ($2100-$44FF) | |
| snesPrgRom | ROM (PC 오프셋) | exec 브레이크포인트에 사용 |
| snesVideoRam | VRAM | **워드 주소** (바이트÷2) |
| snesDebug | 사이드이펙트 없는 읽기 | 레지스터 read 감시 시 |

- 읽기/쓰기는 `emu.read`/`emu.readWord`만 존재한다 (`read16`/`debugRead` 없음).
- 콜백 내부의 `emu.getState().cpu.pc/k`는 nil이거나 0인 경우가 많다. `pcall` 래핑이 필수이며, 정확한 caller PC가 필요하면 GUI 디버거의 Write Breakpoint를 쓴다.
- 활용 패턴: 뱅크 플래그 변수 Write 감시(경합 조건 검출), VRAM 특정 영역 Write 감시(잔류/오염 검출), PPU 레지스터 변화 감지(이전 값과 비교해 노이즈 제거), 훅 진입점 exec 카운팅(발화 횟수 확인).

### 정적/오프라인 검증

- **65816 트레이서**: 에뮬레이터 없이 ROM을 명령 단위로 실행해 특정 VRAM 주소로의 DMA를 감지하는 자체 추적기가 폰트 로딩 경로 분석에 유효하다(목표 VRAM 주소로 필터링).
- **선형 디스어셈블러**: 훅 설치 전후 바이트를 니모닉으로 확인한다.
- 바이트 패치 검증은 SNES 주소→PC 변환 후 hexdump 비교가 기본이다.

### 텍스트박스 오버플로우 QA

오버플로우는 두 종류다. (1) 줄 폭 초과 → 자동 줄바꿈(의도치 않은 줄 수 증가 유발), (2) **페이지 줄 수 초과 → 화면 밖 렌더링(실제 사고)**.

- 박스 치수(폭 타일 수, 페이지당 줄 수)를 엔진에서 확정하고, 번역 전량을 대상으로 **레이아웃 시뮬레이터 기반 전수 검사** 커맨드를 만든다. 패치 ROM을 입력으로 뱅크·주소·폭·줄 수를 리포트한다.
- 규칙 예(폭 20타일·3줄 게임): 한 줄 10글자, `{NL}`는 최대 2개, 초과분은 `{PAGE}`로 분할. 페이지/구분자/박스 태그가 줄 카운터를 리셋하는지는 엔진별로 확인한다.
- 이 검사는 빌드마다 돌려 오버플로우를 0건으로 수렴시키는 게이트로 운용한다. 초기에 다수 검출되면 먼저 창 크기·제어코드·페이지 분할·동적 width/height 계산·VWF/spacing 가능성을 확인하고, 닫힌 제약으로 남은 항목만 번역 축약·줄바꿈 병합으로 줄여 나간다.
- 길이 고정 슬롯(도감 이름·전투 이름·아이템명 등 게임이 바이트 폭을 박아 둔 필드)은 초과 시 빌드 에러로 처리한다. 슬롯별 정확한 폭은 게임에서 실측한다.
- 대사창 크기가 스크립트 파라미터(원문 글자 수 기준)로 박혀 있는 게임은, 훅에서 한글 텍스트를 런타임 스캔해 width/height를 동적 재계산하고 화면 우측 경계 클램핑을 적용한다. 예를 들어 폭 32타일 화면의 특정 대사창에서는 `right_edge > 30이면 좌측 이동` 같은 게임별 타일 좌표 경계가 나올 수 있으나, 이 값은 SFC 일반 규칙이 아니라 해당 화면의 실측값이다.

### HITL(사람 확인) 체크포인트

자동 검증 후 사람이 에뮬레이터로 확인할 최소 체크리스트: 타이틀 메뉴 한글 표시 → 새 게임 첫 대사 → 아이템/마법 설명 → 확장 프리픽스 글자 렌더링 → 세이브 메뉴 → 전투 텍스트 → 화면 전환 후 VRAM 잔류 여부.

### 실기 구동 경로

- **수단**: 플래시 카트로 검증한다. FXPak Pro 같은 고급 카트는 특수 칩 FPGA 재현과 MSU-1을 지원해 검증 범위가 넓다. 보급형 플래시 카트는 특수 칩 게임을 미지원할 수 있으므로 대상 게임의 칩 요구사항을 먼저 확인한다.
- **제약**: FXPak Pro의 ROM 로드 한계는 Krikzz 공식 제품 페이지 기준 128Mbit로 표기되며 실제 구현은 96Mbit(12MB) 수준이다(https://krikzz.com/our-products/cartridges/fxpak-pro.html). 펌웨어 버전별로 상향 가능하니 사용 펌웨어에서 확인 필요하다. ExHiROM 매핑을 지원한다.
- **빌드 포맷에의 영향**: ROM을 4MB에서 6MB ExHiROM으로 확장해도 카트 로드 한계 안이면 무방하나, 내부 헤더의 매핑 모드를 정확히 기재해야 실기가 올바로 해석한다(7절 빌드의 체크섬·헤더 갱신 단계에서 함께 처리). 더 흔한 사고는 **헤더 유무**다 — 512바이트 카피어 헤더가 붙은 `.smc`와 헤더리스 `.sfc`가 섞이면 전 오프셋이 512바이트 어긋난다. 7절에서 정한 헤더리스(No-Intro) 기준을 실기 카트에 올릴 ROM까지 일관되게 유지하라. 패치 단계에서 어긋나면 적용이 실패한다.
- **함정**: 부정확한 에뮬레이터를 기준으로 짠 핵이 정확한 실기에서 안 도는 사례가 있다. MSU-1을 쓰면 순정 카트만으로는 구동할 수 없어 플래시 카트 전용이 되므로, 순정 카트 호환을 유지하려면 도입하지 않는다.
- **중간 검증**: 에뮬레이터와 실기 사이의 중간 검증으로 사이클 정확 FPGA 재구현(SNES 코어)을 쓸 수 있다. 정확도가 높아 부정확한 에뮬레이터를 전제로 만든 패치는 여기서 걸러진다.

## 9. 사례 요약

| 사례 | 텍스트 시스템 | 한글화 접근 |
|------|--------------|------------|
| **SNES 마도물어** | 단일 16×16 텍스트 엔진, 커스텀 타일 인덱스 인코딩, 텍스트 비압축 ROM 직접 읽기 | 폰트 뱅크 전체 교체 + F1/F0 프리픽스 확장, 텍스트 엔진 훅 + 텍스트 블록 클리어, LZ 인터셉트 + OAM DMA로 UI/스프라이트까지 한글화, 포인터 전수 리라이트 |
| **SNES 나조뿌요 1(룰루의 루)** | 2바이트 VRAM 타일맵 쌍 인코딩, LZ 압축 폰트를 WRAM $7F에 디컴프 후 타일 캐시+DMA | 엔진 진입 훅 + VMAIN 훅 + 인라인 패치, WRAM 스테이징(인터리브 2bpp), 4bpp 화면용 투명 페이지 인코딩, 스프라이트 DMA와의 WRAM 충돌 영역 회피가 핵심 제약 |
| **SNES 나조뿌요 2(룰루의 철완번성기)** | ASCII 인코딩 + 16×16 2bpp(2×2 타일), Bank 단위 폰트 ROM에서 문자당 DMA 2회, NMI DMA 큐 | 렌더러 진입점 2개 훅($96CC/$96D3), 폰트 소스 뱅크를 한글 뱅크로 교체, 빈 뱅크 활용. 자매작과 인코딩이 전혀 달라 코덱 재사용 불가 |

같은 플랫폼·같은 개발사라도 텍스트 인코딩(타일 인덱스 / 타일맵 쌍 / ASCII), 폰트 경로(ROM 직접 DMA / WRAM 스테이징), 타일 배치(연속 64B / stride 256)가 전부 다르다. 역공학으로 확정하기 전에는 어떤 가정도 이식하지 않는다.
