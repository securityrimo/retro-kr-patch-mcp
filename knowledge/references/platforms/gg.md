# 게임기어 (Sega Game Gear) 한글 패치 참조

## 1. 플랫폼 개요

게임기어의 CPU는 Zilog Z80A 호환이며 클럭은 3.58MHz(정확히는 3.579545MHz)다. 데이터 버스 8비트, 주소 버스 16비트로 CPU가 직접 보는 주소 공간은 64KB뿐이다. 리틀엔디언이므로 헥스 에디터에서 `00 81`로 보이는 포인터는 `$8100`이다. 16비트 포인터는 `HL`에 싣고 `(HL)`로 역참조하는 코드가 표준 관용구다.

화면 출력은 TMS9918A 계열에서 파생된 세가 VDP가 담당하며, 상용 게임기어/마스터시스템 게임의 일반적인 분석 대상은 **Mode 4**다. VRAM 16KB, 8×8 픽셀 4bpp 타일, 팔레트 2개×16색. 게임기어 팔레트는 색당 2바이트(채널당 4비트, 4096색)로 마스터시스템(색당 1바이트, 64색)과 다르다.

게임기어 특유의 제약은 **가시 영역**이다. 내부 네임테이블은 마스터시스템과 동일한 32×28타일이지만, LCD에는 중앙 160×144픽셀 = **20×18타일**만 표시된다. 스크롤 레지스터가 0일 때 대략 열 6~25, 행 3~20이 보이는 범위다. 텍스트 창 좌표를 계산하거나 타일맵을 덤프해 분석할 때 이 오프셋을 항상 보정해야 한다.

분석 단계에서 자주 쓰는 Z80 헥스 시그니처:

| 패턴 | 의미 |
|------|------|
| `3E xx 32 FF FF` | `LD A,xx / LD ($FFFF),A` — 슬롯 2 뱅크 전환 |
| `D3 BF` / `D3 BE` | VDP 컨트롤/데이터 포트 출력. 렌더링 루틴 단서 |
| `7E 23 B7 C8` | `LD A,(HL) / INC HL / OR A / RET Z` — 문자열 읽기 루프 |
| `5E 23 56 EB` | 16비트 포인터 역참조 — 포인터 테이블 코드 단서 |
| `ED B3` (OTIR) | VRAM 블록 전송 루프 |
| `54 4D 52 20 53 45 47 41` | "TMR SEGA" ROM 헤더 매직 |

## 2. 메모리 맵과 뱅킹

### 2.1 SEGA 매퍼와 16KB 슬롯 3개

ROM은 16KB(0x4000) 뱅크 단위로 잘려 세 슬롯에 매핑된다. 매퍼 제어 레지스터는 RAM 최상단 주소에 있다.

```
0x0000-0x3FFF : 슬롯 0 (레지스터 0xFFFD, 사실상 뱅크 0 고정.
                 0x0000-0x03FF의 첫 1KB는 인터럽트 벡터 보호를 위해 항상 뱅크 0)
0x4000-0x7FFF : 슬롯 1 (레지스터 0xFFFE)
0x8000-0xBFFF : 슬롯 2 (레지스터 0xFFFF) — 데이터 뱅크 전환의 주 통로
0xC000-0xDFFF : 워크 RAM 8KB
0xE000-0xFFFF : RAM 미러 (0xFFFC-0xFFFF는 매퍼 레지스터가 인터셉트)
```

물리 ROM 오프셋 = 뱅크 번호 × 0x4000 + 슬롯 내 오프셋. 디버거에서 본 논리 주소를 ROM 파일 오프셋으로 옮길 때 항상 현재 뱅크를 함께 기록해야 한다. 매퍼는 뱅크 번호의 하위 비트만 보므로 ROM 크기를 넘는 뱅크 번호는 wrap-around된다.

### 2.2 ROM 헤더와 확장

오프셋 0x7FF0에 16바이트 TMR SEGA 헤더가 있다. 0x7FFA에 16비트 체크섬, 0x7FFF 상위 니블에 리전(0x5=GG Japan, 0x7=GG International), 하위 니블에 크기 코드(0x0=256KB, 0x1=512KB, 0x2=1MB)가 들어간다. ROM을 수정하면 빌드 단계에서 체크섬을 재계산한다.

512KB(32뱅크) ROM은 끝에 512KB를 덧붙여 1MB(64뱅크)로 확장할 수 있다. 확장 영역(뱅크 32~63)은 0xFF로 패딩한 뒤 크기 코드를 0x2로 갱신한다. 아래는 한 사례의 구획 예다.

- **뱅크 32**: 한글 렌더 코드(ko_render). 슬롯 2에 매핑해 $8000부터 실행
- **뱅크 33~34**: 한글 글리프 데이터 전용 뱅크. 4bpp 글리프 1자 = 32바이트이므로 수백 자 규모의 서브셋이 16KB 뱅크 하나를 넘어 두 뱅크에 걸친다

코드(고정 슬롯 0)와 글리프 데이터(슬롯 2 경유)가 다른 뱅크에 있으므로 렌더링 중 뱅크 전환이 일어난다. 전환 전 현재 뱅크를 RAM에 보존하고, VDP 접근 구간은 `DI`/`EI`로 인터럽트를 막는다.

## 3. 비디오 시스템

### 3.1 4bpp 인터리브드 비트플레인 타일

타일 1개 = 8×8픽셀 = 32바이트. 행마다 4바이트가 연속 배치되며 각 바이트가 비트플레인 0~3을 맡는다(행 단위 인터리브). 픽셀 (x,y)의 색 인덱스는 네 플레인의 비트 (7-x)를 모은 4비트 값이다.

단색 폰트는 비트플레인 0만 쓰면 된다(인덱스 0=투명, 1=전경). 1bpp 비트맵 8바이트를 4bpp 32바이트로 펼치는 변환은 다음과 같다.

```python
def to_gg_tile(bitmap_8bytes, fg=1):
    """1bpp 8바이트 → GG 4bpp 32바이트 (전경색 = 팔레트 인덱스 fg)"""
    tile = bytearray(32)
    for row in range(8):
        b = bitmap_8bytes[row]
        for plane in range(4):
            tile[row * 4 + plane] = b if (fg >> plane) & 1 else 0
    return bytes(tile)
```

### 3.2 VRAM 레이아웃과 타일 수 한계

```
0x0000-0x37FF : 타일 패턴 (타일 N의 주소 = N × 32)
0x3800-0x3EFF : 네임테이블 32×28 = 896 엔트리 × 2B = 1,792B
0x3F00-0x3FFF : 스프라이트 속성 테이블(SAT)
```

네임테이블이 기본 위치 0x3800에 있을 때 사용 가능한 타일은 **최대 448개**다. 배경·UI가 상당수를 점유하므로 폰트에 쓸 수 있는 타일은 그보다 적다. 한글 완성형 수백 자를 상주시키기는 어렵기 때문에, 대사 본문은 표시 시점에 필요한 글리프만 VRAM에 올리는 동적 전송을 우선 검토한다. 짧은 UI 전용 서브셋은 정적 상주도 가능하므로 화면별 타일 점유를 먼저 산정한다.

네임테이블 엔트리는 16비트 리틀엔디언: 타일 인덱스 9비트(0~511) + H/V 플립 + 팔레트 선택 + 우선순위(priority, 텍스트를 배경 위에 그릴 때 사용). 좌표 (col,row)의 엔트리 주소는 `0x3800 + (row*32 + col)*2`다.

### 3.3 VDP 포트 프로토콜

VRAM 접근은 포트 두 개로만 이루어진다. 0xBF에 주소 하위/상위(쓰기 모드는 상위에 bit6 셋)를 차례로 출력한 뒤, 0xBE로 데이터를 읽고 쓴다(주소 자동 증가). 블록 전송은 `OUTI`/`OTIR` 루프를 쓰되, VDP 쓰기 타이밍을 맞추기 위해 루프 안에 `PUSH IX / POP IX` 같은 지연을 끼우는 패턴이 흔하다. CRAM(팔레트) 쓰기는 주소 상위에 bit7+bit6을 셋한다.

## 4. 텍스트 엔진 패턴

### 4.1 자체 테이블 인코딩

게임기어 게임의 텍스트는 ASCII/Shift-JIS가 아닌 게임 고유 테이블로 인코딩된다. 아래는 한 인코딩 예다. 문자→타일 변환 공식(아래 `바이트+0x54`)·다이그래프 구성·기본 문자 대역은 게임마다 다르므로, 한 게임의 인코딩 표를 다른 게임에 보편 규칙으로 가정하지 말고 렌더 루틴의 변환 코드(여기서는 `ADD A,$54` 한 줄)를 직접 확인해 게임별로 재확정한다.

- **인코딩 공식**: `VDP 타일 인덱스 = 텍스트 바이트 + 0x54`. 렌더 루틴의 `ADD A,$54` 한 줄이 문자→타일 변환의 전부다
- **기본 문자(0x0B~0x4A)**: 64개 순차 매핑. `0x0B`=스페이스, `0x0C~0x25`=A-Z, `0x26~0x3F`=a-z, 이후 구두점(`0x40`=, `0x41`=' `0x42`=? `0x43`=! `0x44`=. `0x47`=~ `0x49`=-)
- **다이그래프(0x4C~0x7E)**: 51개. 1바이트가 두 글자 조합(th, he, an, in, er, on, re, ed, nd, ...)으로 풀린다. 영문 텍스트 압축 수단이며, 추출기는 이 대역을 반드시 2글자로 복원해야 한다
- **제어코드(0xF8~0xFF 상단 대역)**: `0xF8 XX`=변수 치환, `0xFB`=종료 변형, `0xFC`=[WAIT], `0xFD`=줄바꿈, `0xFE`=[END](텍스트 종결자), `0xFF`=[PAGE]. 제어코드 검사는 렌더 진입 직후 `INC A` 순차 비교로 0xFF부터 역순 판별하는 구조다

### 4.2 VWF(가변폭 폰트) 엔진과 2타일 합성

선행 영문 패치가 VWF 엔진을 별도 뱅크에 얹는 패턴이다. 텍스트 엔진이 컨텍스트(텍스트 포인터 BC, 네임테이블 포인터 HL, 슬롯 뱅크)를 RAM($DFxx)에 저장한 뒤 점프해 들어온다. 아래 RAM 주소·뱅크 번호는 한 인스턴스값이고, 메커니즘(2타일 합성·타일 베이스+카운터 상태 공유)은 엔진 단위로 다시 확인한다. 동작 원리:

1. 폰트는 1bpp(글리프당 8바이트 = 폭 1바이트 + 7행)로 ROM에 저장, 최초 사용 시 VRAM에 업로드
2. 글리프 폭은 1~8픽셀 가변. 현재 타일 내 픽셀 위치 p에 폭 w를 더해 세 갈래로 분기한다 — w+p<8이면 현 타일에 합성(fits), w+p=8이면 경계 일치(boundary), **w+p>8이면 글리프가 두 타일에 걸친다(straddle)**
3. straddle 시 넘치는 비트를 overflow 버퍼에 좌측 시프트로 분리해 두고, 다음 호출에서 새 타일의 시작 내용으로 복사한다. 글리프는 SRL 시프트 후 OR 병합으로 합성 버퍼(28바이트, 7행×4플레인)에 누적되고, 완성된 32바이트 타일이 OUTI 루프로 VRAM에 flush된다
4. 출력 타일 인덱스 = 타일 베이스($DFDC: 창 위치에 따라 $70 또는 $9F) + 타일 카운터($DFE8). 다이그래프 바이트는 두 글리프로 분해되어 순차 합성되므로 한 텍스트 바이트가 2타일에 걸치는 일이 흔하다
5. 상태 머신은 RAM 1바이트($DFE6)로 관리: bit7=세션 활성, bit6=새 타일 시작, bit5=straddle 후반부 대기, bit4=타일 확정(네임테이블 포인터 2바이트 되감기 트리거), 하위 니블=픽셀 위치

### 4.3 read-ahead 가드

VWF 엔진은 **다음 텍스트 바이트를 미리 읽어** RAM($DFD8)에 저장하고, 그것이 제어코드(0xFD/0xFE/0xFF)면 폭 8픽셀짜리 wide space 글리프(텍스트 바이트 0x48)를 강제로 흘려 넣는다. 폭이 8이면 합성 여지가 없어 타일 경계가 정확히 떨어지므로, 합성 중이던 부분 타일이 깨끗하게 flush된다.

한글 패치는 이 메커니즘을 확장할 수 있다. 예를 들어 한글 프리픽스를 0x80 이상으로 잡은 경우, **미리 읽은 바이트가 한글 프리픽스 범위이면 wide space로 대체**해 compositing을 강제 완료시킨다. 이 가드가 없으면 VWF 엔진이 한글 프리픽스를 글리프 인덱스로 오인해 잘못된 폰트 데이터를 합성하고, 영문→한글 전환 지점에서 텍스트 포인터 전진량까지 어긋날 수 있다. 프리픽스 범위와 대체 바이트는 대상 엔진의 인코딩과 wide-space 규약으로 다시 정한다.

## 5. 한글 인코딩

### 5.1 2바이트 프리픽스 인코딩

8×8 1타일 완성형 방식의 한글 인코딩 구조:

| 바이트 범위 | 용도 |
|---|---|
| 0x0B~0x7E | 기존 영문 문자 유지 (스페이스, A-Z, a-z, 구두점, 다이그래프) |
| **0x80 XX** | 한글 음절 #0~255 (빈도순 상위) |
| **0x81 XX** | 한글 음절 #256~511 |
| **0x82 XX** | 한글 음절 #512~767 |
| 0xF8 XX | 변수 (원본 유지) |
| 0xFB~0xFF | 제어코드 (원본 유지) |

음절 상한은 「확보한 프리픽스 수 × 두 번째 바이트 256」 공식의 출력이다. 프리픽스 N개면 N×256 음절이고, 미사용 대역에서 프리픽스를 더 확보하면 256 단위로 늘어난다(산정 일반론: `references/strategy/font-strategy.md` 3절). 음절→코드 매핑은 고정 코드표가 아니라 **빌드 시 번역 텍스트의 출현 빈도순으로 동적 생성**한다. 자주 나오는 음절일수록 0x80 프리픽스에 배정되어 통계적 지역성이 생기고, 번역이 바뀌면 매핑도 다시 생성된다. 프리픽스 대역(0x80~0x82)이 원본 인코딩의 미사용 대역이어야 하므로, 원본에서 그 대역을 쓰는 문자(탁음 합성 등)는 사전에 정리해야 한다.

### 5.2 글리프 운용

- 번역 전량에서 실제 사용된 음절만 글리프로 만든다. 한 사례에서는 실사용 음절 수가 산정 상한 안에 들었지만, 게임마다 번역 코퍼스에서 다시 집계한다
- 글리프는 8×8 4bpp 타일(32바이트)로 사전 렌더링해 전용 뱅크에 배치한다. 필요 뱅크 수는 `음절 수 × 32B`로 계산한다
- 한글은 고정폭 8px 1문자=1타일이므로, 가변폭 합성으로 여러 글자를 한 타일에 욱여넣는 영문 VWF 대비 **네임테이블 엔트리를 2~3배 소비**한다. 창 폭 예산(타일 수)을 한글 기준으로 다시 계산해야 한다
- 8×8 한 타일에 초성·중성·종성을 다 그려야 하므로 가독성 한계가 뚜렷하다. 획이 많은 글자를 피하는 번역어 선택도 실질적인 대응 수단이다

## 6. ASM 패치 패턴

### 6.1 트램폴린 → 디스패치 → 렌더 3단 구조

원본 코드를 최소로 건드리면서 한글 경로를 끼워 넣는 구조다. 아래는 한 사례의 바이트 예산이며, 주소·RAM 변수·뱅크 번호는 대상 게임에서 다시 찾아야 한다.

| 위치 | 크기 | 역할 |
|---|---|---|
| 뱅크 0 고정 영역 | 3B | `JP` 목적지 교체 — 텍스트 시작 루틴을 트램폴린으로 리다이렉트 |
| 뱅크 0 고정 영역 | 6B | 트램폴린: 타일 카운터($DFE8) 리셋 후 원래 루틴으로 `JP` |
| 타일 매핑 지점($238A) | 7B | 원본 `LD A,$54 / ADD A,D / ...`를 `JP ko_dispatch` + NOP 4개로 교체 |
| read-ahead 지점($23A6) | 45B | 뱅크 스위치 코드 재작성 + read-ahead 가드(다음 바이트 ≥0x80 → 0x48 대체) |
| 뱅크 0 자유 공간 | 63B | ko_dispatch: D<0x80 → 영문 경로(`ADD A,$54` 복원 후 VWF 진입), D≥0x80 → 한글 경로(슬롯 2 저장 → 두 번째 바이트 읽기 → 뱅크 32 전환 → `JP ko_render`) |
| 뱅크 32 ($8000) | 146B | ko_render: 글리프 VRAM 전송 + 네임테이블 기록 준비 |
| VWF 엔진 리턴($B4F4) | 1B | 리턴 주소 1바이트 패치 — 가드 코드가 2바이트 길어져 리턴 지점이 밀린 것 보정 |
| VWF 초기화($B41D) | 2B | `LD (HL),$00` → NOP 2개 — 엔진 init의 카운터 리셋 제거(한글 렌더러와 카운터 공유 위해) |

원본 영역(뱅크 0·VWF 뱅크) 수정은 수백 바이트 수준이고, ko_render는 확장 뱅크 32에 들어간다. 원본 코드 개입을 작게 유지하고 본체 로직은 확장 뱅크로 빼는 배치가 요점이다.

### 6.2 ko_render의 동작

```
1. BC/HL/DE/슬롯1 뱅크를 $DFxx에 저장
2. VWF 세션 체크($DFE6 bit7): 미시작이면 타일 카운터($DFE8)=0 리셋,
   타일 베이스($DFDC)를 $9F로 고정 초기화 (EN VWF 엔진 init는 네임테이블 포인터 위치 기반 $70/$9F 선택이지만, 한글 렌더 경로는 $9F 고정)
3. 타일 할당: $DFE8 pre-increment 후 타일 = ($DFDC)+($DFE8) — VWF 엔진과 동일 규약
4. VRAM 주소 = 타일 인덱스 × 32 계산, 4bpp 글리프 32B를 OUTI 루프로 전송 (DI/EI 보호)
5. VWF 합성 버퍼($DF90) 클리어, $DFE6=$C0 설정
   → 다음 영문 문자가 새 타일에서 시작하도록 상태를 맞춘다
6. D=타일 인덱스로 레지스터 복원, 원본 네임테이블 기록 루틴으로 JP
```

핵심 설계는 **기존 VWF 엔진과 한글 렌더러가 타일 카운터와 타일 베이스를 공유**해 출력 타일을 순차 할당받게 하는 것이다. 게임의 타일 관리에 그대로 편입되므로 배경 타일과 충돌할 위험을 줄인다. 반대로 카운터 초기화 책임이 양쪽에 분산되면 부팅 직후 첫 한글 문자가 엉뚱한 타일을 덮어쓰는 식의 버그가 난다. 세션 미시작 시 어떤 상태 변수를 초기화해야 하는지는 대상 엔진에서 실측한다.

### 6.3 자유 공간 발굴 주의점

뱅크 0의 NOP(0x00) 연속 영역이라도 코드가 아니라 **데이터로 쓰일 수 있다**. 예컨대 64바이트 NOP 영역 중 앞 32바이트를 게임이 RAM 버퍼 초기화용 제로 데이터로 복사하고 있는 경우가 있다. 이럴 때는 NOP sled 32바이트를 그대로 둔 채 그 뒤에서 새 코드를 시작한다. 자유 공간 후보는 반드시 그 주소를 읽는 코드(`LD HL,주소`)가 없는지 교차 확인한다.

## 7. 텍스트 공간 전략

### 7.1 다중 포인터 테이블

텍스트가 하나의 테이블이 아니라 뱅크별·용도별 복수 테이블로 쪼개져 있는 구조가 흔하다. 여러 테이블이 두 뱅크에 흩어진 배치 예:

| 뱅크 | 테이블 | 논리 base | 내용 |
|------|--------|-----------|------|
| 3 | Main | $8000 | 전투/상태 |
| 3 | Secondary | $8F00 | 몬스터 전투 |
| 3 | Third | $9D40 | 메뉴/UI/크레딧 |
| 5 | Main | $8000 | 스토리/대사 |
| 5 | Secondary | $A000 | 이벤트 |

포인터는 16비트 리틀엔디언이며 값은 **테이블 base 기준 오프셋**이다(논리 주소 = base + 포인터 값). 같은 뱅크의 테이블이라도 base가 제각각이므로, 테이블마다 base와 Z80 진입점을 따로 확정해야 한다. 진입점이 다른 테이블을 하나로 합쳐 처리하면 일부 텍스트만 깨지는 형태로 드러난다.

### 7.2 접미사 공유 트레이드오프

원본 텍스트는 **복수 포인터가 동일 텍스트 블록의 중간 지점을 가리켜 접미사를 공유**하는 최적화를 쓴다(예: "...입니다[END]" 꼬리를 여러 엔트리가 공유). 이를 무시하고 엔트리마다 전문을 펼치면 공간이 폭발한다 — 한 뱅크에서 접미사 공유를 해제했더니 **12KB가 70KB로 불어나** 16KB 뱅크에 들어갈 수 없게 된 경우가 있다.

따라서 한글 빌드는 접미사 공유를 보존해야 한다. 실무 요령:

- 추출 시 어느 포인터들이 같은 블록을 공유하는지 그래프로 기록한다
- 번역 후에도 공유 가능한 꼬리(공통 종결구 + [END])는 다시 공유시킨다
- 미번역 엔트리가 참조하는 바이트는 **보호 바이트**로 표시하고 그 위에 한글 텍스트를 덮지 않는다
- 텍스트 뱅크 안에 게임 로직/레벨 데이터가 섞여 있을 수 있다(뱅크 5의 뱅크 내 오프셋 $3000~$3FFF, 즉 슬롯 2 논리 주소 $B000~$BFFF 구간이 그 예). 텍스트가 아닌 영역은 경계를 박아 절대 침범하지 않는다

## 8. 빌드

배포 포맷(IPS/BPS) 생성 이전에, 빌드 자체는 **ROM 바이너리를 직접 재조립**하는 단일 파이프라인으로 만든다.

```
1. 베이스 ROM 로드 (원본 또는 분석·권리 검토를 통과한 선행 패치판)
2. 번역 데이터 로드 (테이블별 JSON 등, 다중 테이블 인식)
3. 음절 빈도 집계 → 프리픽스 매핑 동적 생성 → 필요한 글리프 렌더링
   (1bpp → 4bpp 변환) → 뱅크 33~34에 삽입
4. Z80 패치 어셈블·삽입 (트램폴린/디스패치/가드/ko_render/리턴 패치)
5. 테이블별 한글 텍스트 인코딩 → 뱅크 재구성:
   원본 뱅크 전체 복사 → 보호 바이트 회피 → 한글 텍스트 배치 → 포인터 테이블 갱신
6. TMR SEGA 헤더 크기 코드·체크섬 재계산
7. 패치 ROM 저장 (배포는 패치 파일로, 원본 ROM 미포함)
```

베이스를 선행 영문 패치판으로 잡을 수 있는지는 별도 판단이다. 확장 영역·VWF 엔진 같은 인프라 재사용 가치가 크고, 허락·크레딧·라이선스 문제가 해결되며, 2단계 적용 절차를 감수할 수 있을 때만 선택한다. 그렇지 않으면 일본판 원본을 베이스로 같은 인프라를 직접 구현한다.

빌드가 결정적(같은 입력 → 같은 출력)이어야 디버깅 중 세이브 스테이트를 재활용할 수 있다. 인코딩 매핑이 빈도순 동적 생성이므로 번역 한 줄만 바뀌어도 매핑 전체가 바뀔 수 있다는 점을 인지하고, 디버깅 세션 중에는 번역을 고정한다.

Z80 패치 코드는 핸드 어셈블보다 어셈블러(WLA-DX 등) 또는 빌드 도구 내장 인코더로 생성하고 단위 테스트를 붙인다. `JR` 상대 오프셋(기준 = 명령 다음 바이트, signed 8비트) 계산 실수가 단골 버그다.

## 9. 검증

### 9.1 Emulicious

수동 분석·검증의 1순위 도구다.

- **Tile Viewer**: 한글 글리프가 VRAM에 올바른 비트플레인 배열로 올라왔는지 시각 확인
- **Memory Editor + VRAM 쓰기 브레이크포인트**: 글자가 깨질 때, 해당 VRAM 주소에 쓰는 순간 멈추고 콜스택을 역추적해 글리프 데이터 문제인지(폰트 변환 버그) 네임테이블 인덱스 문제인지(렌더러 버그) 구분
- 보조로 Meka(VRAM 뷰어), BizHawk(Lua 자동화)도 쓸 수 있다

### 9.2 자동 검증 (에뮬레이터 개조 + 프로그램 제어)

에이전트/스크립트 기반 자동 검증의 요건은 메모리 읽기·입력 주입·스크린샷·세이브 스테이트를 프로그래밍으로 호출할 수 있는 에뮬레이터다. 예컨대 Gearsystem을 개조해 MCP HTTP 서버를 내장하고 JSON-RPC로 조작하는 방식이 있다(업스트림 미지원 — 자체 개조 또는 동등한 자동화 수단이 필요하다). 아래 운용 요점은 그 개조판 기준이다.

- **메모리 읽기**: area는 정수 ID(7=워크 RAM, 8=VRAM, 9=CRAM, 16=전체 ROM). 논리 주소 $DFxx는 RAM area 오프셋 `$DFxx - $C000`으로 변환. 네임테이블은 VRAM 오프셋 $3800에서 1,792바이트. 매퍼 레지스터($FFFD~$FFFF)는 RAM 오프셋 $1FFD~$1FFF로 읽는다
- **디버깅**: 논리 주소 브레이크포인트, 스텝 실행(step_into 과다 호출은 불안정하므로 50회 이내), VDP 쓰기·뱅크 전환 트레이스 로그
- **상태 검사**: Z80 레지스터/VDP 레지스터 조회로 디스패치 분기·뱅크 상태를 직접 확인. $DFE6(세션)/$DFE8(타일 카운터)을 읽으면 한글-영문 혼용 렌더링의 카운터 공유가 깨졌는지 즉시 드러난다
- **자동 진행**: 컨트롤러 입력(press_and_release가 안정적), 배속, 세이브 스테이트(ROM 빌드와 1:1 대응 — 다른 빌드의 스테이트 로드 금지), 스크린샷(base64 PNG)으로 화면 단위 회귀 검증

### 9.3 체크리스트

```
[ ] 한글 글리프 전수 표시 (깨짐·비트플레인 어긋남 없음)
[ ] 영문↔한글 혼용 행에서 타일 경계 정상 (read-ahead 가드 동작)
[ ] 제어코드 전수: [WAIT]/줄바꿈/[END]/[PAGE]/변수 치환
[ ] 텍스트 창 경계 내 수납 (한글 고정폭 기준 줄 폭 재계산 반영)
[ ] 뱅크 전환 글리치 없음 (전투/메뉴/스토리 등 테이블 전환 지점)
[ ] 부팅 직후 첫 텍스트 (카운터/베이스 초기화 경로)
[ ] 장시간 진행 소프트 락 없음 (종료 계열 제어코드 보존 확인)
```

### 9.4 실기 구동 경로

- **수단**: EverDrive GG X7 같은 플래시 카트나 Analogue Pocket 어댑터 경로를 쓴다. 사용 가능한 장비와 펌웨어별 ROM 한계를 프로젝트 시작 시 확인한다.
- **제약**: GG X7의 ROM 한계는 현 펌웨어 기준 최대 1MB(8Mbit)다(모델·펌웨어별로 변동 가능하니 사용 기기에서 확인 필요). Sega·Codemasters·한국식 매퍼를 자동 감지한다.
- **빌드 포맷에 미치는 영향**: 한글 폰트와 스크립트를 확장하다 보면 ROM이 커진다. 타겟 기기의 확인된 상한을 빌드 설정으로 두고, 그 값을 넘으면 빌드를 실패시킨다(예: 1MB 타겟). 접미사 공유 같은 공간 절약은 실기 호환 목표를 지키는 수단이 된다.
- **함정**: 1MB를 초과하면 로드를 거부하거나 오동작한다(거부하는지 부분만 로드하는지는 모델·OS별로 확인 필요). 실물 게임기어는 LCD·전원 회로 노화가 심해 검증 기기 확보 자체가 변수다.
- **중간 검증**: 에뮬레이터와 실기 사이의 중간 검증으로 사이클 정확 FPGA 재구현(Analogue Pocket 등 게임기어 코어)을 쓸 수 있다. 부정확한 에뮬레이터를 전제로 만든 패치가 여기서 걸러진다.

## 10. 사례 요약 (GG 마도물어 1)

일본판 512KB ROM에 대해 1MB로 확장된 선행 영문 패치판이 존재했고, 한글 패치는 그 위에 구축되었다. 영문 패치가 이미 ROM 확장과 가변폭(VWF) 텍스트 엔진을 갖추고 있어 인프라를 재활용한 사례다.

- **인코딩**: 영문 인코딩(0x0B~0x7E, 다이그래프 포함)을 그대로 유지하고, 미사용 대역 0x80~0x82를 한글 프리픽스로 배정한 2바이트 방식. 프리픽스 3개 × 256 슬롯에 빈도순 동적 매핑
- **폰트**: 8×8 1타일 완성형 글리프(4bpp)를 확장 뱅크 33~34에 배치. 고정폭 8px
- **패치 코드**: 트램폴린 → 디스패치 교체 → read-ahead 가드 → ko_dispatch → ko_render(뱅크 32) + VWF 엔진 미세 패치. 원본 영역 수정을 작게 유지하고 본체는 확장 뱅크에 배치
- **공존 설계**: 영문 VWF 엔진과 한글 렌더러가 타일 카운터·타일 베이스를 공유해 출력 타일을 순차 할당. read-ahead 가드가 영문→한글 전환 시 합성을 강제 완료시켜 두 경로가 한 문장 안에서 안전하게 섞인다
- **텍스트**: 복수 포인터 테이블 구조. 접미사 공유 최적화를 보존(해제하면 텍스트가 부풀어 16KB 뱅크를 초과)하고, 미번역 엔트리 참조 바이트와 텍스트 뱅크 내 게임 데이터 구간을 보호하며 재배치
- **빌드**: 단일 CLI 도구가 폰트 생성 → Z80 패치 어셈블 → 다중 테이블 텍스트 삽입 → 체크섬 재계산을 일괄 수행
- **검증**: Emulicious 디버거로 수동 분석, Gearsystem MCP로 메모리/브레이크포인트/입력/스크린샷 자동화

이 사례가 남긴 교훈은 이렇다. 우선 선행 영문 패치가 확장 영역과 현대화된 텍스트 엔진을 이미 갖고 있다면 재사용 후보가 되지만, 베이스 선택은 작업량·배포성·권리 조건을 비교해 결정한다. 또 기존 엔진을 대체하기보다 **상태(카운터·세션 플래그)를 공유하는 병렬 경로**로 붙이면 패치 코드를 작게 유지할 수 있다. 끝으로 read-ahead처럼 엔진이 텍스트를 미리 읽는 구조에서는 새 인코딩 대역이 기존 경로에 새어 들어가는 지점을 가드로 차단해야 한다.
