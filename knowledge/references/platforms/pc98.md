# NEC PC-98

PC-98 계열의 CPU·BIOS·disk 기본 사양은 대상 기종을 고정한 뒤 1차 자료에서 확인한다. 실행 주소, 표시 경로, disk container와 filesystem 계층은 서로 구분한다.

## 1. 실행과 표시 경로

runtime segment:offset을 file 위치로 바꾸려면 실제 loader와 DOS·BIOS I/O가 어느 file·sector를 어느 영역에 적재하는지 연결한다. real-mode 관계를 후기 protected-mode·32-bit 경로나 다른 module에 이식하지 않는다.

text plane, system font·외자, font hardware에서 graphics로 복사하는 경로, 자체 bitmap renderer와 graphics에 구운 글자는 서로 다른 소비자일 수 있다. 한 화면의 외자 PoC나 저장 bitmap 발견을 본문·menu·battle의 공통 font 경로로 확대하지 않는다.

## 2. D88 container

D88은 track·sector metadata를 보존하는 container이며 sector data만 있는 raw image와 같지 않다. track-offset table의 160/164-entry 변형을 첫 offset 하나로 고르지 않고, 각 후보의 전체 nonzero offset·sector chain·선언 disk size가 파일 안에서 일관되는지로 판정한다.

sector의 선언 size code와 header의 data-length field가 다를 수 있고, 후자는 확인한 image에서 신뢰할 수 없는 값일 수도 있다. N, 다음 sector·track 경계와 파일 범위를 교차해 실제 저장 범위를 판정한다. C/H/R/N, sector count, density, deleted/status, data-length field, interleave와 unformatted track을 임의로 정상화하지 않고 raw field를 보존한다. 지원 표현은 대상 image 전수의 무변경 왕복으로 검증한다.

## 3. raw floppy와 HDI 계층

headerless raw floppy는 file size만으로 geometry가 유일하지 않을 수 있다. BPB·media geometry·실제 loader를 교차하고, raw↔D88 변환이 합성하거나 잃는 sector metadata를 배포 조건에 포함한다.

`.hdi`는 통상 header와 geometry를 가진 PC-98 hard-disk image container다. headerless floppy raw가 `.hdi`로 잘못 유통돼도 HDI의 한 형식이 아니라 오표기된 입력이다. 확장자와 내부 구조가 충돌하면 실제 포맷을 다시 판정한다.

HDI header는 container이고 IPL·partition table·FAT는 그 안의 disk content다. HDD 경로를 지원하면 target BIOS, SASI/IDE/SCSI 등 장치 인터페이스, geometry, boot partition과 device identity를 별도 검증한다.

## 4. filesystem과 sector 단위

FAT 종류와 sector·cluster 크기는 disk 용량이나 확장자가 아니라 실제 BPB·cluster count로 판정한다. physical sector와 BPB logical sector가 다르면 어느 계층이 변환하며 loader가 어떤 단위를 소비하는지 확인한다.

파일 성장·이동은 directory·FAT chain·복제 FAT·free allocation과 게임 자체 sector/file table 가운데 실제 소유자를 모두 갱신해야 한다. filesystem 구조가 맞는다는 사실을 raw track 참조나 자체 loader의 완료 증거로 쓰지 않는다.

## 5. 텍스트와 한글 code space

문자 encoding과 script 표현은 PC-98 플랫폼 사실이 아니다. 실제 dispatcher, glyph lookup과 화면 소비로 확정한다. 게임 로컬 한글 code를 이해하지 않는 OS·BIOS·별도 renderer 경로에는 자동 적용되지 않는다.

코드 공간 재사용, 전체 레퍼토리와 활성 작업 집합은 `references/strategy/font-strategy.md`가 판정한다. 표준 decoder 통계로 미사용 영역을 잘못 확정한 국소 사례는 현재 조건이 맞을 때만 `references/tips/pc98.md#pc98-001`에서 읽는다.

## 6. 재빌드와 검증

다음 중 이번 변경이 닿은 경로만 검증한다.

- runtime 주소↔loader↔file·sector 관계와 실제 module identity
- text plane·system font·자체 renderer 가운데 수정한 소비 경로
- D88 variant와 sector metadata의 무변경 왕복
- raw·D88·HDI container와 IPL·partition·filesystem content 계층
- 파일 성장·disk 교환·HDD 설치 뒤 loader buffer와 실제 장치 경로

다른 container로 변환해 배포하면 geometry·sector order와 보호 metadata의 손실 범위를 별도로 검증한다. 한 장치 구성의 부팅 성공을 다른 drive·HDD 경로의 증거로 확대하지 않는다.
