# 드림캐스트 (Sega Dreamcast) 한글 패치 참조

## 1. 플랫폼 개요

| 항목 | 사양 |
|------|------|
| CPU | Hitachi SH-4 (SH7091), 200 MHz, 32비트 RISC |
| 엔디언 | **리틀엔디언** (바이엔디언 칩이나 세가가 리틀엔디언 선택) |
| 명령어 | 고정 16비트 길이, 지연 분기(delay slot) 있음 |
| GPU | PowerVR2 (CLX2), 100 MHz, 타일 기반 지연 렌더링(TBDR) |
| 메인 RAM | 16 MB (`0x0C000000`, P1 가상 주소 `0x8C000000`~) |
| VRAM | 8 MB (64비트 접근 `0x04000000`, 32비트 접근 `0x05000000`) |
| 사운드 | AICA + ARM7, Sound RAM 2 MB |
| 매체 | GD-ROM, 고밀도 영역 약 984 MB |

한글 패치 관점의 핵심 사실은 다음과 같다.

- **리틀엔디언**이다. 같은 세가라도 메가드라이브(68000)·새턴(SH-2)은 빅엔디언이므로 데이터 해석 방식이 완전히 다르다. 주소 `0x8C010000`을 가리키는 포인터는 메모리에 `00 00 01 8C`로 저장된다.
- SH-4 명령어는 고정 2바이트라 디스어셈블리가 단순하다. Ghidra(SH-4 모듈), IDA, Radare2로 분석한다.
- 즉치값이 8비트로 제한되므로 32비트 상수·주소는 코드 근처의 **리터럴 풀**에 두고 `MOV.L @(disp,PC), Rn`으로 로드한다. 문자열 포인터·테이블 주소 추적과 패치 모두 이 패턴을 중심으로 이루어진다.
- `BRA`/`BSR`/`JSR`/`RTS` 등 분기 명령어 다음 1개 명령어는 분기 전에 실행된다(delay slot). 분기 명령을 수정할 때 delay slot을 함께 고려해야 한다. delay slot에는 다른 분기·PC 상대 리터럴 풀 로드(`MOV.L @(disp,PC)`, `MOVA`)·`TRAPA`를 둘 수 없다(illegal slot 예외). delay slot을 재배치할 때 리터럴 풀 로드를 그 자리로 옮기지 않는다.
- 텍스처는 Store Queue(`PREF` 명령, 32바이트 burst)로 VRAM에 전송된다. 폰트 텍스처 로딩 루틴을 추적할 때 이 패턴을 만난다.
- 메인 RAM이 16 MB로 이전 세대 콘솔보다 크지만, 폰트 텍스처·텍스트 데이터 확장 여유는 게임의 로딩 버퍼와 런타임 메모리 배치로 다시 산정한다.

### 가상 주소 변환

| 가상 주소 | 물리 주소 | 설명 |
|-----------|-----------|------|
| `0x8Cxxxxxx` | `0x0Cxxxxxx` | P1 메인 RAM (cacheable) |
| `0xACxxxxxx` | `0x0Cxxxxxx` | P2 메인 RAM (non-cacheable, DMA용) |
| `0xA4xxxxxx` | `0x04xxxxxx` | P2 VRAM |

`physical = virtual & 0x1FFFFFFF` (상위 3비트 제거)로 변환한다.

## 2. 디스크 포맷 (GD-ROM)

### 2.1 이원 세션 구조

GD-ROM은 정확히 **2개 세션**으로 구성된 멀티세션 디스크다.

```
[LD Area (~35MB)] [Buffer Zone] [HD Area (~984MB)]
  Session 1        보안 링        Session 2
  내측 ──────────────────────────────► 외측
```

- **LD Area (Session 1)**: 디스크 내측. 트랙 피치 1.6 μm로 일반 CD 드라이브가 읽을 수 있다. 보통 Track 01(데이터)과 Track 02(경고 음성 CDDA)로 구성된다.
- **Buffer Zone**: LD와 HD 사이의 빈 영역. 일반 드라이브의 HD 접근을 물리적으로 막는 복제 방지 요소다.
- **HD Area (Session 2)**: 디스크 외측. 트랙 피치 약 1.2 μm. 게임 실데이터가 전부 여기 있다. **Track 03은 항상 데이터 트랙**이며 IP.BIN과 ISO9660 파일시스템을 담는다. Track 04 이후는 선택적 CDDA 오디오 트랙이다.

한글 패치에서 건드리는 것은 HD Area의 데이터 트랙뿐이다. CDDA 트랙은 그대로 유지한다.

### 2.2 LBA와 섹터

- HD Area의 Track 03은 **LBA 45000**에서 시작하는 것이 표준이다.
- 데이터 트랙은 Mode 1 섹터를 쓴다. raw 섹터 2352바이트 = Sync 12 + Header 4 + **User Data 2048** + EDC 4 + Reserved 8 + ECC 276. raw 트랙에서 실데이터를 읽으려면 각 섹터의 오프셋 16부터 2048바이트만 추출한다.
- 섹터를 수정하면 EDC(CRC 기반)·ECC(Reed-Solomon)를 재계산해야 한다. 2048바이트 cooked 트랙이면 불필요하다.
- LBA↔MSF 변환: `LBA = (분×60 + 초)×75 + 프레임 − 150`.

### 2.3 GDI 포맷

GDI는 GD-ROM의 트랙 레이아웃을 그대로 보존하는 덤프 포맷이다. `.gdi` 텍스트 파일이 트랙 레이아웃을 기술하고 트랙별 `.bin`/`.raw` 파일이 데이터를 담는다.

```
<총 트랙 수>
<트랙번호> <시작LBA> <트랙타입> <섹터크기> <파일명> <오프셋>
```

| 필드 | 값 |
|------|-----|
| 트랙타입 | `0` = Audio(CDDA), `4` = Data(Mode 1) |
| 섹터크기 | 오디오는 항상 2352, 데이터는 2352(raw) 또는 2048(cooked) |

예시:

```
6
1 0 4 2048 track01.bin 0
2 756 0 2352 track02.raw 0
3 45000 4 2048 track03.bin 0
4 356250 0 2352 track04.raw 0
...
```

트랙 크기는 GDI에 기록되지 않는다. 다음 트랙 시작 LBA와의 차에 섹터 크기를 곱하거나, 트랙 파일 크기로 계산한다.

### 2.4 CDI 포맷 주의

CDI(DiscJuggler)는 단일 파일 이미지로 모든 트랙을 합친다. 원래 700 MB CD용 포맷이라 **HD Area가 잘릴 수 있고**, LBA 재매핑·Pre-Gap 처리 차이가 있어 패치 작업의 기준 포맷으로 부적합하다. 작업은 GDI 기준으로 하고, CDI는 필요할 때만 변환한다.

### 2.5 ISO9660 파일시스템 (Track 03 내부)

```
Track 03 (LBA 45000~)
├─ 섹터 0~15:  System Area = IP.BIN (16섹터, 32,768바이트)
├─ 섹터 16:    PVD (Primary Volume Descriptor)  → LBA 45016
├─ 섹터 17:    Volume Descriptor Set Terminator
├─ Path Table (L-type LE / M-type BE)
├─ Root Directory Record
└─ 파일 데이터 영역 (1ST_READ.BIN 등)
```

- 파일명은 대문자 + `;1` 버전 번호 형식이다 (예: `1ST_READ.BIN;1`).
- LBA·크기 필드는 **Both-Endian**(LE와 BE 양쪽 연속 저장)이다. 갱신 시 양쪽 모두 고쳐야 한다.
- 파일 위치를 옮기면 Directory Record와 Path Table을 모두 갱신해야 하고, 데이터는 2048바이트 섹터 경계에 정렬해야 한다.

### 2.6 IP.BIN

Track 03 첫 16섹터(32 KB)를 차지하는 부트스트랩이다.

| 오프셋 | 필드 | 비고 |
|--------|------|------|
| `0x000` | Hardware ID `"SEGA SEGAKATANA "` | 절대 변경 금지 (BIOS 검증) |
| `0x030` | Area Symbols (J/U/E) | 지역 제한 해제 시 `"JUE"` |
| `0x060` | Boot Filename `"1ST_READ.BIN    "` | 부팅 실행 파일 지정 |
| `0x080` | Game Title (128바이트) | |
| `0x100`~`0x2FF` | TOC 영역 | 트랙 추가·제거 시 갱신 필요 |
| `0x300`~`0x7FFF` | Bootstrap 코드 (SH-4) | 1ST_READ.BIN 로드·점프 담당 |

부팅 순서: BIOS → IP.BIN(`0x8C008000`에 로드) → 1ST_READ.BIN(`0x8C010000`에 로드) → 게임 실행.

## 3. 텍스처/폰트 포맷 (PVR)

드림캐스트 게임은 BIOS 내장 폰트(ROMFONT, JIS X 0208)를 쓰거나, PVR 텍스처에 글리프 아틀라스를 저장하거나, 실행 바이너리에 비트맵 글리프를 직접 내장할 수 있다. PVR 아틀라스를 쓰는 경우 한글 패치의 그래픽 작업은 PVR 디코딩→글리프 교체→재인코딩이다.

### 3.1 파일 구조

```
[GBIX 헤더 (선택, 12바이트)] [PVRT 헤더 (16바이트)] [텍스처 데이터]
```

GBIX(전역 인덱스) 헤더: Magic `GBIX` + Length(보통 8) + Global Index. PVRT 시작 위치는 `0x08 + Length`로 계산한다. 재인코딩 시 **Global Index 값을 보존**해야 한다.

PVRT 헤더:

| 오프셋 | 크기 | 필드 |
|--------|------|------|
| 0x00 | 4 | Magic `PVRT` |
| 0x04 | 4 | Data Size (헤더 이후 데이터 크기) |
| 0x08 | 1 | Pixel Format |
| 0x09 | 1 | Texture Type |
| 0x0C | 2 | Width (LE) |
| 0x0E | 2 | Height (LE) |

### 3.2 픽셀 포맷

| 코드 | 포맷 | BPP | 폰트 용도 |
|------|------|-----|----------|
| 0x00 | ARGB1555 | 16 | 1비트 알파, 안티앨리어싱 불가 — 외곽선 폰트에 흔함 |
| 0x01 | RGB565 | 16 | 알파 없음 |
| 0x02 | ARGB4444 | 16 | 4비트 알파, 반투명 안티앨리어싱 폰트 |
| 0x05 | 4BPP 팔레트 | 4 | 16색 인덱스 |
| 0x06 | 8BPP 팔레트 | 8 | 256색 인덱스, 다단계 안티앨리어싱 폰트 |

### 3.3 텍스처 타입과 Twiddling

| 코드 | 타입 |
|------|------|
| 0x01 | Square Twiddled (정사각형 필수) |
| 0x03 | VQ 압축 |
| 0x05/0x07 | 4BPP/8BPP 팔레트 Twiddled |
| 0x09 | Rectangle (linear, 비정사각형 가능) |
| 0x0D | Rectangle Twiddled (비정사각형 twiddled) |
| 0x10 | Small VQ (≤64×64) |

Twiddled 텍스처는 픽셀이 **Morton code(Z-order)** 순서로 저장된다. (x, y) 좌표의 비트를 교차 배치해 1D 인덱스를 만든다. 헥스 에디터로 보면 패턴이 깨져 보이며, 편집하려면 untwiddle → 픽셀 수정 → twiddle 과정을 거쳐야 한다. twiddling을 빠뜨리면 텍스처가 깨져 출력된다. 비정사각형 twiddled(0x0D)는 짧은 변 기준 정사각형 블록으로 나누어 블록별로 twiddle한다.

크기 제한: 너비·높이는 2의 거듭제곱(8~1024), Square Twiddled는 정사각형 필수.

### 3.4 VQ 압축

PowerVR2가 하드웨어 디코딩하는 손실 압축이다.

```
[Codebook: 256엔트리 × 2×2픽셀 × 2바이트 = 2048바이트]
[인덱스: (width/2) × (height/2) 바이트, twiddled 순서]
```

- 텍스처를 2×2 픽셀 블록으로 나누고 각 블록을 코드북 인덱스 1바이트로 대체한다.
- 256×256 16bpp 기준 128 KB → 18 KB (약 7:1).
- 수정 후에는 **VQ 재인코딩**(k-means 클러스터링 기반)이 필요하다. 대안으로 Square Twiddled로 변환할 수 있으나 파일이 커지고 게임 코드 패치가 필요할 수 있다.
- Small VQ는 16×16=16, 32×32=64, 64×64=128 엔트리로 코드북이 축소된다.

### 3.5 팔레트(CLUT) 렌더링

4BPP/8BPP 텍스처의 픽셀 값은 색상이 아니라 팔레트 인덱스다.

- 팔레트는 PVR 파일에 포함되지 **않는다**. 게임 코드가 별도로 PowerVR2 팔레트 RAM(1024엔트리: 4BPP×64뱅크 또는 8BPP×4뱅크)에 로드하거나, PVP 파일·실행 바이너리 내부에 둔다.
- 팔레트가 실행 바이너리에 박혀 있는 경우가 있으므로, 폰트 텍스처가 팔레트 모드라면 **바이너리에서 팔레트 테이블 위치를 먼저 찾는다**.
- 여러 텍스처가 단일 팔레트를 인덱스 범위를 나누어 공유하기도 한다. 한글 글리프를 렌더링할 때는 원본과 같은 인덱스 범위만 사용해야 다른 텍스처를 깨뜨리지 않는다.
- 8BPP 팔레트 폰트는 ARGB1555 1비트 알파보다 안티앨리어싱 품질이 좋다(다단계 계조).

### 3.6 도구

도구 요건은 PVR 인코딩·twiddle·VQ를 다룰 수 있는 변환기다. PVR↔PNG 변환(pvr2png/png2pvr), PuyoTools(세가 게임 텍스처·아카이브), KallistiOS의 vqenc(VQ 인코딩)·pvrtex가 검증된 사례 도구다. 원본과 **같은 픽셀 포맷·텍스처 타입·해상도·Global Index·Data Size**를 유지해 재인코딩하는 것이 원칙이다.

## 4. 텍스트 엔진 패턴 — 엔진·렌더러가 여러 개로 분리된 경우

드림캐스트 게임은 텍스트 렌더링 경로가 하나가 아닌 경우가 많다. 용도(스토리 대화/UI/옵션·퍼즐)별로 인코딩·폰트 텍스처·렌더러가 따로 존재하므로, **경로를 전부 식별하기 전에는 폰트 교체 범위를 확정할 수 없다**.

### 4.1 다중 경로 예

확인되는 분리 구조 예:

1. **Shift-JIS 경로 (UI·캐릭터 이름)**: Shift-JIS 코드를 flat slot(0~255)으로 변환하는 변환기(히라가나/카타카나/한자별 서브루틴)를 거쳐 `divmod(slot, 16)`으로 행/열→UV를 계산한다. slot 경계값(예: 256)으로 참조 텍스처를 분기한다.
2. **글리프 인덱스 경로 (스토리 대사)**: 텍스트가 u16 LE **글리프 인덱스**로 저장되어 인코딩 변환 없이 직접 텍스처 슬롯을 가리킨다. 제어코드(줄바꿈·프레임 대기·포트레이트 지정·종료 등)가 `0xF000`대 값으로 섞여 있다.
3. **독립 타일 렌더러 (퍼즐·대전 UI 등)**: 모드별 전용 폰트 텍스처를 쓰는 별도 렌더러. 글리프 인덱스 체계가 아니라 텍스처 슬롯을 직접 스프라이트로 그리는 경우도 있다(인플레이스 글리프 교체로 대응).
4. **내장 비트맵 폰트**: 실행 바이너리에 ARGB1555 비트맵 글리프(수십 KB)가 직접 내장되어 실시간 라벨에 쓰이는 경우.

### 4.2 텍스처 디스크립터 테이블

렌더러가 텍스처 크기를 하드코딩하지 않고 **디스크립터 테이블**(`{u32 width, u32 height}` 배열)에서 동적으로 읽는 구조가 있다. 폰트 로더가 텍스처 등록 함수를 호출하며 테이블에 크기를 기록하고, 렌더러들은 테이블에서 읽은 width/height로 UV를 계산한다.

이 구조의 의미: **테이블 기록 지점(로더의 리터럴 풀 상수)만 패치하면 렌더러 전체가 새 텍스처 크기에 자동 적응**한다. 예컨대 512×256 텍스처를 512×512로 키워 한글 슬롯을 늘릴 때, 로더가 height를 로드하는 16비트 리터럴(`0x0100`=256)과 등록 호출의 레지스터 복사 지점을 찾아 수정하면 된다.

### 4.3 분석 전략

1. **경로 인벤토리부터**: 화면별(스토리/메뉴/옵션/모드별 UI)로 어떤 텍스처·어떤 렌더러가 쓰이는지 에뮬레이터 VRAM 뷰어와 디스어셈블리로 전수 조사한다.
2. 테이블 기준 주소의 리틀엔디언 바이트열을 바이너리에서 검색하면 writer(로더)와 reader(렌더러)가 모두 나온다. writer(로더)는 보통 하나뿐이고 reader(렌더러)는 여럿이다.
3. **상태머신 추적**: 스토리 렌더러는 컨트롤러→텍스트 블록→글리프의 다단계 상태머신일 수 있다. 프레임당 1글리프 등록 같은 타이밍 특성이 있으면, 대사 표시 시간이 가장 긴 행의 글리프 수에 비례하는 식의 동작을 GDB 트레이스로 확정한다.
4. 제어코드는 값 범위에 따라 직접 switch 처리와 콜백 디스패치가 나뉘는 2단 구조일 수 있다. 핸들러별 파라미터 수를 표로 확정해야 텍스트 재삽입기가 스트림을 안전하게 파싱한다.
5. **엔진별 인코딩 테이블 분리**: 여러 텍스처가 일부 행만 공유하고 나머지 행 내용이 다르면, 단일 인코딩 테이블로 두 텍스처를 서비스하다 충돌한다. 렌더러별로 인코딩 구조체를 분리하고(퍼즐용/UI용), 각자의 배치 범위(공유 행 + 전용 행)를 명시해 독립 생성한다. 보존해야 할 원본 행(버튼 아이콘·기호 행)은 RGBA 디코딩 후 그대로 유지한다.

## 5. 한글 운용 수치

- **글자 수**: 완성형 2,350자 전체가 아니라 실제 번역 스크립트에 등장하는 글자만 추출한 부분집합이 운용 기준이다. 드림캐스트는 폰트가 텍스처 슬롯 단위로 관리되므로 글리프 예산은 「엔진별 텍스처 슬롯 수(텍스처 면적 ÷ 셀 크기 × 장수, 예약 슬롯 제외)」로 산정하며, 텍스처 확장(높이 패치 + 디스크립터 갱신)이 가능하면 그만큼 늘어난다(`references/strategy/font-strategy.md` 3절의 "렌더러 텍스처 예산" 병목). 한 사례에서는 스토리 번역에 필요한 글자 수를 원본 스토리 폰트 슬롯에 라우팅 테이블 확장 + 텍스처 높이 패치로 수용했고(숫자 등 원본 슬롯은 유지), 상한은 렌더러마다 따로 산정해 퍼즐 폰트·UI 폰트에 각각 별도 예산을 배정했다.
- **글리프 크기**: 16×16(본문)+8×8(보조) 조합에서 시작하되, 640×480 출력과 텍스처 예산이 허용하면 24×24·32×32 셀도 검토할 수 있다. 32×32 셀이면 512×512 텍스처 한 장에 256슬롯(16×16 그리드)이 들어간다.
- **텍스처 용량 참고**: 글리프 16×16이면 512×512 텍스처에 1,024자, 1024×1024에 4,096자가 들어간다. 완성형 2,350자 전부를 넣으려면 16×16 기준 1024×1024가 필요하나, 부분집합 운용의 필요 텍스처 크기는 렌더러별 슬롯 수와 예약 슬롯을 빼고 산정한다.
- **슬롯 부족 대응**: 텍스처 높이 확장(256→512, 디스크립터 테이블 패치 동반), 미사용 텍스처 슬롯 전용(轉用), 다른 폰트 텍스처로의 오버플로우 배치를 조합한다.

## 6. 실행 바이너리 패칭 (1ST_READ.BIN)

1ST_READ.BIN은 게임의 메인 실행 파일이다. 헤더 없는 flat binary(SH-4, 리틀엔디언)로, IP.BIN 부트스트랩이 **`0x8C010000`**(기본값)에 로드한 뒤 그 주소로 점프한다.

### 6.1 주소 변환

```
메모리 주소 = 파일 오프셋 + 0x8C010000
파일 오프셋 = 메모리 주소 − 0x8C010000
```

바이너리 내 포인터가 `0x8Cxxxxxx` 형태면 메인 RAM 주소다. 테이블 주소를 찾을 때는 그 값의 리틀엔디언 바이트열을 검색한다.

### 6.2 스크램블

정식 게임의 1ST_READ.BIN은 GD-ROM에 **스크램블**(XOR 기반)되어 저장되며 BIOS가 로드 시 자동 디스크램블한다. 리핑 이미지는 이미 디스크램블된 상태일 수 있다. 작업 순서는: 디스크램블(필요시) → 분석·수정 → 재스크램블(원본이 스크램블 상태였다면) → 이미지 재삽입. `scramble`/`descramble` 커뮤니티 유틸리티를 쓴다.

### 6.3 흔한 패치 대상

- **텍스트 데이터**: 스토리 대사·UI 문자열이 바이너리 내부에 직접 들어 있는 경우가 많다. 글리프 인덱스 스트림이면 한글 인코딩 테이블에 맞춰 재생성한다.
- **리터럴 풀 상수**: 텍스처 크기·테이블 주소·슬롯 경계값이 코드 근처 리터럴 풀에 있다. 명령어가 아니라 리터럴 풀 데이터를 고치는 패치가 다수다.
- **팔레트 테이블**: 팔레트 모드 폰트의 팔레트가 바이너리 내 고정 오프셋에 박혀 있을 수 있다(256엔트리 ARGB1555 등).
- **내장 비트맵 폰트**: 바이너리에 직접 내장된 글리프 영역을 한글로 교체.
- **크기 변경 시**: 파일 크기를 바꾸면 ISO9660 Directory Record의 Data Length 갱신이 필요하고, 뒤 파일들의 LBA가 밀릴 수 있다. 부분 재배치가 가능한지 먼저 따져 보고, 메타데이터 정합성 검증 비용이 크면 전체 재빌드로 전환한다.

## 7. 빌드 (GDI 재빌드)

### 7.1 인플레이스 교체의 한계

개별 파일을 같은 크기로 교체하는 인플레이스 방식이 가장 단순하지만, 파일이 커지면 불가능하다. 디스크 끝 재배치 + 디렉토리 엔트리 부분 패치는 ISO9660 메타데이터의 일관성이 깨져 미묘한 그래픽 글리치를 유발할 수 있다. 파일 크기가 변하는 패치라면 부분 재배치의 정합성을 증명하거나 **전체 재빌드**를 우선 검토한다.

### 7.2 전체 재빌드 파이프라인

```
1. 파일 수집:   원본 ISO9660에서 전체 파일 목록 파싱 (이름·LBA·크기), LBA순 정렬.
               교체 파일은 새 데이터, 나머지는 원본에서 추출.
2. 재배치:     데이터 트랙 시작 LBA부터 빈틈없이 연속 배치.
               1ST_READ.BIN은 부트 요구사항상 배치 위치 제약 확인 (마지막 배치 사례 있음).
3. 디렉토리:   Directory Record 전체 재생성. 디렉토리 크기는 원본과 동일 유지.
               파일명;1 형식. 엔트리 순서(알파벳순 여부)를 기준 패치와 일치시킨다.
4. PVD:        volume_space_size는 원본 값 유지. root_dir LBA/size 유지.
5. 트랙 쓰기:   재배치 데이터를 데이터 트랙에 기록, 메타데이터 섹터 갱신.
6. EDC/ECC:    raw(2352) 트랙이면 변경 섹터 전체의 EDC/ECC 재계산.
```

주의사항:

- 1ST_READ.BIN의 LBA가 IP.BIN에 하드코딩되어 있을 수 있으므로 확인한다.
- 재빌드 후 트랙 크기가 원본을 초과하면 안 된다(volume_space_size 유지).
- HD Area의 메타데이터 트랙(Track 03)과 파일 데이터 트랙이 분리된 멀티 데이터 트랙 구성이 있다. 이 경우 디렉토리는 Track 03에, 파일 본체는 마지막 데이터 트랙에 있으므로 양쪽을 함께 갱신한다.
- 재빌드 결과에 글리치가 남으면, 정상 동작하는 기준 이미지(원본 또는 기존 패치)와 디렉토리 섹터를 **바이트 단위로 비교**해 패딩·필드 값·정렬·배치 순서의 차이로 원인을 좁힐 수 있다.

### 7.3 도구

gditools(GDI 파싱·추출), mkisofs/genisoimage(ISO9660 생성), cdirip/cdi2gdi(CDI 처리), binhack32(SH-4 바이너리 패치). 단, 재빌드의 세부 제어(LBA 고정, 디렉토리 크기 유지)가 필요하면 자체 빌더를 구현한다.

## 8. 검증 (에뮬레이터)

- **Flycast**: 1차 검증 환경. **GDB 스텁을 지원**하므로 렌더러 상태머신 추적, 제어코드 디스패치 검증, 레지스터 값 확정 같은 동적 분석에 필수다. 정적 분석(디스어셈블리)으로 세운 가설을 GDB 브레이크포인트·트레이스로 확정하는 워크플로우를 쓴다.
- **lxdream**: 보조 검증용.
- 검증 항목: 부팅(IP.BIN 검증 통과), 폰트 텍스처 로드(VRAM 뷰어로 twiddling·팔레트 정상 여부 확인), 전 모드 텍스트 출력(스토리/UI/옵션/퍼즐 각 렌더링 경로별), 대사 타이밍(프레임당 글리프 수 기반 표시 시간), CDDA 재생, 장시간 플레이 안정성.
- 렌더링 경로가 여러 개이므로 **경로별로 화면을 하나씩 확인**해야 한 경로에만 매달리다 다른 엔진의 폰트 누락을 놓치지 않는다.
- 최종적으로 실기 또는 실기에 준하는 정확도의 환경에서 부팅·전 구간 확인을 거친다.

### 실기 구동 경로

수단은 두 갈래다.

- **CD-R 셀프부트**: MIL-CD 부트를 이용한다. 단 후기 본체는 MIL-CD 부트가 제거되어 있다.
- **ODE**: GDEMU, MODE 등.

핵심 제약 — **1GB GD-ROM을 700MB CD-R에 그대로 담을 수 없다.** CDI 변환은 다운샘플링·재배치를 거치며 손실이 발생한다. MIL-CD 부트 가능 본체는 바닥의 원 안 숫자로 식별한다(0 또는 1은 가능, 2 이상은 불가). ODE는 GDI를 1:1 무손실로 받고 본체 리비전과 무관하다.

**빌드 포맷에의 영향**: 작업 기준은 GDI로 둔다. CHD는 무손실 압축 배포 후보가 될 수 있고, CDI 배포는 용량 절감·다운샘플링·후기 본체 MIL-CD 차단 문제를 동반하므로 별도 변환본으로 분리한다. 패치는 GDI 기준으로 만들고 CDI 변환은 사용자 책임으로 둔다(2.4절 CDI 부적합 경고와 직접 연결, 7절 GDI 재빌드 산출물을 그대로 실기에 올린다).

함정 — GDEMU는 Redump와 비호환이라 TOSEC식 GDI가 권장된다. 스왑 방식 GDI는 트랙 오프셋 차이가 있다. CDI 변환 시 CDDA를 지원하지 않는 도구를 쓰면 음악 트랙이 소실된다.

**알려진 한계**: 한 사례에서 **스토리 모드가 실기에서 정상 동작하지 않는 미해결 문제**가 남았다. 나머지 모드는 실기에서 동작했다. 다중 렌더링 경로 중 일부 경로가 실기에서만 어긋날 수 있으므로, 전 경로 실기 검증을 빠뜨리지 않는다.

중간 검증 — 에뮬레이터와 실기 사이에 사이클 정확 FPGA 재구현(드림캐스트 코어가 있으면)을 둘 수 있으나, GD-ROM 기반 기종이므로 최종적으로 실기 ODE 검증이 따로 필요하다.

## 9. 사례 요약 — 드림캐스트 뿌요뿌요~n

- **원본 구조**: 멀티트랙 GDI. Track 03(LBA 45000) 메타데이터, 중간 트랙들은 CDDA BGM, 마지막 데이터 트랙에 파일 본체가 LBA순 연속 배치. 텍스트는 Shift-JIS(UI)와 u16 LE 글리프 인덱스(스토리)의 이중 인코딩.
- **전략 선택**: 기존 영문 패치는 런타임 폰트 렌더링을 우회해 텍스트를 배경 텍스처에 직접 구워넣는 베이킹 방식을 썼지만, 한글 패치는 **원본의 런타임 렌더링 경로를 유지**하고 폰트 텍스처와 텍스트 데이터만 교체하는 방식을 채택했다. 한글은 글자 수가 많아 베이킹보다 런타임 렌더링이 유리하다.
- **폰트 텍스처**: 스토리 대화용 여러 장(512×256 ARGB1555, 32×32 셀), 퍼즐용 512×512 ARGB1555 Twiddled, UI용 512×512 8BPP 팔레트 Twiddled 2장(팔레트는 1ST_READ.BIN 내 ARGB1555 테이블을 두 텍스처가 인덱스 범위 분할 공유). 대전 UI용 256×256은 글리프 체계가 아니라 문구 단위 인플레이스 교체.
- **디스크립터 테이블**: 렌더러가 텍스처 크기를 하드코딩하지 않고 `{width, height}` 디스크립터 테이블에서 동적으로 읽는 구조를 확인. 폰트 로더의 리터럴 풀(height 상수, `0x0100`=256)과 등록 호출 지점을 패치해 한자 텍스처를 512×256→512×512로 확장, 한글 슬롯을 확보했다.
- **인코더 분리**: 퍼즐 렌더러와 UI 렌더러가 다른 텍스처를 쓰는데 일부 행 내용이 달라, 단일 인코딩 테이블 공유 시 원본 글리프(아이콘·기호)가 소실되는 문제가 발생. 렌더러별 인코딩 구조체를 분리하고 각자 배치 범위와 보존 행을 명시해 해결했다.
- **스토리 렌더러**: 3레벨 상태머신(컨트롤러→텍스트 블록→글리프) + 제어코드 파서. 프레임당 1글리프, 행 단위 병렬 렌더링이라 대사 표시 시간은 가장 긴 행의 글리프 수에 비례. 제어코드는 `0xF000`대 값 기준 2단 디스패치(직접 switch + 콜백)이며 핸들러별 파라미터 수를 표로 확정해 재삽입기에 반영했다.
- **빌드**: 개별 파일 교체 + 디렉토리 부분 패치 방식에서 그래픽 글리치가 발생해, 영문 패치와 같은 **전체 재빌드** 방식으로 전환. 잔존 글리치는 정상 동작하는 기준 이미지와의 디렉토리 섹터 바이트 비교로 추적했다.
- **검증**: Flycast + GDB 스텁으로 상태머신·디스패치 가설을 동적 확정하는 워크플로우를 사용했다. 실기에서는 스토리 모드가 정상 동작하지 않았고(미해결) 나머지 모드는 동작했다.
