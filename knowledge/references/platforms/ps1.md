# PS1 (PlayStation) 한글 패치 참조

## 1. 플랫폼 개요

- **CPU**: MIPS R3000A 33MHz, 리틀엔디언. 하드웨어 인터록이 없어 load delay slot이 존재한다. `lw` 직후 1 instruction에서는 로드 결과를 사용할 수 없다 — 이전 레지스터 값이 그대로 읽힌다. branch delay slot도 항상 실행된다.
- **GPU**: 1MB VRAM. 글리프는 LoadImage로 VRAM에 업로드해 스프라이트/텍스처로 그린다. 4bpp/8bpp/16bpp(ABGR1555) 텍스처를 쓴다.
- **CD-ROM**: Mode2/2352 섹터(유저 데이터 2048B), ISO 9660 파일 시스템. XA 오디오와 무비 스트리밍이 같은 드라이브를 공유하므로 패치 코드가 CD를 직접 읽으면 스트리밍과 충돌할 수 있다.
- **RAM**: 메인 2MB. 한글 폰트를 올릴 공간은 EXE 미참조 영역, 디컴프레서 출력 범위의 제로 런(code cave), 디스크 끝 추가 섹터에서 확보한다.
- **실행 파일**: PSX-EXE 포맷. 헤더에 Entry Point, Load Address(통상 0x80010000), text_size, Initial SP가 있다. RAM 주소 → 파일 오프셋 변환은 `file_offset = (ram_addr - load_addr) + 0x800` (0x800 = 2KB 헤더).

## 2. 메모리 맵

| 영역 | 주소 범위 | 캐시 |
|------|-----------|------|
| KSEG0 | 0x80000000~0x9FFFFFFF | I-cache + D-cache |
| KSEG1 | 0xA0000000~0xBFFFFFFF | 없음 (항상 RAM 직접) |

KSEG0와 KSEG1은 같은 물리 RAM의 별칭이다. 0x800C8C10(KSEG0) = 0xA00C8C10(KSEG1). 하드웨어 레지스터는 KSEG1으로 접근한다 (CD 레지스터 0xBF801800 = 물리 0x1F801800).

### I-cache stale 문제와 KSEG1 실행

부트 디컴프레서가 출력 범위(예: 0x80030000~0x80121724)에 쓰는 동안 I-cache 라인이 오염된다. 이후 EXE가 다시 로드되어 RAM에는 패치 코드가 복원돼도 I-cache는 stale 상태로 남아, 훅 분기가 옛 코드로 실행되는 증상이 난다. FlushCache(BIOS A-table 함수 0x44) 자체가 디컴프레서 출력 범위 안에 있으면 신뢰할 수 없다.

해결책은 **훅 코드를 KSEG1 주소로 실행**하는 것이다. KSEG1은 I-cache를 완전히 우회한다. 원래 JAL 사이트를 트램폴린으로 교체한다:

```
lui  $at, 0xA00C        ; cave의 KSEG1 별칭 상위
ori  $at, $at, 0x8C10   ; 하위
jalr $at                ; uncached 실행
(delay slot)            ; 교체로 밀려난 원본 명령 1개
```

트램폴린이 I-cache 라인(16B) 2개에 걸치면 라인별로 stale/fresh가 갈려 반쪽 실행이 일어난다. **패치를 1개 캐시 라인 안에 들어가는 3 instruction으로 줄이고**, 밀려난 원본 명령은 훅 프롤로그에서 보상 실행한다. 적용 전 JAL 사이트 주소를 확인하라 — `addr % 16 <= 4`여야 12B가 한 라인에 들어간다.

### Load delay slot 버그 패턴

```mips
lw   v1, 0(v1)          ; 로드 시작
nop                      ; load delay slot — 반드시 1 cycle 대기
bnez v1, target          ; 이제 올바른 값
```

`nop` 없이 `lw` 직후 `bnez`를 쓰면 v1에 아직 주소(non-zero)가 남아 있어 분기가 항상 성립한다(taken). 에뮬레이터 GDB stepi로 확인 가능한 전형적 버그다.

## 3. 폰트

### BIOS 폰트 (Krom2RawAdd)

PS1 BIOS는 ROM에 SJIS 전체 글리프를 내장하고 BIOS 함수로 노출한다:

- **B(0x51) Krom2RawAdd**: SJIS 코드 → 15×15 1bpp 글리프 데이터 포인터 반환. 실패 시 0xFFFFFFFF.

글리프 포맷은 15 halfword(30바이트)/글리프. 각 halfword의 bit 15..1이 15픽셀(MSB = 왼쪽), bit 0 = 0. BIOS 폰트를 쓰는 게임은 이 1bpp 글리프를 자체 변환 루틴으로 VRAM 텍스처에 올린다. 변환 방식(1bpp→4bpp 룩업, 캐시 슬롯 수, LoadImage 배치)은 게임별 렌더러에서 실측한다.

**활용 가능성**: 게임이 BIOS 폰트를 쓰고 호출 지점이 공통 경로로 수렴한다면 텍스트 엔진 전체(u16 파싱, 1bpp→4bpp 변환, VRAM 업로드, 캐시)를 그대로 두고 Krom2RawAdd 호출부만 훅할 수 있다. 한글 코드 범위면 자체 폰트 포인터를 반환하고, 그 외는 원래 BIOS로 폴스루한다. 호출 경로가 여러 개면 경로별 누락을 따로 처리한다.

**한계**:
- 15×15 1bpp 고정. 복잡한 한글 음절은 가독성이 한계에 가깝다. 부족하면 1bpp를 우회해 4bpp를 직접 공급하는 더 큰 훅이 필요하다.
- BIOS 함수 번호는 자료마다, BIOS 리비전마다 다르게 적혀 있다. 어떤 게임은 B(0x51)로 호출하고, 다른 자료·환경에서는 B(0x58) 등으로 적기도 한다. 어느 한쪽을 "틀린 값"으로 단정해 후보에서 빼지 말고, 자기 게임이 실제로 호출하는 함수 번호를 디스어셈블·디버거로 실측해 확정한다.
- 게임에 따라 폰트 렌더 경로가 여럿일 수 있다. 호출 사이트를 전수 확인해, 프리캐시·UI 등 별도 경로의 글리프가 주 훅에서 누락되지 않게 한다.

### 자체 폰트 경로

TTF에서 15×15 1bpp로 렌더링해 BIOS 출력 포맷과 동일하게 만든다(글리프당 30B). 총 용량은 글리프 수 × 30B로 계산한다. 적재 위치 선택지:

1. **EXE 확장**: 헤더 text_size를 늘려 미참조 영역에 폰트+훅을 포함. 로드 시 자동 적재, 추가 I/O 불필요. 단, 부트 디컴프레서가 해당 영역을 덮는 게임에서는 못 쓴다.
2. **디스크 추가 섹터 + lazy load**: 폰트를 디스크 끝에 두고(사례: LBA 301341, 16섹터, 31KB) 첫 글리프 렌더링 때 CD에서 RAM(사례: 0x80121724)으로 읽는다. 디컴프레서 간섭을 피하지만 CD 프로토콜 직접 제어가 필요하다(§7).

프리렌더 텍스처(BTX 등)에 박힌 UI 문자열은 폰트 훅으로 해결되지 않는다 — 텍스처 자체를 한글로 다시 그려 교체해야 한다.

## 4. 아카이브 포맷 (VDD)

컴파일(Compile)계 PS1 게임의 컨테이너. 디스크 최상위에 FIL_SEQ.VDD(스크립트/시퀀스), FIL_GRP.VDD(그래픽: BTX/FMP/HMD), FIL_MAP.VDD(맵/타일), FIL_SND.VDD(사운드), FIL_DMY.VDD(더미 패딩, 전체 0x00)를 둔다.

```
오프셋 0x00: u32 LE = 엔트리 수
각 엔트리 (24 바이트):
  [0:16]  파일명 (ASCII, null-padded)
  [16:20] 섹터 오프셋 (u32 LE) — ×2048 = 아카이브 내 바이트 오프셋
  [20:24] 파일 크기 (u32 LE)
```

분석 사례에서는 파일 데이터가 테이블 뒤 섹터 5(0x2800)부터 시작했고, 내부 파일 다수가 CNX v2 압축(컴파일 공통 포맷)이었다. CNX v2는 Greedy LZ77이며, 16B 헤더(매직 "CNX\x02", 해제 크기는 offset 12의 BE u32)에 이어 플래그 바이트의 2비트 모드 4개로 리터럴/매치를 구분한다(알고리즘 전체 사양: 같은 압축을 쓰는 새턴 문서 `references/platforms/saturn.md` 5절 참조).

**재계산**: 내부 파일 크기가 변하면 후속 엔트리의 섹터 오프셋이 전부 밀린다. 수정된 파일을 CNX 재압축 → 2048B 섹터 정렬로 재배치 → 엔트리 테이블(섹터 오프셋 + 크기) 재기록 → VDD 풀 리빌드 순서다. EXE 안에도 VDD별 파일 디스크립터 테이블(엔트리당 36B)이 별도로 존재하므로, 아카이브 내 배치가 바뀌면 EXE 쪽 테이블 정합성도 확인한다. EXE 디스크립터 수는 아카이브 실측 엔트리 수와 다를 수 있다(예약·더미 엔트리 포함).

## 5. 텍스트 엔진 패턴

### 인코딩과 바이트 순서

이하는 한 사례의 텍스트 엔진 실측 구조다. 새 게임은 1바이트 경로 유무·바이트 순서를 먼저 실측한다.

이 사례의 본문은 표준 Shift-JIS를 **BE 원시 바이트 [lead, trail]**로 저장한다 (예: "ラ" = 0x8389 → 파일에 `83 89`). 엔진은 `lhu`로 u16 LE를 읽으므로 바이트스왑된 값을 얻고, 글리프 렌더러가 다시 스왑해 표준 SJIS 순서로 복원한 뒤 Krom2RawAdd에 넘긴다. 이 엔진의 인코더는 BE로 출력해야 한다 — 훅은 스왑된 값을 기준으로 매칭하므로 한글만 우연히 맞고, 비한글 SJIS는 전부 깨진다.

이 엔진은 2바이트 단위로 읽는다(`lhu` + `addiu +2`). 1바이트 문자 경로가 없으므로 ASCII를 1바이트로 넣으면 다음 바이트와 합쳐 읽혀 스트림 정렬이 영구적으로 어긋난다. 이 경우 ASCII(0x20~0x7E)는 SJIS 전각 2바이트로 변환한다. 고정길이 교체의 패딩도 0x0000이 아니라 전각 공백(0x8140)을 쓴다 — 이 엔진에서 0x0000은 문자 코드로 해석되어 에러 스프라이트를 띄운다.

### 제어 코드 (사례 게임 실측, LE FF00~FF24)

논리값 FFxx를 LE u16으로 저장하므로 파일에는 `xx FF` 순서로 나타난다. 디스패치는 u16 값 > 0xFEFF 여부로 분기한다.

| 코드 | 파일 바이트 (LE) | 파라미터 | 기능 |
|------|----------------|---------|------|
| FF00 | `00 FF` | 0B | 메시지 끝 |
| FF01 | `01 FF` | 0B (+4B RAM 주소) | 항목 종료 (VM 스크립트) |
| FF02 | `02 FF` | 0B | 개행 |
| FF03 | `03 FF` | 2B | 메시지 블록 시작 (세이브/캐릭터 셀렉트 계열) |
| FF05 | `05 FF` | 2B | 줄 끝 + 딜레이/버튼 대기 |
| FF08 | `08 FF` | 0B (+4B RAM 주소) | 계속 (VM 스크립트) |
| FF0A | `0A FF` | 0B | 메시지 계속 |
| FF0F | `0F FF` | 2B | 변수 표시 시작 |
| FF11 | `11 FF` | 0B | 변수 표시 종료 |
| FF1A | `1A FF` | 0B | 섹션 구분 |
| FF20 | `20 FF` | 8B (4× u16) | 씬/디스플레이 설정 |
| FF23 | `23 FF` | 0B | 다음 페이지 |
| FF24 | `24 FF` | 0B | 메시지 블록 시작 |

사례 메시지 패턴: `{FF24}{FF20:8B}{FF1A}[화자명]{FF02}[대사]{FF05:2B}{FF00}`. 추출/재삽입 시 이 엔진의 종료 제어코드(FF00 등)를 raw 바이트에 포함하고, 절단이 필요하면 종료자 2바이트를 먼저 확보한 뒤 자른다. FF01/FF08 뒤의 4B RAM 주소는 텍스트가 아니므로 패치 대상에서 제외한다.

### SEQ 포인터 테이블 (스토리 스크립트 — 한 사례의 포맷)

스토리 SEQ는 RAM에 고정 베이스로 적재되는 로더블 모듈이다:

```
0x0000  12B     헤더 (3× u32 LE RAM 포인터)
0x000C  ~108B   디버그 템플릿 텍스트 (SJIS)
0x0078  176B    디스패치 테이블 (event_id → 포인터 인덱스, 0xFF=미사용)
0x0128  가변    포인터 테이블 (N× u32 LE 절대 RAM 주소)
이후    가변    텍스트 데이터 (메시지 연속 배치, 갭 0)
끝      444B    MIPS crt0 테일 코드 (+선행 2B 정렬 패딩 가능)
```

- ram_base = header[2] − 0x0C로 유도된다. 파일 내 모든 포인터는 이 베이스 기준 절대 RAM 주소다.
- **메시지가 완전 연속(갭 0)**이므로 1바이트만 늘어도 후속 포인터 전부 재계산이 필요하다. 디스패치 테이블은 인덱스 기반이라 불변.
- **444B 테일 코드**(crt0: BSS 클리어, $sp/$gp 설정, 모듈 엔트리 호출)는 절대 주소를 하드코딩하므로 텍스트가 자라면 새 위치로 복사하고 내부 lui+addiu 주소쌍과 말미의 u32 상수 3개를 시프트량만큼 갱신한다. 테일 코드는 자기 자신·EOF·파일 시작만 참조하고 텍스트 영역은 참조하지 않는다.

### VM 바이트코드 SEQ (메뉴/시스템)

사례의 메뉴 계열 SEQ는 바이트코드·포인터 테이블·텍스트가 인터리브된 구조다. 텍스트 영역에 RAM 포인터 참조가 광범위하게 산재해 포인터 전수 보정 비용이 크므로 **고정길이 교체**(전각 공백 패딩, 초과분 절단)를 우선 검토한다. 파일 끝에 위치한 영역(설명 텍스트 등)은 테일 코드 재배치 조건을 만족하면 성장시킬 수 있다. 추출은 영역 경계를 수동 지정해 바이트코드 오인을 차단한다.

## 6. 한글 매핑 설계 사례

**사례 전략: SJIS 코드 공간 재사용(원문이 SJIS일 때 성립).** 새 인코딩을 만들지 않고 한자 영역 코드에 한글 음절을 대입하면 엔진 무수정으로 동작한다.

- **범위**: 0x8840~0x8EFC. 상위 바이트 7개(0x88~0x8E) × 유효 하위 188개 = **1,316 슬롯**. 이 슬롯 수가 대상 번역의 고유 음절 수를 덮는지는 번역 코퍼스 집계로 확인한다.
- SJIS 유효 하위 바이트는 0x40~0x7E(63개) + 0x80~0xFC(125개). **0x7F는 절대 유효하지 않다.**
- 인덱스 공식 (룩업 테이블 불필요, 순수 산술):

```
hi_offset = H - 0x88                       (0~6)
lo_offset = (L - 0x40) - (L >= 0x80 ? 1 : 0)   (0~187)
glyph_index = hi_offset * 188 + lo_offset       (0~1315)
glyph_ptr = font_base + glyph_index * 30
```

- 구두점은 0x81xx 등 범위 밖 코드를 그대로 써서 BIOS로 폴스루시킨다.
- 음절 할당은 번역문에서 고유 음절을 수집해 유니코드 순(U+AC00부터)으로 정렬 후 인덱스 순서대로 부여한다 — 결정적·재현 가능.
- 훅은 짧다: 상위 바이트가 0x88~0x8E면 인덱스 계산 후 글리프 포인터 반환(`jr $ra`), 아니면 원래 BIOS 트램폴린으로 `j`(JAL 아님 — 동일 $ra 재사용).
- JAL 패치 검증: `jal target` 인코딩은 `0x0C000000 | ((target >> 2) & 0x03FFFFFF)`. 예: target 0x800D0200 → 0x0C034080 → LE 바이트 `80 40 03 0C`. `jal`/`jalr`이 `$ra`에 넣는 복귀값은 호출 명령이 아니라 지연 슬롯 다음 명령 주소(PC+8)이므로, 트램폴린에서 `$ra`를 복귀에 재사용할 때 이 +8을 전제한다.

**알려진 충돌**: 미번역 일본어 중 lead 0x88~0x8E인 한자(예: 掲 0x8C66, 示 0x8EA6)는 훅이 한글로 오인한다. 코드 우회는 불완전하므로, 이 방식에서는 해당 범위 일본어가 패치본에 남지 않도록 번역 커버리지를 검증해야 한다.

## 7. 디스크와 빌드

### 이미지 구조와 재삽입

BIN/CUE는 Mode2/2352 raw 섹터다. 유저 데이터는 섹터당 2048B이고 나머지는 sync/헤더/서브헤더/EDC/ECC. 파일을 수정해 디스크에 재삽입하면 **해당 섹터의 EDC/ECC를 재계산**해야 한다. 빌드는 원본 이미지 대비 diff(변경 섹터만 패치) 방식이 안전하다. LBA가 바뀌지 않는 동일 크기 교체가 가장 단순하고, 크기가 변하면 ISO 9660 디렉토리 레코드의 크기 필드와 후속 파일 LBA까지 갱신해야 한다.

### 폰트 추가 섹터

이미지 끝(기존 총 섹터 수 뒤)에 섹터를 덧붙이면 기존 파일 배치를 건드리지 않고 데이터를 넣을 수 있다. 사례: 총 301,357섹터 이미지에 폰트 바이너리를 LBA 301341부터 16섹터(31KB)로 배치하고 훅이 직접 읽었다.

### CD 하드웨어 직접 제어 (훅에서 폰트 로드)

런타임 라이브러리(Psy-Q CdControl/CdRead)를 훅에서 호출하면 IRQ 핸들러와 충돌한다(INT 가로채기). 부트 시점에는 BIOS B-table CD 함수도 미초기화 상태일 수 있다. 확실한 방법은 **레지스터 직접 제어**다.

베이스 0xBF801800. 1800 bits[1:0]로 Index 선택. Status(모든 Index에서 읽기): bit7 BUSYSTS(명령 처리 중), bit6 DRQSTS, bit5 RSLRRDY, bit4 PRMWRDY. Index 0 쓰기: 1801=COMMAND, 1802=PARAMETER, 1803=Request(0x80=BFRD). Index 1 쓰기: 1802=IRQ Enable, 1803=IRQ Clear(0x1F=전체, 0x40=ClrParam). IRQ flag(Index 1, 1803 읽기) bits[2:0]: 1=DataReady(INT1), 2=Complete(INT2), 3=Acknowledge(INT3), 5=DiskError.

N섹터 읽기 시퀀스:

```
1. I_MASK(0x1F801074) bit2 비활성화 (런타임 IRQ 핸들러 차단)
2. DPCR(0x1F8010F0) |= 0x8000              (DMA3 활성화)
3. SetMode(0x80) → INT3 → ACK
4. SetLoc(min,sec,frame BCD) → INT3 → ACK
5. ReadN() → INT3 → ACK
6. 섹터마다: INT1 대기 → Result 소비 → BFRD(0x80→1803) →
   DMA3 (MADR(0x1F8010B0) ← 목적지 RAM 주소, BCR=0x00010200 [512워드×1블록=2048B],
   CHCR=0x11000000) → CHCR bit24==0 대기 → IRQ ACK(0x1F)
7. Pause() → INT3 → ACK → INT2 → ACK
8. 원래 SetMode 복원 (BUSYSTS 대기 + ClrParam 선행)
```

주의사항:
1. **INT1 먼저, BFRD 나중** — DRQSTS만으로 새 섹터 감지 불가
2. **매 섹터 IRQ ACK 필수** — 안 하면 다음 INT 미발생
3. **Result FIFO 소비 필수** — 남으면 다음 응답과 혼합
4. **새 명령 전 BUSYSTS 대기**
5. DMA 목적지는 4바이트 정렬
6. **Pause의 INT2(완료)까지 대기**하고 SetMode를 보내라 — 감속 중 명령은 무시되어 CD 모드가 미복원된다

부작용: 훅의 직접 CD 읽기는 XA/스트리밍 상태와 간섭할 수 있다(사례: 첫 폰트 로드 시 음악 중단, 씬 전환 후 복구). 로드 시점을 신중히 고르고 모드 복원을 철저히 한다.

## 8. 검증 (에뮬레이터 훅 디버깅)

GDB 스텁을 지원하는 에뮬레이터로 다음을 점검한다.

- **훅 리턴 상태 검증**: 트램폴린 리턴 주소에 BP를 걸고 히트마다 입력 SJIS, 분기 경로(한글/BIOS), 출력 포인터, 보상 레지스터 값을 표로 기록한다. 글리프 주소는 수동 계산(font_base + index×30)과 대조한다.
- **load delay 추적**: `stepi`로 한 명령씩 진행하며 `lw` 직후 레지스터에 옛 값이 남는 것을 직접 관찰한다. 분기가 이상하게 항상 성립한다면(taken) 가장 먼저 의심한다.
- **I-cache stale 판별**: RAM 직접 읽기(패치 코드 존재)와 실제 실행 흐름(stepi PC 추적)이 다르면 I-cache 문제다. KSEG0/KSEG1 양쪽 덤프가 같으면 D-cache는 무관하다.
- **디스크 vs RAM 비교**: 폰트 로드 검증은 디스크 원본 바이트와 RAM 적재 결과를 같은 오프셋에서 비교한다. 불일치하면 CD 명령 주소·인자 순서를 의심한다 (사례: CdControl 주소 오인 + 인자 순서 `(command, param, result)` 3개를 2개로 잘못 호출).
- **함수 주소 확정법**: 후보 함수의 호출 사이트를 전수 수집해 호출 패턴으로 검증한다 (사례: LBA→BCD 변환기 호출 사이트가 전부 직후에 같은 함수를 호출 → CdControl 확정).
- **데이터 파괴 격리 테스트**: 크래시 시 스크립트 패치 제외, 훅 제외, 단일 파일만 패치 같은 격리 빌드로 원인 층을 분리한다. 에뮬레이터의 "write to unknown address" 로그는 OOB 포인터 산술의 단서다.
- **시각 증상 사전**: 에러 스프라이트 연속 표시 = 0x0000 패딩, 전체 스트림 깨짐 = 1바이트 문자로 인한 정렬 어긋남, VRAM 노이즈 = CD 상태 오염으로 텍스처 로드 실패.

### 실기 구동 경로

수단은 세 갈래다.

- **CD-R + 모드칩/스왑**: 구운 CD-R을 모드칩 또는 스왑 트릭으로 부팅한다.
- **소프트 언락**: FreePSXBoot, tonyhax 등 메모리 카드 기반 부트 익스플로잇.
- **ODE**: Xstation, PSIO 등.

기기마다 받아들이는 포맷이 다르다. Xstation은 BIN/CUE(단일·멀티)와 Redump를 드래그앤드롭으로 받지만, **PSIO는 단일 BIN+CUE만 받으므로 멀티 BIN은 병합해야 한다**(CDDA가 있으면 CU2 시트가 추가로 필요하다).

**빌드 포맷에 미치는 영향**: **패치 후 EDC/ECC를 반드시 재생성한다** — 데이터 트랙을 수정하면 불일치가 생기고, 실기와 CD-R에서 디스크가 무효화되거나 행에 빠진다(도구로 EDCRE 등을 쓴다. 7절 디스크와 빌드의 재삽입 산출물과 직결). 배포는 Redump BIN/CUE에 PPF 또는 xdelta 차분을 얹는 관행을 따르고, PSIO용 병합 BIN과 CU2 시트는 사용자 단계 변환으로 분리한다. 일부 게임은 의도적으로 틀린 EDC를 안티카피로 쓴다 — 특정 섹터의 EDC를 비정상값으로 두고 게임이 그 불일치를 정품 검사로 읽으므로, 전 섹터 일괄 재계산은 보호를 깨 락업을 부른다. 이런 게임은 변경 섹터만 재계산하고 의도적 오류 섹터는 raw로 보존한다(`references/strategy/build-and-verify.md`의 EDC/ECC 절).

흔한 함정도 두 가지다. 멀티 BIN을 PSIO에서 못 돌려 "패치 불량"으로 오인하기 쉽고, EDC/ECC 미재생성이 실기 부팅 실패의 최빈 원인이다.

**실기 검증 한계**: 한 사례에서는 패치가 배포되지 않아 실기로는 검증하지 못했다(에뮬레이터 검증까지만).

중간 검증 — 에뮬레이터와 실기 사이에 사이클 정확 FPGA 재구현(PS1 코어가 있으면)을 둘 수 있으나, CD 기반 기종이므로 최종적으로 실기 ODE 검증이 따로 필요하다.

## 9. 사례 요약 (PS1 와쿠와쿠 뿌요뿌요 던전)

1998년 컴파일 발매, Mode2/2352, ISO 9660. 데이터 전체가 VDD 아카이브 5개(FIL_SEQ/FIL_GRP/FIL_MAP/FIL_SND/FIL_DMY 더미 패딩)에 들어 있고 내부 파일은 CNX v2 압축. PSX-EXE는 Load 0x80010000, text_size 0xC0000, Initial SP 0x801FFFF0.

- **텍스트**: 표준 Shift-JIS + LE 제어 코드(FF00~FF24). 다수의 SEQ에 텍스트 엔트리가 흩어져 있다. 스토리 SEQ는 포인터 테이블 기반으로 자유 성장 가능(포인터 재계산 + 444B crt0 테일 코드 재배치), 메뉴 SEQ는 RAM 참조가 광범위해 고정길이 교체. 디컴프레스 버퍼(0x80121724~0x80183A00)에 성장 여유가 충분했다.
- **폰트**: BIOS Krom2RawAdd B(0x51) 사용 확인. 글리프 렌더러의 JAL 1개를 트램폴린으로 교체. 부트 디컴프레서의 I-cache 오염 때문에 EXE 확장안 대신 code cave(0x800C8C10) + KSEG1(0xA00C8C10) uncached 실행 + 트램폴린 3 instruction(캐시 라인 1개) 구성을 채택.
- **한글**: SJIS 0x8840~0x8EFC 재사용, 15×15 1bpp 글리프 30B/자. 폰트는 디스크 끝 LBA 301341에 16섹터(31KB)로 추가하고 첫 렌더링 시 CD 직접 하드웨어 프로토콜로 0x80121724에 lazy load.
- **인코딩 규칙 확립**: SJIS는 BE 원시 바이트, 모든 ASCII 전각 2바이트화, 고정길이 패딩은 전각 공백(0x8140), 절단 시 종료자 2바이트 선확보, 번역 데이터에 FF00/FF01 종료자 포함.
- **잔여 제한**: 미번역 일본어(lead 0x88~0x8E)의 한글 오인은 번역 완성으로만 해결, 첫 폰트 로드 시 CD 스트리밍 일시 중단.
- **검증**: 8절에서 적었듯 배포가 되지 않아 실기로는 검증하지 못했다.
