# 닌텐도 DS (NDS) 한글 패치 참조

닌텐도 DS 게임의 한글 패치 제작에 필요한 플랫폼 사실과 일반 패턴을 정리한다.
수치와 구조는 대상 ROM의 헤더·오버레이·렌더 경로를 실측한 뒤 적용한다.

---

## 1. 플랫폼 개요

| 항목 | 값 |
|------|-----|
| CPU (메인) | ARM946E-S (ARM9) 67MHz, ARMv5TE, 리틀엔디언 |
| CPU (서브) | ARM7TDMI 33MHz, ARMv4T, 리틀엔디언 |
| 메인 RAM | 4MB (0x02000000-0x023FFFFF) |
| 공유 WRAM | 32KB (ARM9/ARM7 분할 설정 가능) |
| ARM7 전용 WRAM | 64KB |
| VRAM | 총 656KB (Bank A~I, 용도별 매핑) |
| OAM | 2KB (메인 1KB + 서브 1KB, 128 스프라이트 × 8B) |
| 팔레트 RAM | 2KB (메인 1KB + 서브 1KB) |
| 2D 엔진 | 메인 엔진(상/하 선택) + 서브 엔진, 각 BG 4개 + OBJ |
| 매체 | ROM 카트리지 (NitroROM 파일시스템) |

### ARM9/ARM7 역할 분담

- **ARM9**: 게임 로직, 2D/3D 그래픽, 텍스트 렌더링의 대부분을 담당한다. 한글 패치의 훅은 거의 전부 ARM9 코드에 설치한다.
- **ARM7**: 사운드, 터치스크린, Wi-Fi, 전원 관리를 담당한다. 텍스트 처리에는 관여하지 않는 것이 보통이다.
- 두 CPU는 IPC(Inter-Processor Communication) FIFO와 공유 WRAM으로 통신한다.

### ARM 명령 세트 요점

- **ARM 모드(32비트)** / **Thumb 모드(16비트)**: NDS 게임은 코드 밀도를 위해 대부분 Thumb으로 컴파일된다. `BX` / `BLX`로 모드 전환하며, 주소 bit0=1이면 Thumb, 0이면 ARM이다.
- **BL 명령**: ARM에서 `BL offset`은 4B(±32MB), Thumb에서 `BL offset`은 4B(2개 halfword 조합, ±4MB). 훅 설치 시 도달 범위를 확인한다. 범위 밖이면 트램폴린(`LDR PC, [PC, #offset]` + 주소 리터럴)을 쓴다.
- **LDR 리터럴**: `LDR Rd, [PC, #imm]`으로 코드 근처의 상수 풀에서 주소·값을 읽는다. 포인터나 함수 주소가 코드에 인라인되는 대표 패턴이다.
- **조건 실행**: ARM 모드는 거의 모든 명령에 조건 접미사를 붙일 수 있다(`LDRNE`, `MOVEQ` 등). 훅에서 원본 명령을 보상 실행할 때 조건 플래그 보존에 주의한다.

## 2. 메모리 맵

| 영역 | 주소 범위 | 크기 | 비고 |
|------|-----------|------|------|
| ARM9 ITCM | 0x00000000 (고정) | 32KB | 명령 캐시 대용, 인터럽트 벡터. CP15로 크기만 설정 가능, base 재배치 불가 |
| ARM9 DTCM | 0x03000000 (SDK 기본, CP15로 재배치 가능) | 16KB | 스택, 빠른 변수. devkitPro는 0x02FF0000 사용 |
| 메인 RAM | 0x02000000-0x023FFFFF | 4MB | ARM9/ARM7 공유 |
| 공유 WRAM | 0x03000000 | 32KB | ARM9/ARM7 분할 |
| ARM9 I/O | 0x04000000 | — | 2D 엔진·DMA·타이머 레지스터 |
| 팔레트 RAM | 0x05000000 | 2KB | 메인 0x000-0x3FF, 서브 0x400-0x7FF |
| VRAM | 0x06000000-0x068FFFFF | 가변 | 뱅크 매핑에 따라 배치 |
| OAM | 0x07000000 | 2KB | 메인 0x000-0x3FF, 서브 0x400-0x7FF |
| GBA 슬롯 | 0x08000000-0x09FFFFFF | 32MB | 카트리지 직접 접근(GBA 호환) |
| ARM9 BIOS | 0xFFFF0000 | 4KB | SWI 함수 |
| ARM7 BIOS | 0x00000000 | 16KB | ARM7 전용 |

### ROM 로딩과 오버레이

NDS ROM의 코드는 부팅 시 RAM에 로드되어 실행된다. ROM 파일에서 직접 실행하는 것이 아니라 **RAM 상의 코드**를 패치한다.

- **ARM9 바이너리**: ROM의 `arm9.bin`이 헤더에 지정된 RAM 주소(보통 0x02000000 부근)에 로드된다.
- **ARM7 바이너리**: `arm7.bin`은 ARM7 전용 영역에 로드된다. 텍스트와 무관한 경우가 대부분이다.
- **오버레이(OVL)**: 메모리 절약을 위해 장면·모듈별로 분리된 코드/데이터 블록이다. 오버레이 테이블(OVT)에 각 오버레이의 RAM 주소·크기·압축 여부가 기록된다. **텍스트 엔진이 특정 오버레이에 들어 있으면 해당 OVL 파일을 패치한다.** 같은 RAM 주소에 다른 오버레이가 교대 로드되므로, 훅 대상 오버레이가 로드된 상태에서만 훅이 유효하다.
- **NitroFS 데이터**: `data/` 디렉토리 아래의 파일들. 스크립트·폰트·그래픽 등 게임 리소스가 파일 단위로 정리되어 있다. 파일명과 디렉토리 구조가 보존되므로 분석이 카트리지 ROM보다 수월하다.

### RAM 주소 ↔ 파일 오프셋 변환

ARM9 바이너리의 경우:
```
file_offset = (ram_addr - arm9_ram_address) + arm9_offset_in_rom
```
`arm9_ram_address`와 `arm9_offset_in_rom`은 ROM 헤더에서 읽는다. 오버레이도 OVT에 기록된 RAM 주소와 파일 오프셋으로 동일하게 변환한다. ROM 분해 도구로 ROM을 분해하면 개별 바이너리 파일의 오프셋 0부터 직접 작업할 수 있다.

## 3. 비디오 시스템과 폰트 제약

### 2D 엔진 구조

NDS에는 **메인 엔진**과 **서브 엔진** 두 개의 독립 2D 엔진이 있다. 각각 상단/하단 화면에 배정된다(POWCNT1 레지스터로 전환 가능).

| 항목 | 메인 엔진 | 서브 엔진 |
|------|-----------|-----------|
| BG 수 | 4 (BG0-BG3) | 4 (BG0-BG3) |
| OBJ | 128 스프라이트 | 128 스프라이트 |
| 3D | BG0에 3D 출력 가능 | 불가 |
| 확장 팔레트 | 지원 | 지원 |
| VRAM 뱅크 | A, B, C, D, E, F, G | C, D, H, I |

### VRAM 뱅크 시스템

VRAM은 9개 뱅크(A~I)로 나뉘며, 각 뱅크를 용도별 슬롯에 매핑해 쓴다. 매핑은 `VRAMCNT_x` 레지스터로 설정한다.

| 뱅크 | 크기 | 주요 용도 |
|------|------|-----------|
| A | 128KB | 메인 BG / 메인 OBJ / 텍스처 |
| B | 128KB | 메인 BG / 메인 OBJ / 텍스처 |
| C | 128KB | 메인 BG / 서브 BG / 텍스처 / ARM7 WRAM |
| D | 128KB | 메인 BG / 서브 OBJ / 텍스처 / ARM7 WRAM |
| E | 64KB | 메인 BG / 메인 OBJ / 텍스처 팔레트 |
| F | 16KB | 메인 BG / 메인 OBJ / 텍스처 팔레트 |
| G | 16KB | 메인 BG / 메인 OBJ / 텍스처 팔레트 |
| H | 32KB | 서브 BG |
| I | 16KB | 서브 BG / 서브 OBJ |

폰트 타일을 어느 VRAM 뱅크에 올릴지는 게임의 BG/OBJ 설정을 실측해 결정한다.

### 타일 포맷

GBA와 동일한 포맷을 사용한다.

| 포맷 | 크기 | 용도 |
|------|------|------|
| 4bpp 8×8 | 32B/타일 | BG 텍스트 (16색 팔레트) |
| 8bpp 8×8 | 64B/타일 | BG/OBJ (256색 / 확장 팔레트) |
| 1D 매핑 OBJ | 가변 | 스프라이트 텍스트 |

- 4bpp 타일은 행당 4바이트, 각 바이트에 2픽셀씩 하위 니블=왼쪽 픽셀, 상위 니블=오른쪽 픽셀로 패킹된다 (GBA/NDS 공통).
- 텍스트 BG는 보통 4bpp 모드(BG 모드 0 텍스트)를 쓴다. 타일맵 엔트리는 16비트: 타일 인덱스 10비트(0~1023) + H/V 플립 + 팔레트 번호 4비트.
- **확장 팔레트**: 슬롯당 256색 × 16슬롯으로 BG/OBJ 색 수를 확대한다. 일부 게임이 텍스트에 확장 팔레트를 쓰면 팔레트 슬롯까지 추적해야 한다.

### 비트맵 BG와 소프트웨어 렌더링

NDS 게임 중 상당수는 타일 BG 대신 **비트맵 BG(모드 3~5) 또는 프레임버퍼에 소프트웨어로 텍스트를 직접 그린다**. 이 경우:

- 폰트 데이터가 VRAM 타일이 아니라 RAM/ROM의 비트맵 글리프로 존재한다.
- 렌더러가 글리프를 픽셀 단위로 프레임버퍼에 블리팅한다.
- 타일 인덱스 기반 인코딩이 아니라 유니코드/Shift-JIS 등 문자 코드 기반이 흔하다.
- 한글 폰트를 같은 포맷의 비트맵으로 만들어 교체·확장하면 된다. 타일 제약이 없어 완성형 2,350자+ 수용이 비교적 용이하다.

### 텍스트 렌더링 방식별 분류

| 방식 | 폰트 위치 | 한글화 접근 |
|------|-----------|------------|
| 타일맵 BG | VRAM 타일 | 타일 교체 / 동적 DMA. SNES·GBA와 유사 |
| 비트맵 BG / 프레임버퍼 | RAM 글리프 테이블 | 글리프 테이블 교체. 해상도 여유가 큼 |
| OBJ 스프라이트 | OBJ VRAM 타일 | 스프라이트 타일 교체 |
| 3D 텍스처 | 텍스처 VRAM | 텍스처 이미지 교체 |

## 4. 텍스트 엔진 패턴

### SDK 라이브러리 기반 구조

NDS 게임 대부분은 닌텐도 SDK(NitroSDK / TWL-SDK)를 사용하며, 텍스트 처리도 SDK 함수 위에 구축된다. SDK 함수 시그니처(특히 `GX_LoadOBJ`, `GX_LoadBG`, `G2_SetBGxxx`, `OS_xxx`, `FS_xxx`)를 식별하면 엔진 분석이 빨라진다.

### 파일 기반 리소스

카트리지 ROM 시대와 달리 NDS는 **NitroFS 파일시스템**으로 리소스를 관리한다. 텍스트·폰트·스크립트가 개별 파일로 분리되어 있는 경우가 많다.

```
data/
  font/
    main_font.NFTR      ← NDS 폰트 리소스
    kanji_12.NFTR
  script/
    event_001.bin        ← 스크립트/텍스트 데이터
    system_msg.bin
  graphic/
    title.NCGR           ← 타일 그래픽
    title.NSCR           ← 타일맵
    title.NCLR           ← 팔레트
```

**핵심**: 많은 NDS 게임에서 텍스트 수정은 ARM 바이너리 해킹이 아니라 **NitroFS 내 데이터 파일 교체**로 해결된다. 바이너리 훅은 인코딩 확장·폰트 경로 변경 등 파일 교체만으로 안 되는 경우에 쓴다.

### Nitro SDK 표준 리소스 포맷

NDS 게임 대부분이 사용하는 닌텐도 SDK 표준 리소스 포맷 목록이다. 매직 시그니처로 식별할 수 있다.

| 포맷 | 매직 | 용도 | 한글화 관련도 |
|------|------|------|-------------|
| NFTR | `RTFN` | 폰트 (글리프+폭+매핑) | ★★★ 한글 글리프 추가 대상 |
| BMG | `MESGbmg1` | 텍스트 메시지 (UTF-16 기반) | ★★★ 텍스트 교체 대상 |
| NARC | `NARC` | 파일 아카이브 (NitroFS 내 하위 묶음) | ★★ 리소스 해제/재조립 |
| NCGR | `RGCN` | 타일 그래픽 (2D 캐릭터) | ★★ 그래픽 텍스트 수정 시 |
| NSCR | `RCSN` | 타일맵 (2D 스크린) | ★ 타일맵 BG 텍스트 |
| NCLR | `RLCN` | 팔레트 | ★ 색상 변경 시 |
| NCER | `RECN` | 스프라이트 셀 (OBJ 배치) | ★ OBJ 기반 텍스트 |
| NANR | `RNAN` | 스프라이트 애니메이션 | — 텍스트 애니메이션 시 |

### BMG (Binary Message Group) 포맷

닌텐도 SDK의 **표준 텍스트 메시지 포맷**이다. 마리오 카트 DS, 포켓몬 다이아몬드/펄 등 SDK 기반 게임 다수가 BMG로 텍스트를 저장한다.

```
MESGbmg1 헤더 (32B)
├─ INF1 (Info): 메시지 인덱스·속성 (각 메시지의 DAT1 내 오프셋)
├─ DAT1 (Data): 문자열 데이터 (보통 UTF-16LE, null-terminated)
└─ MID1 (Message ID): 메시지 ID 배열 (선택적 섹션)
```

**한글화에서의 이점**: BMG는 UTF-16LE 기반이므로 한글 코드포인트(U+AC00~U+D7A3)를 인코딩 훅 없이 직접 넣을 수 있다. 폰트에 한글 글리프만 추가하면 된다. INF1의 오프셋과 DAT1의 문자열만 교체/갱신하면 텍스트 교체가 완료된다.

### NARC (Nitro Archive) 포맷

NitroFS 내에서 여러 파일을 하나로 묶는 아카이브 포맷이다. 스크립트·폰트·그래픽이 NARC 안에 들어 있는 경우가 흔하다.

```
NARC 헤더 (16B, 매직 "NARC", BOM 0xFFFE, 섹션 3개)
├─ BTAF (FATB): 파일 오프셋/크기 테이블
├─ BTNF (FNTB): 파일명 테이블 (생략 가능 — 없으면 인덱스로만 접근)
└─ GMIF (FIMG): 실제 파일 데이터 연결 (4바이트 정렬)
```

NARC 내 파일을 수정하면 BTAF의 오프셋을 재계산해야 한다. Python 라이브러리(ndspy 등)로 프로그래밍 처리할 수 있다.

### NFTR 폰트 포맷

닌텐도 공식 폰트 리소스 포맷이다. 많은 NDS 게임이 이 포맷으로 폰트를 저장한다.

```
RTFN 헤더 (16B)
├─ FINF (Font Info): 인코딩, 기본 폭/높이, 글리프 포맷(1bpp~8bpp)
├─ CGLP (Character Glyph): 글리프 비트맵 배열
├─ CWDH (Character Width): 글리프별 (좌측 여백, 글리프 폭, 전진 폭)
└─ CMAP (Character Map): 문자 코드 → 글리프 인덱스 매핑
    ├─ 타입 0: 연속 범위 (시작 코드~끝 코드, 순차 인덱스)
    ├─ 타입 1: 범위 내 개별 매핑 배열
    └─ 타입 2: 이진 탐색 (코드, 인덱스) 쌍
```

한글 폰트 추가 전략:
1. **CMAP에 한글 범위 추가**: 타입 0으로 U+AC00~U+D7A3(완성형 11,172자) 또는 사용 음절 서브셋을 매핑한다.
2. **CGLP에 글리프 추가**: 기존 글리프 뒤에 한글 글리프를 붙인다. 비트 깊이·크기는 기존 일본어 글리프와 맞춘다.
3. **CWDH에 폭 정보 추가**: 한글은 보통 전각(일본어와 같은 폭)이므로 기존 전각 폭을 복사한다.

NFTR을 쓰지 않는 게임은 자체 폰트 포맷을 역공학해야 한다.

### 인코딩 패턴

| 패턴 | 설명 | 한글 확장 |
|------|------|-----------|
| Shift-JIS | 일본판 표준. 1~2바이트 혼합 | SJIS 한자 영역 재사용 또는 EUC-KR/커스텀 확장 |
| UTF-16 | SDK 문자열 함수 기반. 2바이트 고정(BMP) | 한글 코드포인트(U+AC00~) 그대로 사용 가능 |
| UTF-8 | 일부 게임 | 한글 3바이트로 길이 증가. 포인터 재배치 필요 |
| 커스텀 테이블 | 게임 고유 인덱스 | 테이블에 한글 엔트리 추가 |

**UTF-16을 쓰는 게임이면 인코딩 훅 없이 한글을 직접 넣을 수 있다** — NDS 세대의 가장 큰 이점이다. 이 경우 핵심 작업은 폰트에 한글 글리프를 추가하는 것으로 귀결된다.

### 스크립트 포맷

NDS 게임의 스크립트는 바이너리 명령 스트림 형태가 많다.

```
[opcode 1B~2B][파라미터 가변][opcode][파라미터]...
  ├─ 텍스트 표시: opcode + 문자열 (null-terminated 또는 길이 프리픽스)
  ├─ 분기/조건: opcode + 오프셋/라벨
  ├─ 변수 조작: opcode + 변수ID + 값
  └─ 시스템: 대기, 효과음, 화면 전환 등
```

스크립트 파서를 역공학해 opcode 테이블을 확정한 뒤, 텍스트 opcode만 추출·교체하는 도구를 만든다. 길이 프리픽스 방식이면 한글로 길이가 변해도 파라미터 필드만 갱신하면 되지만, 고정 오프셋 기반이면 후속 명령 전체의 오프셋이 밀려 재배치가 필요하다.

## 5. 한글 인코딩과 운용

### 인코딩별 전략

#### UTF-16 게임 (최선)

엔진이 UTF-16이면 한글 완성형(U+AC00~U+D7A3)을 그대로 넣을 수 있다. 남은 작업:
- 폰트 리소스에 한글 글리프 추가 (NFTR이면 CMAP+CGLP+CWDH 확장)
- 텍스트 데이터 파일에서 일본어를 한글로 교체
- 문자열 길이가 변하면 길이 필드·포인터 갱신

#### Shift-JIS 게임

SJIS 한자 영역(0x889F~0x9FFC, 0xE040~0xEAA4 등)에 한글을 대입하는 PS1 방식을 쓸 수 있다. 또는 엔진을 훅해 별도 한글 코드 범위를 추가한다. NDS는 RAM이 4MB로 넉넉하므로 한글 폰트 전체(완성형 2,350자 기준, 12×12 1bpp = ~41KB)를 상주시킬 수 있다.

#### 커스텀 테이블 게임

게임 고유 문자 테이블을 역공학해 한글 엔트리를 추가한다. 테이블 크기 확장이 필요하면 오버레이나 NitroFS 파일을 수정한다.

### 글리프 예산

NDS는 이전 세대보다 자원이 풍부하다.

| 구성 | 글리프 수 | 메모리 |
|------|-----------|--------|
| 완성형 전체 (U+AC00~U+D7A3) | 11,172 | 12×12 1bpp: ~196KB, 16×16 1bpp: ~349KB |
| KS X 1001 완성형 | 2,350 | 12×12 1bpp: ~41KB, 16×16 1bpp: ~73KB |
| 번역 코퍼스 서브셋 | 수백~2천 | 게임별 산정 |

4MB 메인 RAM에서 수십~수백 KB는 확보 가능한 경우가 많으므로, 8/16비트 세대처럼 극단적인 글리프 절약이 필요하지 않다. 다만 VRAM 타일 방식이면 동시 표시 타일 수 제한이 여전히 병목이 될 수 있다.

### 폰트 크기

| 원본 크기 | 한글 권장 | 비고 |
|-----------|-----------|------|
| 12×12 | 12×12 (동일) | 대사창 기본 크기로 가장 흔함 |
| 10×10 | 10×10 또는 12×12 | 10px는 한글 가독성 한계. 가능하면 12px |
| 16×16 | 16×16 | 여유 있는 크기. 그래픽 텍스트에 흔함 |
| 8×8 | 8×8 | 상태창·축약 UI 전용. 본문에는 부적합 |

## 6. 포인터 체계와 재배치

### 파일 기반 구조의 이점

NDS의 파일시스템 구조 덕분에 포인터 처리가 이전 세대보다 단순한 경우가 많다.

- **파일 내부 포인터**: 스크립트 파일 시작 기준의 상대 오프셋이 보통이다. 파일 크기가 변해도 다른 파일에 영향이 없다.
- **FAT(File Allocation Table)**: NitroFS의 FAT가 각 파일의 ROM 내 시작/끝 오프셋을 관리한다. 파일 크기가 변하면 FAT 엔트리와 후속 파일 오프셋을 갱신해야 한다. ROM 재조립 도구가 자동 처리한다.
- **오버레이 테이블(OVT)**: 오버레이 크기가 변하면 OVT의 크기 필드를 갱신한다.

### 바이너리 내 포인터

ARM9/오버레이 바이너리 안에 텍스트가 인라인되어 있으면, 포인터는 RAM 절대 주소(32비트)다.

```
LDR R0, =0x020A1234    ; 문자열 주소 로드
BL  print_message       ; 출력 함수 호출
```

이런 경우 문자열을 옮기면 상수 풀의 주소 리터럴을 함께 갱신해야 한다. 디스어셈블러/디컴파일러에서 문자열 참조를 전수 조사한다.

### 스크립트 파일 내 포인터

흔한 구조:

```
[헤더: 엔트리 수 N]
[오프셋 테이블: N × u32 LE (파일 시작 기준)]
[텍스트 데이터 영역]
```

한글로 길이가 변하면 오프셋 테이블을 재계산한다. 텍스트가 완전 연속(갭 0)이면 1바이트 증가로도 후속 전체가 밀린다 — PS1 SEQ와 동일한 원리다.

## 7. 빌드

### ROM 구조와 도구

NDS ROM(.nds)은 다음 구조다:

```
[NDS 헤더 (4KB)]
[ARM9 바이너리]
[ARM9 오버레이 테이블 + 오버레이 파일들]
[ARM7 바이너리]
[ARM7 오버레이 테이블 + 오버레이 파일들]
[FNT (File Name Table)]
[FAT (File Allocation Table)]
[배너 (아이콘 + 타이틀)]
[NitroFS 데이터 파일들]
```

### 빌드 파이프라인

```
1. ROM 분해: 원본 ROM에서 arm9.bin, arm7.bin, overlay/, data/, banner를 추출
2. 폰트 생성: 한글 글리프를 게임 폰트 포맷(NFTR 또는 자체)에 맞춰 생성
3. 텍스트 인코딩: 번역 데이터를 게임 스크립트 포맷으로 변환 (포인터·길이 재계산 포함)
4. ARM 훅 (필요시): arm9.bin 또는 overlay 바이너리에 코드 패치 적용
5. 리소스 배치: 수정된 파일을 NitroFS 데이터 영역에 배치
6. ROM 재조립: 분해된 구성 요소를 다시 하나의 ROM으로 조립 (FAT·FNT·헤더 자동 재계산)
7. 패치 생성: 원본 ROM과의 바이너리 diff로 배포용 패치 파일 생성
```

NDS ROM의 파일시스템 구조 덕분에 재조립 시 FAT·FNT·헤더가 자동 재계산되므로, 파일 크기 변경에 따른 수동 오프셋 조정이 불필요하다. 이것이 NDS 패치의 큰 이점이다.

NDS 리소스 포맷(NARC·BMG·NFTR 등)의 프로그래밍 조작에는 Python 라이브러리(ndspy 등)가 적합하다. ARM 크로스 컴파일러/어셈블러(arm-none-eabi-gcc/as)는 훅 코드 작성에, BLZ 압축 해제 도구는 ARM9 바이너리 패치 전처리에 사용한다. 디스어셈블리·디컴파일은 ARM 바이너리 분석 도구(Ghidra 등)로 수행한다. 구체적 도구 선택은 대상 게임의 리소스 구조와 실행 환경에 맞춰 그 시점에 결정한다.

### ARM9 secure area

ARM9 바이너리의 첫 2KB(ROM 오프셋 0x4000~0x47FF)는 **secure area**로 암호화되어 있다. arm9.bin을 패치하려면 먼저 복호화해야 한다. 표준 ROM 분해 도구는 추출 시 자동 복호화하고 재조립 시 자동 재암호화한다. 수동으로 ROM을 직접 편집하는 경우 secure area 처리를 별도로 해야 한다.

### 배너 한글 타이틀

NDS 배너(아이콘+타이틀)는 버전에 따라 지원 언어가 다르다.

| 버전 | 추가 언어 |
|------|-----------|
| v1 (0x0001) | JP, EN, FR, DE, IT, ES |
| v2 (0x0002) | + 중국어 (간체) |
| v3 (0x0003) | + **한국어** (offset 0x0940) |
| v0103 | + DSi 애니메이션 아이콘 |

한글 패치 배포 시 배너의 한글 타이틀 필드도 수정하면 플래시 카트/홈 메뉴에서 한글 제목이 표시된다. 배너 버전이 v1/v2이면 v3으로 올려야 한글 필드가 인식된다.

### 배포

- xdelta3가 NDS 씬의 주류 패치 포맷이다. BPS도 사용 가능하다.
- NDS ROM 자체는 커밋하지 않는다. 패치 파일 + 빌드 스크립트 + 번역 데이터만 배포한다.
- 대상 ROM의 식별은 No-Intro DAT의 CRC32/SHA-1로 한다. 헤더에 게임 코드(4자, 예: `ADAJ`)와 리비전이 있으므로 함께 기록한다.

### 압축 대응

NDS 게임은 파일 단위 압축이 흔하다.

| 압축 | 식별 | 비고 |
|------|------|------|
| LZ77 (닌텐도 표준) | 매직 `0x10` | SDK `MI_UncompressLZ77` |
| LZ11 | 매직 `0x11` | 더 높은 압축률 |
| Huffman | 매직 `0x20`/`0x28` | SDK `MI_UncompressHuffman` |
| RLE | 매직 `0x30` | SDK `MI_UncompressRL` |
| BLZ (Bottom LZ) | arm9.bin 전용 | ARM9 바이너리 뒷부분 압축. 패치 전 해제 필수 |

ARM9 바이너리가 BLZ 압축되어 있으면 패치 전에 해제해야 한다. 빌드 시 해제 상태로 두고 ROM을 재조립해도 동작한다(부팅 시 디컴프레서가 무압축을 감지해 스킵). NitroFS 파일이 LZ77/LZ11로 압축된 경우 해제 → 수정 → 재압축 → 교체 순서를 밟는다. 재압축 후 크기가 원본보다 커질 수 있으므로 ROM 재조립으로 FAT를 갱신한다.

## 8. 검증

### 에뮬레이터 선택

| 에뮬레이터 | 강점 | 약점 |
|-----------|------|------|
| **melonDS** | 높은 정확도, Wi-Fi 에뮬레이션, 오픈소스 | 내장 디버거 기능이 제한적 |
| **DeSmuME** | 강력한 내장 디버거(메모리 뷰어, 디스어셈블러, 브레이크포인트), Lua 스크립팅 | 정확도가 melonDS보다 낮음 |
| **no$gba** | 가장 상세한 디버거(I/O 뷰어, 트레이스, 조건 브레이크포인트) | 클로즈드 소스, 일부 게임 호환성 이슈 |

**권장 워크플로**: 디버거가 강력한 에뮬레이터(조건 브레이크포인트·트레이스 로그·메모리 뷰어 지원)로 분석·디버깅하고, 정확도가 높은 에뮬레이터로 최종 검증한다.

### 에뮬레이터 디버깅 기능

NDS 에뮬레이터가 제공하는 디버깅 기능 중 한글 패치에 특히 유용한 것:

- **조건 브레이크포인트**: `[02012345]=1234h` 같은 메모리 조건으로 특정 텍스트 코드가 처리될 때 멈출 수 있다.
- **트레이스 로그**: 실행 흐름을 파일로 덤프해 텍스트 렌더러 호출 경로를 역추적한다.
- **I/O 맵**: VRAM·DMA·2D 엔진 레지스터 상태를 실시간으로 관찰한다.
- **메모리 뷰어**: ARM9/ARM7 주소 공간, VRAM 각 뱅크를 직접 열람한다.
- **디스어셈블러**: ARM/Thumb 모드 자동 인식. 실행 중 스텝 실행 가능.
- **타일/맵/OAM 뷰어**: 폰트 타일이 올바르게 로드되었는지 시각 확인.
- **스크립팅(Lua 등)**: 메모리 읽기/쓰기, 프레임 콜백, 스크린샷 자동화.

### 자동 검증

- 에뮬레이터 스크립팅으로 텍스트 표시 트리거(메모리 감시) → 스크린샷 → 레퍼런스 이미지와 비교하는 회귀 테스트를 구축할 수 있다.
- 스크립트 전량에 대해 인코딩 테이블 매핑 누락·길이 초과를 빌드 타임에 검증한다.

### 텍스트 QA 체크리스트

```
[ ] 한글 폰트 전수 표시 (깨짐·비트 깊이 불일치 없음)
[ ] 대사창 줄바꿈·페이지 넘김 정상
[ ] 이름 입력 화면 한글 입력 (해당 시)
[ ] 상단/하단 양쪽 화면의 텍스트 정상
[ ] 터치스크린 UI 텍스트 위치·크기 정상
[ ] 세이브/로드 화면 텍스트
[ ] 전투/메뉴/아이템/스킬 등 모든 UI
[ ] 오버레이 교대 로드 후 텍스트 정상 (장면 전환)
[ ] 장시간 진행 후 메모리 누수·크래시 없음
```

### 실기 구동 경로

- **수단**: 플래시 카트(R4 계열, Ace3DS+, DSTT 등)로 검증한다. DSi에서는 TWiLight Menu++/nds-bootstrap으로 SD 카드 직접 실행도 가능하다.
- **제약**: 구형 플래시 카트는 ROM 크기 한계(보통 256MB까지, NDS ROM은 최대 512MB이나 대부분 128MB 이하)나 AP(Anti-Piracy) 패치가 필요할 수 있다. AP 우회는 게임별로 다르다.
- **빌드 포맷에의 영향**: ROM 크기가 늘어나면 플래시 카트 호환성을 확인한다. AP 루틴이 ROM CRC를 검사하는 게임은 패치 후 AP 우회가 추가로 필요할 수 있다.
- **함정**: 에뮬레이터에서 동작하지만 실기에서 화이트스크린이면 AP, ARM9 BLZ 미해제, 또는 헤더 체크섬 불일치를 먼저 의심한다. DSi 전용 게임(TWL)은 DS 플래시 카트로 구동 불가하다.
- **중간 검증**: melonDS의 높은 정확도가 에뮬레이터와 실기 사이의 중간 검증 역할을 한다.

## 9. 사례 요약

NDS 플랫폼은 새로 추가된 문서이며, 선행 한글화 사례가 아직 축적되지 않았다. 프로젝트가 진행되는 대로 이 절에 추가한다.

### 일반 관찰

NDS 한글 패치는 8/16비트 세대와 비교해 다음 차이가 있다.

| 항목 | 8/16비트 세대 | NDS |
|------|--------------|-----|
| 리소스 접근 | ROM 직접 읽기, 바이트 단위 오프셋 | NitroFS 파일시스템, 파일 단위 교체 |
| 인코딩 | 게임 고유 테이블이 대부분 | UTF-16·SJIS 등 표준 인코딩 빈도 높음 |
| 폰트 | VRAM 타일 제한으로 글리프 수 병목 | RAM 여유로 완성형 2,350자+ 수용 가능 |
| 폰트 포맷 | 게임별 역공학 필수 | NFTR 등 공통 포맷 존재 |
| 포인터 | ROM 오프셋 직접 패치 | 파일 내 상대 오프셋 + 재조립 시 자동 FAT 갱신 |
| 코드 패치 | 어셈블리 직접 주입 | 오버레이 시스템으로 모듈 단위 패치 가능 |
| 빌드 도구 | 게임별 자체 빌드 파이프라인 | 표준화된 ROM 분해/재조립 도구 존재 |

이런 특성 덕분에 NDS는 한글 패치 진입 장벽이 상대적으로 낮은 플랫폼이지만, 게임별 스크립트 포맷·압축·AP의 역공학은 여전히 필요하다. strategy 축의 조사 순서·검증 원칙을 출발점으로 삼되, NDS 고유의 파일시스템·오버레이·VRAM 뱅크 구조에 맞춰 적용한다.
