# 세가 새턴 (Sega Saturn) 한글 패치 참조

## 1. 플랫폼 개요

새턴(1994)은 다중 프로세서 아키텍처를 채택한 2D/3D 콘솔이다. 한글 패치 관점에서 핵심 프로세서는 다음과 같다.

| 프로세서 | 역할 | 클럭 |
|----------|------|------|
| Master SH-2 (SH7604) | 메인 CPU, 게임 로직 | 28.6363 MHz |
| Slave SH-2 (SH7604) | 보조 CPU, 병렬 처리 | 28.6363 MHz |
| SH-1 (SH7034) | CD-ROM 컨트롤러 | 20 MHz |
| MC68EC000 | 사운드 CPU (SCSP 제어) | 11.2896 MHz |
| SCU DSP | DMA/연산 보조 | 14.3182 MHz |

SH-2 분석 시 핵심 특성:

- **Big-endian**, 명령어 고정 16비트(2바이트)다. 모든 명령어가 2바이트 경계에 정렬된다.
- 즉시값 로드가 8비트로 제한되어 32비트 상수는 `MOV.L @(disp,PC), Rn`으로 코드 근처의 리터럴 풀에서 읽는다. 주소/상수를 패치할 때 리터럴 풀 위치를 반드시 확인한다. 참조 주소 계산: `(PC & 0xFFFFFFFC) + 4 + disp * 4`.
- `BRA`·`BSR`·`JMP`·`JSR`·`RTS`·`RTE`·`BT/S`·`BF/S`는 delay slot을 가진다(단순 `BT`/`BF`는 갖지 않는다. 레지스터 상대 `BRAF`/`BSRF`도 갖는다). 분기 직후 1개 명령어가 분기 전에 실행되므로 함수 호출 무효화 시 delay slot까지 함께 NOP(`0x0009`) 처리한다. delay slot에는 다른 분기·PC 상대 리터럴 풀 로드(`MOV.L @(disp,PC)`, `MOVA`)·`TRAPA`를 둘 수 없다(illegal slot 예외). delay slot을 재배치할 때 리터럴 풀 로드를 그 자리로 옮기지 않는다.
- 조건 분기 반전: `BT(0x89xx) ↔ BF(0x8Bxx)`, `BT/S(0x8Dxx) ↔ BF/S(0x8Fxx)`.
- Ghidra/IDA 설정: 프로세서 SH-2, Big-endian, 베이스 주소는 게임 바이너리 로드 주소(보통 `0x06010000` 부근).

부팅 흐름: BIOS가 디스크 선두 16섹터의 IP.BIN을 `0x06002000`에 로드해 실행하고, IP.BIN이 헤더에 기록된 1st Read Address/Size에 따라 메인 실행 파일(1ST_READ.BIN 등)을 Work RAM High에 로드한다. 게임 코드 디스어셈블은 이 로드 주소를 기준으로 시작한다.

## 2. 메모리 계층

새턴은 CPU RAM / VDP1 / VDP2의 삼중 메모리 구조를 가지며, 텍스트 데이터는 이 계층을 단계적으로 거친다.

| 주소 범위 | 크기 | 영역 |
|-----------|------|------|
| `0x00200000` - `0x002FFFFF` | 1 MB | Work RAM Low — 저속 DRAM, CD 로딩 버퍼·데이터 테이블 용도 |
| `0x06000000` - `0x060FFFFF` | 1 MB | Work RAM High — SH-2 직접 버스, 게임 코드·스택·메인 데이터 |
| `0x05C00000` - `0x05C7FFFF` | 512 KB | VDP1 VRAM (스프라이트 텍스처 + 커맨드 테이블) |
| `0x05C80000` - `0x05CBFFFF` | 256 KB ×2 | VDP1 프레임버퍼 (더블 버퍼) |
| `0x05E00000` - `0x05E7FFFF` | 512 KB | VDP2 VRAM (128 KB 뱅크 A0/A1/B0/B1) |
| `0x05F00000` - `0x05F00FFF` | 4 KB | VDP2 CRAM (팔레트) |

- CD에서 로드된 실행 파일과 스크립트는 대부분 `0x06000000`대(Work RAM High)에 배치된다.
- 일부 게임은 스크립트 파일을 Work RAM Low(`0x0020xxxx`)에 로드하고, 파일 내부 포인터가 이 RAM 베이스 주소를 그대로 사용한다. 포인터 패턴 분석의 출발점이 된다(6절).
- VDP2 VRAM은 캐시 스루 주소 `0x25E00000`대로도 접근된다. 폰트 데이터 흐름의 전형: CD → 압축 해제 → Work RAM 버퍼 → CPU memcpy로 스테이징 버퍼 → VDP2 VRAM.

## 3. 비디오 시스템과 글리프 한계

### VDP1 / VDP2 역할 분담

- **VDP1**: 스프라이트·폴리곤. VRAM 내 32바이트 커맨드 테이블을 순차 실행한다. 텍스트를 Normal Sprite로 그리는 경로를 가진 게임도 있으나, 분석 사례에서는 대화 텍스트에 사용되지 않았다.
- **VDP2**: 배경 스크롤 플레인. NBG0-NBG3(일반), RBG0/1(회전). 셀(타일) 기반이며 셀 크기는 8x8 또는 16x16, 색상 모드는 16색/256색 등.

대화 텍스트가 어느 하드웨어로 렌더링되는지는 에뮬레이터의 레이어 토글로 확정한다. 분석 사례에서는 NBG3을 비활성화하면 대화 텍스트 전체가 사라져 대화 텍스트 = VDP2 NBG3로 확정했다. 바이너리에 VDP1 텍스트 경로가 존재해도 실제 사용 여부는 별개이므로 반드시 실측한다.

### PND character number 상한과 글리프 슬롯 산정법

VDP2 타일맵의 PND(Pattern Name Data) 엔트리는 16비트이며 구성은 다음과 같다.

```
[15:12] 팔레트 인덱스
[11:0]  character number (한 사례의 PNT 기록 코드가 0x0FFF 마스크를 적용해 12비트 운용)
```

1워드·12비트 character number 설정에서는 참조 가능한 8x8 타일이 최대 4,096개다. 16x16 글리프는 2x2(4타일) 조합이므로, 게임이 예약한 비글리프 타일(ASCII·UI·와이드 문자 등)을 빼고 4로 나눈 값이 16x16 글리프 슬롯 최대치다.

일반식:

```
글리프 슬롯 = (2^(character number 비트 수) − 게임 예약 타일 수) ÷ 글리프당 타일 수
```

(`references/strategy/font-strategy.md` 3절의 "타일 인덱스 폭" 병목.) 예약 타일 수는 게임마다 다르므로 슬롯 수는 게임별로 재산정한다. 이 산정은 1워드 PND 모드 기준이며, 2워드 PND 모드는 더 큰 character number를 지원하므로 비트 수 입력 자체가 달라진다. 폰트 파일을 해당 모드의 타일 상한을 넘게 확장하면 화면이 깨진다.

### 글리프 슬롯 운용 전략

PND character number 상한이 완성형 2,350자보다 작게 나오는 설정이라면 다음 전략을 조합한다.

1. **동적 수집 할당**: 번역 전체에서 실제 사용되는 한글만 수집(정렬된 집합)하여 슬롯에 할당한다. 수요가 슬롯 한계를 넘으면 번역 어휘를 조정해 미할당 0으로 맞춘다.
2. **보존 슬롯 분리**: 게임 UI 전용 픽셀아트(아이콘, 버튼 표기, 게이지 바, 음표 등)는 원본 비트맵을 보존하고 할당에서 제외한다.
3. **재렌더링 자리 유지**: 한국어에서도 쓰는 문자(숫자, 영문, 괄호, 기호)는 한글 폰트로 다시 렌더링하되 동일 타일 코드에 배치해 기존 스크립트 참조를 그대로 유지한다.
4. **슬롯 회수**: 일본어 전용 문자(가나, 한자, 장음 부호)와 미사용 와이드 문자 슬롯을 한글에 재할당한다.
5. **직접 매핑**: 와이드(16x8) 영역에 재렌더링한 문자는 16x16 슬롯을 따로 쓰지 않고 와이드 타일 코드를 직접 사용해 슬롯을 절약한다.
6. **투명 타일 주의**: all-zero 타일이라도 VDP2 배경 패턴이 투명 타일로 참조 중일 수 있다. 덮어쓰면 배경에 글리프가 노출되므로 참조 분석 없이 회수하지 않는다.

폰트 파일의 16x16 글리프는 보통 8x8 타일 4개(TL, TR, BL, BR)로 분해 저장된다. 4bpp 기준 타일당 32바이트, 글리프당 128바이트. 인터리브 배치(앞 절반 타일 = 상반부, 뒤 절반 = 하반부)를 쓰는 게임도 있으므로 블록 구조를 먼저 파악한다.

## 4. 텍스트 엔진 패턴

### 인코딩: 타일 인덱스 직접 참조

Shift-JIS가 아니라 폰트 파일 타일 인덱스를 2바이트 big-endian으로 직접 인코딩하는 게임이 있다. 한 사례의 타일 인덱스 체계:

| 타일 코드 | 타입 | 변환 |
|-----------|------|------|
| `0x0000`-`0x005F` | 8x8 ASCII | 직접 사용 |
| `0x0060`-`0x007F` | 8x8 UI 그래픽 | 직접 사용 |
| `0x0080`-`0x01B5` | 16x8 와이드 문자 (155쌍) | `wide_index = (code - 128) / 2` |
| `0x01B6` 이상 | 16x16 글리프 (2x2 타일) | `glyph_index = (code - 438) / 4` |
| `0xFF00`-`0xFFFF` | 제어 코드 | high byte `0xFF`로 식별 |

SH-2 코드에서 `SHLL8`/`OR`로 lead/trail 바이트를 결합해 16비트 타일 인덱스를 만든다. 초기 분석에서 SJIS 스캐너는 부분 매칭만 되므로, 타일 인덱스 디코더를 만들어야 전체 텍스트가 복원된다.

### SEQ 스크립트 파일 구조

이하는 한 사례의 이벤트 스크립트 포맷(SEQ)이다. 자기기술 메타데이터가 없는 전형적인 실행 전용 포맷이다.

```
┌──────────────────────────────────────┐
│ 스크립트 코드 (0x0000 ~ text_start)  │  ← 텍스트 블록을 가리키는 절대 오프셋 포인터 포함
├──────────────────────────────────────┤
│ 텍스트 블록 영역                      │
│  [gap] [text FF02...FF05] [gap] ...  │
├──────────────────────────────────────┤
│ 이벤트/NPC/맵 데이터 (있는 경우)      │  ← 64KB 초과 파일은 두 번째 뱅크 포인터 포함
└──────────────────────────────────────┘
```

- 헤더(offset 0)에는 2바이트 모듈 ID만 있고 텍스트 시작 오프셋 필드가 없다. 텍스트 영역 감지는 포인터 타겟 역방향 스캔 + 제어코드 밀도 + 2바이트 정렬 검증의 다층 휴리스틱이 필요하다.
- 텍스트 블록 사이에는 고정 패턴의 갭(같은 화자 연속 12바이트, 씬 전환 36바이트)이 삽입되며, 씬 포인터는 갭 내부의 `FF 0F`(씬 시작) 명령을 가리킨다.
- 대사 패턴: `[타일 코드...] FF02 [타일 코드...] FF02 ... FF02 FF05`.
- 코드 영역의 2바이트 필드값이 글리프 코드 범위와 광범위하게 겹쳐 거짓양성을 만든다. 제어코드도 와이드 문자도 없는 추출 엔트리는 강한 노이즈 후보로 분류하되, 자동 삭제 전에 화면 대조나 주변 구조로 한 번 더 확인한다.
- 일부 파일은 `FF00` 구분자만으로 항목을 나열하는 이름 테이블(기술명·아이템명)을 코드/데이터 영역에 둔다. 길이 헤더가 없으므로 원본 길이에 맞춰 패딩하는 고정길이 패치로 처리하면 포인터 수정이 불필요하다.

### 제어 코드 디스패치 (FF00-FF20)

엔진은 16비트 값을 경계값 `0xFEFF`와 비교해 분기한다.

- `> 0xFEFF` (즉 `0xFF00` 이상): 제어 코드. `(code & 0xFF) << 2`를 인덱스로 디스패치 테이블에서 핸들러 포인터(4바이트)를 로드해 JSR.
- `≤ 0xFEFF`: 글리프 코드. 텍스트 오브젝트에 저장 후 렌더링 경로로.

분석 사례의 디스패치 테이블은 FF00-FF20 범위(미정의 코드는 NOP 핸들러)였고, 텍스트 종료(FF00), 줄바꿈(FF02), 렌더링 모드 설정(FF0F/FF10/FF11) 등이 확인됐다. 다른 사례에서는 FF00(줄바꿈/구분), FF02(라인 표시), FF05(블록 종료), FF30(화자/초상화, 3워드 파라미터), FF3D(페이지 전환, 2바이트 페이지 번호 + 4바이트 RAM 포인터) 등 게임별 의미가 달랐다. **제어 코드의 파라미터 개수를 정확히 파악하지 못하면 파서가 파라미터를 글리프로 오인한다.** 패딩 위치도 게임별 종료자 규약으로 정한다. 이 분석 사례처럼 trailing `FFxx` 제어코드 시퀀스가 종료 처리와 함께 소비되는 엔진에서는 패딩을 마지막 비-제어코드 위치(제어코드 시퀀스 앞)에 삽입한다. 일반 판정 절차는 `references/strategy/reinsertion.md` §6.3을 따른다.

### VDP2 NBG 텍스트 렌더링 콜체인

분석 사례에서 확인된 렌더링 콜체인 구조:

```
Master Text Render Dispatcher        ← 텍스트 오브젝트 배열(12슬롯, 슬롯당 ~150B) 순회
  → VDP2 Text Wrapper
    → VDP2 Text Engine (메인 루프)   ← 16비트 BE tile code 읽기, 제어코드/글리프 분기
      → VDP2 Pattern Name Writer    ← tile code를 VDP2 PNT에 기록
      → Bounds Check + Dispatch     ← VDP1 스프라이트 경로 (플래그 기반)
      → Control Code Dispatch Table ← FF00-FF20 핸들러
```

Pattern Name Writer의 처리 단계:

1. 폰트 디스크립터 룩업: 텍스트 오브젝트의 폰트 셋 인덱스로 디스크립터(28바이트 단위 배열)에서 VRAM character base와 맵 베이스를 얻는다.
2. character number 계산: `base 오프셋을 물리주소 보정 후 >>7 (÷128)` 하고 tile code를 그대로 가산한다. 즉 tile code = VDP2 character number의 직접 인덱스다. (보정 상수는 한 사례의 게임 바이너리에서 관측된 값이라 게임마다 다르다. 자기 게임에서 실측한다.)
3. 화면 좌표 → PNT 주소: `(Y/8 * row_stride + X/8) * 2 + map_base`.
4. 렌더링 모드별 쓰기. 16x16 모드는 4개 PNT 엔트리를 쓴다:

```
TL = tile_code + 0,  TR = tile_code + 1
BL = tile_code + 64, BR = tile_code + 65
```

모드(8x8 / 8x16 / 16x16)는 렌더링 시점이 아니라 제어 코드가 사전에 플래그로 설정한다. 따라서 어떤 tile code든 모드만 맞으면 임의 크기로 렌더링되며, 폰트 파일의 타일 배치(인터리브)만 이 오프셋 패턴(+0/+1/+64/+65)과 일치시키면 기존 엔진 코드를 수정 없이 재사용할 수 있다.

## 5. 압축 — CNX v2 (Greedy LZ77)

폰트·스크립트 파일이 CNX v2(컴파일계 게임들이 공유하는 공통 포맷)로 압축된 사례가 확인됐다. 이 절만으로 재구현이 가능하도록 기술한다.

### 컨테이너 헤더 (16바이트)

| 오프셋 | 크기 | 내용 |
|--------|------|------|
| 0 | 4 | 매직 `"CNX\x02"` (BE u32 `0x434E5802`) |
| 12 | 4 | 해제 후 크기 (big-endian u32) |

게임 디컴프레서는 압축 크기 필드를 읽지 않고 플래그 바이트 종결자에 의존한다. 압축 스트림은 오프셋 16부터 시작하며, 해제 루프는 출력 누계가 해제 크기에 도달할 때까지 반복된다.

### 스트림 구조

```
[플래그 바이트] [op0 데이터] [op1 데이터] [op2 데이터] [op3 데이터] [플래그 바이트] ...
```

- 플래그 바이트 1개가 2비트 모드 코드 4개를 담는다. 처리 순서는 하위 비트부터다: bits [1:0] → [3:2] → [5:4] → [7:6] (각 op 처리 후 우측 2비트 시프트).
- 플래그 바이트가 `0x00`이면 스트림 종료.

### 모드별 동작

**모드 0 (`00`) — 스킵/브레이크**

```
skip_count = src[0]            // 포인터 전진 없이 읽음
src += skip_count + 1          // skip_count 바이트 + 자기 자신을 건너뜀
내부 루프 강제 종료 (다음 플래그 바이트로)
```

스트리밍(부분 해제) 모드에서는 모드 0 발생 횟수를 카운터로 사용해 지정 횟수 도달 시 함수를 빠져나간다.

**모드 1 (`01`) — 단일 리터럴**: 소스에서 1바이트 읽어 출력에 쓴다.

**모드 2 (`10`) — LZ 매치**

```
pair     = (src[0] << 8) | src[1]      // 2바이트 big-endian
length   = (pair & 0x1F) + 4           // 4..35
distance = (pair >> 5) + 1             // 1..2048
source   = output_pos - distance
output_pos부터 length 바이트를 source에서 전방(byte-by-byte) 복사
```

전방 복사이므로 자기참조 매치(distance < length)는 앞 distance 바이트의 순환 반복을 생성한다. 압축기에서 매치 검증 시 `src[i % distance]`로 모델링한다.

**모드 3 (`11`) — 다중 리터럴**: 카운트 바이트 1개를 읽고, 그 수만큼 리터럴 바이트를 출력에 복사한다 (카운트 ≤ 0이면 스킵).

### 압축기 파라미터

| 요소 | 값 |
|------|-----|
| 알고리즘 | Greedy LZ77 |
| 최소 매치 길이 | 4 바이트 |
| 최대 매치 길이 | 35 바이트 (5비트 + 4) |
| 윈도우(매치 거리 상한) | 2,048 바이트 |
| 플래그 바이트 | 2비트 코드 ×4 |

### 함정: 거리 상한과 출력 위치

게임 디컴프레서는 링 버퍼를 쓰지 않고 출력 버퍼에서 직접 `output_pos - distance`를 읽는다. 따라서 압축기가 `distance > 현재까지 출력된 바이트 수`인 매치를 생성하면 게임은 출력 버퍼 앞쪽 미정의 메모리를 읽어 데이터가 깨진다. 0으로 초기화된 링 버퍼를 쓰는 자체 디컴프레서로는 라운드트립 테스트가 통과해 버려 이 버그가 은폐된다.

- **올바른 매치 탐색 범위**: `distance ∈ 1..=min(pos, 2048)`.
- 실측 영향: 선두가 0x00 4바이트 이상으로 시작하는 파일(스프라이트·배경 등)이 잘못된 압축기로 재압축 시 조용히 손상됐다.
- 검증은 자체 디컴프레서 라운드트립만으로 부족하며, 게임 디컴프레서의 동작(출력 버퍼 직접 참조)을 그대로 모사한 검증기가 필요하다.

## 6. 포인터 체계 — 다중 패턴 분류와 일괄 수정

번역으로 텍스트 길이가 바뀌면 후속 블록이 이동하지만 스크립트 코드의 포인터는 원래 오프셋을 유지해, 빈 텍스트 박스·소프트락이 발생한다. 포인터가 파일 유형마다 완전히 다른 규약으로 섞여 있으면 유형별 전용 스캐너로 나눠 처리한다. 한 규약의 스캐너로 전부 잡으려 하면 누락이 생긴다.

### 패턴 분류 (사례 기준 5종)

| 유형 | 포인터 패턴 | 길이 | 비고 |
|------|------------|------|------|
| MP (맵 이벤트) | `00 24 XX XX 00 00 00 05` | 8B | `XX XX` = 2바이트 BE 파일 오프셋, RAM 베이스 `0x0024xxxx` |
| PT (파트 이벤트) | `00 25 XX XX 05 00 00 00` | 8B | 서픽스 바이트 순서가 MP와 역순이라 단순 프리픽스 교체 불가 |
| Other (미니게임·던전·엔딩 등) | `00 24 XX XX 00 00 00 05` | 8B | MP와 동일 패턴. 누락 시 해당 콘텐츠만 버그 발생 |
| COMMON (공용 데이터) | `00 20 XX XX` / `00 21 XX XX` | 4B | RAM 베이스 `0x00200000`. 다단계 간접 참조(1차→2차→3차 테이블→텍스트) |
| DIARY (일기) | `00 27 XX XX` (4B BE RAM 주소) | 4B | 제어코드 파라미터 내부에도 포인터가 존재 |

포인터 상위 2바이트는 해당 파일이 로드되는 Work RAM 세그먼트 베이스(`0x0020xxxx`, `0x0024xxxx` 등)이고 하위 2바이트가 파일 내 오프셋이다. 즉 로드 주소를 알면 포인터 패턴을 역으로 도출할 수 있다.

### 일괄 수정 알고리즘

1. **원본** 파일에서 유형별 포인터 패턴을 스캔한다 (패치 후가 아니라 원본 기준).
2. 각 패치의 `(offset, orig_len, new_len)`으로 누적 이동량(shift)을 계산한다.
3. 포인터 타겟이 이동한 경우 새 값으로 갱신한다. 포인터 자체는 패치 영역 앞(스크립트 코드)에 있어 위치 불변.
4. 오버플로우 체크 (2바이트 오프셋은 0x0000-0xFFFF 범위).
5. 엔트리별 4바이트 정렬 패딩: 크기 변화량이 4의 배수가 아니면 공백 타일을 삽입해 맞춘다. 패딩 위치는 4절의 제어코드/종료자 분석과 `references/strategy/reinsertion.md` §6.3 기준으로 정한다.

패치 자체는 오프셋 역순 splice(뒤에서부터 적용)로 수행해 앞쪽 오프셋이 유효하게 유지되도록 한다.

### 거짓양성 방어

포인터 패턴은 짧아서 데이터 영역에서 우연히 출현한다(4바이트 패턴은 거짓양성이 실제 포인터보다 훨씬 많을 수 있다). 필수 방어책:

- **범위 검증**: 타겟이 파일 크기 이내인지.
- **정렬 검증**: 스크립트 포인터가 2바이트 정렬임을 확정한 뒤, 홀수 타겟은 스킵 후보로 배제한다. 한 사례에서는 데이터 영역의 좌표값 37(`0x25`)이 포인터 프리픽스 바이트와 우연히 겹쳐 포인터로 오인돼 맵 전환 코드가 손상됐다.
- **서픽스 검증**: 모호한 매치(뱅크 경계 넘는 경우 등)는 고정 서픽스 일치를 요구한다. 반대로 서픽스 블랙리스트 방식은 거짓음성(NPC 미표시)을 유발해 폐기됐다.

### 듀얼 뱅크 (64KB 초과 파일)

2바이트 오프셋은 64KB까지만 표현하므로, 64KB 초과 파일은 두 번째 뱅크 프리픽스를 쓴다(사례: `0024` = 0x0000-0xFFFF, `0025` = 0x10000 이상, `ptr_val + 0x10000 = 파일 오프셋`). 텍스트 축소/확장으로 타겟이 0x10000 경계를 넘으면 프리픽스 자체를 자동 전환(bank-crossing)해야 한다.

## 7. 디스크 포맷 — ISO 9660, LBA 재배치, EDC/ECC

### BIN/CUE와 섹터 구조

새턴은 Mode 1 데이터 트랙 + CD-DA 오디오 트랙 구성이 일반적이다. BIN 파일은 raw 2352바이트 섹터의 연속이다.

```
오프셋   크기    내용
0x000   12      Sync (00 FF FF FF FF FF FF FF FF FF FF 00)
0x00C   4       Header (분:초:프레임 BCD + 모드 0x01)
0x010   2048    User Data
0x810   4       EDC
0x814   8       Reserved (0x00)
0x81C   276     ECC (P-parity 172 + Q-parity 104)
```

핵심 변환 공식:

```
BIN 오프셋(User Data 시작) = LBA * 2352 + 16
```

CUE의 `INDEX MM:SS:FF`는 `(MM*60+SS)*75+FF` 섹터다. 디스크 레이아웃: LBA 0-15 = IP.BIN(System Area, 32 KB), LBA 16 = ISO 9660 PVD(`"CD001"` 시그니처), LBA 18 이후 파일시스템과 데이터.

### ISO 9660 파일 접근

- PVD(LBA 16) 오프셋 `0x9C`의 Root Directory Record에서 루트 디렉토리 LBA/크기를 얻는다. 멀티바이트 정수는 both-endian(LE 4B + BE 4B) 기록.
- Directory Record: `+0x02` extent LBA(8B both-endian), `+0x0A` 데이터 크기(8B both-endian), `+0x19` 플래그(bit1 = 디렉토리), `+0x21` 파일명(`NAME.EXT;1` 형식).
- 각 파일은 연속 섹터(단일 extent)에 저장된다. 파일 읽기/쓰기는 섹터마다 User Data 2048바이트만 추출/기록한다.

### 파일 재배치 (LBA 갱신)

번역으로 압축 파일이 원본 섹터 할당을 초과하면 그 자리에서 늘릴 수 없다(다음 파일이 바로 이어짐). 해결 절차:

1. CD-DA 트랙 직전의 빈 영역에서 필요한 섹터 수만큼 새 LBA를 확보한다 (가용 여유는 디스크별로 실측한다).
2. 새 LBA에 파일 데이터를 기록한다.
3. ISO 9660 디렉토리 엔트리의 extent LBA와 크기를 both-endian으로 동시 갱신한다.
4. 게임이 자체 파일 크기 테이블을 가진 경우(메인 실행 파일 내 하드코딩) 그 테이블도 패치한다.

크기가 섹터 할당 내에서만 변하면 디렉토리 엔트리의 크기 필드만 갱신하면 된다.

### EDC/ECC 재계산

섹터 User Data를 수정하면 EDC/ECC가 무효화된다.

- **EDC**: 섹터 오프셋 `0x000`-`0x80F`(Sync+Header+User Data, 2064바이트)에 대한 CD-ROM 전용 CRC-32. 다항식 `x^32+x^31+x^16+x^15+x^4+x^3+x+1`, 테이블 구현 시 반전 상수 `0xD8018001`, 결과는 little-endian으로 `0x810`에 기록.
- **ECC**: Reed-Solomon Product Code. P-parity 172바이트(Header+User Data 대상), Q-parity 104바이트(Header+User Data+P 대상).
- 에뮬레이터(Mednafen 등)는 일반적으로 EDC/ECC를 무시하므로 에뮬레이터 검증 단계에서는 생략 가능하다. 실기 구동을 목표로 하면 변경한 섹터의 EDC/ECC를 재계산한다. 단 전 데이터 섹터를 무조건 일괄 재계산하지는 않는다 — 의도적으로 틀린 EDC/ECC를 카피 프로텍션으로 쓰는 게임이 있어, 일괄 재계산이 그 의도적 오류를 정정해 보호 검사를 깨뜨릴 수 있다(`references/strategy/build-and-verify.md`의 EDC/ECC 절 참조). 변경 섹터만 재계산하고 원본의 의도적 오류 섹터는 보존한다.

## 8. 패치 포맷과 빌드

- 빌드 산출물은 패치된 BIN + CUE다. CUE는 원본과 트랙 레이아웃이 동일하므로 자동 생성한다.
- 배포는 **BPS 패치**로 한다. 원본 BIN → 패치 BIN의 diff를 빌드 완료 시 자동 생성한다. 새턴 디스크 이미지는 100 MB 이상이므로 이미지 자체 배포는 부적절하며, BPS는 원본 체크섬 검증을 내장해 잘못된 원본 적용을 차단한다.
- BIN이 트랙 전체(데이터 + CD-DA)를 담은 단일 파일인 경우 BPS도 그 단일 BIN 기준으로 만든다. 원본 덤프의 트랙 분할 방식(단일 BIN vs 트랙별 BIN)이 다르면 패치가 적용되지 않으므로, 대상 원본의 정확한 크기·체크섬을 명시한다.
- 빌드 파이프라인 예시: 번역 JSON 로드 → 사용 한글 수집·글리프 테이블 생성 → 폰트 파일 패치(압축 해제 → 글리프 교체 → 재압축) → 각 스크립트 패치(추출 → splice → 포인터 수정 → 재압축) → 디스크 기록(재배치 포함) → ISO 디렉토리 갱신 → EDC/ECC 재계산 → BPS 생성.

## 9. 검증 — Mednafen 디버깅

- **레이어 토글**: 대화 텍스트의 렌더링 하드웨어(VDP1 스프라이트 vs VDP2 NBG 레이어)를 확정하는 가장 빠른 방법. 레이어를 하나씩 꺼서 텍스트가 사라지는 레이어를 찾는다. VDP2 PND character number 한계가 적용되는지 여부가 여기서 갈리므로 패치 설계 전에 반드시 확인한다.
- **회귀 격리 테스트**: 전체 빌드에서 행(hang)이 발생하면 패치 단계를 하나씩 끄는 격리 빌드(특정 SEQ만 패치, 재압축만 수행 등)로 원인 단계를 좁힌다. 한 사례에서는 CD 행의 원인이 포인터 누락임을 이 방식으로 확정했다.
- **증상별 원인 패턴**:
  - 빈 텍스트 박스 → 포인터 미수정 (타겟이 이동한 블록).
  - 하드락 → 패딩이 제어코드 파라미터로 오인됨, 또는 거짓양성 포인터 수정으로 코드 손상.
  - 특정 콘텐츠(미니게임 등)만 오동작 → 해당 파일 유형의 포인터 패턴 누락.
  - 화면 깨짐 → 폰트 타일 수가 PND 12비트 한계 초과.
  - 재압축 파일만 깨짐 → 압축기 거리 상한 버그(5절).
- 에뮬레이터는 EDC/ECC를 무시하므로 에뮬레이터 통과가 실기 동작을 보장하지 않는다. 실기 검증 전 EDC/ECC 재계산을 확인한다.

### 실기 구동 경로

수단은 두 갈래다.

- **CD-R + 부트 우회**: 외주 링(rim) 프로텍션 때문에 구운 CD-R이 그냥 부팅되지 않는다. 우회 수단은 모드칩, 스왑 트릭(드라이브 마모 위험), Pseudo Saturn Kai 같은 부트 카트 펌웨어가 있다.
- **ODE**: Fenrir, Satiator, MODE 등.

제약 — 리전 락은 프로텍션과 별개 문제다. ODE는 리전 프리를 제공하지만 CD-R 경로는 리전 패치가 필요하다. 이미지 포맷도 ODE별로 갈린다. Fenrir는 멀티트랙을 하나로 합친 단일 BIN과 비호환이라 멀티 BIN+CUE가 필요하고, Satiator는 멀티 BIN을 수용하며, MODE는 Redump 호환이다. PSKai의 확장 RAM 카트 병용 여부와 자동 리전 패치 범위는 확인 필요.

**빌드 포맷에의 영향**: Redump 스타일 멀티트랙 BIN/CUE가 최대공약수다. 단일 병합 BIN은 Fenrir에서 깨지므로 병합본을 배포하지 않는다. 데이터 트랙을 수정한 뒤에는 EDC/ECC를 재계산한다(에뮬레이터는 무시하지만 실기는 필수다. 7절 디스크 포맷의 EDC/ECC 재계산 산출물과 직결). CD-R로 배포한다면 리전 패치 정책을 빌드 단계에서 결정한다.

함정 — 스왑 트릭은 기기 손상 위험이 있어 검증 수단으로 부적합하다. PSKai는 로더 버전별 호환 편차가 있어 "패치 버그"인지 "로더 비호환"인지 구분이 어렵다.

중간 검증: 에뮬레이터와 실기 사이에 사이클 정확 FPGA 재구현(새턴 코어가 있으면)을 둘 수 있으나, CD 기반 기종이므로 최종적으로 실기 ODE 검증이 따로 필요하다.

## 10. 사례 요약

### 새턴 마도물어

- 텍스트: 다수의 SEQ 스크립트, 타일 인덱스 직접 인코딩(2바이트 BE), 대화 렌더링 VDP2 NBG3 (Mednafen 레이어 토글로 확정).
- 폰트: CNX v2 압축 폰트 파일을 확장, PND 12비트 한계 내 글리프 슬롯 + 와이드 영역 회수/직접 매핑으로 수요 흡수.
- 포인터: MP/PT/Other/COMMON/DIARY 5종 패턴을 유형별 스캐너로 일괄 수정. 64KB 초과 파일은 듀얼 뱅크(bank-crossing 자동 전환) 처리.
- 빌드: SEQ 패치, ISO 9660 디렉토리 갱신 + 빈 영역 재배치, 전 데이터 섹터 EDC/ECC 재계산, BPS 자동 생성.
- 검증: Mednafen으로 검증.

### 새턴 와쿠와쿠 뿌요뿌요 던전

- 폰트: 폰트 파일이 이미 12비트 타일 한계에 근접해, 확장 대신 존(Zone) 재활용 전략을 썼다. 슬롯을 보존/재렌더링/회수로 분류하고, 8x16 영역 일부를 16x16 인터리브로 전환해 슬롯을 추가 확보.
- 렌더링: Master Dispatcher → VDP2 Text Engine → Pattern Name Writer 콜체인 전체를 디스어셈블로 확정. 16x16 모드의 PNT 오프셋 패턴(+0/+1/+64/+65)에 맞춰 타일을 배치해 엔진 코드 무수정 재사용.
- 부수 효과 처리: 폰트 타일을 배경 타일맵·블록 그래픽으로 재사용하는 스크립트를 함께 패치(회수된 타일 참조를 공백 타일로 교체). 폰트 타일이 텍스트 외 용도로 참조될 수 있음을 보여주는 사례.
- 검증: Mednafen으로 검증.
