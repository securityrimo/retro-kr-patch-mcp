# 메가드라이브 (Sega Mega Drive / Genesis) 한글 패치 참조

## 1. 플랫폼 개요

메가드라이브의 메인 CPU는 Motorola 68000(M68K)이며 클럭은 7.67MHz(NTSC) / 7.60MHz(PAL)이다. 내부 레지스터는 32비트(데이터 D0~D7, 주소 A0~A7)이지만 외부 데이터 버스는 16비트, 주소 버스는 24비트(실효 주소 공간 16MB)이다. 게임 코드는 대부분 Supervisor 모드에서 실행된다.

보조 프로세서로 Z80(3.58MHz)이 있으며 주로 사운드 드라이버를 실행한다. 텍스트 처리와는 무관하므로 한글 패치에서는 건드릴 일이 거의 없다. 단, 68000이 Z80 주소 공간($A00000~)에 접근하려면 버스 요청($A11100) 절차가 필요하다는 점만 알아두면 된다.

화면 출력은 VDP(세가 315-5313, 초기 기판에는 Yamaha YM7101로 각인)가 전담한다. VRAM 64KB, CRAM 128바이트(4팔레트×16색), 해상도 320×224(H40) 또는 256×224(H32)이다. 모든 그래픽은 8×8 픽셀 4bpp 타일 단위로 처리되므로, 한글 폰트도 결국 8×8 타일의 조합으로 환원된다.

68000의 핵심 성질 두 가지가 패치 작업 전반을 지배한다.

- **빅엔디언**: 다중 바이트 값은 상위 바이트가 낮은 주소에 온다. 헥스 에디터에서 `00 30 12 34`로 보이면 그대로 포인터 `$00301234`다. 바이트 스왑이 필요 없다.
- **워드 정렬**: Word/Long 접근은 짝수 주소에서만 가능하다. 홀수 주소 접근은 Address Error 예외를 일으켜 게임이 죽는다. 훅 코드와 데이터 테이블을 배치할 때 반드시 짝수 정렬을 지킨다.

디스어셈블리에서 자주 만나는 텍스트 관련 패턴은 다음과 같다.

| 패턴 | 의미 |
|------|------|
| `LEA Table(PC), A0` | 문자열/테이블 포인터 로드. 텍스트 데이터 위치 단서 |
| `MOVE.B (A0)+, D0` | 문자열 순차 읽기. 텍스트 파서 루프의 핵심 |
| `CMPI.B #$FF, (A0)` | 문자열 종료 마커 검사 ($FF, $00 등 게임 고유) |
| `JSR (Table,PC,D0.W)` | 점프 테이블 간접 호출. 제어 코드 디스패처에 흔함 |
| `DBRA D7, .loop` | 카운터 루프. 타일 전송 루프에 빈번 |

## 2. 메모리 맵

카트리지 ROM은 $000000부터 직접 매핑된다. 뱅크 스위칭 없이 최대 4MB($000000~$3FFFFF)를 평탄하게 사용할 수 있다. ROM 파일 오프셋 = CPU 주소이므로 분석과 삽입이 단순하다.

```
$000000 ~ $3FFFFF : 카트리지 ROM (최대 4MB, 파일 오프셋과 1:1)
$A00000 ~ $A0FFFF : Z80 주소 공간
$A10000 ~ $A1001F : I/O 레지스터 (컨트롤러 등)
$C00000 ~ $C0000F : VDP 포트
$FF0000 ~ $FFFFFF : 68K 워크 RAM 64KB ($E00000~$FEFFFF는 미러)
```

ROM 내부 구조:

| 범위 | 내용 |
|------|------|
| $000000~$0000FF | 예외 벡터 테이블. $000004 = Reset Vector(코드 진입점), $000078 = VBlank 벡터 |
| $000100~$0001FF | ROM 헤더. $1A4 = ROM 끝 주소, $18E = 체크섬 |
| $000200~ | 게임 코드와 데이터 |

표준 카트리지 매핑에서는 4MB($000000-$3FFFFF)까지 68000이 평탄하게 접근할 수 있다. 원본이 4MB보다 작고 실기/에뮬레이터 호환성을 넓게 유지하고 싶다면, 4MB 이하로 확장해 빈 상위 영역을 한글 폰트·텍스트·훅 코드에 쓰는 안을 우선 검토한다. 확장 시 헤더의 ROM 끝 주소($1A4)를 갱신한다. 4MB를 넘기면 SSF 계열 매퍼 같은 뱅크 전환이 필요하므로, 공간 이득과 호환성 비용을 별도로 판단한다.

체크섬은 $000200부터 ROM 끝까지를 Word 단위로 합산(mod $10000)한 값이며 $18E에 저장된다. ROM을 수정하면 빌드 단계에서 재계산해 써넣는다. 검증하지 않는 에뮬레이터도 있으나 일부 환경에서 부팅이 거부될 수 있다.

```python
def fix_checksum(rom):
    cs = 0
    for i in range(0x200, len(rom), 2):
        cs = (cs + ((rom[i] << 8) | rom[i + 1])) & 0xFFFF
    rom[0x18E] = cs >> 8
    rom[0x18F] = cs & 0xFF
```

## 3. 비디오 시스템

### 3.1 플레인 구조

VDP는 스크롤 가능한 Plane A / Plane B, 스크롤되지 않는 Window Plane, 그리고 스프라이트를 합성해 화면을 만든다. 대화창 텍스트는 게임에 따라 Plane A, Window Plane, 또는 스프라이트로 표시된다(대화 텍스트를 스프라이트로 그리는 게임도 실재한다. 이 경우 훅·교체 대상이 네임테이블이 아니라 OAM/스프라이트 속성 테이블이 된다). **텍스트가 어느 레이어에 그려지는지 파악하는 것이 분석의 첫 단계다.** Window Plane은 고정 표시라 대화창·HUD에 즐겨 쓰인다.

각 플레인은 네임테이블(타일 배치표)을 VRAM에 가지며, 엔트리는 16비트다.

```
비트 15    : Priority
비트 14-13 : 팔레트 번호 (0~3)
비트 12    : V-Flip
비트 11    : H-Flip
비트 10-0  : 타일 인덱스 (0~2047)
```

텍스트 엔진은 결국 "문자 코드 → 타일 인덱스 변환 → 네임테이블 기록"을 수행하는 코드다. 한글화는 이 변환 단계를 가로채는 작업이다.

### 3.2 4bpp 타일

- 타일 1개 = 8×8 픽셀 = 32바이트 (행당 4바이트, 바이트당 2픽셀)
- 니블 하나가 팔레트 인덱스 0~15. 상위 니블 = 좌측 픽셀, 0 = 투명
- 타일 인덱스 N의 VRAM 주소 = N × 32
- VRAM 64KB에 이론상 최대 2048타일. 실제로는 네임테이블·스프라이트 테이블이 공간을 나눠 쓰므로 폰트에 할당 가능한 타일 수는 훨씬 적다

```python
def to_md_tile(pixels_8x8):
    """8x8 픽셀 배열(값 0~15) → 메가드라이브 4bpp 타일 32바이트"""
    tile = bytearray(32)
    for row in range(8):
        for col in range(8):
            off = row * 4 + col // 2
            if col % 2 == 0:
                tile[off] |= (pixels_8x8[row][col] & 0xF) << 4
            else:
                tile[off] |= pixels_8x8[row][col] & 0xF
    return tile
```

ROM 내 폰트가 비압축이면 이 포맷 그대로 연속 저장되어 있고, Kosinski·Nemesis 등 세가 계열 압축이나 게임 고유 압축이 걸려 있을 수도 있다. 타일 뷰어(YY-CHR의 4bpp Sega MD 포맷 등)로 ROM을 훑어 폰트 위치를 시각적으로 찾을 수 있다.

### 3.3 DMA와 타일 동적 로딩

한글 완성형 수백 자를 전부 VRAM에 상주시키기는 어렵다(16×16 기준 300자 × 4타일 = 1200타일). 따라서 **표시 시점에 필요한 글자 타일만 ROM에서 VRAM으로 옮기는 동적 로딩**을 우선 검토한다. 단 UI 전용 소형 서브셋처럼 수요가 작으면 정적 상주도 가능하므로, 폰트 슬롯·네임테이블·스프라이트 사용량을 먼저 산정한다. 전송 수단은 DMA가 표준이지만, 인터럽트를 막고 VDP 데이터 포트에 워드를 직접 써 넣는 CPU 전송도 쓰인다(글자당 소량 전송에 자주 쓰인다). 어느 쪽이든 Active Display를 피해 VBlank·인터럽트 차단 구간에서 수행한다.

DMA 요점:

- 68K 메모리(ROM/RAM) → VRAM 전송. 소스 주소는 바이트 주소 ÷ 2(Word 주소)로 레지스터 $15~$17에 지정, 길이는 Word 단위로 $13~$14에 지정
- 전송 트리거는 Control Port($C00004)에 VRAM Write 커맨드 + DMA 비트를 기록
- VBlank 동안 약 7.5KB(NTSC H40)까지 안전 전송 가능. 한 대화창에 필요한 글자(16×16 기준 보통 15~30자 = 60~120타일 = 약 2~4KB)는 한 프레임 VBlank 안에 들어간다
- Active Display 중 DMA는 화면 깨짐 위험이 있으므로 대화창 갱신은 VBlank에 맞춘다

캐시 전략은 세 가지가 있다.

1. **정적 캐시**: 폰트 전체 또는 소형 서브셋을 VRAM 상주. 글리프 수요가 작을 때만 가능
2. **화면 단위 캐시**: 대화창이 열릴 때 그 대화에 등장하는 고유 글자만 VRAM의 폰트 슬롯 영역에 일괄 DMA. 대화창이 닫히면 해제. 한 화면 고유 글자 수가 슬롯 안에 들면 구현이 단순하다
3. **LRU 캐시**: 고정 슬롯(예: 64자분)을 두고 오래된 글자를 교체. 유연하지만 7.67MHz CPU에서 관리 오버헤드를 따져야 한다

## 4. 텍스트 엔진 패턴

### 4.1 일반 구조

메가드라이브 게임의 텍스트는 표준 인코딩(ASCII, Shift-JIS)이 아닌 게임 고유 문자 테이블을 쓰는 경우가 대부분이다. 인코딩 단위도 게임마다 달라 1바이트 코드, 2바이트(16비트) 타일 인덱스 직접 기록 방식 등이 있다. 일본어 게임은 가나·한자 수백 자를 다루므로 처음부터 16비트 멀티문자 체계인 경우가 많고, 이런 게임은 한글화 난이도가 크게 낮아진다(코드 공간이 이미 넓다).

전형적인 렌더링 흐름:

```
1. 포인터 테이블에서 문자열 주소 획득
2. 1단위(1바이트 또는 1워드)씩 읽기
3. 제어 코드($FFxx 등)면 제어 처리 (줄바꿈, 페이지 넘김, 종료, 이벤트 트리거)
4. 문자 코드 → 타일 인덱스 변환
5. (동적 로딩형이면) 타일을 VRAM에 DMA
6. 네임테이블에 타일 인덱스 기록, 커서 전진
7. 반복
```

### 4.2 2바이트 멀티문자 처리

1바이트 ASCII 엔진에 한글을 얹으려면 바이트 분기(byte dispatch)를 넣는다. 첫 바이트가 $80 이상이면 다음 바이트와 합쳐 2바이트 한글 코드로 해석한다.

```asm
kr_text_render:
    move.b  (a0)+, d0           ; 1바이트 읽기
    beq.s   .end                ; $00 = 종료
    cmpi.b  #$20, d0
    blo.s   .control            ; 제어 코드 — 바이트 값 비교는 무부호 분기(BLO/BHS)가 원칙.
    cmpi.b  #$80, d0            ;   BLT/BGE(부호 비교)를 쓰면 $80~$FF가 음수로 취급돼 오분기한다
    bhs.s   .hangul             ; $80 이상 → 2바이트 한글
    ; --- ASCII: 1타일 처리 ---
    subi.b  #$20, d0
    ext.w   d0                  ; 결과 $00~$5F라 부호 확장 안전
    addi.w  #ASCII_TILE_BASE, d0
    bsr     write_tile_1x1
    bra.s   kr_text_render
.hangul:
    move.b  d0, d1              ; 상위 바이트
    move.b  (a0)+, d2           ; 하위 바이트
    subi.b  #$80, d1
    ext.w   d1                  ; 결과 $00~$7F라 부호 확장 안전
    mulu.w  #$C0, d1
    subi.b  #$40, d2
    andi.w  #$FF, d2            ; 결과가 $80~$BF일 수 있으므로 ext.w(부호 확장) 금지 — 제로 확장
    add.w   d2, d1              ; d1 = 한글 인덱스
    lsl.w   #1, d1              ; 8x16 = 2타일이면 ×2 (16x16이면 lsl #2)
    addi.w  #HANGUL_TILE_BASE, d1
    bsr     write_tile_hangul
    bra.s   kr_text_render
```

원래 16비트 타일 인덱스 방식인 게임이라면 더 간단하다. 미사용 인덱스 대역(예: $0100 이상)을 한글에 배정하고, 렌더링 루틴에 "인덱스가 임계값 이상이면 한글 폰트 경로"라는 분기 하나만 추가하면 된다.

### 4.3 렌더링 루틴 수정 지점 — 훅 기법

원본 루틴을 통째로 재작성하기보다, **문자 코드가 타일 인덱스로 확정되는 지점에 `JMP`/`JSR` 훅을 심어 확장 영역의 새 코드로 우회**시키는 방식을 먼저 검토한다.

1. 디스어셈블러(Ghidra 68000 모듈 등)로 텍스트 파서를 찾는다. VDP Data Port($C00000) 기록, 문자열 순차 읽기 `(A0)+`, 종료 마커 비교가 단서다
2. 문자→타일 변환 직전 명령(6바이트 이상)을 `JMP $3xxxxx`(6바이트: `4E F9` + 32비트 주소)로 덮는다
3. 확장 영역의 훅 코드에서: 한글 코드면 KR 폰트 처리(DMA 로딩 + 네임테이블 기록) 후 원본 루프로 복귀, 아니면 덮어쓴 원본 명령을 재현하고 원래 경로로 복귀
4. 훅 코드는 짝수 주소에 배치하고, 사용한 레지스터는 `MOVEM`으로 보존한다

에뮬레이터 디버거(BlastEm의 GDB 연동, Gens KMod의 VRAM 뷰어)에서 VDP 포트 접근에 브레이크포인트를 걸면 렌더링 루틴을 빠르게 특정할 수 있다.

## 5. 한글 운용 수치

### 5.1 인코딩 방식 선택

- **완성형(사용 글자만)**: 번역 텍스트를 먼저 완성하고 실제 사용된 고유 음절만 추출해 폰트화한다. 선행 사례의 고유 한글은 수백 자 규모였지만, 일반 산정은 `references/strategy/font-strategy.md` 3절을 따른다. 렌더링 로직이 단순(인덱스→타일 매핑)해 기존 엔진 수정이 작아지기 쉽다. 4MB 이하 확장 영역에 폰트·텍스트·훅이 수납되면 완성형이 유력하다.
- **조합형**: 초성 19·중성 21·종성 28의 자모 글리프(벌 변형 포함 약 150~390개)만 저장하고 런타임 조합한다. 폰트 용량은 1/5~1/10이지만 68000 어셈블리로 조합 루틴을 새로 짜야 해 난이도와 디버깅 부담이 크다. ROM 확장이 자유로운 메가드라이브에서는 굳이 택할 이유가 약하다.

완성형 선택 기준: ROM 확장 여유가 있고, 사용 글자가 수백 자 수준이며, VRAM 적재 전략이 성립하면 완성형. 텍스트 분량이 커서 글자 수가 크게 늘거나 ROM/VRAM 확장이 불가능한 사정이 있으면 조합형이나 부분 조합 fallback을 검토한다.

### 5.2 글자 크기와 8×16 권장 근거

| 크기 | 타일 수 | 바이트/자 | 500자 폰트 | 평가 |
|------|---------|-----------|------------|------|
| 8×8 | 1 | 32B | 16KB | 받침 글자 판독 불가. 부적합 |
| 8×16 | 2(세로) | 64B | 32KB | 한글 표현 가능한 최소선. 타일 예산 효율 최고 |
| 16×16 | 4(2×2) | 128B | 64KB | 가독성 표준. 복잡 자모까지 여유 |

**8×16을 우선 검토하는 근거**는 다음과 같다.

- VRAM 타일 소비가 16×16의 절반이다. 대화창 한 화면분 글자 캐시가 절반 타일로 끝나 동적 로딩 DMA량과 폰트 슬롯 점유가 모두 가벼워진다
- 네임테이블 기록이 세로 2엔트리로 끝나 2×2 기록 루틴보다 단순하고, 줄당 표시 글자 수가 두 배(H40 320px 기준 최대 40자 vs 20자)라 한글 번역문 특유의 길이 증가를 흡수하기 쉽다
- 일본어 게임에도 8×16(1×2타일) 폰트 전례가 있어 기존 엔진의 세로 2타일 처리 코드를 재활용할 수 있는 경우가 있다

단, 8×16의 가로 8픽셀은 'ㅙ', 'ㅞ' 등 복합 모음과 겹받침에서 가독성이 급격히 떨어진다. 폰트 시안을 먼저 도트로 떠 보고 판독성이 확보되지 않으면 16×16으로 올린다. 16×16은 16비트 콘솔 한글 패치의 가독성 표준이며, 대화창 줄 수·창 폭이 넉넉한 게임이라면 16×16이 결과물 품질에서 유리하다. 16px 높이의 가변폭 방식을 채택한 사례도 있다.

### 5.3 ROM 공간 산정

- 한글 텍스트는 원문 대비 바이트가 늘어나는 경우가 많다(2바이트 인코딩 시 영문 대비 최대 2배). 원본 위치에 수납되지 않으면 확장 영역으로 재배치(relocate)한다
- 4MB 확장 ROM 기준 배치 예: 원본 2MB 유지, $200000 이후에 포인터 테이블 / 한글 텍스트 / 훅 코드 / 폭 테이블 / 한글 폰트를 구획해 배치
- 폰트 용량: 고정폭 16×16 완성형 500자 = 64KB. 가변폭·서브픽셀 시프트 같은 고급 렌더링을 쓰면 자당 용량이 커진다. 전체 용량은 선택한 렌더링 방식과 글리프 수로 계산하고, 텍스트·훅·테이블까지 합산해 4MB 이하 수납 여부를 판단한다

## 6. 포인터 체계

68000 게임의 텍스트 포인터는 주로 **32비트 빅엔디언 절대 주소**다. ROM이 $000000부터 직접 매핑되므로 포인터 값이 곧 ROM 파일 오프셋이다. 헥스 에디터에서 `00 0A 2A 74`가 보이면 그대로 오프셋 $0A2A74를 가리킨다.

- 포인터 테이블 식별: ROM 후반 텍스트 영역 부근에서 `00 0x XX XX` 꼴의 오름차순 32비트 값이 연속되면 포인터 테이블일 확률이 높다
- 2단 구조도 흔하다: 32비트 절대 포인터가 텍스트 블록을 가리키고, 블록 머리에 16비트 상대 오프셋(블록 시작 기준) 테이블이 따라오는 형태. 이 경우 상대 테이블도 함께 재계산해야 한다
- 뱅크 포인터는 표준 매핑에서는 등장하지 않는다(매퍼 사용 게임 제외)

재배치 시 파이프라인: 새 텍스트를 확장 영역에 순차 기록하면서 각 엔트리의 시작 오프셋을 32비트 빅엔디언으로 포인터 테이블에 다시 써넣는다.

```python
import struct

def rebuild_pointers(rom, table_off, texts, text_off, term=b'\x00'):
    cur = text_off
    for i, data in enumerate(texts):
        struct.pack_into('>I', rom, table_off + i * 4, cur)  # 32-bit BE
        rom[cur:cur + len(data)] = data
        cur += len(data)
        rom[cur:cur + len(term)] = term
        cur += len(term)
    return cur
```

주의: 포인터 테이블을 거치지 않고 코드에 직접 박힌 주소(`LEA addr.L, A0`, `JMP addr.L` 등)도 있다. 텍스트를 옮긴 뒤 옛 주소를 참조하는 코드가 남아 있지 않은지 ROM 전체에서 해당 주소 바이트열을 검색해 확인한다.

## 7. 패치 포맷과 빌드

배포는 **BPS 포맷**으로 한다. IPS는 3바이트 오프셋(16MB 한계)에 체크섬이 없어 잘못된 ROM에 적용해도 경고가 없다. BPS는 소스/타깃/패치 3중 CRC32를 내장해 원본 검증이 자동으로 이루어지고, 가변 길이 인코딩으로 패치 파일도 작다. 4MB 확장처럼 파일 크기가 변하는 패치에도 안전하다.

```bash
# Floating IPS(flips)로 BPS 생성/적용
flips --create --bps original.md patched.md output.bps
flips --apply output.bps original.md result.md
```

빌드는 스크립트로 전 과정을 자동화한다. 표준 파이프라인:

1. 원본(베이스) ROM 로드 — 필요 시 4MB로 확장하고 헤더의 ROM 끝 주소 갱신
2. 번역 텍스트 로드, 사용 글자 추출, 인코딩 테이블 생성
3. 폰트 렌더링(TTF/비트맵 폰트 → 4bpp 타일) 후 확장 영역에 삽입
4. 훅 코드(68K) 삽입, 원본 루틴에 JMP 패치
5. 텍스트 인코딩·삽입, 포인터 테이블 재구성
6. 체크섬($18E) 재계산
7. 패치 ROM 저장, BPS 생성

배포 시 원본 ROM은 절대 포함하지 않는다. 패치 파일과 함께 적용 대상 ROM의 CRC32/MD5/크기를 명시한다.

## 8. 검증

에뮬레이터는 디버깅 기능 기준으로 고른다.

| 에뮬레이터 | 특징 |
|------------|------|
| BlastEm | 고정밀. GDB 프로토콜 원격 디버깅, 68K 브레이크포인트·메모리 감시. 1순위 |
| Gens KMod | VRAM 뷰어·네임테이블 뷰어 내장. 한글 타일이 VRAM에 제대로 올라왔는지 시각 확인에 필수 |
| Exodus | 정밀 에뮬레이션, 상세 디버거 |

검증 절차:

1. **자동 검사 먼저**: 제어 코드 무결성(원문과 번역문의 제어 코드 1:1 대조), 줄 폭 오버플로우(픽셀 폭 기준)를 스크립트로 전수 검사한다. 씬 종료·이벤트 트리거 계열 제어 코드 누락은 소프트 락으로 직결되므로 CRITICAL 0건이 통과 기준이다
2. **실기동 테스트**: 오프닝부터 엔딩까지 주요 장면 체크리스트를 만들어 통과시킨다. 확인 항목 — 글리프 깨짐 없음, 줄바꿈 위치, 텍스트 창 넘침 없음, 대화창 전환 시 스프라이트 잔상 없음, 메뉴 선택 분기 정상, 씬 전환 후 소프트 락 없음
3. **VRAM 검사**: 글자 깨짐이 보이면 Gens KMod VRAM 뷰어로 타일 데이터 자체가 깨졌는지(폰트 변환 버그) 네임테이블 인덱스가 어긋났는지(렌더링 버그) 구분한다
4. 가능하면 실기 또는 정밀 에뮬레이터(BlastEm)에서 최종 확인한다. 관대한 에뮬레이터에서만 돌던 패치가 정밀 환경에서 죽는 경우가 있다

증상별 진단 요령: 버튼 무반응 소프트 락 → 종료 계열 제어 코드 누락. 이벤트 미발동 → 이벤트 트리거 코드 파라미터 훼손. 스프라이트 잔상 → 줄 수 초과로 인한 VRAM 타일 오버플로우. 글자 깨짐 → 문자 매핑 테이블 또는 폰트 변환 오류.

### 실기 구동 경로

- **수단**: 플래시 카트로 구동한다. Mega EverDrive PRO 또는 X7이 표준이다.
- **제약**: 68000이 직접 어드레싱하는 범위는 4MB이고, 이를 초과하는 ROM은 SSF 매퍼(512KB 뱅크 단위)가 있어야 접근된다. X7은 SSF 매핑을 지원하며 현 펌웨어 기준 15MB 수준까지 로드한다(모델·펌웨어별 확인 필요). 본질 제약은 4MB 매퍼 경계이고 총 용량 한계는 부수적이다.
- **빌드 포맷에의 영향**: **4MB를 초과하는 빌드는 ROM 헤더의 시스템 필드($000100)를 `SEGA SSF`로 바꿔야** 매퍼가 활성화된다 — 기본값 `SEGA MEGA DRIVE`로 두면 매퍼가 동작하지 않아 4MB 너머가 안 읽힌다. 이 필드 갱신은 7절 빌드 스크립트의 책임이다(체크섬 재계산과 같은 단계에 둔다). 반대로 4MB 이하로 수납되면 평범한 단일 바이너리가 되어 실기 호환성이 가장 넓다 — 5.3절 공간 산정에서 4MB 선을 넘지 않게 설계하면 매퍼 자체를 피할 수 있다.
- **함정**: SSF 매퍼는 구형 에뮬레이터가 미지원하는 경우가 있어, 실기에서는 도는데 구형 에뮬에서 안 도는 역방향 불일치가 생길 수 있다. 뱅크 스왑을 쓰면 SRAM 세이브 매핑이 어긋나지 않게 주의한다.
- **중간 검증**: 에뮬레이터와 실기 사이의 중간 검증으로 사이클 정확 FPGA 재구현(메가드라이브 코어)을 쓸 수 있다. 부정확한 에뮬레이터를 전제로 만든 패치가 여기서 걸러진다.

## 9. 사례 요약 (메가드라이브 마도물어 1)

일본판 2MB ROM을 기반으로 한 영어 패치판(4MB 확장)이 존재했고, 한글 패치는 그 위에 구축되었다. 영어 패치가 이미 확장 영역($200000~)에 가변폭 폰트 렌더러를 갖추고 있어 이를 재활용한 사례다.

- **텍스트 인코딩**: 원작이 처음부터 16비트 타일 인덱스 방식. 한글은 미사용 대역 $0100 이상에 배정. 멀티바이트 분기를 새로 만들 필요가 없었다
- **훅**: 영어판 렌더링 루틴의 문자 처리 지점 1곳에 `JMP` 훅(확장 영역의 한글 처리 코드로 우회)을 삽입. 타일 인덱스가 $0100 미만이면 기존 영문 경로, 이상이면 한글 경로로 분기
- **폰트**: 도트 한글 폰트를 16px 높이로 렌더링, 가변폭 + 4단계 서브픽셀 시프트 사본 구조. 폭 테이블은 별도로 운용
- **배치**: 확장 영역에 포인터 테이블 / 한글 텍스트 / 훅 코드 / 폭 테이블 / 폰트를 구획 배치해 2MB 확장 공간에 수납
- **포인터**: 32비트 빅엔디언 절대 포인터 테이블 + 블록 내 16비트 상대 오프셋의 2단 구조를 재구성
- **빌드**: 단일 Python 스크립트가 ROM 로드 → 폰트 생성 → 훅 삽입 → 텍스트 인코딩·삽입 → 체크섬 재계산 → 저장을 일괄 수행. 배포는 BPS
- **QA**: 제어 코드 전수 대조와 픽셀 폭 오버플로우 자동 검사를 빌드 게이트로 운용. 씬 종료 코드 누락이 소프트 락의 최다 원인이었다

선행 영어 패치가 있는 게임은 확장 영역 코드, 멀티바이트 처리, 가변폭 렌더링 같은 기반 작업이 이미 되어 있을 수 있다. 그 구조를 분석해 재사용 가치가 크고 라이선스·허락·크레딧 문제가 해결된다면 영어 패치판을 베이스로 삼는 선택지가 있다. 단 배포 패치의 적용 대상이 "원본 + 영문 패치 적용본"이 되어 사용자 적용 절차가 2단계가 될 수 있으므로, 일본판 원본 베이스와 작업량·배포성을 비교해 결정한다.
