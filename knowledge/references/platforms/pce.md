# PC Engine / PCE CD-ROM² 한글 패치 참조

PC Engine(NEC/Hudson, 1987)과 CD-ROM² 확장(1988) 타이틀의 한글화에 필요한 플랫폼 사실을 정리한다. 플랫폼 사실과 특정 게임에서 나온 수치는 구분해 읽고, 텍스트 엔진·로더·디스크 배치는 대상 게임에서 다시 실측한다.

---

## 1. 플랫폼 개요

핵심 칩 구성은 다음과 같다.

| 칩 | 역할 | 비고 |
|---|---|---|
| HuC6280 | CPU + PSG + 타이머 + IRQ + I/O + MMU | 65C02 슈퍼셋, 8비트 |
| HuC6270 (VDC) | Video Display Controller | 64KB VRAM, 타일/스프라이트 |
| HuC6260 (VCE) | Video Color Encoder | 9비트 RGB 팔레트 |
| System Card (CD) | CD-ROM² BIOS HuCard | 부팅 + CD/ADPCM/폰트 라이브러리 |

HuC6280 요점:
- 65C02 슈퍼셋, 리틀 엔디언. 클럭은 1.7897725MHz(저속)/7.15909MHz(고속)이며 `CSL`/`CSH` 명령으로 런타임 전환한다. 많은 게임이 부팅 직후 `CSH`로 7.16MHz로 올리지만, 실제 클럭 상태는 초기화 코드를 확인한다.
- 65C02 확장 외 PCE 전용 명령: `TAM`/`TMA`(MPR 제어), `TII`/`TDD`/`TIN`/`TIA`/`TAI`(7바이트 블록 전송, 사이클 = 17 + 6×len. 방향: TII 양쪽 증가, TDD 양쪽 감소, TIN 소스만 증가, TIA 소스 증가·목적 2주소 교대(16비트 포트용), TAI 소스 2주소 교대·목적 증가. 어느 명령이 보이는지로 데이터 흐름을 읽는다), `ST0`/`ST1`/`ST2`(VDC 직접 쓰기), `CSL`/`CSH`, T 플래그(메모리-메모리 연산).
- 스택은 `$2100-$21FF`다(MPR1=$F8 work RAM 전제). 65C02식 `$01XX`로 쓰지 않는다.
- 6502와 달리 `abs,X` 페이지 경계 크로싱 패널티(+1 사이클)가 없다.

CD-ROM² 시스템은 System Card HuCard로 부팅하며 카드 요구사항에 따라 RAM 모델이 다르다: 원본 CD-ROM²는 일반 64KB + ADPCM 64KB + 세이브 2KB, Super CD-ROM²(Card 3.0)는 일반 256KB, Arcade Card는 Super 구성에 2MB 확장 RAM을 더한다. BIOS는 `$E000-$FFFF`에 점프 테이블(항목당 3바이트)로 노출되고 `cd_read`($E009), `ex_getfnt`($E060), `ex_joysns`($E063), `ex_vsync`($E07B) 등을 제공한다. 단, 게임이 BIOS CD 읽기를 쓰지 않고 자체 로더를 복제해 넣은 경우가 있으므로 섹터 배치를 바꿀 때는 자체 로더의 하드코딩 섹터 번호를 추적해야 한다. 카드 요구사항은 이름·릴리즈 연도·폴더명으로 추정하지 말고 부팅 요구, 에뮬레이터 모듈, 초기 로더, Arcade Card I/O 접근 여부로 확인한다.

ADPCM 음성은 64KB 전용 RAM(CPU 주소 공간 밖)에서 약 32kHz OKI 4비트 포맷으로 재생된다. BGM은 CDDA 오디오 트랙으로 공급되는 경우가 많아 텍스트 패치에서 건드리지 않는 경우가 많지만, 트랙 구성은 CUE와 게임 로더에서 확인한다.

## 2. 메모리 맵과 MPR 페이징

CPU 논리 주소는 16비트(64KB), 물리 주소는 21비트(2MB)다. 논리 공간을 8KB 단위 8페이지로 나누고, MPR0~MPR7 레지스터가 각 페이지의 물리 뱅크(상위 8비트)를 결정한다.

| 논리 주소 | 페이지 | MPR | TAM/TMA 마스크 |
|---|---|---|---|
| `$0000-$1FFF` | 0 | MPR0 | `%00000001` |
| `$2000-$3FFF` | 1 | MPR1 | `%00000010` |
| `$4000-$5FFF` | 2 | MPR2 | `%00000100` |
| `$6000-$7FFF` | 3 | MPR3 | `%00001000` |
| `$8000-$9FFF` | 4 | MPR4 | `%00010000` |
| `$A000-$BFFF` | 5 | MPR5 | `%00100000` |
| `$C000-$DFFF` | 6 | MPR6 | `%01000000` |
| `$E000-$FFFF` | 7 | MPR7 | `%10000000` |

`TAM #$ii`로 A를 마스크 비트의 MPR에 쓰고, `TMA #$ii`로 읽는다. RESET 시에는 MPR7만 `$00`(ROM 뱅크 0)으로 설정되어 ROM 선두 뱅크의 `$E000+offset`이 매핑되고, 그 안의 인터럽트 벡터(`$FFF6-$FFFF`)로 부팅한다. 나머지 MPR은 초기화되지 않으므로 부트 코드가 직접 설정한다.

물리 뱅크 레이아웃(뱅크 1개 = 8KB):

| 뱅크 | 영역 |
|---|---|
| `$00-$7F` | HuCard ROM (최대 1MB) |
| `$68-$87` | CD 일반 RAM (Super CD-ROM², 256KB) — 원본 CD-ROM²는 `$80-$87` 64KB |
| `$F7` | BRAM (세이브 2KB) |
| `$F8` | Work RAM (8KB, `$F9-$FB`는 미러) |
| `$FF` | 하드웨어 페이지 (VDC/VCE/PSG/Timer/I/O/IRQ) |

CD 모드의 `$E000-$FFFF`는 System Card BIOS 점프 테이블의 안정적인 entry anchor다. 다만 BIOS ROM bank·함수 본체·데이터가 항상 MPR7에만 보인다고 가정하지 않는다. 게임이나 BIOS가 MPR을 바꾼 뒤에는 다른 논리 슬롯에도 System Card bank가 노출될 수 있으므로, BIOS 경로 분석은 호출 주소와 함께 현재 MPR0-MPR7 상태를 기록한다.

### 포인터의 $8000 기준 제약

CD 게임의 스크립트 데이터는 8KB 페이지 단위로 특정 논리 슬롯에 swap-in되는 구조가 흔하며, 이 경우 스크립트 포인터 테이블의 값은 파일 오프셋이 아니라 **매핑된 상태의 CPU 버스 주소**다. 텍스트 region의 포인터가 모두 `$8000` 기준(MPR4 슬롯 매핑 전제)인 구조에서는 파일 오프셋 = `섹션 베이스 + (포인터 값 − $8000)`으로 환산한다. 이 구조에서 따라오는 제약은 다음과 같다.

- 페이로드는 매핑 슬롯의 가시 범위를 벗어날 수 없다. 예컨대 text group 1(16KB, 여러 region 공유)과 text group 2(약 24.5KB, 최대 region 단독)처럼 그룹별 매핑 슬롯 크기가 고정 한도가 된다.
- 한글 번역으로 텍스트가 일본어 대비 1.2~1.5배 부풀면 이 한도를 위협한다. 대응책은 DTE/사전 압축, 미사용 영역 재활용, 뱅크 추가(인터프리터 수정 동반) 순으로 비용이 커진다.
- 번역 후 페이로드를 재배치하면 포인터 테이블(low 바이트 배열 + high 바이트 배열로 분리된 형태가 흔하다)을 `$8000` 기준으로 재계산해야 한다.

## 3. 비디오 시스템

VDC(HuC6270)는 자체 VRAM 64KB를 가지며 16비트 워드 단위로 접근한다(VRAM 1주소 = 2바이트). 백그라운드 1레이어 + 스프라이트 64개, 픽셀은 4bpp(16색 팔레트, BG/스프라이트 각 16개)다.

- 타일: 8×8, 4bpp 플레이너. 16×16 글리프 1자 = 타일 4개 = 128바이트.
- BAT(타일맵) 엔트리는 16비트 — 비트 0-11 타일 인덱스(4096슬롯), 비트 12-15 팔레트 번호.
- VDC 접근은 `$FF:0000`(레지스터 셀렉트/상태), `$FF:0002/0003`(데이터 LSB/MSB). MAWR($00)에 주소를 쓰고 VWR($02)로 데이터를 흘린다. `TIA src,$0002,len` 블록 전송은 VRAM 대량 쓰기 경로로 자주 쓰인다.
- VCE 팔레트는 512 엔트리 × 9비트(3R/3G/3B). 표준 해상도는 256×224가 가장 흔하다.

텍스트 렌더에서 중요한 것: 글리프는 1bpp로 받아 4bpp로 expand한 뒤 VRAM에 전송하는 패턴이 일반적이다. 1bpp 바이트를 두 플레인에 동시에 쓰면 색 인덱스 3이 된다.

## 4. 디스크 포맷

PCE CD는 BIN/CUE 멀티트랙 이미지가 표준이다. 데이터 트랙은 MODE1/2352, BGM은 CDDA 오디오 트랙이다. 데이터 트랙이 하나라고 가정하지 말고 CUE에서 모든 MODE1/2352 트랙을 열거한다. 그중 실제 로더가 참조하는 데이터 트랙만 수정하고, 나머지 데이터 트랙과 오디오 트랙은 byte-exact로 보존한다. 트랙 번호나 파일명보다 CUE의 track type과 게임 로더의 참조가 우선이다.

MODE1/2352 섹터(2352바이트) 구조:

```
┌─────────┬────────┬─────────────────┬─────┬──────────┬─────┐
│ Sync 12 │ Hdr 4  │ User data 2048  │ EDC4│ Reserved8│ECC276
└─────────┴────────┴─────────────────┴─────┴──────────┴─────┘
      0       12        16        2064  2068      2076  2352
```

- Sync: `00 FF FF FF FF FF FF FF FF FF FF 00`
- Header: MM:SS:FF(BCD) + 모드 바이트(`01`)
- 섹터 N의 사용자 데이터 파일 오프셋 = `N × 2352 + 16` (BIN 파일 기준). 논리(2048바이트 기준) 오프셋 = `N × 0x800`.

### EDC/ECC 재계산

데이터 트랙 섹터를 수정하면 해당 섹터의 EDC/ECC를 재계산해야 한다. 생략하면 실기와 엄격한 에뮬레이터에서 거부될 수 있다(Mednafen은 비교적 관대하지만 의존하지 않는다).

- **EDC**: 4바이트 CRC32. normal polynomial은 `0x8001801B`이고, reflected/right-shift table 구현 상수는 `0xD8018001`이다. 입력은 섹터 선두 2064바이트(sync + header + user data), 결과는 오프셋 2064-2067에 리틀 엔디언 저장.
- **ECC**: 276바이트 Reed-Solomon Product Code(P-패리티 172바이트 + Q-패리티 104바이트). 입력은 header + user data + EDC + reserved 8바이트. 알고리즘은 ECMA-130 Annex A.

## 5. 텍스트 엔진 패턴

### 5.1 op-code 디스패치 분석법

PCE 텍스트 인터프리터에서 자주 보이는 패턴 중 하나는 "바이트가 임계값 미만이면 op-code, 이상이면 문자"인 구조다. 던전 엔진에서 확인되는 구조:

- op-code 영역은 임계값 미만의 연속 코드 구간(예: `0x00~0x32`)이고, 그 위는 미사용, 0x80 이상은 표준 Shift-JIS 리드 바이트(다음 바이트와 페어)다.
- 디스패처는 op 바이트를 인덱스로 **low 바이트 핸들러 테이블 + high 바이트 핸들러 테이블** 두 평행 배열에서 핸들러 주소를 조립하고, 자기 수정 코드로 간접 점프한다. `LDA table_lo,Y / STA operand / LDA table_hi,Y / STA operand+1` 패턴을 찾으면 테이블 위치가 확정된다. 임계값 판정 + 핸들러 테이블 인덱싱은 PCE 텍스트 엔진 분석의 유력한 출발점이지만, low/high 평행 배열 + 자기수정 점프는 여러 구현 중 하나다. 다른 게임은 단일 주소 테이블이나 점프 테이블 등 다른 형태를 쓰므로 디스패처 형태는 게임별로 확인한다.
- op 전부가 고유 핸들러인 경우가 있다 — 공유 핸들러가 없으면 핸들러 주소를 정렬해 op별 의미 분석의 출발점으로 삼는다.

전수 덤프에서 식별되는 op-code 예:

| op | 이름 | op | 이름 |
|---|---|---|---|
| 0x00 | leftbox | 0x09 | yesno_prompt |
| 0x01 | rightbox | 0x0A | end |
| 0x02 | centerbox | 0x1D | color |
| 0x03 | br | 0x2C | jump |
| 0x04 | delay | 0x2D | wait_for_voice |
| 0x05 | wait | 0x31 | (페이싱) |
| 0x06 | clear_and_close | 0x32 | (상태머신 셀렉터) |
| 0x07 | clear | | |

`end`(0x0A)는 스크립트 종료 + 콜 스택 pop이다. 소수 엔트리짜리 ret-stack과 인덱스 변수를 RAM에 두고, 서브 스크립트가 끝나면 부모 스크립트 포인터를 복원한다. 빈 스택이면 태스크 종료 처리로 빠진다.

### 5.2 빈도 통계로 의미를 추정하는 기법

핸들러 디스어셈블만으로 의미가 안 잡히는 op는 전수 덤프의 통계로 좁힌다. 실전에서 통한 추론 순서:

1. **빈도 자체가 신원이다.** end의 출현 수는 스크립트 엔트리 수와 정확히 일치한다(엔트리당 종결 1회). br·clear처럼 엔트리당 1회 이상 나오는 op는 줄바꿈/클리어 같은 핵심 렌더 명령 후보가 된다.
2. **인자 분포를 본다.** delay(0x04)의 인자가 `0x1E`(30)에 몰리면 "60Hz 프레임 단위" 가설이 선다(30프레임 = 0.5초). 2바이트 인자 op의 second byte가 대부분 0이면 first byte가 주 카운터, second는 미세조정이라는 구조가 보인다(예컨대 first byte × 10프레임 대기로 확정되는 식).
3. **주변 컨텍스트를 본다.** 반복적으로 `{op}{박스 op}{clear}<텍스트>` 위치에 나오는 op는 화자/씬 식별자 후보이고, 텍스트 사이에 끼는 op는 표정 전환·SE 트리거 후보가 된다. 인자 값별로 직후 텍스트를 모아 보면 화자 매핑이 잡힌다.
4. 빈도가 0회인 op도 핸들러 자체는 존재할 수 있는데, 미사용 op라면 패치에서 무시해도 된다.
5. **음성 동기 op는 보존한다.** wait_for_voice(0x2D)는 ADPCM 음성 종료까지 박스 진행을 멈추는 마커다. 제거하면 음성 도중 박스가 넘어간다. 의미를 모르는 op도 인자 포함 바이트 시퀀스 보존을 우선한다 — 핸들러가 BIOS 태스크 스케줄러 호출이어도 바이트가 같으면 동작이 보존된다.

### 5.3 텍스트 박스 시스템

박스 3종 사양 예(고정폭 전제 빌드):

| 박스 | 가로 px | 세로 px | 줄 수 | 줄 높이 |
|---|---|---|---|---|
| leftbox | 144 | 48 | 3 | 16 |
| rightbox | 144 | 48 | 3 | 16 |
| centerbox | 304 | 48 | 3 | 16 |

16×16 고정폭 셀 기준 줄당 한도는 left/right 9자, center 19자다. 박스 종류는 VRAM 베이스와 줄 간 점프 거리만 다르고 좌표 시스템은 공통이다. 커서는 박스 타입별 X/Y 배열(RAM 3바이트씩)로 관리되며, 박스 전환 op(0x00/01/02)는 타입 인덱스만 바꾸고 커서를 리셋하지 않는다 — 리셋은 clear가 한다. 원본에 dynamic word wrap이 없고 스크립트가 고정폭을 전제로 op_br을 미리 박아둔 구조라면, 한글 텍스트도 줄당 글자 수·3줄 한도를 빌드 도구의 정적 검사(overflow check) 대상으로 삼는다.

### 5.4 인라인 리터럴 주의

스크립트 region 밖, 핸들러 코드 옆에 일본어 문자열이 직접 박혀 있는 경우가 있다. 층수 표시("階", "地下")·빈 슬롯 표시("未使用")·전각 숫자 테이블("０１…９", 20바이트) 같은 문자열이 코드 뱅크 안에 인라인으로 존재하면서 `TII` 복사 + 숫자 테이블 fetch로 출력되는 식이다. region 번역만으로는 안 잡히므로 별도 바이트 패치 카탈로그가 필요하다. 한글도 2바이트/자라서 같은 길이로 fit하는 경우가 많다("階"→"층", "地下"→"지하").

## 6. 한글 표시 경로

### 6.1 원본 글리프 경로: BIOS `ex_getfnt`

BIOS 폰트에 의존하는 게임에서는 일본어 글리프가 게임 데이터가 아니라 **System Card BIOS의 `ex_getfnt`($E060)**에서 온다. 자체 폰트를 디스크에 내장한 게임도 있으므로 글리프 출처를 먼저 실측한다. 호출 규약:

- 입력: `_AL`($F8) = SJIS trail, `_AH`($F9) = SJIS lead, `_BL/_BH`($FA/$FB) = 결과 버퍼 주소, `_DH`($FF) = 컬러 속성(0 = 단색)
- 출력: `_BX`가 가리키는 RAM에 16×16 1bpp 글리프 32바이트

게임 코드는 SJIS 코드를 룩업 테이블 없이 BIOS에 그대로 던지고, 받은 32바이트를 1bpp→4bpp로 expand(같은 바이트를 두 플레인에 복사)해 128바이트 = 4타일을 만들고, IRQ 컨텍스트에서 `TIA`로 VWR에 DMA한다.

한글 글리프는 어떤 System Card BIOS 버전에도 없다. 따라서 한글 표시의 본질은 **`ex_getfnt` 호출을 가로채 ROM에 임베드한 한글 글리프를 같은 규약으로 돌려주는 것**이다.

### 6.2 최소 PoC: $E060 훅

BIOS `ex_getfnt`에 의존하는 게임에서 통하는 최소 개입 경로:

1. 한글 글리프를 16×16 1bpp 32바이트(1행 = 2바이트, MSB 우선)로 만들어 데이터 트랙의 미사용 영역에 임베드한다.
2. 텍스트 인터프리터의 `JSR $E060` 호출 지점(게임 내 1곳) 3바이트를 훅 함수 호출로 교체한다.
3. 훅 함수(약 30줄 ASM)는 lead 바이트가 한글 영역(예: SJIS 사용자 정의 영역 `$F0+`)이면 임베드 글리프를 `_BX` 버퍼에 `TII`로 복사하고, 아니면 `JMP $E060`으로 BIOS에 넘긴다.
4. 스크립트의 SJIS 한 글자를 한글 코드로 바꿔 빌드하면 expand/DMA는 원본 코드가 그대로 처리해 한글이 화면에 출력된다.

훅 함수를 호출 지점과 같은 뱅크의 free 영역에 두면 뱅크 트램폴린(`TMA`/`PHA`/`TAM` 보존 패턴)이 필요 없다. 인코딩은 SJIS 사용자 정의 영역(`$F040-$FCFC`) 또는 EUC-KR lead(`$B0-$C8`, SJIS lead 범위 밖)를 쓸 수 있는데, 어느 쪽이든 `>= $80` 분기를 통과하므로 인터프리터의 리드 바이트 인식은 그대로 동작하고 훅에서만 구분하면 된다.

### 6.3 확장

- 훅의 단일 글리프 복사를 테이블 룩업(lead/trail → 글리프 인덱스)으로 바꾸면 풀 한글로 확장된다.
- 글리프 시트는 번역에 실제 등장하는 글자만 서브셋(1500~2000자 × 32바이트 = 48~64KB)으로 줄이고, 8KB 뱅크 단위로 배치해 훅에서 MPR swap으로 접근한다. 완성형 2350자 풀셋은 약 75KB다.
- 16×16 고정폭이면 원본 엔진(3줄 한도, expand, DMA)을 그대로 쓴다. 가변폭이 필요하면 BIOS 경로를 완전히 버리고 자체 1bpp 합성 엔진을 끼우는 대규모 개조(리터럴 판정 임계값 변경, 리터럴 핸들러 교체, 전송 큐·IRQ DMA 교체 등 5곳 이상 훅)가 필요하므로, 고정폭으로 먼저 검증하고 판단한다.

## 7. 빌드 (BIN/CUE 재구성)

1. 원본 BIN/CUE에서 모든 데이터 트랙(MODE1/2352)을 열거하고, 각 트랙의 섹터별 사용자 데이터 2048바이트를 추출한다(`N × 2352 + 16`).
2. 게임 로더가 참조하는 데이터 트랙의 추출본에만 스크립트 페이로드 교체·포인터 재계산($8000 기준)·훅/글리프 바이트 패치를 적용한다.
3. 변경된 섹터만 EDC/ECC를 재계산해 2352바이트 섹터로 되감는다. 어떤 섹터가 변했는지 추적하는 구조로 만들면 재계산 범위와 diff 검증이 쉬워진다.
4. 수정한 데이터 트랙만 신규로 쓰고, 수정하지 않은 데이터 트랙과 오디오 트랙(CDDA BGM)은 원본 그대로 복사 또는 참조하는 CUE를 생성한다. 멀티트랙 BIN을 합본 single-bin으로 낼 수도 있다.
5. 산출물의 CRC32/MD5와 변경 섹터 목록을 기록한다.

게임이 자체 CD 로더를 쓰는 경우 섹터 배치를 바꾸면 로더의 하드코딩 섹터 번호도 갱신해야 한다. 기존 섹터 안에서만 바이트를 바꾸는 패치라면 이 문제가 없다.

## 8. 검증 (에뮬레이터 디버깅)

검증 요건은 HuC6280 스텝 실행·RAM/VRAM watch·breakpoint를 갖춘 에뮬레이터 디버거다. Mednafen 내장 디버거가 이 요건을 충족하며, 아래는 그 기준의 운용 요점이다.

- `mednafen out.cue`로 부팅한다. System Card BIOS 파일(`syscard3.pce` 등)을 Mednafen BIOS 디렉터리에 배치해야 하며, 게임 첫 섹터 시그니처가 요구하는 카드 버전과 일치해야 한다.
- Mednafen 내장 디버거로 HuC6280 스텝 실행, RAM/VRAM watch, 훅 지점 breakpoint를 건다. BIOS 글리프 호출 지점과 VRAM 전송 루틴에 breakpoint를 걸면 글리프 데이터가 버퍼→expand→VRAM으로 흐르는 것을 직접 확인할 수 있다.
- 시각 검증 경로: 부팅 화면 → 첫 플레이 가능 지점 → 텍스트 박스 출력 확인. 박스 폭/줄 한도 초과는 에뮬레이터에서 글자가 박스 밖으로 흘러나오는 형태로 드러나므로 빌드 단계의 정적 overflow 검사와 이중으로 잡는다.
- Mednafen은 EDC/ECC 불일치에 관대한 편이므로 Mednafen 부팅 성공만으로 디스크 무결성을 보증하지 말고, EDC 검증 도구와 실기(또는 엄격한 에뮬레이터)로 교차 확인한다.

### 실기 구동 경로

CD-R 굽기, ODE, FPGA 세 경로가 있다.

- **CD-R 굽기**: 플랫폼 공통의 강한 디스크 인증 장치는 흔치 않지만, 이것이 모든 타이틀의 CD-R 부팅을 보장하지는 않는다. INDEX/pregap/CUE 재현, 변경 섹터의 EDC/ECC, 타이틀별 디스크 구조 검사를 보존·측정한다. 게임에 맞는 시스템 카드도 본체에 필요하다.
- **Turbo EverDrive PRO**: HuCard 슬롯에서 CD 이미지를 구동한다. BIOS 파일이 필요하다.
- **Super SD System 3**: BIN/CUE·ISO+CUE를 받고 시스템 카드를 내장한다.

Turbo EverDrive PRO의 CD 코어는 CD-ROM² 유닛이 붙은 본체(Duo 계열)에서 버스 충돌로 동작하지 않고, CD-DA는 모노만 낸다는 제약이 있다.

**빌드 포맷에 미치는 영향**: 오디오+데이터 혼합 구조에서 INDEX·pregap을 보존해야 하므로 **멀티트랙 BIN/CUE 유지를 우선 선택한다**. 데이터 트랙만 ISO로 뽑으면 CDDA가 들어간 게임이 깨진다. 굽기용과 ODE용이 같은 BIN/CUE를 공유할 수 있으면 포맷 변환 비용이 낮다(7절 BIN/CUE 재구성 산출물을 실기 검증 대상으로 삼는다).

자주 빠지는 함정으로, 게임이 요구하는 시스템 카드 버전을 명시해야 한다. 실기 드라이브(특히 Duo)는 노화로 CD-R 인식이 실패하기 쉬운데, 이를 "패치 문제"로 오인하기 쉽다.

에뮬레이터와 실기 사이에는 사이클 정확 FPGA 재구현(PC 엔진 코어)을 중간 검증으로 둘 수 있다. 다만 CD 기반 기종이므로 FPGA를 통과해도 최종적으로 실기 ODE(또는 CD-R) 검증이 따로 필요하다.

## 9. 사례 요약 — PCE 마도물어 1

- CD-ROM² 타이틀. 멀티트랙 BIN/CUE 중 게임 로더가 참조하는 데이터는 Track 02(MODE1/2352)에 있고, 별도 MODE1 데이터 트랙은 System Card BIOS 메타/크레딧 계열로 식별되어 패치 대상에서 제외했다. 오디오는 CDDA 트랙(prologue 포함). 패치는 Track 02만 수정하고 나머지 트랙은 보존.
- 게임이 BIOS `cd_read`를 쓰지 않고 자체 CD 로더를 내장. 메인 던전 블록은 단일 섹터 구간에서 통째로 CD RAM에 로드된 후 8KB 페이지 단위로 MPR swap-in.
- 텍스트 엔진: 임계값 미만 연속 op-code(예: `0x00~0x32`), 0x80 이상은 표준 Shift-JIS 페어. low/high 평행 핸들러 테이블 + 자기 수정 간접 디스패치. 전수 덤프의 op 빈도 통계로 의미를 역추론(빈도 = 신원, 인자 분포, 주변 컨텍스트).
- 박스 3종(left/right 144×48, center 304×48, 3줄×16px). 포인터는 전부 `$8000` 기준이며 text group 1/group 2의 매핑 슬롯 크기가 한글 페이로드의 강한 사이즈 한도.
- 한글 PoC: 인터프리터의 `JSR $E060`(BIOS `ex_getfnt`) 단일 호출 지점을 같은 뱅크 free 영역의 훅 함수로 교체하고, 16×16 1bpp 32바이트 글리프를 ROM에 임베드해 SJIS 사용자 정의 영역 코드로 매핑. expand/4타일 VRAM DMA는 원본 코드를 그대로 재사용. Mednafen으로 검증.
- 음성 동기 마커(wait_for_voice)와 미해석 op의 바이트 시퀀스는 전량 보존. 일본어 음성은 유지하고 한글 자막만 교체하는 정책.
