# 빌드와 검증

한글 패치의 빌드 파이프라인 설계, 패치 포맷 선택, 무결성 처리, 에뮬레이터 검증, 텍스트 QA, 이슈 관리를 다룬다.
플랫폼별 체크섬 위치·메모리 맵·디버거 세부는 `references/platforms/` 해당 문서를 본다.

---

## 1. 빌드 파이프라인 원칙

### 원본 ROM 비포함

원본 ROM·디스크 이미지는 저작물이다. 저장소에 절대 커밋하지 않는다.

- 원본은 로컬 경로(예: `roms/` — `.gitignore` 등록)에 두고 빌드 시 경로 인자나 설정으로 주입한다.
- 배포 산출물은 **패치 파일만**이다. 패치된 ROM 자체를 배포하지 않는다.
- 원본을 식별할 해시(SHA-1, MD5, CRC32)는 문서에 기록한다. 해시는 저작물이 아니므로 커밋해도 된다.

### 단방향 재현 가능 빌드

빌드는 다음 단방향 흐름으로 설계한다.

```
원본 ROM ──extract──▶ 중간 자산 ──(번역·편집)──▶ 중간 자산 ──build──▶ 패치 ROM ──diff──▶ 패치 파일
```

- **매 빌드는 원본에서 처음부터 재현된다.** 패치된 ROM을 다시 열어 누적 수정하는 방식은 금지한다. 누적 수정은 재현 불가능하고 버그가 어느 단계에서 들어갔는지 추적할 수 없다.
- 빌드 도구의 수정이 곧 패치의 수정이다. ROM에 손으로 바이트를 찍지 않는다. 헥스 에디터로 찾은 수정은 빌드 스크립트의 코드로 옮긴다.
- 빌드 한 번으로 패치 ROM과 패치 파일이 모두 나오게 한다. CD 매체라면 한 빌드 명령이 ROM 재조립 → EDC/ECC 재계산 → 차분 패치(BPS 등) 생성까지 한 번에 수행하도록 묶는다.

### 추출 자산의 중간 포맷

사람(또는 번역 에이전트)이 편집하는 것은 중간 포맷뿐이다. 바이너리를 직접 편집하지 않는다.

- **텍스트**: 주소·원문·번역을 묶은 JSON이 표준이다. 제어 코드는 `{br}`, `{page}` 같은 가독 태그로 이스케이프한다(표기 규약 정본: `references/strategy/text-extraction.md` §4.4).

```json
{
  "entry_id": 42,
  "file_offset": "0x1D8FDA",
  "text": "ぁんた　だれ？{br}そこをどきなさい！",
  "ko": "너 누구야?{br}거기서 비켜!"
}
```

- **폰트**: 글리프 이미지(PNG)나 렌더링 파라미터. 빌드가 타일 포맷으로 변환한다.
- **그래픽**: 추출된 이미지 파일. 빌드가 원래 포맷으로 재압축·재배치한다.

중간 포맷 ↔ 바이너리 변환은 라운드트립 테스트로 보증한다. 원본에서 추출한 자산을 무수정 재삽입했을 때 원본과 동일한 바이트가 나와야 한다.

### 빌드 타임 검증 내장

빌드가 곧 1차 QA다. 다음 검사를 빌드 단계에 내장하고, 위반 시 빌드를 실패시킨다.

- 인코딩 가능 여부(번역문에 폰트에 없는 글자가 있는가)
- 글리프 수요가 산정된 공급을 넘지 않는가
- 텍스트 박스 오버플로우(5절)
- 길이·용량 제약(고정 길이 필드 초과, 뱅크 용량 초과)
- 포인터 테이블 정합성(개수·범위)
- PoC 게이트에서 확정한 전제(경로, 인코딩 영역, 재배치 정책, 압축 라운드트립)가 깨지지 않았는가

빌드가 통과한 ROM만 에뮬레이터 검증(4절)으로 보낸다.

## 2. 패치 포맷 선택

### 매체별 결정표

| 매체 | 1순위 | 대안 | 비고 |
|------|-------|------|------|
| 카트리지 ROM (SNES·GG·MD·PCE(HuCard) 등) | **BPS** | IPS | BPS는 원본 체크섬 내장 — 잘못된 원본 적용을 차단 |
| 플로피·디스크 파일 단위 (PC-98 등) | 파일별 BPS/xdelta | 파일 교체 스크립트 | 이미지 내 파일을 추출·교체하는 빌드와 병행 |
| CD 이미지 (PS1·새턴) | **xdelta** 또는 BPS | 트랙별 분할 패치 | 수백 MB 이미지 — 포맷의 크기 한계 확인 |
| GD-ROM (드림캐스트) | 트랙 파일별 패치 | 이미지 재빌드 배포 도구 | 멀티 트랙 — 트랙 단위로 diff |

### 포맷별 특성

| 포맷 | 원본 검증 | 크기 한계 | 특징 |
|------|----------|----------|------|
| IPS | 없음 | 16MB (24비트 오프셋) | 가장 단순. 잘못된 ROM에 적용해도 에러 없이 망가짐 |
| BPS | CRC32 내장 (원본·결과 모두) | 사실상 없음 | 카트리지 ROM 표준. 델타 인코딩으로 작음 |
| xdelta (VCDIFF) | 구현·옵션 의존 — 별도 해시 검증 필요 | 사실상 없음 | 대용량 디스크 이미지에 적합 |

원본 검증이 없는 IPS는 피한다. 사용자가 리비전이 다른 ROM·헤더 유무가 다른 ROM에 적용하는 사고가 흔하다. xdelta는 RFC 3284의 VCDIFF 스트림을 쓰는 구현 계열이라 대용량에는 좋지만, 배포 포맷만 믿지 말고 적용 스크립트·README·체크섬 파일에서 원본 해시를 별도로 검증한다.

외부 사양 기준선은 다음처럼 잡는다. ISO 9660 파일시스템의 볼륨·디렉토리·파일 배치 판단은 ECMA-119를, CD-ROM raw 섹터·트랙·EDC/ECC 판단은 ECMA-130을, xdelta/VCDIFF 계열 차분 스트림 판단은 RFC 3284를 기준으로 확인한다. 단 콘솔 부트로더와 게임 자체 로더는 이 표준 위에 게임별 제약을 얹을 수 있으므로, 표준 사양 확인 뒤에도 실행 파일의 LBA/크기/로더 참조를 별도로 역추적한다.

### 인플레이스 수정 vs 이미지 재빌드

디스크 매체에서 파일 크기가 변하면 두 갈래가 있다.

- **인플레이스 수정**: 크기가 같거나 줄어들면 제자리 교체. 메타데이터 갱신 최소.
- **전체 재빌드**: 파일시스템(ISO 9660)을 통째로 재생성하고 모든 파일을 재배치.

부분 수정은 메타데이터 정합성이 깨지기 쉽다. 한 사례에서 개별 파일 교체·재배치 방식은 디렉토리를 부분 수정하다가 그래픽 글리치를 유발했다. 반면 선행 영문 패치는 9백여 개 파일 전체를 재배치하는 전체 재빌드로 문제가 없었다. **파일 크기가 늘어나는 디스크 패치는 먼저 부분 재배치가 안전하게 닫히는지 증명하고, 그 증명이 어렵거나 부트·파일시스템 메타데이터가 여러 곳에 걸리면 전체 재빌드를 선택한다.** 재빌드 시 주의점:

- 부트 로더가 특정 파일의 LBA를 하드코딩할 수 있다 — 부트 요구사항(메인 실행 파일의 배치 위치 등)을 먼저 확인한다.
- 디렉토리 레코드 포맷(패딩·필드 값·정렬 순서)을 원본 또는 동작이 검증된 선행 패치와 바이트 레벨로 일치시킨다.
- 총 볼륨 크기를 원본과 유지할 수 있으면 유지한다.

## 3. 무결성 처리

### 내부 체크섬

플랫폼에 따라 ROM 헤더에 체크섬이 있고, 부트 코드나 게임이 이를 검증하기도 한다.

| 플랫폼 계열 | 헤더 체크섬 | 대응 |
|------------|------------|------|
| SNES | 있음 (16비트 보수쌍) | 빌드 마지막에 재계산 |
| 메가드라이브 | 있음 (16비트 합) | 빌드 마지막에 재계산 |
| 새턴·PS1 | 없음 | 매직 문자열·지역 코드만 보존 |

체크섬 재계산은 **빌드 파이프라인의 마지막 단계**에 자동화한다. 정확한 오프셋·알고리즘은 `references/platforms/` 해당 플랫폼 문서를 본다.

게임 자체의 자기 무결성 검사(실행 파일 CRC 등)는 드물지만, 의심되면 부트 직후 초기화 코드에서 큰 메모리 블록을 순회·누적 합산하는 루프와 비교 실패 시 무한 루프로 빠지는 패턴을 역분석으로 확인한다.

### CD 매체의 EDC/ECC

CD-ROM Mode 1 섹터(2352바이트)에는 섹터 단위 오류 감지(EDC, CD 전용 CRC-32 변형)와 오류 정정(ECC, Reed-Solomon Product Code) 코드가 내장되어 있다.

```
Mode 1 섹터 (2352바이트):
[000-00B] Sync Pattern   — 수정 금지
[00C-00F] Header         — 수정 금지 (MSF 주소, 모드)
[010-80F] User Data      — 수정 가능 영역 (2048바이트)
[810-813] EDC            — 수정 시 재계산 대상
[814-81B] Reserved       — 항상 0x00
[81C-92F] ECC (P+Q)      — 수정 시 재계산 대상
```

재계산 필요 여부는 실행 환경으로 갈린다.

| 환경 | EDC/ECC | 이유 |
|------|---------|------|
| 에뮬레이터 | **불필요** | 주요 에뮬레이터는 user data 2048바이트만 추출하고 EDC/ECC를 무시한다 |
| 실기 (CD-R·ODE(Optical Drive Emulator)) | **필수** | 드라이브·CD 서브시스템이 검증한다. 틀리면 읽기 오류 |

실기 대응을 포기하지 않는 한 빌드에 EDC/ECC 재계산을 내장하는 것이 안전하다. 알고리즘은 ECMA-130 표준에 정의되어 있으나 인터리빙이 까다로우므로 검증된 참조 구현(ECM, edcre, ECCRegen 계열)을 쓰거나 포팅한다. **포팅 시 정확성 검증법**: 원본 이미지의 정상 섹터에 재계산을 적용해 원본 바이트와 일치하는지 단위 테스트로 확인한다. 원본 전체가 곧 테스트 벡터다.

단, 전 섹터를 무조건 재계산하면 안 되는 경우가 있다. 일부 게임은 의도적으로 틀린 EDC/ECC를 카피 프로텍션으로 쓴다 — 특정 섹터의 EDC/ECC를 일부러 비정상값으로 두고, 게임 코드가 그 불일치를 읽어 정품을 판별한다. 전 섹터 일괄 재계산은 이 의도적 오류를 "정정"해 보호 검사가 복제본으로 판단하게 만들어 락업을 일으킨다. 따라서 **변경한 섹터만 재계산하고 원본의 의도적 오류 섹터는 그대로 보존**한다(raw 기록, 또는 의도적 오류를 건드리지 않는 재계산 도구를 쓴다). 어떤 섹터가 의도적 오류인지는 원본의 기존 EDC/ECC와 정상 계산값을 비교해 식별한다.

### 파일시스템 메타데이터

ISO 9660에는 파일시스템 레벨 체크섬이 없다. 대신 파일을 수정하면 메타데이터 정합성을 맞춰야 한다.

| 시나리오 | 갱신 항목 |
|----------|----------|
| 내용만 수정 (크기 불변) | 없음 (CD라면 EDC/ECC만) |
| 크기 축소 / 같은 섹터 수 내 증가 | Directory Record의 Data Length (리틀/빅 엔디언 양쪽) |
| 섹터 수 초과 증가 | 파일 재배치 + LBA·Length 갱신 → 부분 재배치 검증이 어렵다면 전체 재빌드 검토 (2절) |
| 파일 추가/삭제 | Directory Record + Path Table + Volume Space Size |

부트 헤더가 메인 실행 파일의 크기·로드 주소를 들고 있는 플랫폼(새턴의 1st Read Size 등)은 실행 파일 크기 변경 시 함께 갱신한다. 세부는 해당 플랫폼 문서를 본다.

### 패치 적용 대상 검증

- 패치가 전제하는 원본의 해시와 리전·리비전을 README에 명시한다.
- BPS처럼 원본 체크섬을 내장한 포맷을 우선 쓰고, xdelta처럼 구현·옵션에 따라 검증 보장이 달라지는 포맷은 적용 스크립트나 별도 해시 검사로 원본 불일치를 차단한다.
- 빌드 도구도 입력 ROM의 해시를 검사해 잘못된 원본으로 빌드하는 사고를 막는다.

## 4. 에뮬레이터 검증

### 수동 플레이 체크리스트

매 빌드 후 최소 스모크 테스트를 돌린다. 게임별로 체크리스트를 만들어 두고 같은 경로를 반복한다.

```
[ ] 부팅·타이틀 화면 도달
[ ] 새 게임 시작 — 첫 대사 박스 정상 표시
[ ] 한글 렌더링 (자모 깨짐·팔레트 오염·잔상 없음)
[ ] 메뉴 열기·닫기
[ ] 세이브 → 로드
[ ] 이름 입력·사용자 문자열 저장/재표시 (해당 시)
[ ] 전투(또는 핵심 시스템) 1회 진입·종료
[ ] 분기 루트가 있으면 루트별 시작 지점 각각 확인
```

마지막 항목은 실제 사고에서 나온 교훈이다. 한 사례에서 한 주인공 루트는 정상인데 다른 루트는 시작 즉시 화면이 깨졌다 — 특정 루트만 쓰는 텍스트 인덱스 경로에 빌드 버그가 숨어 있었다. 분기·캐릭터별 진입점은 전부 별개 테스트 대상이다.

세이브스테이트는 ROM 빌드와 1:1로 묶인다. 다른 빌드에서 만든 스테이트를 로드하면 불일치로 오판하므로, 빌드를 갈아끼우면 스테이트도 다시 만든다.

### Lua 스크립팅 자동화

Lua 스크립팅을 지원하는 에뮬레이터(Mesen2 계열 등)는 GUI 없이 자동 검증을 돌릴 수 있다.

- **headless 테스트러너**: 스크립트가 조건을 검사하고 종료 코드(pass/fail)를 반환한다. CI나 빌드 후크에 연결할 수 있다.
- **메모리 콜백**: 특정 주소의 읽기·쓰기·실행에 콜백을 걸어 텍스트 엔진의 동작을 계측한다. 프레임 이벤트 콜백으로 매 프레임 상태를 샘플링한다.
- **함정**: 콜백 등록 시 CPU 종류·메모리 영역 파라미터를 생략하면 다른 콘솔 기본값이 적용되어 콜백이 영원히 발화하지 않는 API가 있다(한 에뮬레이터에서 실제로 겪은 함정). 파라미터는 명시적으로 지정한다. VRAM이 워드 주소인 플랫폼은 바이트 주소를 2로 나눠 넘긴다.

API 세부와 플랫폼별 주소 변환은 해당 `references/platforms/` 문서를 본다.

### MCP 연동 자동화

에뮬레이터에 메모리 읽기·입력 주입·스크린샷을 프로그래밍으로 호출할 수단(MCP(Model Context Protocol) 서버 등)을 붙이면 에이전트가 직접 검증 루프를 돌릴 수 있다. 다음을 에이전트 호출로 수행할 수 있다.

- 메모리 영역(워크 RAM·VRAM·CRAM·ROM) 읽기/쓰기 — 타일맵·매퍼 레지스터를 직접 들여다본다
- 컨트롤러 입력 주입 — 사람 없이 게임을 진행시킨다 (press_and_release가 가장 안정적)
- 스크린샷 캡처 — 렌더링 결과를 이미지로 회수해 비교한다
- 세이브스테이트 저장/로드 — 검증 지점으로 즉시 점프한다
- 배속 — 진행 구간을 빨리 감는다

검증 루프의 기본형: 스테이트 로드 → 입력 주입 → 수 프레임 진행 → 메모리/스크린샷 검사 → 판정. 도구 호출 규약(파라미터 타입, 주소 표기, 호출 빈도 한계)은 에뮬레이터마다 다르므로 별도 가이드로 정리해 둔다.

### 자동화는 공짜가 아니다 — 선행 비용과 한계

위 자동화는 매끄럽게 들리지만, 현실에서 거저 주어지지 않는다. 도입 전에 다음 비용·한계를 정직하게 잡고 시작한다.

- **인터페이스 자체가 없는 경우가 많다.** 공개 에뮬레이터에 메모리 읽기/쓰기·입력 주입·브레이크포인트·스크린샷·트레이스 같은 디버거·자동화 인터페이스가 빠져 있는 경우가 흔하다. 그러면 자체 개조가 선행 과제가 된다 — HTTP/스크립팅 서버를 내장하거나, 소스에 후킹을 추가하거나, niche fork를 직접 소스 빌드한다. 이는 1회성이지만 공짜가 아닌 선행 비용이며, 에뮬레이터가 업데이트되면 개조를 다시 머지해야 하는 유지 부담이 따라온다. 어떤 에뮬레이터·디버거가 현재 최선인지는 빠르게 바뀌므로, 고정된 도구 목록에 기대지 말고 작업 시점에 최신 인프라를 조사해 고른 뒤 부족하면 그 위에서 clone·개조한다.
- **API에는 함정이 깔려 있다.** 파라미터 타입(정수 vs 문자열)을 한 군데 틀리면 호출이 크래시로 끝난다. 스텝·호출을 과다 반복하면 세션 자체가 불안정해지므로 주기적으로 재시작하는 루틴을 검증 루프에 넣는다. 세이브스테이트는 빌드 버전과 1:1로 묶여(수동 검증 절과 동일) 다른 빌드에서는 로드되지 않으니, 빌드를 갈아끼우면 스테이트도 다시 만든다.
- **화면 검증은 끝까지 자동화되지 않는다.** golden screenshot 회귀는 폰트를 미세 조정할 때마다(행간·커닝·안티앨리어싱) 기준 이미지를 갱신해야 한다. 자동 비교는 "달라졌다"까지만 말해 줄 뿐, 최종 "이게 맞다"는 판정은 사람 눈이 한다 — 완전 자동화되지 않는다.
- **자동 시나리오는 좁은 길만 덮는다.** 자동 트레이스·재생 시나리오는 고정 입력과 초기화 경로만 커버한다. 플레이어가 실제로 만드는 다양한 입력 조합에서의 렌더링 검증은 HITL(사람 확인) 영역으로 남는다. 자동화로 회귀를 거르고, 사람 확인으로 커버리지를 메우는 분담으로 설계한다.

### 브레이크포인트 추적

런타임 버그는 텍스트 엔진의 호출 지점에 브레이크포인트를 걸어 추적한다.

- **실행(exec) 브레이크포인트**: 텍스트 렌더링 진입점, 포인터 룩업 루틴에 건다. 히트 시 레지스터(텍스트 인덱스, 포인터 값)를 읽어 어떤 데이터로 호출됐는지 확인한다.
- **쓰기(write) 브레이크포인트**: VRAM 전송 레지스터, 텍스트 버퍼 주소에 걸어 "누가 이 값을 썼는가"를 역추적한다.
- **주소 미러링 주의**: 브레이크포인트 주소가 코드의 실제 호출 주소와 같은 표기인지 확인한다. 한 사례에서 뱅크 미러링 때문에 `$00:xxxx`에 건 브레이크포인트는 히트하지 않고 `$80:xxxx`로 걸어야 잡혔다. 플랫폼별 미러링 규칙은 해당 플랫폼 문서를 본다.
- **트레이스 로그**: 브레이크포인트로 좁히기 전, VDP/PPU 쓰기·뱅크 전환 트레이스를 켜서 이상 동작의 전후 맥락을 확보한다. 단 트레이스는 실시간 시각화가 아니라 사후 재구성 자료다. "어디서 VRAM에 썼나"라는 단발 기록만으로는 부족하고, 직전의 뱅크 전환과 직후의 다음 쓰기를 함께 엮는 후처리를 거쳐야 DMA·렌더 경로를 식별할 수 있다. 게다가 버퍼 한계로 장시간 기록은 어려우니, 의심 구간을 좁힌 뒤 짧게 켜는 식으로 쓴다.
- **스텝 실행**: step into/over/out으로 호출 체인을 따라간다. 원인 루틴을 찾으면 디스어셈블리와 대조해 데이터 문제인지 코드 문제인지 가른다.

## 5. 텍스트 QA

### 오버플로우 시뮬레이터

번역문이 텍스트 박스를 넘는지는 **빌드 타임에 전수 검출**한다. 에뮬레이터에서 눈으로 찾는 것은 최후 수단이다.

원리: 게임의 렌더러를 빌드 도구 안에 모델링한다.

1. 게임의 줄바꿈·폭 계산 규칙을 역분석한다 — 글리프별 폭(고정폭이면 상수, VWF면 폭 테이블), 박스 폭, 페이지당 최대 줄 수, 줄/페이지 카운터를 리셋하는 제어 코드.
2. 번역문을 인코딩한 뒤 같은 규칙으로 시뮬레이션한다 — 글리프 폭을 합산해 줄 폭을 구하고, 제어 태그로 줄·페이지를 분할한다.
3. 한도 초과를 빌드 에러(또는 리포트)로 출력한다 — 위반한 항목의 주소·실측 폭·줄 수를 함께 찍는다.

### 두 가지 오버플로우

| 종류 | 증상 | 위험도 |
|------|------|--------|
| 줄 폭 초과 | 자동 줄바꿈 발생 → 의도치 않은 줄 수 증가 | 간접적 (아래로 전이) |
| 페이지 줄 수 초과 | 박스 밖·화면 밖 렌더링, 하단 텍스트 짤림 | **직접적 — 실제 사고** |

줄 폭 초과 자체는 에러가 아닐 수 있으나 줄 수 초과의 원인이 되므로, 시뮬레이터는 자동 줄바꿈까지 반영한 **최종 줄 수**로 판정한다.

제어 태그의 카운터 리셋 규칙을 정확히 모델링한다. 페이지 넘김 태그는 줄 카운터를 리셋하고, 박스 전환 태그는 박스 전체를 리셋하는 식이다. 같은 게임 안에서도 텍스트 타입(대화·메뉴·도감·아이템 이름)마다 박스 치수·렌더러·제약이 다르므로 타입별 제약표를 먼저 만든다. 한 사례의 제약표 발췌:

| 텍스트 타입 | 폭 한도 | 줄 수 | 페이지 분할 |
|------------|---------|-------|------------|
| 대화/전투 | 20타일 | 3줄 | 가능 |
| 일기 | 30타일 | 5줄 | 불가 |
| 도감 이름 | 16바이트 (인코딩 후) | 1줄 | 불가 |
| 아이템 이름 | 10바이트 고정 | 1줄 | 불가 |

(수치는 해당 게임 고유 값이다. 게임마다 역분석으로 새로 채운다.)

### 전수 검사와 번역자 가이드

- `verify` 류 서브커맨드로 패치 ROM(또는 번역 JSON 전체)을 일괄 검사하고, 위반 항목의 위치·수치를 리포트한다. 빌드 직후 항상 실행한다.
- 시뮬레이터의 규칙을 번역자용 가이드 문서로 변환해 배포한다 — "한 줄 최대 N자, 줄바꿈 태그 M개 이하, 초과 시 페이지 분할" 수준의 실용 규칙과 OK/NG 예시. 원문(일문)의 줄바꿈·페이지 패턴을 따라가는 것이 가장 안전하다는 점도 함께 적는다.
- 고정 길이 필드(이름류)는 빌드 에러로 강제한다. 이름 입력을 지원하는 게임은 입력 UI의 최대 길이, 저장 인코딩, 재표시 화면별 폭 한도를 같은 검사에 묶는다. 느슨한 제약(스크롤 뷰어 등)은 경고로 둔다.

## 6. 이슈 관리

### QA 라운드

검증은 라운드 단위로 돈다.

```
빌드 → 플레이테스트 → 이슈 수집(스크린샷 필수) → 근본 원인 분석 → 수정 → 다음 라운드에서 재검증
```

- 이슈마다 ID(A, B, C, …)를 부여하고 라운드 문서 한 편에 모은다. 증상·스크린샷·추정 원인·확정 원인·수정 파일·상태를 한 이슈 블록에 기록한다.
- 상태는 명시적으로 관리한다: **수정 완료 / 수정 후 재확인 필요 / 버그 아님 / 범위 밖**.
- "수정 후 재확인 필요"가 중요하다. 한 근본 원인이 여러 증상을 만들 수 있어서, 원인 하나를 고치면 다른 이슈가 같이 사라지는지 다음 라운드에서 확인한다. 한 사례에서는 공백 문자 인코딩 버그 하나가 "대사창 검은 블록", "텍스트 경계 초과" 등 복수 이슈의 공통 원인이었다.
- "버그 아님"(원본 동작 — 증거를 들어 확정)과 "범위 밖"(미번역 메뉴 등)은 명시해 닫는다. 닫지 않으면 라운드마다 다시 올라온다.

개별 이슈의 재현 압축·격리 기법·메커니즘 확정·이슈 스키마(기각 가설 보존, 추정/확정 분리)·"버그 아님" 판정법·회귀 묶음 운영은 `references/strategy/debugging.md`를 따른다.

### 라운드 종료 판정

수정 후에는 같은 지점을 재캡처해 전후를 대조한다. 에뮬레이터 자동화(4절)의 스크린샷 도구와 세이브스테이트를 쓰면 같은 장면 재현이 빨라진다. 라운드 종료 판정은 HITL이다: 자동 검사 전부 통과 + 라운드 이슈 전부 닫힘 + 사람 눈으로 필수 경로 커버리지 매트릭스 통과를 모두 만족해야 다음 단계(배포 후보)로 넘긴다. 배포 후보에는 알려진 치명 문제를 남기지 않는다. 플레이어가 같은 패치를 다시 시도하지 않을 수 있으므로, "드물게 깨짐"이나 "문맥상 오해 가능"도 재현성과 영향도를 따져 블로커로 승격한다.
