# 그래픽 텍스트 (베이크드 텍스트) 처리 전략

텍스트가 폰트 렌더 경로를 거치지 않고 그래픽 자산에 직접 구워져(베이크드) 있을 때의 처리 전략이다. 메뉴 라벨, 타이틀, 월드맵 지명, HUD처럼 글자가 배경과 한 장의 이미지로 합쳐져 저장된 경우로, 초기 조사 질문 6(`references/strategy/initial-survey.md`)에서 "이미지에 구워진 텍스트"로 판정된 자산이 이 문서의 입력이다. 폰트 훅은 여기서 무력하다. 바꿀 코드 경로가 없고 픽셀만 있다. 파이프라인은 네 단계다: **전수 인벤토리 → 클린 배경 복원 → 한글 조판 → 재인코딩·검증.**

경계: 텍스처 컨테이너의 디코딩·재인코딩 세부(twiddle, VQ, 압축, 헤더 메타)는 플랫폼 문서가 정본이다(예: PVR은 `references/platforms/dreamcast.md` §3). 교체 후 화면이 깨질 때의 격리·원인 확정은 `references/strategy/debugging.md`(§3 그래픽 깨짐 결정 트리)를 따른다. 본문 사례의 수치·좌표·개수는 해당 게임의 실측값이다. 새 게임에 이식하지 말고 원칙만 가져간다.

## 1. 베이크드 텍스트 인벤토리

첫 단계는 번역이기 전에 **전수 발견**이다. 베이크드 텍스트는 한곳에 모여 있지 않다. 메뉴·HUD·타이틀·월드맵, 심지어 실행 바이너리 내부까지 흩어져 있다(§5). 일부만 찾아 교체하면 QA마다 일문 그래픽이 산발적으로 발견되어 발견→교체→재검증 사이클이 끝나지 않는다.

인벤토리의 규율은 **분모 고정**이다. "조사했는데 텍스트 없음"과 "아직 안 봄"을 구분해 기록해야 전수 주장이 성립한다. 전체 후보 수(분모)와 판정 완료 수(분자)를 함께 적어 진행률과 완전성을 둘 다 추적한다. 발견 방법은 세 가지이고 상황에 따라 조합한다.

### 1.1 전수 디코딩 — 선행 패치가 없을 때

그래픽 컨테이너 전체를 디코딩해 사람 눈으로 거른다. 한 사례에서는 그래픽 파일을 전부 후보 폭(이미지 너비) 여러 값으로 일괄 디코딩해 이미지로 출력하고, 육안 분류로 **텍스트 있음 확정 / 불명 / 텍스트 없음**의 세 갈래로 판정을 마쳤다(불명은 닫지 않고 보류로 남긴다). 폭을 모르는 포맷이라도 후보 폭 몇 개로 디코딩하면 "텍스트가 있는가"의 판정에는 충분하다. 정확한 폭 확정은 텍스트가 있는 파일에만 들이면 된다. 확정 목록에는 노출 빈도·중요도 기준의 **우선순위 표**를 붙여 작업 순서를 정한다(상시 노출 HUD·메뉴가 상위, 일회성 화면이 하위).

### 1.2 영문 패치 diff — 선행 패치가 있을 때

영문 패치가 고친 그래픽 목록이 곧 베이크드 텍스트 목록의 초안이다.

- 원본과 영문 패치본을 스프라이트 단위로 비교하는 diff 도구로 변동률 높은 자산을 식별한 사례.
- 일문판/영문판 파일 단위 diff로 베이크드 텍스트 후보 텍스처를 추린 사례.

주의: 영문 패치가 안 고친 텍스트(놓쳤거나 의도적으로 남긴 것)는 diff에 잡히지 않는다. diff 결과는 하한선이지 전수가 아니다. 전수 확신이 필요하면 1.1 또는 1.3으로 보완한다.

### 1.3 HITL VRAM write 브레이크포인트 덤프

파일 쪽에서 못 찾으면 화면 쪽에서 거슬러 간다. 의심 화면이 뜰 때 VRAM에 write 브레이크포인트를 걸어 데이터 소스를 역추적하는 HITL 관측이다(요청서 형식은 debugging.md §4). 함정 하나가 결정적이다: **표시 시점이 아니라 로드 시점을 잡아라.** 한 사례에서 베이크드 텍스트는 상주 아틀라스(resident atlas)에 포함되어, 화면에 표시되기 한참 전인 씬 로드 시점에 이미 VRAM에 올라가 있었다. 표시 순간에 write 브레이크포인트를 걸면 빗나간다. 파일 읽기 직후의 VRAM 전송 시점에 걸어야 잡을 수 있다.

### 1.4 카탈로그 스키마

발견 결과는 기계가 읽을 수 있는 카탈로그로 고정한다. 이 카탈로그가 이후 전 단계(복원·조판·검증)의 단일 입력이고, 바운딩 박스는 §4의 검증 마스크로 재사용된다. 등재 단위는 텍스처가 아니라 **텍스트 블록**이다. 한 텍스처에 라벨 여러 개가 들어 있으면 라벨마다 한 행이다. 최소 스키마:

| 필드 | 내용 |
|------|------|
| 파일 | 컨테이너 내 경로·인덱스 |
| 좌표 | 텍스트 영역 바운딩 박스 (x, y, w, h) — 픽셀 단위 실측 |
| 포맷 | 픽셀 포맷(bpp)·팔레트·셀 구조 |
| 원문/번역 | 전사된 원문, 확정 한글 번역 |
| 상태 | 미착수 / 배경 복원 / 조판 / 검증 완료 (불명은 보류로 보존) |
| 우선순위 | 노출 빈도·중요도 등급 |

### 1.5 어노테이션 도구

요건: 이미지 위에 바운딩 박스를 그리고 라벨을 붙여 좌표를 내보낼 수 있을 것. 사례: Label Studio. 한 사례에서는 헥스 덤프만 보고 추정했던 텍스트 좌표가 라벨링 실측으로 크게 다른 값으로 교정됐다. 추정 좌표로 구현부터 시작하면 그대로 재작업이 된다(§5).

- 함정: 도구가 좌표를 픽셀 대신 **백분율로 내보내는** 경우가 있다. 이미지 크기를 곱해 픽셀로 환산한 값을 카탈로그에 넣는다.
- 규모가 수십 장 수준이면 전용 도구 없이 **스프레드시트 + 크롭 이미지 폴더**로 충분하다. 도구 도입 비용이 본전을 못 뽑는다.

### 1.6 OCR 보조

검출기(텍스트 영역 위치)와 인식기(박스 안 글자 전사)의 **2단 구성**으로 육안 분류를 보조한다. 인식기 단독으로는 "어디에 글자가 있는가"를 풀지 못한다. 사례로 manga-ocr를 발견 보조로 썼다. 목적이 recall(놓침 최소화)이므로 오탐은 사람이 거르면 되지만, **전사 확정은 반드시 사람이 한다.** 베이크드 텍스트는 저해상도·장식 글꼴이라 OCR 오독이 잦고, 오독된 원문은 그대로 오역이 된다.

한계: 글리프가 아틀라스에 쪼개진 **타일 시트 상태에는 무력하다.** 인접 픽셀이 한 글자를 이루지 않기 때문이다. OCR은 인게임 스크린샷에 돌리고, 거기서 찾은 텍스트를 자산으로 역추적한다.

검출기 출력은 버리지 말고 **어노테이션 도구(§1.5)의 프리어노테이션으로 넘긴다.** 1차 박스를 이미지 위에 미리 얹어 주면 사람은 빈 화면에 박스를 처음부터 그리는 대신 기계 제안을 보고 영역을 수정·추가·삭제만 한다. 검출기는 recall(놓침 최소)을 노려 느슨하게 잡고, 정밀 좌표·누락·오탐 정리는 사람이 맡는다. 검출기 단독 정확도가 낮아도 사람의 보정 비용이 크게 준다. 이렇게 확정한 박스가 §1.4 카탈로그의 좌표 필드이자 §4의 검증 마스크가 된다.

## 2. 클린 배경 복원 — 판단 트리

한글을 얹기 전에 일문 텍스트를 지운 배경(클린 플레이트)을 만들어야 한다. 이 문서의 기술적 핵심이다. 시작 전에 먼저 확인할 것이 있다. **배경이 단색·평탄이면 복원할 것 없이 영역 전체를 배경색으로 클리어하는 것이 기본값이다.** 복원할 디테일이 없으면 클리어가 곧 클린 플레이트다. 트리는 그다음이며, 위쪽이 싸고 정확하다.

```
① 텍스트가 별도 레이어/스프라이트인가 ──예──▶ 레이어 토글 스크린샷 = 클린 플레이트
        │아니오
② 같은 배경의 변형이 2장 이상인가 ────예──▶ 다중 변형 픽셀별 max 합성
        │아니오
③ 배경이 반복 타일 / 단순 그라데이션인가 ─예─▶ 타일 복사-스탬프 / 행 단위 보간 인페인팅
        │아니오
④ (최후) content-aware fill 초안 + 원본 팔레트 재양자화
```

### ① 레이어 토글

텍스트가 배경과 다른 레이어(별도 BG 플레인, 스프라이트)에 그려진다면 배경 쪽은 이미 깨끗하다. 에뮬레이터의 레이어 토글로 텍스트 레이어만 끈 스크린샷이 곧 클린 플레이트다. 비용이 0에 가까우므로 항상 먼저 시도한다(debugging.md §2의 레이어 토글과 같은 조작이다).

### ② 다중 변형 픽셀별 최댓값 합성

같은 배경 위에 서로 다른 텍스트가 얹힌 변형 블록(메뉴 라벨 모음, 탭 모음 등)이 둘 이상 있을 때만 쓸 수 있는 기법이다(변형이 없으면 ③·④로 간다).

원리: 텍스트 픽셀이 배경보다 **어두운 값**(낮은 인덱스·밝기)이라면, 변형들을 픽셀별 최댓값(max)으로 합성했을 때 각 위치에서 "그 자리에 텍스트가 없는 변형"의 배경 값이 살아남는다. 결과물이 텍스트가 소거된 배경 템플릿이다. (인덱스드 팔레트에서는 비트 OR을 쓰면 안 된다. 다비트 값에서 OR은 max와 달라 팔레트에 없던 인덱스를 만든다. 반드시 max를 쓴다.)

절차:

1. 변형 블록들을 같은 좌표계로 정렬한다.
2. 픽셀별 max로 템플릿을 만든다. **4bpp는 니블 단위, 8bpp는 바이트 단위.**
3. 대상 영역 **전체**를 템플릿으로 덮어쓴다(일문 텍스트 소거).
4. 한글 글리프의 **비제로 픽셀만** 그 위에 오버레이한다. 글자 사이 배경 픽셀은 건드리지 않아 한글이 배경에 자연스럽게 녹는다.

사례: 한 게임 — 메뉴 라벨 5블록, 이름 입력 3블록(4bpp, 니블 max). 그 이식판 — 탭 5종, 파일 3종, 월드맵 10블록(8bpp, 바이트 max).

**함정: 4bpp는 반드시 니블 단위로 합성한다.** 4bpp는 한 바이트에 픽셀 두 개가 packed되어 있다. 바이트 단위로 max를 취하면 두 픽셀이 한 덩어리로 묶여 값이 큰 쪽 바이트가 통째로 남으므로, 투명(0) 니블이 이웃 배경 니블을 덮어쓴다. 배경이 점점이 파괴된다. 구체 예:

- 변형 A의 바이트 `0xC0` = 픽셀 (12, 0) — 둘째 픽셀에 텍스트.
- 변형 B의 바이트 `0x0C` = 픽셀 (0, 12) — 첫째 픽셀에 텍스트.
- 바이트 max → `0xC0` = (12, 0): B가 가진 둘째 픽셀의 배경 값 12가 사라진다. 니블 max → (12, 12) = `0xCC`: 양쪽 배경이 모두 산다.

비교는 픽셀 단위로 한다.

전제와 한계:

- 텍스트가 배경보다 어두운 값이어야 한다. 반대(밝은 텍스트)면 min으로 뒤집으면 같은 논리가 선다.
- 변형이 3장 이상이면 사실상 전 영역이 복원된다. 2장뿐이면 두 텍스트가 겹치는 위치는 복원되지 않는다. 잔여 불일치 영역만 ③·④로 마저 채운다.
- 일반명으로는 사진 합성의 median stacking 계열 기법이다. 다만 게임 그래픽은 변형 간 배경이 **비트 단위로 일치**하므로(노이즈 없음) median보다 max(또는 최빈값) 합성이 더 강력하다. 한 변형이라도 깨끗한 값이 있으면 그 값이 그대로 답이다.

### ③ 타일 복사-스탬프 / 행 단위 보간 인페인팅

변형이 1장뿐일 때의 수동·반자동 복원이다. 배경 구조에 따라 갈린다.

- **반복 타일 배경**: 텍스트가 없는 구간에서 온전한 타일 주기를 찾아 복사-스탬프한다. 픽셀아트의 반복 배경에서는 보존성이 높은 방법이다. 보간이나 자동 인페인팅에서 생기는 흐릿함 없이 원본 픽셀을 그대로 가져온다.
- **그라데이션 배경**: 한 사례를 보자. 변형이 1장뿐인 그라데이션 배경에서, 행(row) 단위로 어두운 픽셀을 텍스트로 분류하고 그 좌우의 배경 픽셀을 선형 보간해 메웠다. 텍스트 외곽선의 번짐(안티앨리어싱 잔여)이 분류를 빠져나가 얼룩이 남는 부분은, 바운딩 박스 밖을 원본 픽셀로 되돌려 마무리했다.

### ④ content-aware fill (최후 수단)

범용 이미지 편집기의 content-aware fill 또는 AI 인페인팅으로 초안을 만든다. 단 **원본 팔레트로의 재양자화가 필수다.** AI 인페인팅은 사진을 전제로 설계되어 팔레트에 없는 중간색을 만들고 디더링 패턴을 뭉갠다. 인덱스드 픽셀아트에서 재양자화 없이 쓰면 색 수 폭발과 디더 파괴로 오히려 더 티가 난다. 초안 → 원본 팔레트 강제 매핑(인덱스 모드 강제) → 디더·가장자리 수동 보정 순으로 정리하고, §4의 팔레트 보존 검사로 확정한다.

### 보조 패턴 — 클린 플레이트 자산화

한 번 복원한 배경은 빌드마다 재사용할 수 있게 자산화하되, **공개 저장소에 커밋할지 여부는 저작물 정책을 먼저 따른다**(`references/strategy/project-conventions.md` §6). 원본 그래픽에서 파생된 클린 플레이트는 기본적으로 커밋하지 않고, gitignore된 로컬 캐시나 빌드 출력 디렉토리에 둔다. 저장소에는 복원 절차, 바운딩 박스, 마스크, 스크립트, 사람이 그린 한글 오버레이처럼 재현에 필요한 최소 입력만 둔다. 권리상 문제가 없거나 비공개 작업 저장소라 커밋을 허용하는 경우에만 바이너리 클린 플레이트를 추적한다.

복원을 빌드 때마다 재계산하면 느릴 뿐 아니라 입력 변화에 따라 출력이 흔들려 회귀 비교가 무의미해진다. 따라서 권장 구조는 "복원 산출물은 로컬 캐시, 빌드는 캐시가 없으면 원본에서 재생성, 검증은 해시와 마스크로 고정"이다. 부수 이득으로 클린 플레이트 자체를 사람이 한 번 검수해 두면 이후에는 조판 결과만 검증하면 된다.

## 3. 한글 조판

클린 플레이트 위에 한글 글리프를 얹는다. 폰트·인코딩 전반은 `references/strategy/font-strategy.md`가 정본이고, 여기서는 베이크드 텍스트 특유의 제약(픽셀 고정 영역, 제한 팔레트, 원본 스타일 모사)만 다룬다.

### 폰트 소스

OFL(SIL Open Font License) 한글 비트맵 계열 폰트(Galmuri, dalmoori, 둥근모 등)를 글리프 소스로 쓴다. 도트가 살아 있는 비트맵 폰트라야 저해상도에서 AA 없이 깨끗하게 렌더된다. 배포 전 라이선스 조건(고지 의무, 파생 폰트 이름 조항)을 확인한다.

### 크기 한계

- **8px**: 한글 표현의 최소선. 받침 있는 글자부터 획이 뭉개진다. 짧은 라벨에 한정한다.
- **10~12px**: 실용선. 받침 가독성이 확보된다.
- 같은 높이라면 한글은 받침 때문에 일문 가나보다 빽빽하다. 원본 라벨에 세로 여유가 있으면 한 단계 키우는 쪽이 낫다.

### 원본 스타일 표

교체 전에 원본 텍스트의 스타일을 표로 추출해 카탈로그에 붙인다: 획 굵기(px), 본문 색(팔레트 인덱스), 외곽선 색·두께, 그림자 방향·색, 셀·행 높이. **새 색 추가는 금물이다.** 한글은 원본 텍스트가 쓰던 팔레트 인덱스만으로 조판한다. 새 인덱스를 쓰면 그 인덱스를 공유하는 다른 그래픽이 깨지거나, 팔레트가 외부(코드·별도 파일)에 있는 포맷에서는 의도한 색이 나오지 않는다.

### 외곽선·안티앨리어싱 원칙

- **1px 외곽선이 표준이다.** 복원된 배경 위에서 가독성을 만드는 주 수단이다. 단 원본이 외곽선 없는 스타일이면 원본을 따른다.
- **12px 이하 글리프에는 AA를 생략한다.** 그 크기에서 중간색은 획을 또렷하게 하는 게 아니라 뭉갠다.
- AA를 쓸 때도 제한 팔레트 안의 중간색 1~2단계로 한정한다.

### 자간·폭 대응

전각 한글이 원본 라벨 폭을 넘칠 때의 대응 순서:

1. **자간 축소** — 글리프 간격을 1px씩 줄인다. 손실이 가장 작다.
2. **장체** — 글리프 자체를 좁게 변형한다.
3. **축약 번역** — 번역을 줄인다. 번역 품질 비용이 있으므로 마지막이다.

반대로 한글이 원본 라벨보다 짧으면 공백 삽입으로 폭 균형을 맞춘다. 2글자 라벨을 "상 태"로 조판한 사례. 폭이 어긋나면 인접 UI 요소와의 정렬이 깨진다.

### 팔레트 제약 실사례

- 세이브 슬롯 UI가 슬롯마다 팔레트를 48 오프셋으로 어긋나게 적용한 사례. 한 슬롯 기준으로 고른 인덱스가 다른 슬롯에서는 다른 색이 된다. 글리프가 쓰는 인덱스가 전 슬롯의 오프셋 위치에서 의도한 색이 되는지 확인해야 했다.
- 배경 그라데이션이 점유한 예약 인덱스 구간을 건너뛰고(skip) 글리프 색을 배정한 사례.

두 수치(48, 예약 구간) 모두 해당 게임 실측값이다. 새 게임에서는 팔레트 인덱스의 용도 지도를 다시 그린다.

### 시각 일관성의 한계 인정

원본과 픽셀 단위로 동일한 인상을 전 자산에서 재현하는 것은 목표가 아니다. 원본 베이크드 글자는 디자이너가 픽셀을 손으로 다듬은 결과라 폰트 렌더와 미세한 무게·곡률이 다르다. 한 사례에서는 AA 렌더한 한글이 원본의 굵은 일문 글자 대비 얇게 보였고, 해결하지 못한 채 **미완 과제로 기록**했다. 한계는 기록한다. 숨기면 후속 라운드에서 다시 입력으로 잡아야 할 것을 놓치고, QA에서 버그로 되돌아온다.

## 4. 재인코딩·검증

### 포맷 보존 원칙

재인코딩 산출물은 원본과 **동일한 픽셀 포맷·텍스처 타입·해상도·팔레트 인덱스 체계**를 유지한다. 렌더러가 원본 포맷을 전제로 동작하므로, "더 좋은 포맷"으로의 임의 변환은 곧 코드 패치 의무를 낳는다. 압축·twiddle 재적용, 헤더 메타(전역 인덱스 등) 보존 같은 컨테이너 세부는 플랫폼 문서를 따른다(PVR은 `references/platforms/dreamcast.md` §3).

### 자산 레벨 자동 검증

인게임 확인 전에 자산 단계에서 기계 검사를 돌린다. 빌드에 내장하면 가장 싼 회귀가 된다(`references/strategy/build-and-verify.md`).

- **어노테이션 좌표 = 검증 마스크.** §1 카탈로그의 바운딩 박스를 그대로 검증 마스크로 쓴다: 의도한 영역 **밖**의 픽셀이 원본과 1픽셀이라도 다르면 실패. 실측 결함 2건이 이 검사 계열로 잡혔다 — 4bpp에서 홀수 폭 영역 쓰기가 경계 바이트의 이웃 니블(영역 밖 픽셀)을 파괴한 결함, 클리어 영역끼리 겹쳐 먼저 그린 이웃 라벨을 침식한 결함. 후자 때문에 **영역 겹침 금지 테스트**(카탈로그의 박스끼리 교차 검사)를 반드시 함께 둔다. 마스크 검증의 전제는 좌표가 실측이라는 것이다. 추정 좌표로는 겹침을 잡지 못한다(§1.5).
- **고유 색 수·팔레트 보존 체크.** 편집 후 고유 색 수가 원본보다 늘었으면 팔레트 이탈이다(§2 ④의 재양자화 누락이 전형). 인덱스드 포맷이면 사용 인덱스 집합이 원본의 부분집합인지 검사한다.

### 인게임 검증

- **세이브스테이트 VRAM 캐시 함정.** 세이브스테이트로 해당 화면에 직행하면 스테이트에 저장된 구 VRAM이 복원되어 교체 자산이 보이지 않는다. "패치가 안 먹었다"는 오판의 단골 원인이다. 자산 교체 확인은 **재시작 후** 실제 로드 경로로 그 화면에 도달해 스크린샷을 찍는다.
- **비교 PNG 자동 생성 관행.** 빌드가 자산별 교체 전/후 비교 이미지를 자동 생성해 두면 육안 검수와 이슈 보고가 한 장으로 끝난다.
- **공유 자산 비파괴.** 같은 타일·텍스처가 텍스트 외 용도로 참조될 수 있다. 한 사례에서는 폰트 타일 일부가 배경 장식으로도 참조되고 있어 그 슬롯을 덮으면 배경이 깨졌다. 충돌 슬롯을 피해 **글자 수를 축소**(번역 단축)하는 쪽으로 해결했다. 교체 대상의 참조처가 텍스트뿐인지 read 브레이크포인트로 먼저 확인한다(debugging.md §4).
- **변형 전수 출현 확인.** §2 ②로 복원한 변형 블록들은 게임 안에서 각 변형이 실제로 나오는 화면을 전부 밟아 확인한다. 변형 하나만 보고 끝내면 나머지 변형의 조판 오류를 놓친다.

### 완료 기준

- 카탈로그의 전 항목이 "검증 완료" 상태다. 불명 항목은 사유와 함께 보류로 남기되 0이 목표다.
- 자산 레벨 검사(마스크 밖 무변경·영역 겹침 금지·팔레트 보존)가 빌드 게이트에 들어가 있다.
- 변형·공유 자산이 걸린 항목은 관련 화면 전부의 재시작 후 스크린샷이 확보되어 있다.

## 5. 함정 모음

- **텍스처 교체만으로 불가능한 공유 셀 충돌.** 보너스 라벨 사례: 라벨들이 타일 시트를 공유하는데 한 타일이 3개 라벨에서 서로 다른 글리프를 요구했다. 시트를 어떻게 그려도 동시 만족이 불가능하다. 런타임 타일 리매퍼를 새로 짜는 복잡한 해법 대신, 라벨이 타일 인덱스를 얻는 데이터 경로를 추적해 **최소 데이터 패치**로 해결했다. 그래픽 문제처럼 보여도 정답이 데이터 패치인 경우가 있다.
- **포맷 불명 파일의 성급한 판정 금지.** 모든 후보 폭으로 디코딩해도 깨져 보인다고 "그래픽 아님"으로 단정하지 마라. 한 파일 안에 폭이 다른 섹션이 섞인 **다중 섹션 혼합 폭** 파일일 수 있다. 섹션별로 폭을 바꿔 다시 디코딩해 보고, 그래도 불명이면 카탈로그에 "불명"으로 보존하고 닫지 않는다.
- **추정 좌표 선구현의 재작업.** 좌표를 어노테이션으로 실측 확정하기 전에 교체 코드를 구현하면 재작업이다. 추정 좌표로 구현했다가 실측으로 크게 다른 값으로 교정된 사례가 있다(§1.5). 좌표 확정이 구현보다 먼저다.
- **베이크 시점 오인.** 표시 시점에 write 브레이크포인트를 걸면 상주 아틀라스에서 빗나간다. 로드 시점을 잡는다(§1.3).
- **실행 바이너리 내장 비트맵.** 베이크드 텍스트는 텍스처 파일 밖에도 있다. 한 사례에서는 실행 바이너리(부트 실행 파일) 내부에 적지 않은 양의 비트맵 글리프가 직접 내장되어 있었다. 인벤토리가 그래픽 파일·디렉토리만 훑으면 놓친다.
- **에뮬레이터 텍스처 팩은 패치가 아니다.** 에뮬레이터의 커스텀 텍스처 로딩(텍스처 팩)으로 그래픽을 갈아끼우는 경로는 특정 에뮬레이터 종속이라, 실기와 타 에뮬레이터에서 무효다. 검수·시안 용도로는 유용하지만 "패치 배포"가 아니라 별도 보조 산출물로 구분해 표기한다.
