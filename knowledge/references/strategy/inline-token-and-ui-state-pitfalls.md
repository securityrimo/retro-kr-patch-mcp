# 인라인 토큰·UI 상태 함정 6종 (2026-07-08/09 Hikaru no Go 2 실전 교훈)

live-verification-pitfalls.md의 3종 이후 발견된 함정들. 전부 "ROM 바이트는
멀쩡한데 실제 플레이에서만 드러나는" 부류라 사용자 리포트 없이는 못 잡는다.

## 함정 4: 예산 슬랙 패딩을 인라인 토큰 앞에 채움 (가변폭 빈칸)

- 증상: 이름/변수 치환 토큰(예: 플레이어 이름) **앞이나 뒤에 빈칸이 대사마다
  들쭉날쭉**하게 생긴다.
- 원인: 번역문이 원문보다 짧을 때 예산 슬랙을 **토큰 바로 옆**에 공백(0x20 등)으로
  채우는 구현이 흔한데, 어느 쪽에 채우느냐에 따라 토큰 앞/뒤에 가변폭 여백이
  생긴다. 번역 길이는 문장마다 다르므로 여백 폭도 문장마다 달라져 "들쭉날쭉"으로
  보인다.
- 교훈: 인라인 토큰(이름/변수/제어코드) 옆에는 **패딩을 두지 말 것**. 슬랙은
  항상 토큰에서 먼 쪽(문장 경계, 이전 절 뒤)에 몰아서 채운다.
  `[패딩...][번역문][보존토큰]` 형태로 — 토큰 직전에서 패딩이 끝나야
  토큰과 텍스트 사이 간격이 항상 0이 된다.
- 재사용 가능한 감사법: 원본 ROM에서 토큰 시퀀스를 SJIS 2바이트 정확 워킹으로
  추출(`0x81-9F`/`0xE0-FC` 리드바이트 스킵) → 패치본과 순서 비교(전수 일치
  필수) + 토큰 직전 연속 패딩바이트 개수 히스토그램을 내서 큰 값이 있으면 회귀.

## 함정 5: 고정폭 슬롯에 반각 패딩 사용 금지 (경칭·이름 테이블)

- 증상: 토큰 조합이 오분절되어 엉뚱한 코드로 렌더(`함정1`의 변형).
- 원인: 엔진이 고정폭(예: 2바이트×N워드)으로 파싱하는 슬롯(경칭 테이블 등)에
  반각 공백으로 패딩하면 정렬이 깨져 하위바이트가 다음 필드로 밀린다.
- 교훈: 고정폭 정렬 슬롯은 **전각(2바이트) 패딩만** 사용. 반각 1바이트 패딩
  절대 금지 — 애초에 "이게 고정폭 슬롯인가"부터 실측(디스어셈블 또는
  다른 길이 값 주입 후 렌더 비교)으로 확인하고 시작할 것.

## 함정 6: 문자 선택/입력 그리드는 "1문자=1타일"이 아닐 수 있다

- 증상: 그리드의 SJIS 코드 배열(데이터)만 바꿨는데 화면 그림은 그대로.
- 원인: 데이터 테이블(코드 배열)과 화면에 그려지는 그래픽이 **서로 다른
  리소스**일 수 있다 — 데이터는 "미리보기/입력값 표시"용 표준 폰트 렌더러가
  읽고, 화면의 선택 그리드 자체는 별도 사전렌더 BG 타일일 수 있음.
  게다가 그 BG 타일도 문자마다 폭이 고정 1타일이 아니라 **1~3타일 가변**일
  수 있다(글자 획 복잡도에 따라). 스크린엔트리 전수 스캔에서 "고유 타일 수 >
  문자 수"가 나오면 이 함정이다.
- 절차: `gfxlib.gba.screen_entries(vram, scrbase, size)`로 전수 파싱 →
  라스터 순서(ty 오름차순, tx 오름차순, 빈 칸 타일 제외)를 원본 데이터 테이블
  순서와 1:1 대조해 "테이블 인덱스 ↔ 실제 사용 타일 목록(1~3개)" 매핑을
  먼저 확정한 뒤에 재작화. 매핑 확정 전 그래픽부터 덮어쓰지 말 것.

## 함정 7: 같은 라벨의 "선택/하이라이트" 상태가 별도 에셋일 수 있다

- 증상: 탭/버튼의 기본(비선택) 상태는 번역됐는데, 선택(하이라이트)했을 때만
  원문이 다시 나온다.
- 원인: 비선택 상태와 선택 상태가 **완전히 다른 그래픽 리소스**(다른 ROM
  주소, 심지어 다른 레이어 — 예: 비선택=OBJ 스프라이트, 선택=BG 타일 폰트,
  혹은 그 반대)로 구현된 경우가 있다. 번역 시 한쪽만 찾고 "완료"로 오판하기
  쉽다.
- 절차: UI 요소를 번역하기 전에 **모든 상호작용 상태**(기본/선택/비활성/눌림)를
  라이브로 순회하며 각 상태에서 `resolve_tile(x,y,layer=...)`을 BG0~3과 OBJ
  전부에 시도해 실제로 어느 레이어·어느 타일이 그 픽셀을 그리는지 확인한다.
  한 레이어에서 blank(전부 0)가 나오면 다른 레이어를 반드시 마저 확인.

## 함정 8: 멀티에이전트 동시 작업 시 프로세스 소유권 불분명

- 증상: 여러 서브에이전트가 각자 mgba/Xvfb를 띄우며 GDB 고정 포트(2345)나
  단일 emucap 포트를 서로 점유해 라이브 검증이 막힘.
- 교훈: 각 에이전트는 **자기 소유 프로세스만** 정리해야 하며(광역 pkill 금지),
  "이번 세션에서 내가 띄운 게 맞는지" 확신이 없는 프로세스는 임의로 죽이지
  말고 상위(사용자/코디네이터) 승인을 받는다. 여러 에이전트가 동일 ROM
  파일을 in-place로 동시에 쓰면 레이스로 서로의 수정을 덮어쓰니, 각자
  전용 사본에서 작업 후 diff/오프셋만 보고해 상위에서 병합한다.

## 함정 9: 호스트 슬롯 재배정으로 세이브 파괴

- 증상: 패치를 재빌드할 때마다 기존 세이브의 캐릭터 이름 등이 다른 한글로
  깨진다(예: "쿠난민"→"켜껄명").
- 원인: "음절 → host SJIS 코드" 배정을 매 빌드마다 집합 순회 순서 등으로
  재계산하면, SRAM/세이브 파일에 저장된 옛 코드값이 새 빌드에서는 다른
  글리프를 가리키게 된다. 텍스트·그래픽이 아무리 정확해도 **세이브
  호환성**은 별도로 챙겨야 하는 축이다.
- 교훈: 배포된(사용자가 실제 플레이 중인) ROM에서 글리프 비트맵을 역대조해
  "음절 ↔ host 코드" 매핑을 **동결 레지스트리**(JSON)로 한 번 저장하고,
  이후 모든 빌드는 이 레지스트리를 최우선 조회하며 **신규 음절만** 빈
  슬롯에 추가한다. 기존 배정은 절대 재계산하지 않는다 — "한 번 정해지면
  영구"가 세이브 호환 프로젝트의 기본 규칙.

## 자산 노트: 초소형 도트 폰트

8×8처럼 아주 작은 셀에 범용 TTF(예: 상용 고딕체)를 그냥 축소 렌더하면
받침 있는 한글은 얼룩으로 뭉갠다. **Galmuri 폰트 패밀리**(7/9/11/14px,
SIL OFL, github.com/quiple/galmuri)는 애초에 이 해상도를 겨냥해 설계돼
받침 있는 음절도 판독 가능하게 나온다 — 8×8은 Galmuri7(내장 비트맵 strike
10×10, `ImageFont.truetype(path, 10)`로만 로드됨), 10×10은 Galmuri9 계열을
우선 시도할 것.
