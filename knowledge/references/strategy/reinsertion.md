# 재삽입 (Reinsertion)

번역된 한글 텍스트를 바이너리에 다시 넣는 단계의 전략 문서다. 한글 번역은 원문(대개 일본어 또는 영어)과 바이트 길이가 거의 항상 달라지므로, 재삽입의 본질은 "길이가 달라진 데이터를 게임이 깨지지 않게 수용시키는 일"이다. 정책 결정 → 포인터 재배치 → 훅 설계 → 공간 확보 → 엔진 보호 순으로 다룬다. 아래 서술은 일본어 원문을 전제로 한다.

## 1. 재삽입 정책 결정

엔트리(문자열) 단위로 세 가지 정책 중 하나를 적용한다. 파일 단위가 아니라 **엔트리 단위로 정책을 섞는 하이브리드가 기본 선택지**다. 정책은 번역 품질, 포인터 규약의 확실성, 저장 공간, 로더·버퍼 한계를 함께 보고 정한다. 길이를 줄이는 것은 가장 쉬운 해결처럼 보이지만, 대사 본문처럼 품질 영향이 큰 항목에서는 먼저 재배치·성장 가능성을 검토한다. 반대로 이름, 메뉴 항목, 아이템명처럼 고정 슬롯 자체가 UI 규약인 항목은 길이 보존이 맞을 수 있다.

### 1.1 길이 보존 (고정길이 패치)

원본 바이트 수를 그대로 유지한다. 짧으면 패딩, 길면 잘라낸다(truncate). 포인터를 전혀 건드리지 않으므로 위험이 낮다.

- 패딩 값은 게임이 공백·무동작으로 처리한다고 실측한 코드를 쓴다(타일 인덱스 인코딩 게임은 공백 타일 코드, SJIS 계열은 전각 공백 `0x8140` 등). `0x00`은 에러 문자나 종료로 해석되는 경우가 많으므로 기본 패딩으로 쓰지 않는다. 단, 대상 엔진에서 `0x00`이 실제 공백/무동작이고 종료·파라미터와 충돌하지 않는다고 검증되면 그 게임의 패딩 정책으로 기록해 사용할 수 있다.
- 패딩 위치는 종료자·trailing 제어코드 규약을 먼저 분석해 정한다. 많은 엔진에서는 trailing 제어코드 앞에 넣는 것이 안전하지만, 종료자 뒤 바이트를 읽지 않는 엔진이나 고정 레코드 구조처럼 다른 규칙을 가진 포맷도 있으므로 6.3절의 판정 기준을 따른다.
- truncate 시에도 종료 제어코드는 보존한다: `budget = 원본길이 - 종료코드 크기`까지만 텍스트를 채우고(멀티바이트 문자 경계 보호), 종료코드를 다시 붙인다.
- 변형으로 **종료코드 보존 모드**가 있다(이름은 자유): 종료코드까지만 쓰고 나머지 원본 바이트를 그대로 둔다. 게임이 종료코드에서 읽기를 멈추므로 동작은 같고, 다른 문자열과 연결 렌더링되는 엔트리에서 안전하다.

### 1.2 완전 재배치

원본 위치의 텍스트를 버리고, 번역문을 자유 공간(확장 뱅크, 빈 영역)에 새로 배치한 뒤 해당 텍스트를 가리키는 포인터를 새 주소로 리라이트한다. 길이 제약이 사라지는 대신 **그 텍스트를 가리키는 포인터를 전부 찾아야** 한다. 하나라도 놓치면 게임이 옛 위치의 쓰레기 데이터를 읽는다.

### 1.3 제자리 성장 (in-place growth) + 포인터 재계산

파일 안에서 텍스트 블록을 splice(잘라내고 새 바이트로 치환)하여 파일 자체가 늘거나 줄게 두고, 뒤로 밀린 모든 데이터에 대한 포인터를 누적 이동량으로 보정한다. 텍스트가 한 영역에 모여 있고 포인터 규약이 파악된 파일에서 가장 번역 품질이 좋다. 단, 텍스트 뒤에 위치 의존 데이터(코드 스텁 등)가 따라오면 그것도 함께 재배치해야 한다(2.4절).

### 1.4 선택 기준

| 조건 | 권장 정책 |
|------|----------|
| 포인터 규약이 완전히 파악됨 + 텍스트가 연속 영역에 모여 있음 | 제자리 성장 + 포인터 재계산 |
| 포인터가 바이트코드·데이터 레코드 사이에 산재, 전수 파악이 어려움 | 길이 보존 |
| 텍스트가 코드 영역에 박혀 있고 주변에 공간이 없음 | 완전 재배치 (확장 영역으로) |
| 메뉴·아이템명 등 짧은 고정폭 문자열 | 길이 보존 (품질 손실 적음) |
| 대사 본문 (길이 변동이 크고 품질이 중요) | 제자리 성장 또는 완전 재배치 |

실측 사례에서 나타난 조합은 이렇다.

- 메시지가 **갭 0바이트로 완전 연속**인 스토리 스크립트는 1바이트만 늘어도 후속 포인터 전부를 보정해야 한다. 이때 포인터 테이블이 한 곳에 깨끗이 모여 있으면 제자리 성장이 맞다. 반면 같은 게임의 메뉴 스크립트는 RAM 포인터가 바이트코드·아이템 레코드·자기참조에 산재하기 쉬워, 앞쪽 텍스트 영역은 길이 보존, 맨 뒤 영역(뒤에 코드 스텁만 있음)만 성장을 허용하는 식으로 갈린다.
- 일반 대사는 splice + 포인터 자동 보정, 기술명 테이블·메뉴 엔트리처럼 짧은 고정폭 항목은 엔트리별 고정길이 플래그로 처리한 하이브리드 사례가 있다.
- 텍스트 뱅크 인플레이스 수용과 확장 뱅크 재배치를 혼합해, "들어가면 제자리, 넘치면 재배치"를 엔트리별로 빌드 파이프라인이 자동 판정하게 할 수 있다.

정책은 빌드 도구의 엔트리 속성(JSON 플래그 등)으로 표현해, 같은 파이프라인이 엔트리마다 다른 정책을 적용할 수 있게 만든다.

재배치 정책을 본격 채택하기 전에는 `references/strategy/poc.md`의 재배치·길이 게이트를 대표 텍스트 1단위로 닫는다. 표본에서 고정 슬롯 초과가 드물고, 포인터 규약이 단순하며, 빌드 게이트가 초과 항목을 확실히 잡으면 별도 PoC 없이 전수 검사로 넘어갈 수 있다. 반대로 표본에서 over-read, 다음 엔트리 침범, 내부 점프, tail code, 파일 크기 증가가 보이면 전량 번역 전에 재배치 가능성을 먼저 증명한다.

## 2. 포인터 재배치 알고리즘

제자리 성장 정책의 핵심 알고리즘이다. 입력은 패치 목록 `(offset, orig_len, new_bytes)`, 출력은 splice된 버퍼와 보정된 포인터들이다.

### 2.1 역순 splice + 누적 shift

```
# 1. 패치를 오프셋 오름차순으로 정렬
patches.sort(by=offset)

# 2. 누적 이동량(shift) 테이블 구축 — 포인터 보정에 사용
shifts = []          # [(경계 오프셋, 누적 delta)]
cum = 0
for p in patches:
    cum += len(p.new_bytes) - p.orig_len
    shifts.append((p.offset + p.orig_len, cum))   # 원본 좌표 기준 경계

# 3. splice는 오프셋 "역순"으로 적용
#    뒤에서부터 치환하면 앞쪽 패치의 원본 오프셋이 끝까지 유효하다
for p in reversed(patches):
    buf[p.offset : p.offset + p.orig_len] = p.new_bytes

# 4. 포인터 보정
def lookup_shift(target):       # target은 원본 좌표
    delta = 0
    for (boundary, cum) in shifts:
        if target >= boundary: delta = cum
        else: break
    return delta

for ptr_loc in pointer_locations:
    target = read_ptr(buf, ptr_loc) - ram_base    # RAM 주소형이면 베이스 차감
    new_target = target + lookup_shift(target)
    write_ptr(buf, ptr_loc, new_target + ram_base)
```

요점:

- **역순 패치**: 앞에서부터 치환하면 매 패치마다 뒤쪽 오프셋이 밀려 원본 좌표가 무효화된다. 역순이면 모든 패치를 원본 좌표 그대로 적용할 수 있다.
- **shift 테이블은 원본 좌표 기준**으로 만든다. 포인터 값도 원본 좌표이므로 그대로 조회 가능하다.
- 포인터가 RAM 절대 주소면 `파일 오프셋 = 포인터 값 - ram_base`로 환산 후 동일하게 처리한다. ram_base는 파일 헤더나 로더 코드에서 확정한다.
- 포인터 자체가 패치 영역보다 앞(스크립트 코드부)에 있으면 포인터의 **위치**는 변하지 않고 **값**만 갱신하면 된다. 포인터 테이블 자체가 패치 영역 뒤에 있으면 테이블 위치 이동도 반영해야 한다.

### 2.2 정렬 패딩

엔진이 텍스트 블록의 정렬(2바이트·4바이트)을 가정하는 경우, 엔트리별 delta가 정렬 단위의 배수가 되도록 공백 타일을 삽입한다. 예컨대 4바이트 정렬을 가정하는 엔진이면 엔트리마다 `delta % 4 != 0`일 때 공백 타일을 채워 정렬을 유지한다. 이 패딩도 게임별 종료자 규약에 맞춰 넣는다(6.3절).

### 2.3 검증 단계

포인터 보정은 조용히 틀리기 쉬우므로 빌드 시 자동 검증을 박는다.

1. **범위 체크**: 보정 후 값이 포인터 폭(16비트면 0x0000-0xFFFF)을 넘치면 빌드 에러. 뱅크 경계를 넘는 경우는 별도 처리(3.3절).
2. **타겟 유효성**: 보정된 포인터가 가리키는 위치에 기대 패턴(씬 시작 명령, 제어코드 등)이 실제로 있는지 확인한다.
3. **통계 비교**: 스캔된 포인터 수를 빌드마다 출력하고 이전 빌드와 비교한다. 급감하면 스캔 누락, 급증하면 false positive 의심.
4. **이동량 0 스킵**: shift가 0인 포인터는 건드리지 않는다(불필요한 쓰기를 줄여 diff 검토가 쉬워진다).
5. **라운드트립**: 패치된 파일을 다시 덤프해 번역문이 의도한 위치에서 디코딩되는지 확인한다.

### 2.4 텍스트 뒤의 위치 의존 데이터 (tail code)

텍스트 영역 뒤에 **절대 주소를 하드코딩한 코드 스텁**이 따라오는 파일 포맷이 있다. 예컨대 스크립트 파일 마지막 메시지 뒤에 MIPS crt0 스텁(BSS 클리어·스택 설정·모듈 엔트리 호출)이 붙고, 스텁 내부의 자기참조·EOF 참조가 전부 절대 RAM 주소인 경우가 있다. 텍스트가 자라면:

1. 스텁을 새 위치(성장한 텍스트 끝 + 정렬)로 복사한다.
2. 스텁 안의 주소 상수 쌍(MIPS면 `lui`+`addiu`)을 스캔해 전체 이동량만큼 재계산한다.
3. 스텁 끝의 내장 데이터 상수(u32 주소들)도 갱신한다.

이 교훈은 일반화할 수 있다. splice 전에 "텍스트 뒤에 무엇이 있는가"를 반드시 확인하고, 위치 의존 코드·데이터가 있으면 재배치 대상에 포함한다. 디스패치 테이블처럼 **인덱스 기반**(오프셋이 아닌 번호) 구조는 갱신이 필요 없다. 무엇이 오프셋이고 무엇이 인덱스인지 구분하는 것이 분석의 핵심이다.

## 3. 다중 포인터 패턴

한 파일·한 게임 안에 여러 포인터 규약이 공존하는 것이 보통이다. 전체 스크립트의 포인터를 **여러 패턴으로 분류**해 일괄 수정한 사례가 다중 포인터 정리의 예다.

### 3.1 패턴 분류 예

| 패턴 | 시그니처 | 포인터 형식 | 대상 |
|------|----------|------------|------|
| 1 | `00 24 XX XX` + 서픽스 `00 00 00 05` | 2바이트 BE 파일 오프셋 | 맵 이벤트 |
| 2 | `00 25 XX XX` + 서픽스 `05 00 00 00` | 2바이트 BE 파일 오프셋 | 파트 이벤트 (서픽스 바이트 순서가 패턴 1과 역순) |
| 3 | `00 20 XX XX` / `00 21 XX XX` | RAM 주소 (base 차감 필요) | 공용 스크립트 |

(발췌 — 전체 패턴 표는 `references/platforms/saturn.md` 6절 참조)

같은 게임인데도 명령어 프리픽스, 서픽스 바이트 순서, 오프셋형/RAM 주소형, 2바이트/4바이트가 전부 섞여 있을 수 있다. **파일 종류별로 패턴을 실측 확정한 뒤, 패턴마다 전용 스캐너를 만들어 합산**한다. 단일 만능 스캐너를 만들려 하면 false positive가 급증한다.

### 3.2 False positive / false negative 제어

좌표·수치 데이터가 우연히 포인터 시그니처와 일치하는 일이 실제로 일어난다(예: 좌표값 37 = `0x25`가 포인터 프리픽스로 오인). 방어책:

- **서픽스 검증**: 프리픽스만이 아니라 뒤따르는 고정 바이트열까지 일치해야 포인터로 인정. 단, 위험 구간(뱅크 경계 횡단 등)에만 선별 적용한다.
- **정렬 검증**: 타겟이 홀수 등 불가능한 값이면 스킵.
- **타겟 내용 검증**: 가리키는 위치에 기대 명령(씬 시작 코드 등)이 있는지 확인.
- 반대 방향 교훈: 검증을 과하게 걸면 **진짜 포인터를 놓쳐**(false negative) NPC 미표시 같은 조용한 버그가 난다. 서픽스 블랙리스트가 false negative를 유발하면 제거해야 한다. 검증 규칙 추가·제거 시 반드시 포인터 수 통계와 실기 증상으로 회귀 확인한다.

### 3.3 뱅크 경계 횡단

16비트 포인터 + 뱅크 프리픽스 구조에서 텍스트 축소·성장으로 타겟이 뱅크 경계(0x10000)를 넘으면, 오프셋 값만 아니라 **프리픽스(뱅크 선택 명령)도 전환**해야 한다. 이때가 false positive 검증을 강하게 걸어야 하는 지점이다.

### 3.4 문자열 중간을 가리키는 포인터

포인터가 항상 문자열의 머리를 가리키는 것은 아니다. 이벤트 스크립트가 조건 분기(이미 본 텍스트 스킵, 응답 점프)를 위해 **텍스트 체인 중간으로 직접 점프**하는 포인터를 가지기도 한다(이른바 "팬텀 포인터" — 뱅크 내 자기참조 mid-chain 포인터를 별도로 추가 등록해야 한다). 번역문은 원문과 내부 구조가 달라 "원문 오프셋 +N" 식 단순 환산이 불가능하므로:

1. **구조 마커 카운팅**: 원문에서 해당 중간 지점 앞에 나온 특정 제어코드(박스 태그 등) 개수를 세고, 번역문에서 같은 인덱스의 제어코드 위치로 매핑한다.
2. **제어코드 경계 매핑**: 중간 지점이 제어코드 직후라면, 같은 순번의 제어코드 직후로 매핑한다. 이때 글리프 조합용 프리픽스 바이트는 구조 분리자가 아니므로 카운트에서 제외한다.
3. **Fallback**: 매핑 불가능하면 그 체인만 원문을 원위치에 보존한다.

포인터 카탈로그(어떤 위치의 포인터가 어떤 형식으로 어디를 가리키는지의 전수 목록)를 데이터로 관리하고, 카탈로그에 있는데 번역 엔트리가 없는 포인터를 빌드 시 검출하는 구조가 안전하다.

## 4. ASM 훅 설계

폰트·인코딩을 바꾸면 원본 텍스트 엔진이 새 인코딩을 모르므로, 엔진에 한글 경로를 끼워 넣는 훅이 필요하다. 확립된 형태는 **트램폴린 → 디스패치 → 렌더의 3층 구조**다.

### 4.1 3층 구조

```
[원본 엔진 코드]
   문자 처리 지점의 명령 몇 바이트를 점프로 교체   ← 1층: 트램폴린 (원본 영역, 최소 바이트)
        │
        ▼
[디스패치]  (원본 뱅크의 자유 공간)              ← 2층
   현재 바이트가 한글 프리픽스 범위인가?
   ├─ 아니오 → 덮어쓴 원본 명령을 재현하고 원래 흐름으로 복귀
   └─ 예     → 두 번째 바이트 읽기, 뱅크 전환, 렌더로 점프
        │
        ▼
[한글 렌더]  (확장 뱅크 / 프리 스페이스)          ← 3층
   글리프 주소 계산 → VRAM 전송 → 엔진 상태 동기화 → 원본 흐름으로 복귀
```

층을 나누는 이유는 공간 제약이다. 원본 코드 영역은 몇 바이트밖에 못 건드리고(트램폴린), 원본 뱅크의 자유 공간은 작으며(디스패치), 부피가 큰 렌더 로직은 확장 영역에 둔다. 코드 개입은 원본 영역(트램폴린 리다이렉트·본체, 디스패치 진입 교체, read-ahead 가드, 디스패치 본체, 미세 보정)과 확장 뱅크 렌더 코드로 나뉜다. 게임 전체 한글화에 필요한 코드 개입을 수백 바이트 수준으로 작게 유지할 수 있고, 작게 유지하면 원본 영역을 덜 건드리게 되어 회귀 범위가 좁아진다.

훅 개수는 게임 구조에 따라 다르다. 단일 텍스트 엔진 게임은 디스패치 훅 한둘로 끝나지만, UI가 별도 렌더러를 쓰면 화면별 훅이 추가된다(엔진 훅 외에 압축 해제 인터셉트, OAM(스프라이트 테이블) DMA 훅, 인라인 바이트 패치 등이 더해질 수 있다). 먼저 메인 엔진 훅으로 본문을 뚫고, UI는 화면 단위로 추가해 나간다.

### 4.2 레지스터·플래그 보존 체크리스트

훅 코드는 원본 흐름의 한가운데에 삽입되므로, 복귀 시점에 CPU 상태가 원본과 완전히 같아야 한다. 체크리스트:

- [ ] **사용하는 레지스터는 전부 저장/복원** (스택 push/pop 또는 예약 RAM에 저장). 복귀 직전 pop 순서가 push의 역순인지 확인.
- [ ] **플래그 레지스터 보존**: 비교·산술을 수행하면 플래그가 바뀐다. 원본의 후속 코드가 플래그에 의존하면(직후가 조건 분기면 반드시) 플래그도 저장/복원한다.
- [ ] **모드 플래그 복원** (65816): 훅 안에서 누산기/인덱스 폭(M/X)을 바꿨으면 복귀 전에 원래 폭으로. immediate 로드 명령의 바이트 수가 M 플래그에 따라 달라지므로, 폭이 어긋나면 이후 코드 전체가 어긋난 채 디코딩된다.
- [ ] **뱅크 레지스터 복원**: 데이터 뱅크/프로그램 뱅크를 바꿨으면 복원. 뱅크 스위칭 플랫폼(Z80 매퍼)은 슬롯의 원래 뱅크 번호를 저장해 두고 되돌린다.
- [ ] **덮어쓴 원본 명령 재현**: 트램폴린이 지운 원본 바이트를 훅 끝에서 그대로 실행하거나 동등한 효과를 재현한 뒤 "원본 다음 명령" 주소로 복귀한다. 덮어쓴 구간에 **상대 분기가 포함되면 오프셋 재계산** 필수.
- [ ] **인터럽트 보호**: VRAM 포트 접근 등 원자성이 필요한 구간은 인터럽트 금지/허용으로 감싼다(Z80 DI/EI). 인터럽트 금지 상태로 복귀하지 않도록 짝을 확인.
- [ ] **delay slot** (SH-2 등): 분기 명령을 교체할 때 delay slot의 명령이 분기 전에 실행된다는 점을 반영한다.
- [ ] **스택 밸런스**: 훅 진입·이탈 경로 전부(분기 포함)에서 push/pop 짝이 맞는지 경로별로 센다.
- [ ] **가드의 위치**: 호출 조건을 판별하는 가드 코드는 **서브루틴 호출(원본 루틴 재호출 등)보다 앞**에 둔다. 호출된 루틴이 판별에 쓰는 레지스터·dp 변수를 덮어쓸 수 있다(상점 화면 훅 등에서 실제로 겪는 함정).

### 4.3 설치 안전장치

- **사전 조건 검증 쓰기**: 훅 설치 코드는 "원본 위치에 기대 바이트가 정확히 있는가"(Bytes 검증), "자유 공간이 정말 비어 있는가"(0xFF/0x00 충진 검증)를 확인하고 불일치 시 빌드를 중단한다. 모든 ROM 쓰기를 이 검증을 강제하는 단일 경로로만 수행하면, 영역 충돌과 오설치를 빌드 단계에서 잡을 수 있다.
- **동적 주소 체인**: 확장 뱅크 안에 폰트·리맵 테이블·훅 코드·자유 공간을 순서대로 배치할 때, 각 단계가 끝 주소를 반환해 다음 단계의 시작 주소로 넘기는 파이프라인으로 만들면 데이터 크기 변화에 주소가 자동 추종한다. 하드코딩 주소 상수는 글리프 수가 바뀔 때마다 어긋난다.

### 4.4 ASM은 바이트가 아니라 명령어로 emit한다

훅 코드를 raw 바이트 배열로 직접 쓰지 않는다. 명령어를 타입으로 표현하는 enum과 라벨 기반 어셈블러로 조립하거나, 검증 가능한 외부 어셈블러를 빌드 파이프라인에서 호출한다. 핵심은 사람이 헥스 배열을 손계산하지 않는 것이다. 자체 인코더를 만들 때는 `references/strategy/initial-survey.md` 5절의 도구 선택 기준과 맞춰, 같은 명령어 테이블을 디코더(분석)와 인코더(패치 생성)가 공유하게 한다.

바이트를 직접 손코딩하면(`[0xA9, 0x01, 0xF0, 0x07, …]` 같은 헥스 배열) 다음 문제가 생긴다.

- **분기 오프셋 손계산**: 상대 분기의 오프셋은 명령어 위치에 의존한다. 코드 한 줄을 넣거나 빼면 그 뒤 모든 분기 오프셋이 바뀌어 전수 재계산해야 하고, 한 곳만 틀려도 이후가 전부 어긋난다. 가장 흔한 버그 원인이다.
- **라벨 이동에 취약**: 확장 뱅크 배치가 바뀌면(글리프 수 변동 등) 점프 대상 주소가 이동하는데, 하드코딩 주소는 전부 붕괴한다(4.3절 동적 주소 체인과 같은 문제다).
- **모드 의존 인코딩 오류**: 65816 immediate 로드처럼 명령어 길이가 CPU 모드 플래그(M/X)에 따라 달라지는 경우, 손코딩은 모드를 추적하지 못해 길이를 틀리고 이후 전체가 어긋난 채 디코딩된다.
- **리뷰·검증 불가**: 헥스 배열은 무엇을 하는 코드인지 읽을 수 없어 사람도 도구도 검증하지 못한다.

명령어 enum + 라벨 어셈블러로 emit하면 이 실패들이 구조적으로 사라진다.

- **라벨 자동 해결**: 분기 대상을 라벨로 지정하면 어셈블러가 최종 배치 기준으로 오프셋을 계산한다. 코드를 추가·삭제해도 다시 emit하면 끝이다.
- **타입 안전**: 잘못된 피연산자·주소 모드를 빌드 단계에서 막는다.
- **디코더와 테이블 공유**: 인코더는 디코더의 역방향이라 명령어 테이블 하나만 유지하면 분석과 생성이 항상 일치한다.
- **단위 테스트 가능**: emit한 바이트를 디코더로 되읽어 의도한 명령과 같은지 라운드트립으로 검증한다. 인코더의 정확성이 테스트로 증명된다.
- **결정적 빌드**: 자체 인코더를 쓰든 외부 어셈블러를 쓰든, 같은 입력에서 같은 바이트가 나오고 그 바이트를 디코더/테스트로 검증할 수 있어야 한다.

기성 어셈블러(WLA-DX 등)를 호출하는 선택지도 있고, 자체 인코더를 빌드 도구에 내장하는 선택지도 있다. 어느 쪽이든 버전·입력·출력 검증을 고정하고, raw 바이트 손코딩만은 피한다.

## 5. 공간 확보

늘어난 텍스트·글리프·훅 코드를 둘 공간의 확보 수단. 플랫폼별 허용 여부와 구체 절차는 `references/platforms/` 아래 해당 플랫폼 문서를 따른다.

### 5.1 기존 공간 재활용 (확장 불가하거나 불필요할 때)

- **텍스트 블록 사이 gap·미사용 영역**: 단, 메시지가 갭 0으로 완전 연속인 포맷도 있다(스토리 스크립트에서 관찰된 사례가 있다). 패딩 흡수가 가능한지부터 실측한다.
- **해방 영역 재활용**: 재배치로 비워진 원본 텍스트 영역을 2-pass로 재활용한다(뱅크 자유 공간이 빠듯하면, 타 뱅크 이전 분을 먼저 빼내고 비워진 텍스트 영역을 뱅크 내 잔류분에 재할당하는 식).
- **압축률 개선분**: 한글 글리프는 한자보다 단순해 재압축 시 원본보다 작아지기도 한다. 압축 포맷 게임에서는 공간 문제가 자연 해소되는 경우가 있다.
- **자유 공간 검증**: 0x00/0xFF 연속 영역이라도 데이터로 쓰일 수 있다. NOP처럼 보이는 영역의 일부를 게임이 RAM 초기화용 제로 데이터로 복사하고 있는 경우가 있다. 해당 주소를 읽는 코드가 없는지 교차 확인 후에만 사용한다.

### 5.2 ROM 확장 / 뱅크 추가 (카트리지 매체)

ROM 파일 끝에 영역을 덧붙이고 헤더의 크기 필드를 갱신한다(예: ROM 용량을 키운 뒤 확장 뱅크에 폰트와 렌더 코드를 배치하거나, 확장 뱅크에 재배치 텍스트·별도 뱅크에 글리프와 훅 코드를 나눠 둔다). 매퍼/메모리 맵 한계, 헤더 체크섬, 미러링 영향은 플랫폼 문서에서 확인한다.

### 5.3 디스크 빈 영역 재배치 (CD 매체)

파일이 원본 섹터 할당을 초과하면 디스크의 빈 영역으로 파일째 옮기고 파일시스템 메타데이터를 갱신한다. 예컨대 CD-DA 트랙 직전 빈 영역으로 자동 재배치하고 ISO 9660 디렉토리 엔트리의 LBA와 크기를 동시 패치하며, EDC/ECC 재계산까지 빌드에 포함한다. 실행 파일이 파일 크기 테이블을 따로 들고 있으면(로딩 버퍼 산정용) 그것도 갱신해야 한다.

### 5.4 RAM 버퍼 한계 확인

파일을 키우기 전에 게임이 그 파일을 올리는 **RAM 버퍼의 크기**를 확인한다. 디컴프 버퍼가 최대 파일보다 충분히 크면 성장을 수용하지만, 버퍼가 빠듯한 게임은 파일 성장 자체가 막힌다. 버퍼 한계가 곧 정책 결정(1절)의 입력이 된다.

## 6. 엔진 보호

원본 엔진은 원문 인코딩과 원문 길이를 전제로 만들어져 있다. 그 전제가 깨지는 지점을 찾아 방어하지 않으면, 패치가 "대체로 동작하는데 가끔 깨지는" 상태가 된다.

### 6.1 read-ahead 가드

텍스트 엔진이 현재 바이트를 처리하면서 **다음 바이트를 프리페치**해 제어코드 여부를 판정하는 구조가 흔하다. 2바이트 한글 코드를 도입하면 두 번째 바이트가 프리페치 판정에 걸려 오동작한다(제어코드로 오인 → 조기 flush·종료). 대응은 read-ahead 지점에 가드를 삽입해, 프리페치된 바이트가 한글 프리픽스 범위(≥0x80)면 와이드 스페이스 코드로 대체해 판정을 무해화하는 것이다. 엔진 디스어셈블 시 "현재 위치 +1을 읽는 코드"를 전부 찾아 같은 검토를 한다.

### 6.2 VRAM/타일 잔류 클리어

한글은 원문보다 타일 소비가 많아(고정폭 16px, 글자 수 증가) 원본 엔진의 클리어 범위를 벗어난 영역에 글리프가 남는다. 증상은 "이전 대사의 글자가 다음 화면에 유출". 대응은 텍스트 영역 풀 클리어 훅을 추가하는 것인데, 클리어 시작 행을 한 행 늦게 잡으면 특정 상황에서만 잔류가 재발하므로 **클리어 범위는 경계 행/열까지 실측으로 확정**해야 한다. 같은 이유로 네임테이블 기반 플랫폼에서는 한글(1글자=1타일)이 가변폭 원문 대비 2-3배의 네임테이블 엔트리를 소비해 텍스트 윈도우 경계를 넘어 인접 그래픽(초상화 등)을 침범할 수 있다. 윈도우별 행·열 한계를 파악하고 번역 길이를 빌드 시 검증한다. 레이아웃 계산을 코드에 직접 박기보다 글자별 픽셀 폭 테이블을 입력으로 받는 줄바꿈·배치 루틴으로 분리해 두면, 인코딩이 바뀌어도 폭 테이블 교체만으로 같은 렌더러를 재사용할 수 있다(공개 ROM해킹 프레임워크에서 관찰되는 설계 옵션이다).

### 6.3 패딩·정렬 바이트의 위치

고정길이 패딩(1.1절)과 정렬 패딩(2.2절)은 대상 엔진의 종료자·trailing 제어코드 규약에 맞춰 삽입한다. 다음 순서로 판정한다.

1. 종료 코드가 나오면 엔진이 즉시 읽기를 멈추는지, 아니면 뒤따르는 파라미터/제어코드 시퀀스까지 소비하는지 디스어셈블과 브레이크포인트로 확인한다.
2. trailing 제어코드가 종료자와 한 덩어리로 소비된다면 패딩은 그 시퀀스 앞에 넣는다. 정렬 패딩이 종료 제어코드 뒤에 들어가면 하드락이 발생할 수 있으므로, 바이트열 끝에서 역방향으로 제어코드 쌍을 전부 건너뛴 위치에 삽입한다.
3. 종료자 뒤 바이트를 전혀 읽지 않는 고정 레코드라면, 종료자 뒤 남은 영역을 패딩으로 쓸 수 있다. 단 포인터·주소 등 비텍스트 데이터가 이어지면 덮어쓰지 않는다.

판정 결과는 엔트리 스키마나 빌드 설정에 `pad_byte`, `pad_position`, `terminator_policy` 같은 데이터로 고정해, 문자열마다 임의 판단하지 않게 한다.

### 6.4 상태 변수 경합

훅이 설정한 상태 플래그(뱅크 선택, 모드 등)를 엔진의 다른 경로가 **재초기화로 덮어쓰는** 경합이 있다. 예컨대 한글 뱅크 플래그를 훅이 세팅한 뒤, 특정 연출 경로의 텍스트 재초기화가 그 변수를 덮으면 폰트 뱅크가 폴백되는 깨짐이 발생한다. 훅이 기존 RAM 변수를 용도 변경(재활용)할 때는 그 변수에 쓰는 코드 전부를 교차 검색하고, 가능하면 전용 미사용 RAM을 쓴다. 디버깅은 해당 주소 쓰기 브레이크포인트가 가장 빠르다.

### 6.5 공유 자원의 초기화 책임

한글 렌더러가 원본 엔진과 자원(타일 카운터, 합성 버퍼, 타일 베이스)을 공유하는 설계에서는 **초기화 책임을 한쪽에 명확히 귀속**시킨다. 한글 렌더러가 엔진의 타일 카운터·타일 베이스를 공유해 출력 타일을 순차 할당받는 구조라면, 세션 미시작 상태에서 베이스 초기화를 빠뜨릴 때 첫 한글 문자가 stale 값으로 엉뚱한 타일을 덮어쓸 수 있다. 렌더러 진입 시 세션 플래그를 검사해 미시작이면 카운터와 베이스를 함께 초기화하고, 이탈 시 엔진 상태 변수를 "다음 원문 문자가 새 타일에서 시작"하도록 맞춰 둔다. 반대로 엔진 init 쪽의 중복 리셋이 한글 카운터를 망가뜨리면 그 리셋을 NOP 처리하는 식으로, 초기화 지점을 하나로 만든다.

### 6.6 인코딩 누락의 조기 검출

번역문에 인코딩 테이블에 없는 문자가 있으면 **빌드 에러로 즉시 실패**시킨다. 누락을 조용히 건너뛰면(silent skip) 해당 구간의 원문 바이트가 미패치 상태로 한글 훅을 통과해 프리픽스 오감지·깨진 글자로 나타나는데, 증상 위치가 원인 위치와 멀어 추적이 매우 어렵다(bisect로만 원인을 좁힐 수 있는 함정).

### 6.7 사용자 문자열 저장·재표시

이름 입력, 저장명, 점수 등록처럼 플레이어가 만든 문자열은 일반 번역 텍스트와 다르다. 입력 UI가 허용하는 문자 집합, 저장 버퍼의 바이트 길이, 세이브 데이터 인코딩, 이후 화면의 재표시 경로가 모두 맞아야 한다. 한글 입력을 지원한다면 입력 단계에서 선택 가능한 글자를 `font-strategy.md`의 글리프·코드 테이블과 같은 소스에서 만들고, 저장 시에는 종료자·패딩·최대 길이를 고정 슬롯 규약으로 검증한다. 기본 이름만 번역하고 실제 한글 입력은 범위 밖으로 둘 때도, 저장 화면·대화 변수 삽입·상태창이 그 기본 이름을 같은 코드 경로로 다시 표시하는지 확인한다.
