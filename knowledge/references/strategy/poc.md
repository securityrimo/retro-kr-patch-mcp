# PoC 전략

## 1. 역할

PoC(proof of concept)는 본 구현 전에 배포 가능성을 뒤집을 수 있는 불확실성을 작은 증거로 판정하는 게이트다. 쉬운 성공을 보여 주는 데 목적이 있는 것이 아니라, 가장 비싼 실패의 전제가 대상 리비전과 실제 소비 경로에서 성립하는지 판정한다.

모든 게이트를 반복하지 않는다. 정적 분석·계산이나 같은 리비전과 동일한 검증 조건에서 얻은 동등한 증거가 이미 위험을 해소했다면 그 근거를 재사용하고 PoC를 생략한다. 반대로 남은 불확실성이 되돌리기 비싼 작업의 전제라면 해당 작업보다 먼저 게이트를 통과해야 한다.

각 PoC는 수행 전에 다음을 정한다.

- 구분하려는 가정과 발동시킨 위험 신호
- 성공·실패·불명확의 판정 기준
- 각 결과가 승인하거나 폐기할 후속 선택지

결과가 어느 쪽이어도 다음 선택을 줄이지 못한다면 질문을 더 좁힌다.

## 2. 게이트 층

| 게이트 | 발동 조건 | 판정 질문 |
|---|---|---|
| 가시성 | 목표 리비전·렌더 경로에서 한글 픽셀 도달이 동등한 런타임 증거로 확인되지 않음 | 한글 글리프가 게임의 실제 데이터 경로를 지나 화면까지 도달하는가 |
| 대표 E2E | 대표 텍스트의 추출→변환→재삽입→소비 경로가 아직 검증되지 않았고 규모 확대가 되돌리기 비쌈 | 실제 빌드 파이프라인의 한 단위가 끝까지 보존되는가 |
| 조건부 | 특정 위험 신호가 있고 정적 증거만으로 통과·기각할 수 없음 | 그 위험이 선택한 설계를 뒤집는가 |

한 게이트의 통과를 다른 게이트로 확대하지 않는다. 가시성은 글리프 예산·인코딩 공간·재배치·압축을 자동으로 증명하지 않고, 대표 E2E도 검증하지 않은 다른 렌더 경로나 자산 종류를 증명하지 않는다.

## 3. 가시성 게이트

표적은 가장 빨리 보이는 화면이 아니라 한글화의 주 대상 렌더 경로를 대표하는 화면으로 고른다. 대상 경로가 여러 개인지 불명확하면 경로별 소비자를 먼저 분리한다. 게임 데이터를 통과하지 않는 스크린샷 합성, 외부 자막이나 실행 환경의 오버레이는 가시성 증거가 아니다.

가시성 게이트는 다음을 모두 만족해야 통과한다.

1. 목표 한글 글리프가 의도한 위치와 자형으로 표시된다.
2. 해당 글리프가 대상 리비전의 실제 저장·적재·선택·표시 경로를 통과했음이 연결된다.
3. 같은 화면의 주변 텍스트·UI와 진입·이탈 동작에 새 회귀가 없다.
4. 증명한 경로와 증명하지 않은 범위가 기록된다.

같은 리비전·같은 렌더 경로에서 위 조건을 충족하는 기존 런타임 증거가 있으면 반복하지 않는다.

## 4. 대표 텍스트 1단위 E2E 게이트

대표 단위는 대사 블록, 메뉴 화면, 페이지처럼 실제 빌드가 처리하는 최소 단위다. 길이·제어 토큰·포인터·압축·패딩 중 목표 범위에 실제로 존재하는 제약을 포함해야 하며, 그 제약을 우회한 짧은 문자열은 대표 증거가 아니다.

해당 변환과 제약이 존재하는 범위에서 다음을 확인한다.

1. 원문 추출과 무수정 라운드트립 또는 선언한 의미 동등성
2. 문자·토큰의 보존·재계산 정책과 글리프 대응
3. 길이·경계·참조·컨테이너 제약을 지킨 재삽입
4. 빌드 산출물의 실제 적재와 소비
5. 대표 경로의 표시와 주변 동작 회귀 부재

같은 경로와 조건을 위 기준으로 이미 검증했다면 기존 증거를 재사용한다. 통과 뒤 본 구현으로 승격하는 것은 임시 실험물이 아니라 확정된 규칙, 판정 근거와 재현 가능한 검증이다.

## 5. 조건부 게이트

조건부 게이트는 아래 위험이 실제로 관측되고 동등한 증거가 없을 때만 켠다.

| 위험 | 발동 조건 | 통과 기준 |
|---|---|---|
| 복수 소비 경로 | 목표 화면들이 서로 다른 글리프·텍스처·렌더러를 쓸 가능성이 있음 | 배포 범위의 각 화면이 어느 소비 경로와 제약에 속하는지 확인됨 |
| 인코딩·제어 충돌 | 새 코드 영역이나 문자 폭이 기존 파서·제어 토큰과 충돌할 수 있음 | 목표 코퍼스가 모호성 없이 디코드되고 비텍스트·제어 경로를 침범하지 않음 |
| 저장·활성 자산 예산 | 글리프·텍스처의 저장 공간이나 동시 메모리·VRAM 공급이 수요에 근접함 | 선언한 배포 수요가 공급과 자산 수명 요건 안에서 충돌 없이 소비됨 |
| 길이·재배치 | 번역 성장으로 경계·참조·정렬·후속 데이터가 이동할 수 있음 | 대표 변경이 모든 관련 참조와 경계를 보존한 채 적재·소비됨 |
| 압축·컨테이너 | 수정 자산이 압축·패킹돼 있고 크기나 표현이 달라질 수 있음 | 무수정 동등성과 수정 후 컨테이너 유효성·게임 소비 호환성이 모두 성립함 |
| 그래픽 텍스트 | 배포 범위의 문자가 그래픽 픽셀로 저장됨 | `references/strategy/graphics-text.md`의 완료 조건이 대표 경로에서 성립함 |
| 레이아웃·시간 | 표시 영역, 페이지 진행, 음성·이벤트 동기가 번역 길이에 의존함 | 대표 장면이 선언한 공간·진행·동기 제약을 지킴 |
| 사용자 문자열 | 사용자가 만든 문자열이 저장되고 다른 화면에서 다시 소비됨 | 입력 가능 집합, 저장 표현, 재표시와 길이 조건이 배포 범위에서 함께 동작함 |

위험 신호가 없거나 계산·정적 근거로 판정 가능하면 게이트를 만들지 않고 그 근거를 기록한다. 런타임 자산의 추가·성장·이동·재패킹이나 소비 규약 변경이 포함되면 `references/strategy/runtime-assets.md`에서 아직 입증되지 않은 연결을 해당 게이트의 통과 기준으로 삼는다.

## 6. 판정과 산출물

게이트 결과는 **통과 / 실패 / 불명확**으로 판정한다.

- 통과는 선언한 범위의 후속 구현만 승인한다.
- 실패는 깨진 전제를 폐기하거나 설계를 바꿀 때까지 의존 작업을 막는다.
- 불명확은 증거가 부족한 상태이며 통과로 간주하지 않는다. 판정을 가르는 더 작은 질문으로 분해한다.

기록에는 발동 위험, 표적의 대표성, 사전 기준, 실제 증거, 증명·비증명 범위, 기각한 가정과 다음 선택을 연결한다. 위험 신호가 있었지만 정적 근거나 동등한 기존 증거로 게이트를 생략했다면 그 판정 근거를 남긴다. 기록이 구분할 의미와 검증 규칙은 `references/conventions/project-records.md` §4가 소유한다.
