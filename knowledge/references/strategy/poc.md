# PoC 전략

## 1. 역할과 철학

PoC(proof of concept)는 본 구현 전에 가장 비싼 실패를 작게 확인하는 장치다. 한글 패치에서 가장 먼저 닫아야 할 질문은 "이 게임의 실제 데이터 경로를 통해 한글이 화면에 도달하는가"다. 그러나 이 질문은 하나의 실험으로 항상 닫히지 않는다. 글리프 포맷, 문자 코드, 렌더 경로, 저장 공간, 활성 VRAM, 포인터 재배치, 그래픽 텍스트, 이름 입력 같은 위험은 서로 독립적으로 실패할 수 있다.

PoC의 방향은 닫기 쉬운 위험을 골라 증명하는 "저위험한 길 찾기"가 아니다. 배포 가능한 한글 패치(0원칙 충족, SKILL.md 핵심 불변식)라는 반드시 도달해야 할 목표가 있고, PoC는 그 목표를 가장 크게 위협하는 블로커가 풀릴 수 있는지를 먼저 직면하는 탐사다. "가장 비싼 실패를 작게 확인한다"는 가장 결정적인 블로커를 먼저 친다는 뜻이지, 증명하기 싼 위험으로 그 블로커를 대체한다는 뜻이 아니다.

따라서 PoC의 기본 형태는 **게이트 묶음**이다. 모든 게이트를 항상 수행하지 않는다. 먼저 정적 조사와 계산으로 위험을 낮추고, 남은 불확실성이 본 구현을 뒤집을 만큼 크거나 되돌리기 비싼 작업의 전제일 때만 해당 PoC를 수행한다. 이 문서의 목적은 "어떤 PoC가 있는가"보다 **언제 어떤 PoC를 켜야 하는가**를 정하는 것이다.

PoC의 가치는 가능성을 보여 주는 데서 끝나지 않고, 결과가 후속 선택지를 줄이는 데 있다. 성공하면 본 구현으로 승격할 지식이 무엇인지, 실패하면 폐기할 가정이 무엇인지, 애매하면 어떤 더 작은 확인으로 쪼갤지를 실험 전에 의식한다. 결과가 어느 쪽으로 나와도 후속 판단이 나아지지 않는다면, PoC를 만들기 전에 질문을 다시 좁힌다.

선행 사례는 위험 신호의 목록이지 새 게임의 규칙이 아니다. 사례에서 겪은 문제가 다른 게임에서도 반복될 수는 있지만, 존재 자체는 대상 게임에서 다시 확인해야 한다. 반대로 사례에 없던 문제가 새 게임에서 나올 수도 있다. 문서는 사례를 모르는 독자도 판단할 수 있게 질문, 신호, 통과 기준을 완결된 형태로 적는다.

## 2. 최소 게이트와 조건부 게이트

PoC는 두 층으로 나눈다.

| 층 | 수행 조건 | 목적 |
|----|----------|------|
| **최소 가시성 게이트** | 거의 모든 프로젝트에서 수행 | 한글 글리프가 실제 게임 데이터 경로를 지나 화면에 보일 수 있음을 확인 |
| **조건부 게이트** | 위험 신호가 있을 때만 수행 | 경로 분기, 인코딩, 수용량, 동적 적재, 재배치, 그래픽 텍스트, 이름 입력 같은 특정 위험을 닫음 |

최소 가시성 게이트는 흔히 "한글 1자 출력"으로 구현된다. 하지만 이 이름을 과대해석하면 안 된다. 기존 글리프 하나를 한글로 바꿔 보이는 실험은 글리프 포맷과 일부 렌더 경로를 확인할 수 있지만, 새 한글 코드 영역이 안전한지, 수백 자 글리프가 들어갈지, 대표 번역문이 포인터와 압축을 거쳐 다시 들어갈지는 증명하지 않는다.

조건부 게이트는 필요한 만큼만 켠다. 위험 신호가 계산과 문서 조사로 충분히 해소되면 PoC를 만들지 않는다. 반대로 한 게이트가 실패하면 전체 조사를 다시 시작하지 말고, 실패한 전제 하나만 바꿔 다시 확인한다.

## 3. 최소 가시성 게이트

### 3.1 검증 질문

최소 가시성 게이트가 닫아야 할 질문은 다음 하나다.

> 대상 게임의 실제 데이터 경로를 통해 한글 글리프가 화면 픽셀까지 도달하는가?

여기서 "실제 데이터 경로"란 원본 게임이 읽는 데이터, 원본 또는 패치된 텍스트 엔진, 원본 또는 패치된 글리프 공급 경로, 원본 영상 출력 경로를 말한다. 스크린샷 합성, 에뮬레이터 오버레이, 외부 자막처럼 게임 데이터 경로를 통과하지 않는 출력은 PoC가 아니다.

### 3.2 표적 선택

한글 1자는 "가장 빨리 눈에 보이는 곳"보다 **가장 싸게 재현 가능하고 목표 경로를 대표하는 곳**에서 띄운다. 여기서 "싸게"는 대표성에 종속된 수단이다. 재현이 싸다는 이유로 목표 경로의 블로커를 비껴가는 표적을 고르면, 그 PoC는 위험을 닫은 것이 아니라 미룬 것이다. 타이틀 화면의 글자가 본문 대화와 다른 폰트·텍스처·스프라이트 경로를 쓰면, 타이틀 PoC는 타이틀 경로만 증명한다. 반대로 초반 대사 한 줄이 본문 엔진의 공용 경로를 통과한다면 그쪽이 더 좋은 표적이다.

표적 선택 기준은 두 층이다. 1차 기준으로 후보를 좁힌다.

1. 한글화의 주 대상 경로와 같은 렌더러를 쓰는가 — 목표 경로의 블로커를 실제로 지나는가.
2. 실패했을 때 글리프 포맷, 코드 매핑, 렌더 경로 중 어디서 실패했는지 관찰할 수 있는가.

1차 기준을 만족하는 후보가 여럿일 때만 다음을 동점 처리 기준으로 쓴다. 재현 난이도나 패치 크기는 표적을 고르는 1차 기준이 아니다.

3. 원본에서 반복 재현하기 쉬운가.
4. 패치 범위가 작고 되돌리기 쉬운가.

대상 경로가 여러 개인지 모르면 먼저 5절의 경로 판별 게이트를 수행한다.

### 3.3 구현 패턴

낮은 비용부터 검토하되, 목표 경로의 블로커를 실제로 지나는 패턴 안에서 가장 싼 것을 고른다.

| 패턴 | 적용 조건 | 증명하는 것 | 증명하지 않는 것 |
|------|----------|-------------|------------------|
| 기존 글리프 1자 교체 | 게임 안에 폰트 데이터가 있고 셀 크기가 한글 표현에 충분 | 글리프 위치·포맷, 해당 렌더 경로의 픽셀 도달 | 새 한글 인코딩 공간, 전체 글리프 수용량 |
| 기존 텍스트 바이트를 교체 글리프 코드로 변경 | 텍스트 바이트 위치와 기존 코드 매핑을 알고 있음 | 실제 텍스트 데이터가 해당 글리프를 참조해 표시됨 | 새 prefix·2바이트 코드 안전성 |
| 글리프 fetch 훅 | BIOS/시스템 폰트 또는 내장 fetch 루틴을 경유 | 원본 fetch 이후 변환·전송 경로 재사용 가능성 | 전체 코드 영역, 전체 글리프 저장 공간 |
| 자체 디스패치 마이크로 PoC | 기존 코드 체계와 한글 코드가 양립하지 않음 | 새 분기와 최소 렌더 경로가 동작함 | 본문 전체 엔진 안정성 |

자체 디스패치는 비용이 본 구현에 가까우므로 더 작게 쪼갠다. 먼저 임의 글리프를 목표 타일·버퍼에 올리는지 확인하고, 그다음 실제 텍스트 바이트나 제어 흐름에서 호출되는지 확인한다.

### 3.4 통과 기준

최소 가시성 게이트는 다음을 모두 만족해야 통과다.

1. 목표 한글 글자가 의도한 위치에 올바른 자형으로 보인다.
2. 같은 화면의 주변 텍스트·UI·그래픽이 새로 깨지지 않는다.
3. 부팅, 장면 도달, 해당 화면 이탈이 멈춤 없이 동작한다.
4. 무엇을 증명했고 무엇을 증명하지 않았는지 문서에 남긴다.

화면에 글자가 보였다는 사실만으로 글리프 예산, 인코딩 예산, 재배치 가능성을 통과 처리하지 않는다.

## 4. 대표 텍스트 1단위 E2E 게이트

최소 가시성 게이트가 통과해도 바로 전량 번역에 들어가지 않는다. 본 구현 전에 대표 텍스트 1단위를 추출부터 재삽입까지 한 번 통과시킨다. 이 게이트는 대개 수행하는 것이 좋다. 전량 번역과 전체 엔진 패치가 되돌리기 비싼 작업이기 때문이다.

대표 단위는 대사 1블록, 메뉴 1화면, 튜토리얼 1페이지처럼 실제 빌드 파이프라인이 처리할 최소 단위다. 표적은 너무 쉬운 짧은 문자열 하나가 아니라 제어코드, 줄바꿈, 포인터, 압축, 패딩 중 최소 하나가 포함된 현실적인 항목을 고른다.

검증 범위:

1. 원문 추출과 무수정 라운드트립.
2. 번역문에서 고유 한글 음절 수집.
3. 음절→코드→글리프 슬롯 테이블 생성.
4. 텍스트 인코딩과 제어코드 보존.
5. 길이 보존, 재배치, 포인터 보정, 재압축 중 해당 단위에 필요한 처리.
6. 빌드 산출물의 화면 검증.

통과하면 본 구현에 승격되는 것은 임시 PoC 코드가 아니라 지식과 파이프라인 규약이다. 수동 헥스 편집, 임시 글리프 파일, 하드코딩 오프셋은 본 구현의 입력이 될 수 없다.

## 5. 조건부 PoC 판정표

조건부 PoC는 다음 표로 판단한다.

| 게이트 | 위험 신호 | 가장 싼 1차 확인 | PoC를 수행하는 시점 | 통과 기준 |
|--------|----------|------------------|---------------------|----------|
| 경로 판별 | 본문·메뉴·전투·이름 입력·팝업이 다른 폰트나 레이어를 쓰는 정황 | 서로 다른 화면에서 다른 글리프 마커 또는 레이어·VRAM 관찰 | 목표 경로를 고르기 전 | 각 화면이 어떤 폰트·텍스처·루틴을 쓰는지 분리됨 |
| 인코딩 운반 | 새 prefix, 2바이트 코드, 사용자 영역을 쓰려 함 | 전 코퍼스 byte 분포와 제어코드 분기 확인 | 기존 코드 교체를 넘어 새 코드 영역을 쓰기 전 | 새 코드가 텍스트로 디코드되고 제어코드·비텍스트 데이터와 충돌하지 않음 |
| 저장 글리프 예산 | 표본·초벌 번역의 고유 음절 수가 슬롯·공간 상한에 접근 | 병목 공식 계산, 표본·초벌 번역 기반 예상 수요 집계 | 전체 폰트 생성 또는 전량 번역 착수 전 | 표본 수요의 초과 위험이 낮다고 판단되거나, 폰트 전략에서 다룰 확장·회수·축소 후보가 정해짐 |
| 활성 메모리·VRAM 예산 | 전 글리프 상주를 전제로 두면 슬롯·VRAM 초과 조짐이 있거나 화면별 캐시가 필요해 보임 | 한 화면 고유 음절 수와 사용 가능한 슬롯 산정 | 동적 로딩·LRU·페이지 캐시를 구현하기 전 | 대표 화면에서 필요한 글리프가 충돌 없이 표시됨 |
| 슬롯 회수·예약 확인 | 빈 슬롯처럼 보이는 영역을 한글 글리프로 재사용하려 함 | 타일맵/OAM/레이어 사용 현황과 참조 검색 | 회수 후보를 실제 빌드 입력으로 넣기 전 | 회수 슬롯이 목표 화면과 관련 화면에서 비글리프 자원을 덮지 않음 |
| 재배치·길이 | 대표 번역·표본이 고정 슬롯을 반복해서 넘거나 포인터·내부 점프가 보임 | 표본 엔트리 길이 분포, 포인터 카탈로그, tail data 확인 | 전량 재삽입 정책을 정하기 전 | 대표 엔트리에서 늘어난 텍스트가 포인터·정렬·파일 크기 제약을 지키며 로드됨 |
| 압축·컨테이너 | 폰트·스크립트·그래픽이 압축되어 있고 수정 후 크기 변화가 예상됨 | 해제·무수정 재압축 라운드트립 | 압축 자산을 패치 입력으로 삼기 전 | 대표 수정본에서 라운드트립과 재압축·공간 처리 경로가 성립함 |
| 그래픽 텍스트 | 텍스트가 폰트가 아니라 이미지에 구워져 있음 | 후보 이미지 디코딩, 레이어 토글, VRAM 소스 역추적 | 해당 화면을 한글화 범위에 포함하기 전 | 클린 배경, 한글 조판, 재인코딩 후 대표 목표 화면이 정상 표시됨 |
| 레이아웃·타이밍 | 고정 창 크기, 자동 진행, 음성 동기, 프레임당 글리프 소비가 보임 | 표시 폭·줄 수·진행 조건 계산 | 번역 길이 정책을 확정하기 전 | 대표 장면에서 글자 넘침·잔상·대기 시간 문제가 없음 |
| 이름 입력·사용자 문자열 | 플레이어가 이름·점수·저장명을 입력하고 이후 화면에 재사용함 | 라벨 번역인지 실제 문자 입력인지 구분 | 한글 입력 지원을 목표 범위에 넣기 전 | 대표 입력 경로에서 입력 가능한 문자, 저장 인코딩, 재표시 경로, 길이 제한이 함께 동작함 |

표의 게이트는 체크리스트가 아니라 분기표다. 위험 신호가 없으면 하지 않는다. 위험 신호가 있어도 정적 분석과 계산으로 충분히 닫히면 PoC 대신 그 근거를 문서화한다.

### 이름 입력 게이트의 후속 라우팅

이름 입력은 세 경우로 나눈다.

1. **라벨만 번역**: 화면에 "이름" 같은 고정 라벨만 있고 실제 입력 문자는 원문 그대로 두는 범위라면 일반 텍스트 또는 그래픽 텍스트 작업으로 처리한다.
2. **기본 이름만 번역**: 기본 주인공명·저장명처럼 게임이 미리 가진 문자열만 바꾸고 사용자가 새 한글을 입력하지 않는다면 고정 슬롯 검증과 재표시 경로 검증으로 닫는다.
3. **실제 한글 입력 지원**: 사용자가 한글을 선택·입력하고 저장한 뒤 다른 화면에서 다시 표시해야 한다면 별도 사용자 문자열 사양을 만든다.

실제 한글 입력 지원은 한 문서의 책임으로 끝나지 않는다. `font-strategy.md`는 입력 가능한 문자 집합, 글리프 수요, 코드 배정을 정한다. `reinsertion.md`는 입력 UI 훅, 저장 인코딩, 변수 삽입·재표시 경로를 다룬다. `build-and-verify.md`는 최장 입력, 저장·로드, 이름이 등장하는 모든 화면의 오버플로우와 깨짐을 회귀 항목으로 묶는다. 한글 입력을 범위 밖으로 둘 수도 있지만, 그 경우에도 "기본 이름만 번역"인지 "원문 입력 UI 유지"인지 범위 결정을 문서화한다.

## 6. 실패 신호와 대응

PoC 실패는 보통 구현을 더 밀어붙이라는 신호가 아니라 전제를 바꾸라는 신호다.

### 글리프가 깨져 보인다

글리프 포맷, 비트플레인 배열, 행 순서, 엔디언, 압축 해제가 틀렸을 가능성이 높다. 같은 글리프 1자 분량을 여러 후보 디코더로 이미지화해 눈으로 확인하고, 화면에 주입한 데이터가 기대 바이트와 일치하는지 비교한다.

### 글리프는 메모리에 있는데 화면에 없다

VRAM이나 폰트 버퍼까지 도달했다는 사실만으로 통과가 아니다. 타일맵, OAM, 텍스처 좌표, 팔레트, 레이어 우선순위를 확인한다. 글리프 데이터 문제와 참조 인덱스 문제를 분리한다.

### 한 경로는 통과했지만 다른 화면이 깨진다

렌더 경로가 복수일 수 있다. 통과한 경로의 결론을 전체 게임에 확장하지 말고 경로 판별 게이트를 켠다. 목표 범위를 본문, 메뉴, 전투, 팝업, 그래픽 텍스트, 이름 입력 등으로 나눈다.

### 예상 수요가 글리프 예산을 위협한다

PoC 단계에서는 전체 수요를 확정했다고 보지 않는다. 표본·초벌 번역에서 고유 음절 수가 예산에 가까우면 `font-strategy.md`의 글리프 예산과 서브셋 산정으로 돌아가 공급 재산정, 슬롯 회수, 저장·활성 슬롯 확장, 동적 적재 가능성을 검토한다. 실제 번역 수요가 빌드에서 공급을 넘는 경우는 폰트 전략의 빌드 게이트와 수요 조정 절에서 미할당 0이 될 때까지 처리한다. PoC 산출물에는 확정 판정이 아니라 위험 신호, 사용한 표본, 가장 싼 다음 확인을 남긴다.

### 길이를 늘리면 오버리드·멈춤·다음 엔트리 침범이 난다

고정 슬롯 모델을 의심한다. 포인터 테이블, 엔트리 경계, 종료 코드, trailing 제어코드, tail data를 확인한다. 대사 본문처럼 품질이 중요한 텍스트는 단축을 기본 해결책으로 두지 말고 재배치와 포인터 보정을 먼저 검토한다. 단, 짧은 메뉴·이름·UI 슬롯처럼 고정폭 자체가 의미인 항목은 길이 보존이 맞을 수 있다.

## 7. 산출물

PoC가 끝나면 다음을 남긴다.

1. 수행한 게이트와 수행하지 않은 게이트의 이유.
2. 표적 화면, 표적 텍스트, 표적 글리프 또는 코드의 근거.
3. 증명한 범위와 증명하지 않은 범위.
4. 실패한 가정과 바꾼 설계.
5. 결과가 열거나 닫은 후속 선택지: 바로 본 구현으로 갈 수 있는 것, 폰트 전략·재삽입·압축 등으로 되돌릴 것, 더 작은 확인으로 쪼갤 것.
6. 본 구현으로 승격할 지식: 글리프 포맷, 인코딩 영역, 훅 좌표, 포인터 규약, 수용량 상한, 빌드 게이트.
7. 본 구현으로 승격하지 않을 임시 산출물: 수동 편집 파일, 실험용 오프셋, 단발성 스크립트.

이 산출물이 있어야 다음 사람이 이전 여정을 몰라도 현재 설계를 이해할 수 있다. 문서는 과거 작업의 일지가 아니라 현재 적용할 판단 기준이어야 한다.
