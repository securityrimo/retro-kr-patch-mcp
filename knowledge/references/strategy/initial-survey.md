# 초기 조사 전략

한글 패치 프로젝트의 첫 단계인 초기 조사를 어떻게 설계하고 수행하는지 정리한다. 플랫폼을 불문하고 적용되는 단계 전략이며, 플랫폼 고유 세부는 `references/platforms/` 아래 각 문서로 안내한다.

---

## 1. 초기 조사의 목표

초기 조사의 목표는 게임 전체를 역공학하는 것이 아니다. **본 구현 전에 닫아야 할 PoC 게이트가 무엇인지 판정하고, 최소 가시성 PoC를 설계할 만큼의 지식을 확보하는 것**이다. 구체적으로 다음 여섯 질문에 답할 수 있으면 초기 조사는 끝난다.

1. **매체**: 게임 데이터를 꺼내고, 수정하고, 되넣어서 다시 구동할 수 있는가?
2. **텍스트 위치**: 본문 텍스트가 어디에(파일·오프셋), 어떤 인코딩으로 들어 있는가?
3. **폰트 경로**: 글리프 데이터가 어디 있고(ROM 내장 / BIOS / 파일), 어떤 비트 포맷이며, 어떤 루틴이 그것을 읽어 그리는가?
4. **훅 지점**: 텍스트 렌더 경로에 한글 분기를 끼워 넣을 단일 지점 또는 최소 훅 집합이 존재하는가?
5. **수용량**: 한글 글리프(수백~수천 자)와 늘어난 텍스트를 담을 공간이 어디에 얼마나 있는가?
6. **베이크드 텍스트**: 텍스트가 폰트 렌더 경로로 그려지는가, 그래픽에 구워져(베이크드) 있는가. 그 비율도 함께 본다. 이미지에 구워진 텍스트는 폰트 훅이 아니라 그래픽 교체 파이프라인(`references/strategy/graphics-text.md`)으로 분기한다.

이 여섯에 답하는 데 불필요한 역공학(전투 로직, 사운드, 그래픽 압축 전반)은 초기 조사에서 의도적으로 제외한다. 조사 범위를 좁게 유지하는 것이 초기 조사의 핵심 규율이다. 단, 텍스트나 폰트가 압축돼 있다면 그 압축만은 범위에 들어온다. 풀지 못하면 2번·3번에 답할 수 없기 때문이다.

PoC의 정의도 여기서 고정한다. 기본 게이트는 **게임의 실제 데이터 경로를 통해 한글 글리프를 에뮬레이터 화면에 표시하는 최소 개입**이다. 원본 폰트 경로에 훅 하나를 거는 방식이 가장 작은 경우가 많지만, 그것만이 PoC는 아니다. 원본 글리프 1개를 한글 비트맵으로 교체해도 렌더 경로와 포맷을 검증할 수 있고, 원본 경로가 BIOS/고정 폰트라 직접 글리프를 공급할 수 없으면 최소 자체 디스패치가 필요할 수 있다. 단, 이 기본 게이트가 새 인코딩 공간, 전체 글리프 수용량, 재배치 가능성, 이름 입력 같은 주변 경로까지 증명하지는 않는다. 그런 항목은 위험 신호가 있을 때만 조건부 PoC로 분리한다. 스크린샷 합성이나 에뮬레이터 오버레이처럼 게임 데이터 경로를 지나지 않는 출력은 PoC가 아니다. 경로별 판정 기준은 `references/strategy/poc.md`를 따른다.

## 2. 조사 순서

조사는 바깥에서 안쪽으로 진행한다: **매체 → 파일시스템 → 코드 → 폰트 → 텍스트 엔진.** 각 단계는 다음 단계의 전제이며, 단계마다 산출물을 남겨야 뒤 단계와 후속 작업이 그 위에 쌓인다.

### 2.1 매체

이미지 포맷을 식별하고 무결성을 고정한다.

- 컨테이너 식별: 카트리지 ROM(헤더·매퍼·뱅킹), CD(BIN/CUE, GDI 등 트랙 구조), 플로피(D88, HDM 등).
- 해시(CRC32/MD5)로 원본이 알려진 보존 덤프와 일치하는지 확인한다. 출처 불명 덤프 위에서 분석을 시작하면 이후 모든 디프가 오염된다.
- **round-trip 검증**: 추출 → 무수정 재조립 → 원본과 바이트 일치를 가장 먼저 확인한다. 되넣지 못하는 추출은 분석용으로만 가치가 있고 패치 제작으로는 이어지지 않는다.
- CD 매체는 어느 트랙이 데이터이고 어느 트랙이 오디오인지 구분하고, 수정 대상을 데이터 트랙으로 한정할 수 있는지 본다. 섹터 단위 오류 정정(EDC/ECC) 재계산이 필요한 포맷인지도 이 단계에서 확정한다.
- **실기 구동 경로**: 이 플랫폼에서 패치본을 실기로 돌릴 경로와 그 제약(용량·매퍼·매체·부트 조건)을 초기에 확인한다. 실기 경로가 빌드 포맷 선택에 역영향을 주기 때문이다. 예를 들어 드림캐스트는 무손실 실기 구동을 위해 GDI 기준으로 빌드해야 하고, 메가드라이브는 4MB 초과 시 ROM 헤더를 SSF로 바꿔야 매퍼가 활성화된다. 이를 늦게 알면 빌드를 다시 짜는 비싼 대가를 치른다. 세부는 해당 플랫폼 문서를 본다.

산출물: 매체 메타 기록(트랙/헤더 구조, 해시), 추출·재조립 도구 또는 절차, 실기 구동 경로 메모(수단·용량/매퍼/매체/부트 제약·빌드 포맷에의 영향).

### 2.2 파일시스템

매체 안의 논리 구조를 파악한다.

- 표준 파일시스템(ISO9660, FAT 계열)인지, 게임 자체 로더가 섹터 위치를 하드코딩하는지 식별한다. 자체 로더라면 파일 크기·위치를 바꾸는 순간 로더도 패치 대상이 된다. 이 사실을 일찍 알수록 좋다.
- 파일(또는 데이터 블록) 목록을 만들고 각각의 역할을 가설로 붙인다: 실행 코드, 스크립트, 폰트, 그래픽, 사운드. 크기·엔트로피(압축 여부)·반복 패턴이 1차 단서다. 압축이 의심되면 식별·해제·재압축 절차는 `references/strategy/compression.md`를 따른다.

산출물: 파일/블록 목록과 역할 가설표.

### 2.3 코드

실행 코드를 읽을 수 있는 상태를 만든다.

- 엔트리 포인트(리셋 벡터, 부트 헤더)와 대상 ISA를 확정한다.
- 디스어셈블 환경을 구축한다(§5). 뱅킹·페이징이 있는 플랫폼이라면 "파일 오프셋 ↔ CPU 주소" 변환 규칙을 이 단계에서 함수 하나로 고정하고 전 도구가 그것만 쓰게 한다. 주소 변환 실수는 초기 조사에서 가장 흔하고 가장 오래 사람을 괴롭히는 오류다.
- 메모리 맵 초안을 만든다: ROM/RAM/VRAM 영역, 뱅크 매핑, BIOS·시스템 콜 영역.

산출물: 동작하는 디스어셈블 도구, 메모리 맵 초안.

### 2.4 폰트

글리프 데이터와 그 소비처를 찾는다.

- 폰트 위치 유형: ROM/파일 내장 시트, BIOS 폰트 호출(이 경우 게임 안에 글리프가 없다), 폰트 ROM 하드웨어. 어느 유형인지에 따라 한글 글리프 공급 전략이 완전히 달라진다.
- 타일/비트 포맷(1bpp·2bpp·4bpp, 플레인 구성, 크기 8×8/8×16/16×16)을 확정하고 시트 전체를 이미지로 덤프해 눈으로 확인한다. 폰트 덤프가 눈으로 읽히는 순간이 보통 초기 조사의 첫 가시적 성과다.
- 글리프 수와 문자 배열 순서를 기록한다. 이 순서가 곧 텍스트 인코딩 테이블의 단서가 된다.

산출물: 폰트 시트 이미지 덤프, 포맷 명세, 글리프-문자 대응 초안.

### 2.5 텍스트 엔진

텍스트가 저장에서 화면까지 가는 전 경로를 파악한다(접근법은 §3).

- 인코딩 식별: 표준(SJIS 등) / 폰트 시트 인덱스 직접 참조 / 자체 테이블. 1바이트·2바이트 혼용 여부와 2바이트 진입 프리픽스.
- 포인터 체계: 문장 시작 주소를 담은 테이블의 위치와 포인터의 기준(절대/상대/뱅크 내).
- 제어 코드(op-code) 목록: 줄바꿈, 페이지 전환, 선택지, 변수 삽입, 지연 등.
- 렌더 콜체인: 텍스트 바이트 fetch → 디코드 → 글리프 lookup → VRAM/프레임버퍼 쓰기. 이 체인 위에서 훅 지점을 고른다.

산출물: 대표 텍스트 블록의 구조화 덤프, 텍스트 엔진 분석 문서, 훅 후보 지점, 수행해야 할 PoC 게이트 목록. 전량 텍스트 덤프와 무손실 라운드트립 파서는 PoC 게이트 이후 본 추출 단계의 산출물로 확장한다.

순서를 지키는 이유는 의존성 때문이다. 파일을 꺼내지 못하면 코드를 못 읽고, 코드를 못 읽으면 폰트 루틴을 못 찾고, 폰트 루틴 없이는 텍스트 엔진의 끝단(렌더)을 특정할 수 없다. 거꾸로 텍스트 엔진부터 덤비면 매 단계에서 앞 단계 미비로 되돌아오게 된다.

## 3. 텍스트 엔진 역공학 접근법

네 가지 기법을 상황에 따라 조합한다. 대개 ①로 데이터를 잡고, ②·③으로 코드를 잡으며, ④로 나머지 의미를 채운다. ①~④가 개별 단서를 잡는 기법이라면, 그 단서들을 앵커에서 출발해 텍스트 엔진 전체 호출 그래프로 점진 종합하는 절차가 §3.5다. 정밀 한글화에서 "어디에 훅을 걸어야 모든 텍스트가 통과하는가"는 그 그래프에서 답이 나온다.

### 3.1 문자열 검색 → 포인터 역추적

게임 화면에서 확인한 문장을 후보 인코딩(SJIS 등)으로 바이너리에서 검색한다. 표준 인코딩이 아니면 상대 검색(문자 간 코드 간격이 가나·알파벳 순서와 같은 등차를 이루는 패턴 탐색)을 쓴다. 문자열 위치를 찾으면 **그 주소(또는 그 주소를 만들 수 있는 값)를 가리키는 바이트열을 다시 검색**해 포인터 테이블을 찾는다. 포인터 테이블이 잡히면 텍스트 블록의 전체 범위·항목 수·정렬 규칙이 한 번에 드러난다. 뱅킹 플랫폼에서는 포인터가 CPU 주소 기준임에 주의한다. 파일 오프셋 그대로 검색하면 영영 안 나온다.

### 3.2 디스어셈블 + 정적 트레이스

두 방향이 있다.

- **순방향**: 포인터 테이블을 읽는 코드를 찾아 fetch → 디스패치 → 렌더로 따라 내려간다. 제어 코드 분기(점프 테이블)가 나오면 그것이 op-code 디스패처다.
- **역방향**: 폰트 데이터 주소나 VRAM 쓰기 지점을 참조하는 코드에서 거슬러 올라간다. 렌더 끝단은 하드웨어 레지스터 접근이라 식별이 쉽다.

뱅킹·페이징 플랫폼에서는 디스어셈블만으로 부족하고, 뱅크 레지스터 상태를 정적으로 시뮬레이션하며 따라가야 분기 너머의 코드를 올바른 주소로 읽을 수 있다. 한 사례에서는 페이징 레지스터 정적 인터프리터를 디스어셈블러에 붙여 op-code 핸들러의 코드 위치를 빠짐없이 매핑했다. xref(상호 참조)와 콜그래프 생성을 도구에 넣어두면 정적 트레이스 효율이 크게 오른다.

### 3.3 에뮬레이터 브레이크포인트

디버거 내장 에뮬레이터에서 동적으로 확인한다.

- 텍스트 바이트가 있는 주소에 **read watch** → 걸리는 순간의 PC가 fetch 루틴이다.
- VRAM(또는 폰트 버퍼)에 **write watch** → 렌더 루틴이다.
- 콜스택을 거슬러 두 지점 사이의 체인을 채운다.

정적 분석으로 세운 가설을 동적으로 확정하는 용도로 특히 강하다. 간접 점프·런타임 디스패치처럼 정적으로 끊기는 지점도 동적으로는 한 번에 뚫린다. 어느 에뮬레이터가 디버깅에 적합한지는 플랫폼 문서를 참조한다.

### 3.4 op-code 빈도 통계로 의미 추정

제어 코드의 코드 위치는 찾았지만 의미를 모를 때, 또는 코드 분석 전에 데이터만으로 가설을 세울 때 쓴다. 전체 텍스트 코퍼스에서 각 코드의 출현 빈도·위치 분포·인접 패턴을 집계한다.

- 모든 문장 끝에 나오는 코드 → 종료/대기.
- 화자 이름 직후 → 개행 또는 창 전환.
- 항상 짝수 개 인자 바이트를 동반 → 파라미터 있는 코드.
- 선택지 화면 텍스트에만 등장 → 선택지 마커.

통계로 세운 의미 가설은 반드시 ②·③으로 핸들러 코드를 읽어 확정한다. 가설 단계의 라벨을 사실처럼 굳히면 번역 단계에서 제어 코드를 잘못 보존해 게임이 멈추는 사고로 이어진다.

### 3.5 콜그래프 점진 재구성

정밀한 한글화(원본 엔진에 한글 분기를 새로 까는 다중 훅)에서는 "어느 지점에 훅을 걸어야 모든 텍스트가 그곳을 통과하는가"를 알아야 하고, 그 답은 텍스트 엔진의 전체 제어 흐름 지도에서 나온다. 그러나 게임 전체를 디스어셈블하는 것은 §1의 범위 규율에 어긋난다. 대신 **확실히 아는 한 지점(앵커)에서 시작해 호출 그래프를 한 홉씩 넓혀, 텍스트 경로에 도달 가능한 부분만 닫는다.** §3.1~§3.4의 기법을 도구로 쓰는 반복 절차이며, 상용 역공학 도구의 콜그래프 뷰가 현재 함수 기준 1레벨만 보여 주고 한 레벨씩 펼치게 하는 것과 같은 작업 패턴이다. 자체 도구 환경에서는 §5의 xref·콜그래프 출력으로 동일하게 수행한다.

**앵커 선정.** 앵커는 정적 스캔으로 한 번에 찾을 수 있는 고정점을 고른다. 후보는 엔진의 양 끝단이다: 출력 끝단(VRAM/프레임버퍼 쓰기 레지스터, BIOS 폰트 호출, 폰트 데이터 read 참조)과 입력 끝단(포인터 테이블을 읽어 문자 바이트를 페치하는 지점, §3.1의 역추적 결과). 하드웨어·BIOS 종단점은 레지스터 주소·콜 번호가 고정이라 식별이 쉬워 역방향 앵커로 좋다. 데이터에서 코드로는 참조 역추적(그 주소·문자열을 참조하는 코드를 찾기)으로 건너간다. 어느 노드를 앵커로 삼을지는 fan-in/fan-out의 비대칭으로 판단한다. fan-in이 큰 노드는 공용 저수준 유틸(글리프 1자 출력, 전송 래퍼)이어서 모든 출력 경로가 수렴하므로 최적의 역방향 앵커이고, fan-out이 큰 노드는 상위 디스패처·파서다. 한 사례에서는 BIOS 폰트 호출(코드로 글리프 비트맵 포인터를 받는 지점)이 가장 강한 출력측 앵커였고, 다른 사례에서는 레이어 토글로 대화 텍스트가 실리는 배경면을 먼저 확정한 뒤 그 면에 패턴을 쓰는 루틴을 종단 앵커로 잡았다.

**fan-in / fan-out 한-홉 확장.** 앵커에서 양방향으로 한 단계씩 넓힌다. fan-in(역방향)은 이 함수·데이터를 누가 호출/참조하는가(xref, §5), fan-out(순방향)은 이 함수가 무엇을 호출하는가다. 새로 나온 노드 중 텍스트 경로에 관여하는 것만 미탐색 경계(frontier)에 넣고 경계가 빌 때까지 반복한다. 동적 역추적의 대표 절차는 앵커 주소(텍스트 RAM 사본, VRAM 타일, 폰트 버퍼)에 watch를 걸고(§3.3) 걸리는 순간의 출처 주소로 브레이크포인트를 한 단계씩 위로 옮기는 것이다. IO 포트·VRAM은 프레임마다 수백~수천 번 접근되므로 조건부 브레이크포인트(누산기가 추적 중인 글자 코드일 때만 정지)가 필요해지는 경우가 많다. VRAM 쓰기가 DMA 일괄 전송이거나 로드 후 갱신만이면 watch가 엉뚱한 타이밍에 걸리니, 앵커가 압축·DMA를 경유하면 그 디코더·전송 경로부터 따로 연다(`references/strategy/compression.md`). 6502 계열에서 간접 인덱스 주소지정(`LDA ($nn),Y`)이 보이면 텍스트·포인터 루틴일 가능성이 높다. 압축·DTE(2글자를 1바이트로 묶는 사전 압축) 경로 등 예외가 있으니 단서로만 쓴다.

호출자 수 자체가 함수의 성격을 가르는 신호다. 호출자가 적으면 그 전부를 개별 라우팅 분기로 해석하고, 많으면 범용 진입점으로 분류해 개별 추적을 접는다. 한 사례에서는 렌더링 루틴의 호출자가 정확히 4곳이어서 각각을 일반 문자·프리픽스 핸들러 분기로 해석했고, 같은 게임의 범용 슬롯 큐 루틴은 호출자가 70곳 안팎이어서 범용으로 분류했다. fan-in 전수 분석은 "이 함수가 찾는 경로인가"를 확정하는 데뿐 아니라 배제하는 데도 같은 무게로 쓰인다. 단, 호출자를 전부 열거했을 때만 배제가 확정된다. 다른 사례에서는 범용 스프라이트 드로우 루틴의 호출자 12곳을 전수 분석해 글리프의 동적 렌더링이 그 루틴을 거치지 않음을 확인하고 다른 경로로 옮겼다.

**함수 기능 가설·검증.** 새로 발견한 노드에 잠정 이름·기능 가설을 붙인다. 단서는 일관된다: BIOS·시스템 콜(플랫폼별 공개 콜 표가 있어 정체가 거의 확정된다), 하드웨어 레지스터 접근(영상 칩=그래픽, 사운드 칩=오디오, DMA 레지스터=전송), 인자 개수·전달 방식, 호출 빈도(프레임당 수십 회=렌더·입력 핫패스, 드물게=초기화·이벤트), 호출 컨텍스트(fan-in 노드의 성격에서 역추론). 같은 함수는 늘 같은 setup/teardown 관용구로 둘러싸여 호출되므로, 닮은 코드를 그 함수로 단정하기보다 호출 사이트들이 공통으로 무엇을 하는가로 정체를 역증명한다(`references/strategy/debugging.md` §6). 한 사례에서는 한 함수의 호출 사이트 17곳이 모두 직전에 같은 좌표 변환을 한다는 공통 패턴으로 그 함수를 확정했다. 단일 디스어셈블 인상보다 호출 패턴 전수가 강한 증거였다. 가설은 정적 재확인 또는 동적 브레이크포인트로 검증해 확정하고, 미확정은 가설 태그로 보존한다(op 중립 명명 `opNN`의 함수 레벨 확장). 검증 안 된 이름을 사실로 굳히지 않는다. 가설명은 분석을 오도한다(`references/strategy/debugging.md` §6). 한 사례에서는 한 핸들러 군을 "서브스크립트 호출자"로 명명하고 인자를 뱅크 ID로 해석했다가 페이징 시뮬레이션으로 그 인자가 BIOS 태스크 스케줄러 파라미터임이 드러나 명명을 정정했고, 다른 사례에서는 타일 포맷을 표준 인터리브 2bpp로 가정했다가 레이아웃 4종 × 디코더 3종을 실제로 렌더해 비교한 뒤에야 플레이너 2bpp + 256바이트 stride로 확정했다. 모든 정정이 더 강한 증거(동적 트레이스, 실렌더 비교)로 이뤄졌다. 같은 개발사·엔진 계보의 다른 게임에서 확정한 시그니처를 가져와 대조하는 것은 유효하지만, 일치는 사전 확률일 뿐이니 가설로 두고 자기 게임에서 실측해 확정한다(§4).

**수렴 판단 — "충분한 시점".** 정적 분석만으로는 간접 분기 때문에 그래프 완전성이 원리적으로 보장되지 않는다. 목표는 전체 ROM 그래프가 아니라 텍스트 경로에 도달 가능한 부분그래프를 닫는 것이다. 닫혔다고 볼 기준은 셋이다: 입력(포인터 테이블 읽기)부터 출력(VRAM/프레임버퍼 쓰기)까지 끊김 없이 연결되고, 훅 후보 지점의 미식별 호출자가 0이며, 제어코드 디스패처의 핸들러 경계가 전수 확정된다. 공통 논리는 "카운트가 닫혔다"는 것이다: 호출자를 다 셌거나, 핸들러를 다 매핑했거나, 입력에서 출력까지 한 줄로 이어졌을 때 멈춘다. 반대로 간접 참조라 정적으로 닫히지 않는 항목은 "동적 확인 필요"로 명시하고 미해결로 남긴다. 닫히지 않은 곳을 닫혔다고 적지 않는다. 한 사례에서는 단일 렌더러에 더해 그 호출자와 텍스트 위치 전진 지점을 전수로 세어 더 들어올 경로가 없음을 확인한 시점에 수렴했고, 다른 사례에서는 op 핸들러를 전부 매핑하고 한 번도 안 쓰인 op까지 식별해 경계를 닫았다.

**순서도 산출물과 디스패처 인식.** 재구성의 종착 목표는 대개 디스패처와 핸들러 테이블이다. 선행 사례의 텍스트 엔진은 바이트코드 인터프리터 형태가 많았고, 중앙 루틴이 op 1바이트를 fetch → 핸들러 테이블 인덱싱 → 분기하는 구조가 자주 보였다. 인식 시그니처는 fan-out이 비정상적으로 큰 단일 함수(호출/점프 그래프에서 후속이 가장 많은 노드가 디스패처, 그 직접 후속들이 핸들러) + fetch-decode-dispatch 루프 + 핸들러들의 공통 복귀다. 디스패처가 닫히면 제어코드 전모가 드러나 충분에 도달한다. 호출 그래프는 산출물로 종합한다(노드=함수·op 핸들러, 엣지=호출·디스패치). 실무에서 쓰인 형식은 핸들러 PC 매핑표(op → 핸들러 주소 + 인자 수·상태 저장 위치), ASCII 콜체인 다이어그램(fetch → decode → dispatch → render), 호출자 목록표(함수 → 호출 사이트 N곳 + 방식·조건), 데이터 흐름도(매체 → 디컴프 → 작업 RAM → VRAM, 주소 병기) 네 가지다. 한 사례의 콜체인이 순서도의 예다: 텍스트 렌더 디스패처 → 텍스트 엔진(타일 코드의 제어/글리프 분기) → 패턴 네임 라이터 + 제어코드 디스패치 테이블. 이 그래프가 훅 지점 선정(`references/strategy/reinsertion.md` §4)의 직접 입력이며, 그래프가 단일 엔진이면 디스패치 훅 1~2개로 끝나지만 UI가 별도 렌더러를 쓰는 다중 경로면 화면별 훅이 추가된다.

**정적·동적 결합과 레트로 특이사항.** 정적 xref로 뼈대를 잡고 간접 점프에서 끊기는 지점을 동적으로 메워 다시 정적 그래프에 합류시킨다(§3.3, §5). recursive descent로 골격을 잡다 막힌 간접 분기를 만나면 그 명령에 실행 브레이크포인트를 걸어 실제 행선지·레지스터·콜스택을 기록하고 recursive의 엔트리 목록에 추가한다. 밟은 경로만 진실이므로 전 대화창·전투·메뉴를 실제로 거치는 입력 커버리지가 동적 완전성을 좌우한다. 점프 테이블에서 끊기면 인덱스 값이 어디서 왔는지 역추적해 경계 검사(`CMP #n` 뒤의 기본 분기)를 찾고 테이블 베이스와 엔트리 크기를 구해 `[base, base+n)`를 전수 등록한다. 레트로 플랫폼에서 정적 콜그래프를 깨는 세 요인을 특히 경계한다:

- **뱅킹·페이징**: 같은 논리 주소가 시점마다 다른 물리 코드를 가리킨다. 분기 시점의 뱅크 레지스터 상태를 따라가야 행선지가 정해지므로 정적 디스어셈블러에 뱅크 상태 시뮬레이터를 붙인다. 한 사례에서는 페이징 레지스터(MPR) 상태를 명령마다 추적하는 인터프리터로 뱅크 경유 호출을 해소해 op 핸들러를 전부 매핑했다.
- **자기수정 코드**: 매체의 바이트가 실행 시점의 코드와 다를 수 있어 정적 기준이 어긋난다. 핸들러 주소를 테이블에서 조립해 코드 영역에 써 넣고 점프하는 패턴(저바이트·고바이트를 각각 로드·저장한 뒤 간접 점프)을 만나면 자기수정 후의 실제 명령을 동적으로 확인한다. 한 사례의 디스패처가 이 형태였다.
- **분할·평행 점프 테이블과 RTS 트릭**: 6502·65816 계열은 16비트 행선지를 저바이트 테이블과 고바이트 테이블로 나눠 두거나 행선지−1을 스택에 넣고 `RTS`로 점프한다. 어느 쪽도 분기 명령이 행선지를 직접 참조하지 않아 핸들러들이 콜그래프에서 고아가 되므로, 디스패처·`RTS` 지점에 브레이크포인트를 거는 동적 추적이 핵심 대응이 된다. 에뮬레이터의 코드/데이터 로거로 점프 목적지와 뱅크 상태를 실행 중에 수집하는 것이 대표 우회다.

세 요인이 겹치면 정적 콜그래프의 가치가 급락하고 에뮬레이터 트레이스 로거가 1차 도구가 된다. 뱅크 상태를 모른 채 "파일 오프셋 = 논리 주소"로 가정하면 포인터가 엉뚱한 뱅크를 가리켜 깨진다(`references/strategy/reinsertion.md`의 뱅크 경계 횡단).

## 4. 선행 패치 활용

대상 게임에 영문(또는 타 언어) 팬 패치가 존재하면 초기 조사를 크게 단축할 수 있다. 선행 패치는 "텍스트 엔진이 어디 있고 무엇을 고쳐야 하는지"에 대한 검증된 답안지다.

- **디프 분석**: 원본과 패치본을 바이트 디프하면 변경 클러스터가 나온다. 큰 클러스터는 재삽입된 텍스트·폰트이고, 작은 산발 클러스터가 엔진 훅이다. 훅 위치를 디스어셈블하면 그들이 찾아낸 렌더 경로를 그대로 얻는다.
- **문제 지도**: 선행 패치가 해결한 문제(가변폭 폰트 도입, 포인터 재배치, 공간 확장)는 한글 패치도 부딪힐 문제다. 그들의 해법 구조를 읽으면 설계 시간을 아낀다. 한 사례에서는 영문 패치의 가변폭 폰트 엔진을 완전 역공학한 뒤 그 규약(타일 할당, 상태 머신)에 호환되도록 한글 렌더러를 설계했다.
- **빌드 도구**: 선행 패치 제작자가 공개한 추출·빌드 도구가 있으면 초기 덤프 검증용으로 쓴다. 단 일회성 검증에 한정하고 파이프라인 의존은 피한다(§5). 자체 덤프와 선행 도구 덤프를 교차 검증하면 양쪽의 파싱 오류를 서로 잡아낼 수 있다.

주의할 점 세 가지. 첫째, 선행 도구의 라이선스(GPL 등)가 자기 코드의 배포 조건에 미치는 영향을 확인한다. 둘째, 선행 패치의 알려진 버그가 같은 기법을 쓰는 한글 패치에 그대로 상속될 수 있으니 버그 목록도 같이 조사한다. 셋째, 영문 패치의 전제(1바이트 문자, 반각 폭, 적은 글리프 수)는 한글에 맞지 않는 경우가 많다. 영문 패치가 "가능함"을 증명해도 그 구조를 그대로 쓸 수 있다는 뜻은 아니다. 한 사례에서는 영문 가변폭 엔진이 여러 문자를 타일 하나에 합성하는 반면 한글 고정폭은 1문자 1타일이어서, 같은 텍스트 창에서 영문 대비 2~3배의 타일을 소비해 창 경계를 넘는 문제가 새로 발생했다.

선행 패치가 없는 경우에도, 같은 개발사·같은 시기의 다른 게임 패치 사례나 같은 시리즈의 타 플랫폼판 분석이 텍스트 엔진 구조의 사전 확률(prior)을 제공한다. 같은 엔진 계보의 게임은 제어 코드 체계와 스크립트 구조가 닮는 경향이 있다.

## 5. 디스어셈블러 전략

대상 ISA의 디스어셈블 환경은 **기성 도구, 얇은 래퍼, 자체 구현 중에서 게임의 주소공간 복잡도에 맞춰 고른다.** 자체 구현은 강력하지만 기본값은 아니다. 다음 조건 중 둘 이상이 맞으면 자체 디스어셈블러 또는 기성 디코더 위의 전용 래퍼를 만든다.

- **플랫폼 결합이 필요함**: 범용 디스어셈블러가 뱅킹·페이징·주소 변환을 표현하지 못한다. 매퍼 시뮬레이션을 디코더나 래퍼에 붙여야 "파일 오프셋의 이 바이트가 CPU 주소 어디로 보이는가"를 도구가 일관되게 답한다.
- **파이프라인 통합 이득이 큼**: 추출 → 분석 → 패치 빌드가 한 코드베이스에 있으면 디스어셈블 결과(주소·라벨)를 패치 생성 코드가 직접 소비한다. 분석 도구와 빌드 도구가 분리되어 수동 전사 오류가 반복되면 통합 가치가 커진다.
- **인코더 겸용이 필요함**: 훅 ASM을 코드로 조립하고 라벨 해결까지 자동화해야 한다면 디코더/인코더 테이블을 공유하는 편이 낫다.
- **범위를 좁힐 수 있음**: 게임이 실제로 쓰는 명령어 부분집합만 정확하면 되는 상황이어야 한다. 전체 ISA 완비가 목표가 아니므로 미지 opcode는 즉시 에러로 드러내고 그때 추가한다.

반대로 플랫폼용 디버거·디스어셈블러가 주소 변환과 xref를 충분히 제공하고, 훅이 작아 수동 어셈블의 위험이 낮다면 기성 도구 결과를 검증 기록과 함께 쓰는 편이 낫다. 어느 경로를 택하든 주소 변환 함수는 프로젝트 안에서 하나로 고정하고, 분석 결과가 빌드 스크립트와 같은 기준을 쓰게 한다.

### 트레이스 모드: linear와 recursive

두 모드를 모두 갖추고 교차 사용한다.

- **Linear sweep**: 시작 주소부터 끝까지 순서대로 디코드. 코드 영역을 통째로 훑기에 좋지만, 코드 사이에 박힌 데이터(점프 테이블, 인라인 리터럴)를 명령어로 오해해 그 뒤가 전부 어긋난다.
- **Recursive descent**: 엔트리에서 분기·호출을 따라가며 도달 가능한 코드만 디코드. 데이터를 건너뛰어 정확하지만, 간접 점프(레지스터 경유, 점프 테이블)에서 추적이 끊겨 커버리지에 구멍이 난다.

실전 운용은 recursive로 뼈대를 잡고, 끊긴 구간을 linear로 훑어 후보를 찾고, 동적 분석(§3.3)으로 간접 점프의 실제 행선지를 채워 recursive의 엔트리 목록에 추가하는 반복이다. ISA 특성도 고려한다. 고정 길이 명령어 ISA는 linear의 오정렬 위험이 작고, 가변 길이 ISA나 실행 모드 플래그에 따라 명령어 길이가 달라지는 ISA는 recursive에 모드 상태 추적까지 얹어야 한다(해당 플랫폼 문서 참조).

디스어셈블러 위에 xref 수집과 콜그래프 출력을 얹으면 §3.2의 정적 트레이스가 도구 지원 작업이 된다. 함수 단위 호출 그래프는 "폰트 루틴을 누가 부르는가"를 한 번의 질의로 답해 준다.

## 6. 산출물

초기 조사가 끝나면 다음 세 가지가 남아야 한다. PoC 전에는 대표 표본, 확정 근거, 필요한 PoC 게이트 판정을 남기는 것이 목표이고, 전량 덤프·전량 라운드트립은 PoC 게이트 이후 본 추출 단계에서 확장한다. 이 산출물들이 이후 전 단계(엔진 개조, 번역, 빌드)의 입력이다.

초기 조사 산출물은 자료 모음이 아니라 후속 작업을 고르는 판정표다. 각 산출물에는 확인한 사실과 남은 가설을 구분하고, 그 결과 어떤 선택지가 강해졌는지 또는 약해졌는지를 드러낸다. 조사 결과가 PoC 표적, 폰트 전략, 추출 범위, 재삽입 위험 중 어느 것도 좁히지 못한다면 조사를 계속 넓히기보다 질문을 더 작게 다시 잡는다.

### 6.1 아키텍처 문서

플랫폼·게임의 구조 지식을 문서로 고정한다. 핵심 규율은 **가설과 사실의 구분 표기**다. 검증 전 추정은 가설로 표시하고, 디스어셈블·실측으로 확인되면 검증 근거(주소, 디스어셈블 결과, 실측값)와 함께 사실로 승격한다. 가설을 사실처럼 적어 두면 몇 주 뒤의 자신이 그 위에 잘못된 설계를 쌓는다. 분석 노트는 날짜순으로 따로 누적하고, 아키텍처 문서는 현재 시점의 결론만 유지한다.

정밀 한글화(다중 훅)를 계획한다면 텍스트 엔진의 **제어 흐름/호출 그래프**(§3.5)도 이 문서의 산출물에 포함한다. 노드는 함수·op 핸들러, 엣지는 호출·디스패치이며, 이 그래프가 훅 지점 선정의 직접 입력이 된다. 기존 폰트 경로에 훅 하나만 거는 최소 개입이면 단일 렌더러 경로만 확인해도 충분하므로 전체 그래프는 필요 없다.

### 6.2 텍스트 맵

텍스트의 물리적 구조를 기계가 읽을 수 있는 형태로 정리한다.

- 대표 텍스트 블록의 위치(파일·오프셋·크기)와 포인터 테이블 위치.
- 인코딩 테이블(바이트 ↔ 문자 대응).
- 제어 코드 목록과 각각의 의미·인자 구조(확정/추정 구분).
- 대표 표본의 구조화 덤프. 전량 구조화 덤프는 필요한 PoC 게이트가 닫힌 뒤 번역 작업의 직접 입력으로 만든다.

대표 표본 덤프도 round-trip 가능해야 한다. 덤프를 무수정 재조립했을 때 원본 바이트가 나와야 파서가 옳다는 증명이 된다.

### 6.3 실현 가능성 평가

go/no-go 판단 문서다. 다음을 정량적으로 적는다.

- **글리프 수용량**: 인코딩 여유 슬롯 수와 표본·초벌 번역 또는 장르별 추정에서 나온 예상 한글 음절 수요의 비교. 전 음절(현대 한글 11,172자 기준, 실무 기준선은 KS X 1001 2,350자)이 안 들어가면 사용 음절 부분집합 전략의 성립 여부.
- **공간 여유**: 늘어날 텍스트와 폰트 시트를 담을 빈 공간(미사용 뱅크, 확장 가능 영역, 추가 섹터)의 위치와 크기.
- **훅 가능성**: 렌더 경로의 훅 후보 지점과 필요한 개입 규모 추정.
- **위험 목록**: 항목별 위험도와 우회안(텍스트가 원문보다 길어질 때의 처리, 창 크기 제약, 압축 재구현 필요 여부 등).

최소 가시성 PoC와 필요한 조건부 게이트를 마치고 이 평가가 긍정이면 초기 조사는 종료하고 본 작업 계획으로 넘어간다. 평가 항목 중 "높음" 위험에 우회안이 없으면, 그 위험 하나만 추가 조사하는 식으로 조사를 연장한다. 전면 조사로 되돌아가지 않는다.
