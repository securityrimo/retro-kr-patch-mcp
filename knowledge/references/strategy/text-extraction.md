# 텍스트 추출 전략

원본 게임 바이너리에서 번역 대상 텍스트를 빠짐없이, 무손실로 끄집어내는 단계의 전략을 다룬다. 추출은 한글 패치 전체 파이프라인의 입력을 만드는 단계이므로, 여기서의 누락·오염은 번역과 재삽입 전 단계로 전파된다. 목표는 세 가지다.

1. **완전성** — 번역해야 할 텍스트를 모두 찾는다. 포인터 테이블 밖에 숨은 인라인 리터럴까지 포함한다.
2. **무손실** — 추출 결과만으로 원본 바이트를 1:1 복원할 수 있어야 한다. 라운드트립 검증으로 보장한다.
3. **재현성** — 추출은 스크립트화하여 언제든 같은 결과를 다시 만들 수 있어야 한다. 수작업 덤프는 검증 불가능하다.

CPU 엔디언, 뱅킹 방식, 주소 환산 공식 같은 플랫폼 세부는 `references/platforms/`의 해당 플랫폼 문서를 따른다. 이 문서는 플랫폼에 의존하지 않는 단계와 판단 기준만 다룬다.

## 1. 포인터 테이블 발견

레트로 게임의 대사·메시지는 **포인터 테이블 + 가변 길이 문자열** 구조로 저장되는 경우가 많다. 포인터 테이블을 찾으면 텍스트의 위치, 엔트리 수, 경계를 한 번에 확정할 수 있으므로 유력한 첫 탐색 대상이다. 다만 블록 마커, 고정 길이 슬롯, 코드 안 리터럴만 있는 영역도 있으므로 포인터 테이블 부재를 곧 텍스트 부재로 보지 않는다.

### 1.1 포인터 폭과 저장 형태

| 폭 | 자주 보이는 용도 | 값의 의미 |
|----|------------|-----------|
| 16비트 | 8비트 CPU(Z80, 65xx 계열), 뱅크 내 참조 | 뱅크/슬롯 상대 주소 또는 테이블 기준 오프셋 |
| 24비트 | 뱅크+주소를 한 번에 담는 long 포인터 | 뱅크 번호 + 뱅크 내 주소 |
| 32비트 | 16/32비트 CPU(68000, MIPS 등) | 절대 ROM 주소 또는 RAM 로드 주소 |

엔디언은 CPU를 따른다. 같은 게임 안에서 두 단계 구조를 쓰기도 한다. 예컨대 32비트 빅엔디언 절대 주소 테이블이 텍스트 블록을 가리키고, 각 블록의 머리에 다시 16비트 상대 오프셋 테이블이 붙어 블록 내 개별 메시지를 가리키는 식이다. 이 경우 외측 테이블만 찾고 끝내면 블록 내부 엔트리 구분을 놓친다.

### 1.2 스캔 휴리스틱

포인터 테이블의 통계적 특징은 다음과 같다. 이를 그대로 스캐너 조건으로 옮긴다.

1. **연속 증가**: 텍스트가 저장 순서대로 배치되면 포인터 값도 (NULL과 중복을 빼고) 거의 단조 증가한다.
2. **합리적 범위**: 포인터 값이 ROM/메모리 맵의 유효 범위 안에 있다. 뱅크 상대 포인터라면 뱅크 크기 미만, 슬롯 기준 포인터라면 슬롯 주소 범위 안.
3. **간격 분포**: 인접 포인터의 차이가 문자열 길이로 그럴듯한 범위(수 바이트~수백 바이트)에 모인다. 차이가 전부 동일하면 고정 길이 레코드 배열(텍스트가 아닐 수 있음)이다.
4. **첫 포인터 = 테이블 크기**: 텍스트가 테이블 직후에 시작하는 배치에서는 첫 포인터 값이 곧 테이블 바이트 크기(엔트리 수 × 포인터 폭)와 일치한다. 강력한 확증 신호다.
5. **카운트 헤더**: 테이블 직전 1~2바이트가 엔트리 수와 일치하는 경우가 있다(테이블 앞 1바이트가 카운트인 게임이 있다).

```python
import struct

def find_pointer_tables(rom: bytes, min_entries: int = 8,
                        lo: int = 0x0000, hi: int = 0xFFFF) -> list[dict]:
    """연속 증가하는 16비트 LE 값 시퀀스를 포인터 테이블 후보로 수집"""
    candidates = []
    for offset in range(0, len(rom) - min_entries * 2, 2):
        ptrs = [struct.unpack_from("<H", rom, offset + i * 2)[0]
                for i in range(min_entries)]
        if not all(lo <= p <= hi or p == 0 for p in ptrs):
            continue
        nonzero = [p for p in ptrs if p]
        if nonzero != sorted(nonzero):        # 단조 증가 위반
            continue
        gaps = [b - a for a, b in zip(nonzero, nonzero[1:])]
        if not gaps or not all(0 <= g <= 512 for g in gaps):
            continue
        if len(set(nonzero)) <= min_entries // 2:  # 중복 과다
            continue
        candidates.append({"offset": offset, "sample": ptrs})
    return candidates
```

24/32비트 변형은 워드 폭과 엔디언만 바꾸면 된다. 후보가 많이 나오면 "첫 포인터 = 테이블 크기" 조건과 포인터가 가리키는 위치에 종료 바이트(1.4 참조)가 규칙적으로 나타나는지로 거른다.

### 1.3 기준 주소·뱅크 보정

포인터 값은 파일 오프셋이 아니다. **포인터 값 → 파일 오프셋 환산 공식을 확정하는 것**이 테이블 발견의 후반부 절반이다. 기준점은 네 종류가 있다.

| 기준점 | 환산 | 전형 |
|--------|------|------|
| 절대 주소 | 그대로 (또는 헤더 크기 차감) | 32비트 절대 ROM 주소 |
| 뱅크/테이블 상대 | `테이블(뱅크) 시작 + 포인터` | 뱅크 시작 기준 오프셋 |
| 슬롯(매핑 주소) 기준 | `섹션 파일 베이스 + (포인터 − 슬롯 베이스)` | 모든 포인터가 매핑 슬롯 기준일 때, 파일 오프셋 = 그룹 베이스 + (P − 슬롯 베이스) |
| RAM 로드 주소 | `포인터 − 로드 베이스 + 파일 내 위치` | 파일 헤더의 u32가 RAM 주소인 경우 |

뱅크 스위칭 플랫폼에서는 "이 테이블이 사용될 때 어느 뱅크가 어느 슬롯에 매핑되는가"까지 알아야 환산이 닫힌다. 같은 16비트 값이라도 매핑 상태에 따라 다른 물리 위치를 가리킨다.

기준점을 모를 때는 역방향으로 확정한다. 화면에 표시되는 것이 확실한 텍스트의 파일 오프셋을 먼저 찾고(2장), 그 오프셋을 네 가지 기준 각각으로 환산한 후보 값을 ROM에서 검색한다. 검색이 적중하는 기준이 답이고, 적중 위치가 곧 포인터 테이블이다.

### 1.4 변형 패턴

- **NULL/기본값 엔트리**: 0x0000이나 특정 기본 주소가 반복되는 엔트리는 빈 슬롯이다. 상당수 엔트리가 동일 기본값을 가리키는 빈 엔트리이거나, 중복 포인터 자체가 기본 메시지 공유 수단인 경우가 있다. 엔트리 번호를 보존해야 하므로 빈 슬롯도 건너뛰지 말고 기록한다.
- **포인터 테이블이 없는 영역**: 블록 구분 마커로 연속 배열되는 텍스트도 있다. 포인터 없이 2바이트 블록 마커가 엔트리 경계가 되는 영역(예: 튜토리얼·힌트 텍스트)이 그렇다. 이 경우 영역 시작·끝을 지정하고 마커 스캔으로 추출한다(4.2 방법 분류 참조).
- **고정 길이 레코드**: 아이템·캐릭터 이름은 포인터 없이 N바이트 고정 슬롯 배열로 저장되는 경우가 많다. 슬롯 크기와 패딩 바이트(0x00/0x20/0xFF)를 함께 기록한다.

## 2. 인코딩 해독

포인터가 가리키는 곳의 바이트가 어떤 문자인지 알아내는 단계다. 표준 인코딩 후보는 플랫폼과 출신 지역으로 좁힌다. 일본 게임이면 Shift-JIS가 1순위(아래 판별 예시는 SJIS 기준), 서양 게임이면 ASCII/Latin 계열이다. 인코딩은 크게 세 부류다.

1. **표준 멀티바이트** — Shift-JIS가 대표. CD 기반·후기 플랫폼에서 흔하다.
2. **커스텀 타일 인덱스** — 바이트(또는 워드) 값이 VRAM에 로드된 폰트 타일 번호. 카트리지 게임의 대다수.
3. **혼합** — 제어코드 영역 + 표준 인코딩 텍스트. 예컨대 0x80 미만은 op-code, 0x80 이상은 표준 Shift-JIS 페어로 가르는 형태가 있다.

### 2.1 표준 인코딩 판별

Shift-JIS는 lead(0x81–0x9F, 0xE0–0xFC)/trail(0x40–0x7E, 0x80–0xFC) 범위가 고정이므로 통계 판별이 쉽다.

```python
def looks_like_sjis(data: bytes, threshold: int = 10) -> bool:
    i = pairs = 0
    while i < len(data) - 1:
        b1, b2 = data[i], data[i + 1]
        if (0x81 <= b1 <= 0x9F or 0xE0 <= b1 <= 0xFC) and \
           (0x40 <= b2 <= 0x7E or 0x80 <= b2 <= 0xFC):
            pairs += 1; i += 2
        else:
            i += 1
    return pairs > threshold
```

판별되면 표준 코드 테이블을 그대로 쓰면 되고, 전각 스페이스(81 40)·괄호류(81 75/81 76) 같은 빈출 코드가 덤프에 자연스럽게 나타나는지로 재확인한다.

### 2.2 커스텀 문자 테이블 복원

표준 인코딩이 아니면 게임 고유의 바이트↔문자 매핑을 직접 복원한다. 다음 기법을 조합한다.

**(a) VRAM 타일 뷰어 대조.** 디버그 기능이 있는 에뮬레이터에서 대사 표시 장면을 멈추고 타일 뷰어로 글리프와 타일 인덱스를 대조한다. 화면의 글자 → 타일 번호 → ROM의 동일 바이트 시퀀스 검색 순서로 매핑이 하나씩 확정된다. 가장 확실한 방법이다.

**(b) 알려진 텍스트 검색.** 타이틀 화면 문구, 주인공 이름처럼 내용이 확실한 짧은 텍스트의 타일 번호 시퀀스를 ROM 전체에서 검색한다. 적중 위치 주변이 텍스트 영역이다.

**(c) 상대 검색(relative search).** 인코딩 테이블이 문자 배열 순서(50음도, 알파벳 순)를 유지한다는 가정 하에, 문자 간 값 차이 패턴으로 검색한다. 첫 글자의 바이트 값(base)이 확정되면 배열 순서를 따라 테이블 전체를 한 번에 유추할 수 있다.

```python
def relative_search(rom: bytes, text: str) -> list[tuple[int, int]]:
    """문자 간 상대 차이로 패턴 검색. 반환: (오프셋, base값)"""
    diffs = [ord(b) - ord(a) for a, b in zip(text, text[1:])]
    hits = []
    for off in range(len(rom) - len(text) + 1):
        if all((rom[off+i+1] - rom[off+i]) % 256 == d % 256
               for i, d in enumerate(diffs)):
            hits.append((off, (rom[off] - ord(text[0])) % 256))
    return hits
```

상대 검색은 배열 순서가 깨진 커스텀 테이블, 압축 텍스트에서는 실패하며 3글자 미만 검색어는 오탐이 많다.

**(d) 글리프 시트 전사.** 폰트 시트를 타일 인덱스 순으로 이미지 덤프하고(`references/strategy/initial-survey.md` §2.4), 각 인덱스의 글리프가 어떤 문자인지 사람이 전사하면 그 전사가 곧 바이트↔문자 테이블이 된다. 배열이 알려진 순서(50음도·가나다)가 아니어서 상대 검색(c)이 통하지 않을 때의 기본 경로다. OCR을 1차 후보로 돌려 표를 미리 채우되, 폰트 글리프는 저해상도·장식체라 오독이 잦으므로 확정은 사람이 한다(`references/strategy/graphics-text.md` §1.6의 검출/인식 분리와 동형). 미해독 칸은 `<XX>`로 두고 화면 대조로 메우는 반복은 (a)와 같다.

복원한 매핑은 ROM 해킹 커뮤니티 표준인 `.tbl` 파일로 관리한다(`바이트값=문자`, 한 줄 1매핑, 제어코드는 `[NAME]` 표기). 테이블이 부분적이어도 일단 덤프해 보고, `<XX>`로 표시되는 미해독 바이트를 화면 대조로 채워 넣는 반복으로 완성도를 올린다.

### 2.3 워드 단위 타일 인덱스

16비트 CPU 게임은 글자 하나를 2바이트 워드로 저장하기도 한다. 빅엔디언 16비트 타일 인덱스(예: 0x0000–0x01DF가 문자, 0xFF00 이상이 제어코드)인 경우도 있고, 타일 번호 1바이트 + 속성 1바이트 쌍인 경우도 있다. 바이트 단위로 디코딩하면 절반이 깨져 보이므로, 빈도 분석에서 2바이트 주기성이 보이면 워드 단위로 다시 본다.

### 2.4 다이그래프/멀티바이트 압축 식별

용량 절약을 위해 1바이트가 2글자(DTE) 또는 단어(MTE)로 확장되는 압축이 흔하다. 식별 단서는 다음과 같다.

- 문자 테이블을 다 채웠는데도 특정 고빈도 바이트에서 디코딩이 계속 깨진다.
- 그 바이트가 특정 값 구간(예: 0xA0 이상)에 모여 있다.
- 같은 게임의 영문 패치본이 있다면, 패치본 스크립트와 길이를 비교했을 때 원본 쪽 바이트 수가 일관되게 짧다.

DTE 후보 바이트를 화면 대조로 풀면 "그 바이트 = 빈출 2글자 조합"이 확인된다. `.tbl`의 다중 문자 매핑(`F0=のは` 형태)으로 기록하면 디코더 수정 없이 처리된다. 영문 패치본을 입수할 수 있으면 패치 제작자가 남긴 테이블/사전 파일이 가장 빠른 지름길이므로 먼저 찾아본다.

## 3. 제어코드 해독

문자가 아닌 바이트(줄바꿈, 박스 전환, 대기, 변수 삽입, 효과음 트리거)를 식별하고 의미와 인자 크기를 확정하는 단계다. **인자 크기(arity)를 틀리면 이후 모든 바이트의 경계가 밀려 추출 전체가 오염되므로**, 제어코드 해독의 최종 목표는 "코드별 인자 크기 표"의 완성이다. 다음 3단계로 진행한다.

### 3.1 1단계 — 빈도 통계

텍스트 영역의 바이트 히스토그램을 만든다. 제어코드는 두 가지 통계 신호를 낸다.

- **값 구간 분리**: 문자 인덱스와 겹치지 않는 가장자리 구간(0xF0–0xFF, 또는 0x00–0x3F 하단)에 몰린다. 하단 구간(예: 0x00–0x32)이 op로, 상단 구간(예: 0xF8–0xFF)이 제어코드로 쓰이는 식이다.
- **빈도 역전**: 자연어 문자라면 나올 수 없는 빈도로 등장한다. 텍스트 영역의 상위 빈도 바이트가 전부 제어코드(줄바꿈, 종료, 명령 접두사)로 드러난 사례가 있다.

종료 코드는 별도 신호로 확정한다. **각 포인터가 가리키는 문자열의 끝 직전 위치마다 공통으로 나타나는 바이트**가 종료자다. 포인터 간격과 종료자 위치가 전 엔트리에서 정합하면 그 바이트로 문자열 경계를 닫을 수 있다. 종료자가 단일 바이트가 아니라 2바이트 조합인 경우(접두사+종료)도 있으므로 종료 직전 바이트의 분포도 함께 본다.

**통계의 범위 — "0회"는 전 코퍼스 합산으로만 확정한다.** 어떤 바이트 값이 안 쓰인다는 판정(가장자리 빈 구간, 제어코드만 점유한 대역, 한글 인코딩의 빈 prefix 후보)은 **단일 파일 분포로 내리지 않는다.** 한 파일에서 0회라도 다른 디스크·실행 파일에서 외자(gaiji)나 실사용 코드로 쓰이는 경우가 흔하고, 그러면 그 값을 비었다고 오인한다. 모든 디스크·모든 실행 파일을 합산한 전수 통계로 0회임을 확인한 뒤에야 빈 값으로 확정한다. 이 합산 통계는 한글 인코딩의 빈 prefix 선정에도 그대로 입력이 된다(`references/strategy/font-strategy.md` §2.1·§2.2 — 단일 파일 분포가 prefix 가설을 거짓음성으로 만든 사례).

### 3.2 2단계 — 화면 대조

후보 코드의 의미를 에뮬레이터 화면과 대조해 부여한다. 특정 대사의 덤프와 실제 표시를 나란히 놓고:

- 화면 줄바꿈 위치와 일치하는 바이트 → 줄바꿈(br)
- 메시지 박스가 닫히고 새로 열리는 지점의 바이트 → clear/박스 전환
- 입력 대기 커서가 뜨는 지점 → wait
- 숫자·이름이 표시되는데 덤프에 그 글자가 없는 지점 → 변수 삽입(인자 있는 코드일 가능성 높음)

인자 크기 가설은 이 단계에서 세운다. 코드 직후 1~2바이트가 문자 테이블로 디코딩되지 않는 값이면 인자로 가정하고, 가정을 적용한 재덤프에서 나머지 텍스트가 깨끗하게 읽히는지로 검증한다. 같은 코드의 인자 값 분포(예: 대기 코드의 인자가 30, 240 같은 프레임 수로 해석되는가)도 의미 추정의 근거가 된다.

### 3.3 3단계 — 디스패처 역추적

확정이 필요한 코드(특히 사용 빈도가 낮아 화면 대조가 어려운 것)는 텍스트 출력 루틴을 디스어셈블해서 닫는다. 찾는 패턴은 **op 바이트로 핸들러 주소 테이블을 인덱싱하는 디스패처**다. 8비트 CPU에서는 핸들러 주소의 low 바이트 테이블과 high 바이트 테이블이 평행하게 놓이는 형태가 자주 보이는 신호다.

디스패처를 찾으면 다음이 한 번에 확정된다.

- **op 범위**: 디스패치 직전의 비교 명령(`CMP #$80` 류)이 문자/제어코드 경계를 그대로 알려준다.
- **arity**: 각 핸들러가 스크립트 포인터에서 인자를 몇 번 읽는지(간접 로드 + 포인터 전진 호출 횟수) 세면 인자 크기가 코드로 증명된다.
- **미사용 op**: 핸들러는 정의돼 있지만 실제 스크립트에 0회 등장하는 코드를 구분할 수 있다.

op 핸들러를 평행 테이블에서 역추적해 arity를 전수 확정해, 추출 엔트리 전체에서 미해독 바이트가 0건으로 수렴한 사례가 있다. 디스패처 역추적은 비용이 크지만, 통계·화면 대조로 대부분을 채운 뒤 남은 모호한 코드만 정밀 타격하는 식으로 쓰면 효율적이다.

### 3.4 제어코드 사양표

해독 결과는 코드·인자 크기·종료자 여부·의미를 한 표로 고정한다. 추출기와 재삽입기가 모두 이 표를 단일 진실로 참조한다. FF-prefix 2바이트 제어코드를 쓰는 포맷의 표 구성 예는 다음과 같다.

| 코드 | 인자 | 종료자 | 의미 |
|------|------|--------|------|
| FF00 | 0B | ✓ | 메시지 종료 |
| FF02 | 0B | | 줄바꿈 |
| FF03 | 2B | | 메시지 블록 시작 |
| FF05 | 2B | | 딜레이/버튼 대기 |
| FF20 | 8B | | 씬 설정 |
| … | | | |

의미를 모르는 코드는 `opNN` 같은 중립 이름으로 등록하고 **바이트 시퀀스를 통째 보존**한다. 추출·재삽입 관점에서는 의미보다 경계(인자 크기)가 중요하다. 의미 해독은 번역 품질(화자 구분, 페이싱 보존)을 위해 점진적으로 보강하면 된다.

### 3.5 검출 오류 유형론

포인터·인코딩·제어코드 해독은 결국 "이 바이트가 텍스트인가"라는 판정의 연속이다. 이 판정은 두 방향으로 틀린다. 비텍스트를 텍스트로 잡는 **거짓양성**(추출물에 노이즈가 섞임)과, 진짜 텍스트를 놓치는 **거짓음성**(번역 대상이 누락됨)이다. 둘은 원인이 다르므로 따로 진단한다. 이 소절은 앞 세 장에서 나온 판정 실패를 유형으로 정리하고, 유형별 신호와 대응을 묶는다.

#### 3.5.1 거짓양성 — 비텍스트를 텍스트로

원인은 대개 **값의 우연한 겹침**이다. 텍스트가 아닌 바이트가 텍스트의 통계적 특징을 우연히 만족한다.

- **문자 인덱스 범위와 코드/데이터 값의 우연 일치** — 타일 인덱스 인코딩에서 글리프 코드 범위가 카운터·오프셋·플래그 같은 비텍스트 2바이트 필드값과 겹치면, 그 필드가 문자로 디코딩된다. 글리프 타일 코드 범위와 2바이트 필드값이 겹치면 1차 추출의 절반 이상이 외자(한 글자짜리) 노이즈가 될 수 있고, 특정 히라가나 타일이 흔한 카운터/오프셋 값과 정확히 겹치기도 한다.
- **포인터 거짓양성** — 짧은 패턴으로 포인터 테이블을 스캔하면 우연 출현이 폭증해 후보 매칭의 대부분이 거짓양성이 되기도 한다. 더 위험한 변종은 **좌표·수치 데이터가 포인터로 오인**되는 경우다. 예컨대 좌표값 37(0x25)이 포인터로 환산되어 잘못된 위치를 가리키면 실제 크래시를 낼 수 있다.
- **제어코드 바이트의 데이터 출현** — 화자 마커나 박스 코드 같은 제어 바이트가 비텍스트 데이터 구조 안에 우연히 나타나면, 그 지점을 텍스트 시작으로 오인한다. 이 오인은 거짓양성인 동시에, 진짜 텍스트 시작을 가려 **거짓음성을 유발**한다(3.5.2 참조).
- **VM 바이트코드·스프라이트 파라미터의 텍스트 오인** — 스크립트 VM 바이트코드나 스프라이트 좌표/속성 블록이 인코딩 휴리스틱을 통과한다. 컷씬·데모 영역의 비텍스트 바이트열이 대량 FP를 낸 사례가 있다.

거짓양성 차단의 필터 전략은 다음을 조합한다.

- **영역 한정** — 텍스트 영역을 수동 지정 테이블로 못 박고, 그 밖은 스캔하지 않는다(4.1 ④와 동일 원칙). 자동 탐지가 닿지 않는 비텍스트 영역을 통째로 배제하는 가장 강한 필터다.
- **정렬 검증** — 환산된 타겟이 인코딩 단위에 정렬되지 않으면(예: 워드 인코딩인데 홀수 타겟) 스킵한다. 우연 매칭은 정렬을 자주 깬다.
- **밀도 기반** — 진짜 텍스트 엔트리는 제어코드와 근접 글리프가 동반된다. "제어코드도 와이드 문자도 없는 엔트리 = 노이즈" 규칙으로 외자 FP를 걷어낸다.
- **선행 패치를 외부 근거로 대조** — 같은 게임의 영문 패치가 리버스 엔지니어링으로 텍스트 영역을 선언했다면 강한 후보 근거가 된다. 다만 검증 없이 받아들이지 말고 원본 ROM과 diff, 실행 경로, 라운드트립 결과로 다시 확인한다. 선행 패치가 디스어셈블리에 "이 영역은 코드/데이터가 아니라 텍스트"라고 표시한 지시문(어셈블러마다 이름이 다르다)은 텍스트 영역 지도의 입력이 될 수 있다.

**과잉 필터 경고**: 위 규칙은 진짜 텍스트도 같이 자를 수 있다. 특히 "외자·짧은 엔트리 = 노이즈"는 한자만으로 이뤄진 짧은 메뉴 항목을 오인한다. "中断"·"装備" 같은 짧은 메뉴 엔트리는 외자처럼 보여도 진짜 텍스트다. 필터는 후보 강등(검토 대기)에 쓰고, 자동 삭제에는 신중해야 한다.

#### 3.5.2 거짓음성 — 진짜 텍스트를 놓침

원인은 **틀린 모델로 스캔**하거나 **스캔 범위를 좁게 잡은** 데 있다. 거짓양성과 달리 추출물만 봐서는 드러나지 않으므로(없는 것은 보이지 않는다) 별도의 교차 신호가 필요하다.

- **표준 스캐너의 부분 매칭 실패** — 인코딩이 SJIS 같은 표준이 아니라 타일 인덱스 직접 참조나 게임 자체 테이블이면, SJIS 스캐너는 진짜 텍스트를 못 잡고 우연히 유효 SJIS로 해석되는 바이트만 노이즈로 토해낸다. 자체 테이블에 다이그래프·DTE 압축까지 얹히면 매칭이 더 어긋난다. **표준 인코딩을 가정한 스캐너가 0건이나 노이즈만 낸다고 텍스트가 없다고 단정하지 않는다. 인코딩 가정 자체를 의심한다.** (타일 인덱스 인코딩, 자체 테이블+다이그래프에서 실제로 이렇게 나타난 사례가 있다.)
- **포인터 타겟 앞의 순차 실행 텍스트** — 추출 시작점을 `min(포인터 타겟)`으로 잡으면, 포인터가 가리키지 않고 코드가 순차 실행으로 진입하는 프롤로그 대사가 시작점 앞에 남아 누락된다. 역방향 스캔으로 시작점 앞의 엔트리를 추가 복구한다.
- **코드 영역의 이름 테이블** — 기술명·아이템명 테이블이 텍스트 영역 밖 코드 구역에 박혀 있으면 영역 한정 필터에 함께 잘린다. 역방향 런 스캔으로 복구한다.
- **종료자 오인 잘림** — 종료자가 아닌 코드를 종료로 오인하면 그 뒤 갭 안의 텍스트가 잘려 나간다. 예컨대 메시지 블록 시작 코드를 종료로 오인하면 블록 내 텍스트를 놓친다.
- **인라인 리터럴** — 포인터 체계 밖, 코드 한가운데 직접 박힌 문자열은 스캐너가 잡지 못한다(6.3 참조). 핸들러 디스어셈블로만 발견된 사례가 있다.
- **종료자 방식 차이** — 같은 게임의 일/영 ROM이 종료 방식을 달리하면(예: JP는 2바이트 조합 종료, EN은 1바이트), 한쪽 방식을 가정한 덤프가 다른 쪽에서 엉킨다. 종료자 폭을 모르면 경계가 통째로 어긋난다.
- **과잉 필터가 만든 거짓음성** — 3.5.1의 필터를 너무 세게 걸면 거짓음성이 생긴다. suffix 블랙리스트가 진짜 포인터를 걸러내 일부 NPC 대사가 미표시되는 식이다. 필터 강화는 항상 FN을 새로 만들 수 있다.

거짓음성은 **느슨한 재스캔**(필터를 끈 전수 스캔)과 **역방향 스캔**(시작점 앞·코드 영역 훑기), 그리고 **외부 기준 대조**(선행 패치 엔트리 수, 영문 diff, 화면 도달 체크리스트)로 잡는다.

#### 3.5.3 제어코드 방식별 검출기 영향

제어코드가 텍스트 스트림에 어떻게 섞이는지는 검출기 설계와 오류 양상을 직접 좌우한다. 방식별로 검증 신호도 다르다.

| 제어코드 방식 | 검출기 영향 | 검증 신호 |
|---------------|-------------|-----------|
| 인라인 1바이트 op(가변 인자) | 텍스트와 op가 같은 스트림에 섞임. arity 테이블이 틀리면 바이트 경계가 어긋나 이후 전부 오독 | raw dump의 unknown_byte 0건·truncated 0건으로 op 경계 모델의 정확성 증명 |
| FF-prefix 2바이트 | 자기 식별되므로 정렬 스캔+lo바이트 통계로 텍스트/비텍스트 판별에 활용 가능. 단 파라미터 크기 맵이 없으면 파라미터를 글리프로 오독 | 파라미터 크기 맵 확정, 코드 뒤따르는 RAM 주소를 raw에서 제외 |
| 자체 테이블+다이그래프 1바이트 | 표준 가정(바이트 길이=글자 수, SJIS 범위)이 전부 무효. 이 방식에선 표준 SJIS 스캐너 매칭 0건 | 외부 신호 강제 — 빈도 분석, 영문 diff, VRAM 뷰어 대조 |
| 16비트 정렬 읽기 | 해당 엔진이 2바이트씩 읽는 것으로 확인되면 1바이트 제어코드를 그대로 내보낼 때 정렬이 어긋남 | 모든 요소를 엔진의 읽기 폭에 맞춰 패딩 |

요지: 제어코드가 **자기 식별되면**(FF-prefix) 정렬·통계가 검출을 돕지만, **인라인이면**(1바이트 op) arity 표가 단일 실패점이 되고, **자체 인코딩이면** 표준 도구가 전부 무력해져 외부 신호가 강제된다. 정렬 단위가 고정된 플랫폼에서는 모든 추출 요소의 폭을 그 단위에 맞춘다.

#### 3.5.4 "검출 완료" 판정

거짓음성은 정의상 보이지 않으므로 "다 찾았다"는 별도의 근거로 증명해야 한다. 다음 중 하나 이상을 충족시킨다.

- **전수 스캔+다중 크로스체크** — 대상 시퀀스를 전수 처리해 추출/텍스트 없음으로 양분하고, 필터를 끈 느슨한 스캔으로 거짓음성 0을 확인하고, 경계를 ±32바이트 확장해 잘림이 없는지 검사한다.
- **수렴+잔여 분석** — 1차 추출을 필터링으로 줄인 뒤, 남은 외자 엔트리를 전수 분석해 더 줄일 노이즈가 수확체감에 이른 시점에서 종료한다. 줄이는 작업은 잔여를 설명할 수 있을 때 멈춘다.
- **선행 패치와 정합성 대조** — 영문 패치가 확정한 텍스트 영역·엔트리 수에 우리 추출이 정합하면 완전성의 외부 근거가 된다. 불일치는 어느 한쪽이 틀렸다는 결론이 아니라, 종료자·영역 분할·리비전 차이를 재검토하라는 신호다.
- **미해독 0건 수렴** — 미해독 바이트 플래그가 붙은 엔트리를 0으로 수렴시키는 것(5.2)은 인자 크기 표와 문자 테이블이 맞다는 증거이자, 그 영역 안에서는 놓친 바이트가 없다는 증거다.
- **영역 단위 차등 신뢰도** — 전 영역을 똑같이 "완료"로 닫지 않는다. 검증이 덜 된 뱅크/영역은 "낮은 신뢰도" 꼬리표를 붙여 이월하고 다음 라운드에서 닫는다.

## 4. 추출 파이프라인

해독 결과를 바탕으로 전수 추출을 자동화한다.

### 4.1 덤프 도구 설계 원칙

1. **raw 보존**: 모든 엔트리에 대해 파일 바이트를 그대로(hex 문자열로) 저장한다. 디코딩 결과는 raw의 파생물이며, 디코더 버그가 있어도 raw가 있으면 재덤프 없이 복구된다.
2. **종료 코드 포함**: raw에는 종료 제어코드까지 포함한다. 재삽입 시 경계 모호성이 사라진다.
3. **포함/제외 규칙 명문화**: 제어코드 뒤 인자 중 "번역과 무관하고 재배치 시 값이 달라지는 것"(예: 스크립트 VM으로 제어를 넘기는 코드 뒤의 RAM 주소)은 raw에서 제외하고, 텍스트 레이아웃에 영향을 주는 인자(블록 시작 파라미터, 딜레이 값)는 포함한다. 어느 쪽이든 규칙을 코드 단위로 문서화해 추출기와 재삽입기가 동일하게 따른다.
4. **영역 밖 무시**: 스캔 범위를 확정된 텍스트 영역으로 한정해 바이트코드·그래픽 데이터의 우연한 일치(false positive)를 차단한다. 추출 대상 테이블에 등재되지 않은 파일/영역 = 텍스트 없음으로 명시적으로 분류한다.

### 4.2 추출 방법 분류

대상 영역마다 구조가 다르므로 추출 방법을 유형화해 영역별로 배정한다. 다음 4분류는 초기 분류 틀로 쓸 수 있지만, 대상 게임의 실제 저장 구조에 맞춰 더 쪼개거나 합친다.

| 방법 | 구조 | 추출 로직 |
|------|------|-----------|
| A. 포인터 테이블 기반 | 테이블 + 가변 길이 문자열 | 포인터 순회, 각 대상에서 종료자까지 읽기 |
| B. 영역 + 종료자 스캔 | 포인터 없음, 텍스트가 종료자로 연쇄 | 영역 내 전진 스캔, 텍스트 시작 바이트 감지 → 종료자까지 |
| C. 블록 마커 기반 | 시작 마커(+인자)와 종료자가 쌍 | 마커에서 엔트리 시작, 종료자에서 닫기 |
| D. 단순 영역 스캔 | 좁은 영역에 종료자 구분만 | 영역 내 종료자 분할 |

하나의 게임 안에 여러 방법이 공존하는 경우가 많다(본편 대사는 A, 메뉴는 B, 시스템 메시지는 C/D처럼 갈릴 수 있다). 추출기 진입점에서 영역 식별 → 방법 디스패치 구조로 짠다.

### 4.3 JSON 스키마

추출 산출물은 테이블(영역) 단위 JSON으로 저장한다. 최소 스키마는 다음과 같다.

```json
{
  "table_id": "bank03_dialog",
  "source": {
    "pointer_table": "0x0C000",
    "entry_count": 256,
    "base": "0x0C000",
    "method": "A",
    "terminators": ["FE"]
  },
  "entries": [
    {
      "entry_id": 0,
      "ptr_value": "0x0200",
      "file_offset": "0x0C200",
      "raw_hex": "2D0D72FD16379B3E0A1B2CFE",
      "text": "오프닝 첫 대사{br}둘째 줄{end}",
      "flags": []
    },
    {
      "entry_id": 1,
      "ptr_value": null,
      "file_offset": null,
      "raw_hex": null,
      "text": null,
      "flags": ["null_entry"]
    },
    {
      "entry_id": 2,
      "ptr_value": "0x0200",
      "file_offset": "0x0C200",
      "raw_hex": null,
      "text": null,
      "flags": ["dup_of:0"]
    }
  ]
}
```

규약:

- **table_id**: 영역(뱅크/파일/그룹)을 식별하는 안정적 문자열. 텍스트 맵(6장)의 키와 일치시킨다.
- **entry_id**: 포인터 테이블 내 인덱스. **테이블의 슬롯 순서를 그대로 보존한다.** 빈 슬롯도 번호를 차지해야 재삽입 시 포인터 테이블을 그대로 재구성할 수 있다.
- **원문(text)**: 디코딩 결과 + 제어코드 표기를 합친 문자열. 번역자는 이 필드만 보고 작업한다.
- **flags**: `null_entry`(빈 슬롯), `dup_of:N`(중복 포인터 — 본문은 N번에만 저장하고 참조로 대체), `has_unknown`(미해독 바이트 포함) 등.

### 4.4 제어코드 표기 규약

text 필드 안의 제어코드는 **역파싱 가능한 단일 규약**으로 표기한다. 권장 형식:

- 인자 없는 코드: `{br}`, `{wait}`, `{end}`
- 인자 있는 코드: `{delay:1E}`, `{face:02,05}` — 인자는 hex, 콤마 구분
- 미해독 코드: `{op27:0B}` — 중립 이름 + 인자 그대로
- 미해독 단일 바이트(문자 테이블 누락): `{raw:XX}`

규칙은 세 가지다. (1) 여는 문자 `{`와 닫는 문자 `}`는 문자 테이블에 등장하지 않는 것으로 고르고, 충돌 시 이스케이프를 정의한다. (2) 표기 ↔ 바이트 변환은 전단사여야 한다. 같은 바이트 시퀀스가 두 가지로 표기되거나, 같은 표기가 두 가지 바이트로 인코딩되면 라운드트립이 깨진다. (3) 번역자가 옮겨 적기 쉽도록 표기는 짧게 유지한다. 번역 단계에서 제어코드 누락·오타는 가장 흔한 오류원이므로, 재삽입기에서 원문과 번역문의 제어코드 집합 비교 검사를 함께 둔다.

## 5. 라운드트립 검증

추출이 무손실임을 증명하는 단계다. 검증 없는 추출 결과는 신뢰할 수 없다.

### 5.1 기본 불변식

핵심 property는 하나다.

```
encode(parse(text)) == raw_hex == 원본 파일의 해당 바이트
```

전 엔트리에 대해 세 값의 일치를 확인한다. text를 역파싱해 재인코딩한 바이트가 raw_hex와 같고, raw_hex가 원본 파일의 file_offset 위치 바이트와 같아야 한다. **불일치는 경고가 아니라 빌드 중단(bail) 사유다.** 이 검사를 빌드 파이프라인에 내장하면 raw가 파일과 어긋나는 순간 빌드가 실패한다.

### 5.2 부가 불변식

- **미해독 0건 수렴**: 미해독 바이트 플래그가 붙은 엔트리 수를 추적하고 0으로 수렴시킨다. 전 영역에서 미해독 바이트·잘림이 0건으로 수렴하면 파서 모델(op 경계 + 인코딩 분리)이 정확함을 실데이터로 증명한 셈이다. 미해독이 남아 있다는 것은 인자 크기 표나 문자 테이블 어딘가가 틀렸다는 뜻이다.
- **경계 정합**: 엔트리 i의 끝 오프셋이 엔트리 i+1의 시작과 일치하거나(연속 배치), 간격이 패딩 바이트로만 채워져 있어야 한다. 설명되지 않는 틈은 놓친 텍스트이거나 경계 오류다.
- **포인터 역산 일치**: file_offset을 환산 공식의 역으로 돌렸을 때 ptr_value가 재현돼야 한다. 기준 주소 공식의 검산이다.
- **엔트리 수 보존**: 추출 엔트리 수(빈 슬롯 포함)가 포인터 테이블 엔트리 수와 일치해야 한다.

### 5.3 property 테스트

위 불변식을 일회성 검사가 아니라 자동 테스트로 고정한다.

```python
def test_roundtrip(dump: dict, rom: bytes):
    for e in dump["entries"]:
        if "null_entry" in e["flags"] or e["raw_hex"] is None:
            continue
        raw = bytes.fromhex(e["raw_hex"])
        off = int(e["file_offset"], 16)
        tag = f"{dump['table_id']}#{e['entry_id']}"
        assert rom[off:off + len(raw)] == raw, f"{tag}: raw/파일 불일치"
        assert encode(parse(e["text"])) == raw, f"{tag}: 라운드트립 실패"
```

추출기·테이블·인자 크기 표를 수정할 때마다 전체 테스트를 재실행한다. 표기 규약 파서에는 무작위 제어코드/문자 조합을 생성해 `parse(format(x)) == x`를 확인하는 생성 기반 property 테스트를 추가하면 이스케이프 누락 같은 규약 구멍을 조기에 잡는다.

### 5.4 교차 검증

같은 게임의 공식 영문판이나 팬 영문 패치본이 있으면 강력한 교차 검증 자료가 된다. 두 빌드를 같은 추출기로 덤프해 엔트리 수·제어코드 구조·포인터 패턴을 비교한다. 구조가 일치하면 추출 모델의 신뢰도가 올라가고, 다른 부분(제어코드 값 변경, 종료 패턴 단순화, 텍스트 재배치)은 그 자체로 "패치가 무엇을 건드렸는가"라는 재삽입 전략의 입력이 된다. 일/영 두 ROM의 포인터 테이블 비교로 종료 코드 체계 차이와 텍스트 재배치 양상을 확인할 수 있다.

## 6. 텍스트 맵 문서화

추출이 끝나면 게임 전체의 텍스트 분포를 한 장의 지도로 고정한다. 텍스트 맵은 추출 단계의 최종 산출물이자 번역·재삽입 단계의 진입 문서다.

### 6.1 영역별 현황표

영역(뱅크/파일/그룹) 단위로 다음 컬럼을 채운다.

| 영역 | 논리 주소 | 파일 오프셋 | 유형 | 엔트리 | 인코딩 | 방법 | 상태 |
|------|-----------|------------|------|--------|--------|------|------|
| 대화 뱅크 | (플랫폼 표기) | 0x091A0–0x0CDD0 | 대화 | 435 | 2바이트 타일 | A | 추출 완료 |
| 스토리 뱅크 | … | 0x2C080–0x2D6D9 | 컷씬 대사 | 101(유니크) | 2바이트 타일 | A | 추출 완료 |
| 튜토리얼 뱅크 | … | 0x7C425–0x7F18E | 힌트 | 290 | 2바이트 타일 | C | 추출 완료 |

(형식 예시이며 수치는 임의값) 합계 행으로 **총 번역 대상 엔트리 수**를 명시한다. 이 숫자가 번역 작업량 산정과 진행률 추적의 분모가 된다.

각 영역에는 현황표 외에 다음을 함께 기록한다.

- 포인터 테이블 위치·엔트리 수·기준 주소 공식
- 종료자와 영역 고유 제어코드(전 영역 공통 사양표 + 영역별 추가분 형태로)
- 엔트리 구조 다이어그램(선행 바이트, 화자 코드, 본문, 종료 시퀀스 순서)
- 추출 산출 파일 경로(JSON/TSV)

### 6.2 비텍스트 영역과 프리 스페이스

텍스트 맵에는 **텍스트가 없는 영역도 명시적으로** 기록한다. "조사했고 텍스트 없음"과 "아직 안 봄"의 구분이 완전성 주장의 근거다. 또한 패딩으로 채워진 빈 영역(프리 스페이스)의 위치와 크기를 함께 기록한다. 추출 단계의 부산물이지만 재삽입 단계에서 확장 텍스트·한글 폰트·후킹 코드의 배치 후보가 되는 핵심 정보다. 비교 대상 영문 패치가 있다면 그 패치가 덮어쓴 영역 목록을 강한 후보 근거로 삼되, 원본 참조·런타임 접근·크기 한도 검증으로 다시 닫는다.

### 6.3 인라인 리터럴

포인터 테이블 체계 밖, **코드 한가운데 박힌 문자열**을 별도 절로 카탈로그화한다. 층 표시·시스템 라벨·숫자 글자 테이블처럼 핸들러 루틴이 직접 참조하는 리터럴이 자주 보이는 형태다. 예컨대 층 표시 핸들러를 분석하다 코드 뱅크 안에서 여러 인라인 일본어 리터럴(층 접미사, 빈 슬롯 라벨, 전각 숫자 테이블)을 발견하고, 영문 패치의 보존 영역과 대조해 진짜 게임 데이터임을 확정하는 식이다. 이런 리터럴은 일반 추출 파이프라인에 잡히지 않으므로 누락되기 쉽다. 탐지 방법:

- 텍스트 영역 밖 전체에 대해 인코딩 패턴(SJIS 페어 등) 휴리스틱 스캔. 오탐이 많으므로 후보 목록으로만
- 변수 삽입·라벨 출력 계열 제어코드의 핸들러를 디스어셈블해 참조하는 데이터 주소 추적. 정확
- 영문 패치본이 있으면 그 패치의 변경 바이트 중 텍스트 영역 밖 위치 검토

인라인 리터럴은 오프셋·현재 바이트·바이트 길이·사용 루틴을 기록하고, 번역 시 길이 제약(같은 바이트 수 안에 들어가야 하는지)을 함께 명시한다.

### 6.4 문서 위치와 갱신 규율

텍스트 맵은 추출기 코드와 같은 저장소에 두고, 추출기·테이블이 바뀌어 수치가 달라지면 함께 갱신한다. 라운드트립 테스트(5장)가 통과하는 추출 결과와 텍스트 맵의 수치가 일치하는 상태가 "추출 단계 완료"의 정의다. 이후 번역(스크립트 작업)과 재삽입(인코딩·포인터 재계산·공간 배치) 단계는 이 문서와 JSON 산출물만을 입력으로 받는다.
