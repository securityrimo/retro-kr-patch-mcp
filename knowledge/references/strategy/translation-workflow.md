# 번역 워크플로우 전략

추출된 텍스트를 번역 적격·빌드 적격 상태로 승격할 때는 현재 맥락과 승인 결정이 충분한지, 보호 정보와 소비 제약을 지켰는지, 사람만 결정할 의미 문제가 남지 않았는지를 분리해 판정한다. 번역 수단이나 작업 순서 자체를 통과 근거로 삼지 않는다.

---

## 1. 상태와 적격성

번역 적격과 빌드 적격은 다른 게이트다.

- **번역 적격**은 해당 단위의 의미를 판정하는 데 필요한 장면·화자·기능 맥락이 확정되고, 적용할 승인 용어·문체 결정이 식별된 상태다. 필요한 맥락이 확정되지 않은 단위는 추측으로 채우지 않고 번역 적격으로 승격하지 않는다.
- **빌드 적격**은 번역 적격 조건에 더해 보호 정보와 실제 소비 제약을 통과하고, 고위험 의미 판단이 모두 해소·승인된 상태다.

적격성의 단위는 실제 빌드 선택 경계에 맞춰 선언한다. 한 단위의 미해결 상태를 이유 없이 더 넓은 범위로 확대하지도, 미해결 단위를 선택 범위에 조용히 섞지도 않는다. 검수나 실행 QA에서 결함이 발견되면 영향을 받는 단위의 적격성을 취소하고 해당 판단 게이트로 되돌린다.

완료율의 분모는 확인된 번역 대상과 근거가 있는 제외 범주로 선언한다. 새 추출물과 기존 자산의 개수 차이는 조사할 후보 변화이지 곧바로 누락 번역 수가 아니다. 추가·삭제·중복·비텍스트 여부를 판정하기 전에는 그 차이를 완료율이나 누락 수로 승격하지 않는다.

자산 상태와 보호 정보가 보존해야 할 의미는 `references/conventions/translation-artifacts.md`가, 공통 빌드 입력 판정은 `references/strategy/build-and-verify.md` §1이 소유한다.

## 2. 컨텍스트 확정

컨텍스트는 추출 텍스트를 실제 장면·화면·기능과 연결해 해석 후보를 좁히는 근거다. 전량 추출 완료를 기다릴 필요는 없지만, 번역할 범위의 분모와 소비 맥락은 확인돼 있어야 한다.

### 2.1 확정 판정

컨텍스트가 확정됐다는 것은 대상 단위의 의미를 바꿀 수 있는 미확정 요소가 더는 남지 않았거나, 남은 불확실성이 번역 결과에 영향을 주지 않음이 확인됐다는 뜻이다. 필요한 요소는 게임과 문자열 역할에 따라 다르며, 모든 프로젝트에 같은 문서 묶음이나 체크리스트를 강제하지 않는다.

의미를 바꿀 수 있는 불확실성이 남으면 장면 확인·추가 추출·소비 경로 분석으로 되돌린다. 불확실성을 번역자의 일반지식이나 자연스러운 추측으로 대신하지 않는다.

대상 단위 밖의 서사·루트·관계 정보가 현재 의미를 바꿀 수 있으면 그 의존 범위도 먼저 확인한다. 서사 중심 범위의 본편 대사는 바깥 맥락의 영향 여부를 판정할 수 있도록 전체 줄거리·분기와 주요 관계 변화의 윤곽을 먼저 확보한다. 후반의 정체 공개, 분기별 사건 순서, 관계 변화처럼 국소 대사만으로 판정할 수 없는 작품은 관련 전체 서사·루트 구조를 재구성한 뒤 영향받는 배치를 번역 적격으로 승격한다. 반대로 독립된 UI나 장면처럼 바깥 맥락이 결과를 바꾸지 않음이 확인됐다면 전 작품의 스토리 파악을 선행 조건으로 만들지 않는다.

### 2.2 근거의 범위

원본 추출물과 실제 소비 장면을 기준선으로 삼는다. 현재 의미·순서·화자·용어의 적용 범위가 확정되지 않았다면 그 질문을 판정할 수 있는 외부 자료를 조사한다. 이미 판정된 질문 때문에 모든 자료군을 의례적으로 검색하지는 않는다.

- 공식 매뉴얼·사이트·가이드 같은 1차 자료는 설정, 명칭, 기능 설명의 근거가 될 수 있다. 공식 한국어판이나 다른 공식 현지화는 정착 표기와 해석의 후보지만, 작품·리비전·지역·시리즈 연속성이 맞는지 확인해야 한다.
- 공략·워크스루는 장면 순서, 분기 조건, 등장 인물과 시스템 기능을 복원하는 근거다. 원문 표현이나 화자 의도를 확정하는 번역 원문으로 쓰지 않는다.
- 선행 현지화·팬 패치·역공학 기록은 해석 후보와 기술 단서를 제공한다. 원문에 없는 보강·생략·시점 변경이 있을 수 있으므로 원본과 실제 장면에 다시 대조한다.

외부 자료로도 불확실성이 해소되지 않으면 추출 코퍼스 독해, 해당 루트 실행, 추가 추출·역공학으로 되돌린다. 출처를 찾았다는 사실만으로 적용 범위가 확인되지 않은 해석을 승인하지 않는다.

새 근거가 기존 컨텍스트를 뒤집으면 현재 판단을 고치고 영향받는 번역의 적격성을 재평가한다. 폐기된 해석을 현재 사실과 함께 보존하지 않는다.

### 2.3 논리 순서와 저장 순서

저장 순서가 플레이·장면 순서라는 보장은 없다. 의미 판정에 필요하면 번역용 논리 뷰를 장면·화면·화자 관계에 맞춰 구성하되, 그 재배열을 추출 기준선이나 물리 배치 변경으로 확장하지 않는다. 동일 원문도 소비 장면이나 화자가 다르면 같은 번역을 강제하지 않는다.

### 2.4 화자와 문체 맥락

화자·청자·관계·장면이 번역을 바꾸는 텍스트는 그 근거를 컨텍스트에 포함한다. 제어 신호·화면·대화 순서처럼 확인된 근거와 어조에서 추정한 정보를 같은 확신도로 다루지 않는다. 추정이 결과를 바꿀 수 있으면 사람 판단 대상으로 남기고, 해소 전에는 해당 단위를 번역 적격으로 승격하지 않는다.

관계나 장면에 따라 문체가 달라질 수 있으므로 하나의 화자 규칙을 모든 상황에 일반화하지 않는다. 차이가 번역을 바꾸면 화자→청자 관계와 서사 시점까지 승인 범위를 나눈다. 그 표현이 표나 다른 구조인지는 프로젝트가 정하며, 승인 문체의 범위와 예외는 §3의 SSOT에서 관리한다.

## 3. 승인 용어·문체 SSOT

고유명사·시스템 용어·반복 표현·화자 문체처럼 번역 전반에 재사용되는 결정은 하나의 승인 SSOT가 소유한다. 최소한 승인 결과, 적용 범위, 근거, 검토 상태를 구분할 수 있어야 한다. 이 정보가 보존해야 할 의미와 검증 규칙은 `references/conventions/translation-artifacts.md`가 소유한다.

- 작성자와 검토자는 승인 SSOT를 입력으로 사용하며, 개별 번역에서 조용히 다른 표기나 문체를 채택하지 않는다.
- 같은 원어라도 대상·화자·장면이 다르면 적용 범위를 나눈다. 반대로 같은 대상을 이유 없이 여러 표기로 갈라놓지 않는다.
- 신규 용어, 출처 충돌, 문체 예외가 번역 결과를 바꿀 수 있으면 승인 전까지 영향받는 단위를 적격으로 승격하지 않는다.
- 시리즈나 선행 현지화의 결정은 후보일 뿐이다. 대상 작품과 현재 맥락에서 다시 승인하지 않은 결정을 상속하지 않는다.
- 대상 작품의 다른 정식판·이식판이나 같은 시리즈의 인명·지명·아이템·시스템 용어를 새로 승인할 때는 적용 가능한 공식 한국어 표기나 정식 계승 표기가 있는지 먼저 확인한다. 확인된 공식 표기도 원문의 의미와 대상 작품의 연속성에 맞는 범위에서만 채택한다.
- 모든 프로젝트에 통하는 고정 출처 순위나 문체 우선순위를 두지 않는다. 무엇을 권위로 인정할지는 대상과 번역 목표에 맞춰 범위를 선언한다.

자동 검사는 SSOT 위반 후보를 만들 수 있지만 신규 표기를 승인하거나 충돌을 해소할 수 없다. 기계 판정이 불가능한 활용·생략·의역은 의미 검토로 보낸다.

### 3.1 Agent 배치 요건

Agent·LLM 번역이나 병렬 검토를 쓸 때는 작업 단위마다 **판정에 필요한 맥락이 확정되고 독립 검토가 가능해야 한다**. 전체 코퍼스가 이 요건을 한 단위로 만족하지 못하면 배치로 나누고, 만족하면 분할을 의무화하지 않는다. 배치 크기나 Agent 수는 고정하지 않으며, 장면·화자·기능 경계를 자르거나 검토자가 전체 관계를 잃을 만큼 큰 배치는 허용하지 않는다.

각 배치는 다음 입력의 같은 스냅샷을 번역자와 검토자에게 제공한다.

- 안정 ID, 원문과 보호된 raw byte
- 가역 제어 토큰과 미확정 raw token
- 확정된 제어 토큰의 대상 소비 의미와 `references/strategy/text-extraction.md` §4.4의 적용 정책
- 장면·화자·인접 문맥, 원본 소비 순서와 의미를 바꾸는 전역 서사·루트 의존
- 승인 용어·문체 SSOT와 적용 범위
- 길이·레이아웃·인코딩 등 실제 소비 제약

Agent 출력은 역문·판단 근거·질문과 제안 상태만 바꾸며 source identity, raw byte와 보호 토큰을 다시 만들거나 정규화하지 않는다. 배치는 번역 편의를 위한 논리 단위일 뿐 원본 엔트리 경계·순서·물리 배치를 새로 정의하지 않는다.

새 용어·문체 결정이나 원문 해석이 배치 사이에서 바뀌면 영향받는 이전 배치도 같은 SSOT로 다시 검증한다. 병렬 결과는 안정 ID와 기준선이 일치할 때만 합치고, 같은 엔트리·용어·보호 정보의 충돌을 마지막 writer 승리로 해소하지 않는다. 구체 상태와 보호 필드 의미는 `references/conventions/translation-artifacts.md`가 소유한다.

## 4. 보호 정보와 소비 제약

번역 가능한 내용과 보존해야 할 구조 정보를 분리한다. 원본 바이트·식별자·포인터 근거·경계 정보처럼 추출 기준선을 구성하는 값은 번역 과정에서 바꾸지 않는다.

제어 토큰은 실제 소비 의미를 확인해 보존, 조판 범위 내 이동, 재계산, 대상 의미로 변환, 수정 금지처럼 정책을 나눈다. 종료·이벤트·변수 삽입처럼 의미나 실행을 바꾸는 토큰은 순서와 인자까지 보존 정책에 포함하며, 의미가 미확정이면 가역 형태로 보존한다. 토큰의 구체 표현은 `references/conventions/translation-artifacts.md` §3이, 가역성과 정책 기준은 `references/strategy/text-extraction.md` §4.4가 소유한다.

길이·폭·줄 수·슬롯·인코딩·삽입 변수 조합은 언어 일반값이 아니라 실제 소비자가 요구하는 제약으로 판정한다. 제약 위반을 번역 품질 문제로 가장하거나, 의미 축약만으로 자동 해소하지 않는다. 번역 조정과 수용량 변경 중 어느 경로가 맞는지는 위반 원인과 영향으로 선택한다.

보호 정보나 소비 제약을 위반한 단위는 사람의 취향 판단으로 통과시킬 수 없으며 빌드 적격에서 차단한다. 제약 자체가 미확정이면 검사를 생략하지 않고 소비 경로 분석으로 되돌린다.

## 5. 생성·검토·승인의 분리

결정적으로 검사할 수 있는 위반과 의미 판단이 필요한 문제를 같은 검수 단계로 합치지 않는다. 보호 정보·상태·인코딩·소비 제약 검사는 의미 검토나 사람 승인으로 대체할 수 없고, 의미 검토도 결정적 검사의 통과만으로 대체할 수 없다.

고위험 의미 판단은 특정 품질 항목의 고정 우선순위가 아니라, 틀렸을 때 플레이어의 이해·선택·진행·인물 관계·작품 의도를 실질적으로 바꿀 수 있는지로 판정한다. 원문 해석 충돌, 신규 표기, 말장난·문화 요소의 의역, 화자·관계 불확실성처럼 영향이 큰 선택은 가능한 해석과 근거를 남기고 사람이 승인한다. 미해결 고위험 판단이 하나라도 남은 단위는 빌드 적격으로 승격하지 않는다.

### LLM 초벌을 쓰는 조건부 흐름

LLM이 초벌을 만들면 다음 판단 사슬을 반드시 적용한다.

```text
LLM 초벌 → 독립 2차 LLM 리뷰 → 사람에게 옵션·근거 제시 → 사람 승인
                  ↘ 결정적 위반은 해당 게이트로 반환
```

2차 리뷰는 초벌의 결론을 전제로 승인하는 절차가 아니다. 초벌과 분리된 검토 맥락에서 원문, 승인 SSOT, 화자·장면 근거, 보호·소비 제약을 다시 대조한다. 의미·표기·문체에 선택지가 있으면 하나를 조용히 덮어쓰지 않고 가능한 옵션, 각 옵션의 근거와 영향, 권고를 사람에게 제시한다. 최종 승인 책임은 사람에게 있으며, 승인 전에는 영향받는 단위를 빌드 적격으로 승격하지 않는다.

독립성은 특정 모델·서비스·실행 방식이 아니라 초벌의 추론과 결론을 답으로 전제하지 않는 별도 검토를 뜻한다. 이 규칙은 LLM 초벌을 선택한 경우에만 발동하며, LLM을 쓰지 않는 프로젝트에 특정 도구를 도입하도록 강제하지 않는다.

---

## 6. 적격 판정

다음 조건을 모두 만족한 범위만 빌드 입력으로 승격한다.

1. 번역 결과를 바꿀 수 있는 컨텍스트 불확실성이 없다.
2. 적용되는 용어·문체 결정이 승인 SSOT와 일치한다.
3. 보호 토큰과 실제 소비 제약을 모두 통과한다.
4. 고위험 의미 판단이 해소되고 필요한 사람 승인을 받았다.
5. LLM 초벌을 썼다면 독립 2차 LLM 리뷰와 옵션·근거 제시를 거쳤다.

어느 조건이 실패했는지에 따라 컨텍스트 조사, SSOT 승인, 보호·소비 제약 해소, 의미 판단으로 되돌린다. 이를 하나의 포괄적 "번역 완료" 상태로 뭉개지 않는다.

## 7. 검수 대시보드 (도구)

§2.3(논리 순서)·§2.4(화자·문체 맥락)·표기 불일치 자동검출를 사람이 한 화면에서 수행하는 내부 IP 도구. `retro-kr-patch-tools` MCP의 `review_dashboard(project_dir, action)`로 **프로젝트별** 기동한다(ROM 무관). 소스 `dashboard/`, 상세 `dashboard/README.md`.

### 원칙
- **편집 원본은 단일 JSON.** 대시보드·빌더가 같은 `translations_json`을 읽고 쓴다. OVERRIDES 같은 2차 소스를 두지 않는다(단일 진실).
- **논리 재배열만.** 대본 뷰의 순서 재정렬은 §2.3 그대로 읽기·작업 순서 재배열이며 ROM 물리 배치·raw 기준선을 건드리지 않는다.
- **자기소유 프로세스만.** `.krpatch/dashboard.pid`로 자기 인스턴스만 관리(광역 kill 금지).
- **코덱 선택.** 프로젝트 코덱(`codec_module`) 있으면 빌드와 100% 동일 로직으로 글리프 미리보기·바이트예산을 계산하고, 없으면 텍스트 전용으로 강등.

### 대본(스토리순) 뷰 — 순서 재정렬은 ROM 아키텍처로 갈린다
- **인라인-바이트코드형**(대사가 스크립트 명령 스트림에 끼워짐): `offset` 순 ≈ 실행순. 별도 포인터 해석 불요.
- **포인터-테이블형**(대사 뱅크 + 코드/테이블 포인터 참조, 다수 JRPG): `pointer` 모드가 ROM 전역을 훑어 각 문자열을 겨냥하는 최소 포인터 위치순으로 정렬(≈ 스크립트 순서). 미참조 문자열은 offset순 뒤로, 커버리지를 표기.
- **한계 명시.** 진짜 분기(전투 분기·조건부 이벤트)까지 따라가려면 ROM별 스크립트 VM 디스어셈블이 필요하며 범용 불가. 두 근사(offset/pointer) + 장면 그룹핑까지가 도구의 책임. 그 이상은 §2.3의 원칙대로 워크스루 병행.
- **화자 묶음.** 이름표 휴리스틱(짧고 부호 없이 개행 종료·가시글자 ≤4자)으로 화자+본문을 대본처럼 묶어, 라인마다 원문을 병기해 문맥·오역을 대조한다(§2.4 화자·문체 맥락의 시각 보조).

### 검출·게이트
- 필터: 미번역 / 원문 한자·가나 잔류 / 표기 불일치(같은 원문 → 다른 번역, 표기 불일치 자동검출) / 재검토 / 수동 / 예산초과.
- 저장 시 코덱으로 인코딩·예산을 즉시 검증하고, 초과·인코딩불가는 강제확인을 요구(빌드 길이 검증과 동일 기준).

## 8. 장면-인지 2단계 번역 파이프라인 (도구)

문자열 1개씩 독립 번역(단순 배치)은 **화자·맥락·줄간 일관성**을 잃는다(같은 이름 다르게 표기, 반말/존댓말 혼선, 제어토큰 유실). `retro-kr-patch-tools` MCP `translate_pipeline(project_dir, action)`로 §2(컨텍스트 확정)·§2.4(화자)·표기 불일치 검출·바이트예산 검증 루프을 한 파이프라인에 묶는다(ROM 무관, codec-driven). 소스 `dashboard/translate.py`.

### 3단계
- **① Context Aggregation.** 연속 문자열을 대화 창(장면)으로 묶어 한 프롬프트로. 장면 경계=offset gap 초과 또는 이름표(＿없는 `이름＠`) 경계, 하드캡 ~24줄. 창 = LLM에 주는 문맥 단위.
- **② Stage A (화자/맥락 초벌).** 창 전체를 JSON으로 주고 이름표로 화자 식별 → 화자별 말투 일관 + 자연스러운 초벌. 제어토큰 위치·개수 100% 보존 지시.
- **③ Stage B (자수 최적화 루프).** 줄별로 코덱 `encoded_len`으로 바이트예산·불가문자·제어토큰 멀티셋을 기계검증 → 위반 줄만 피드백 재시도(최대 3회). 바이트예산 검증 루프를 LLM 인루프로 구현. 3회 실패 시 baseline 유지(안전, 절대 파손 금지).

### 이름표 고정 (프로젝트별, game-independent)
- **glossary.json = 프로젝트별 이름/용어 SSOT.** `action=glossary`가 웹지식 + **ROM 실제 이름표 grounding**(빈도순 스캔)으로 부트스트랩. 게임마다 독립.
- **이름표는 자유번역 금지.** Stage A 출력을 glossary로 덮어써 화자명을 장면 전체에서 고정(첫등장-일관). 이게 없으면 일관성 도구가 오히려 불일치를 만든다(母親→어미/어멈 실사례).
- **예산-맞춤 별칭.** 이름표 예산은 극히 작다(예: 4바이트=한글 1자). `action=labels`가 예산 초과 이름표를 한국 통용 별칭으로 축약하되, **불확실하면 `label_questions.json`으로 분리해 사용자에게 확인**(임의 축약 금지). 확정 후 glossary 반영.
- **진짜 이름표 판별.** `이름＠` 형태(＠개행) + 다음 줄이 `＿`들여쓴 대사(인접성) + 시스템/색코드 잡음 배제. 카운터·메뉴 텍스트를 이름으로 오인해 고정하는 것을 막는다.

### 순서·게이트
- 권장 순서: glossary → labels →(사용자 확인)→ run. `run scope=all`은 전수(백그라운드), `run --start 0xADDR`는 창 단위(동기·검수용). `apply=True`만 JSON 갱신(백업 `.bak-pre-pipeline`).
- 리포트(`pipeline_report.json`)로 baseline↔pipeline 라인 대조·변경/실패 표기 → §7 대시보드에서 잔여 불일치·실패 줄 사람 검수.
- 적용 후 **제어토큰 무결성 전수 감사**(kr↔jp 멀티셋 일치)로 종단자 유실=스크립트 파손을 차단. 종단자(＄/＊/￥)는 필수, ＿(들여쓰기) 유실은 미관(예산 여유 시 복원).
