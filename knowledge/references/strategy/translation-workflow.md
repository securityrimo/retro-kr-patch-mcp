# 번역 워크플로우 전략

레트로 게임 한글 패치에서 추출된 스크립트를 번역하고 검수해 패치 빌드 가능한 상태까지 끌어올리는 작업 흐름을 정리한다. 핵심은 네 가지다. 번역 착수 전에 게임 유형에 맞는 컨텍스트(서사 게임은 장면·화자·관계, 퍼즐·액션·UI 중심 게임은 시스템 용어·메뉴 기능·상태 메시지)를 먼저 확보하고, 진척 상태를 디렉토리 위치로 표현해 어떤 파일이 어느 단계인지 한눈에 보이게 하고, 용어집을 단일 진실 원천으로 삼아 표기를 통일하며, LLM 초벌·LLM 교차 리뷰·사람 최종 검수의 분업으로 분량과 품질을 동시에 잡는다. 사례 수치는 일본어 원문·텍스트 중심 선행 프로젝트에서 가져온 참고값이다. 원문 언어가 다르면 용어집 열 구성·참조 우선순위·길이 비율을 원문 언어 기준으로 치환한다.

---

## 1. 5단계 진척 구조

번역 파일의 진척 상태를 별도 메타데이터가 아니라 **디렉토리 위치**로 표현한다. 파일을 단계 디렉토리 사이에서 `git mv`로 옮기는 것 자체가 상태 전이이며, 이력은 git 로그에 남는다.

```
assets/translations/scripts/
├── raw/                  # 추출 직후, 번역 미착수
├── in_progress/          # LLM 또는 사람이 번역 작업 중
├── needs_review/         # 초벌 완료, 자동 검증 통과, LLM 교차 리뷰 대기
├── needs_human_review/   # LLM 리뷰로 해소 안 되는 판단 필요 항목
└── complete/             # 사람 승인 완료, 패치 빌드 입력 자격
```

### 각 단계의 시맨틱과 이동 조건

| 단계 | 의미 | 다음 단계로 가는 조건 |
|---|---|---|
| `raw/` | 추출 스크립트가 ROM과 1:1 대응. `ko` 필드 비어 있음 | 배치 계획에 따라 작업 착수 시 `in_progress/`로 이동 (이동만, 내용 무변경) |
| `in_progress/` | `ko`/`status`/`notes` 채워 넣는 본 작업 | 전 엔트리 번역 + 자동 검증 스크립트 통과 |
| `needs_review/` | 초벌 품질은 확보, 표기·말투 교차 검토 대기 | LLM 교차 리뷰에서 지적 사항 0이면 사람 검수로, 기계 판단 불가 항목이 있으면 `needs_human_review/`로 |
| `needs_human_review/` | 말장난 의역, 표기 충돌, 문화 요소 등 사람 판단 필요 | 사람이 항목별 결정을 `notes`에 기록하고 반영 |
| `complete/` | 최종 확정. 빌드 파이프라인이 이 디렉토리만 읽는다 | (종착) 이후 수정은 재검수로 되돌린다 |

운영 원칙:

- **이동과 내용 수정을 같은 커밋에 섞지 않는다.** 배치당 커밋을 3개로 나눈 사례가 있다: ① `raw/ → in_progress/` 이동만, ② 번역 본 작업, ③ 검증 통과 후 `needs_review/`로 이동. 이렇게 나누면 diff 리뷰가 단순해진다. 보호 필드 무결성도 git diff로 기계 검사할 수 있다.
- **`raw/`는 불변 기준선이다.** 번역 단계에서 원문(`text`, `raw_hex`), 포인터 값, 오프셋 등 추출기가 채운 필드는 절대 수정하지 않는다. 번역자가 만질 수 있는 필드를 `ko`·`status`·`notes` 3개로 제한하고, 검증 스크립트가 `raw/` 기준선과 byte 단위로 대조해 위반을 잡아낸 사례가 있다. 보호 필드 동일성·제어코드 집합/순서 일관성·길이 예산을 빌드 입력 자격 검사로 삼아, 하나라도 어긋나면 빌드를 실패시킨다. 이것이 「오역·깨짐 = 블로커」를 번역자의 주의력이 아니라 빌드 게이트로 강제하는 핵심이다(검출 게이트 전체는 §5 「자동 검출 게이트」). 스크립트 파일의 스키마와 제어코드 표기의 정본은 `references/strategy/text-extraction.md`(§4.3–4.4)다.
- **파일 안의 엔트리별 `status`와 디렉토리 단계는 별개 축이다.** 디렉토리는 파일 전체의 단계를, 엔트리 `status`(`untranslated`/`in_progress`/`done`/`needs_review`/`needs_human_review`)는 개별 문장의 상태를 나타낸다. 파일 하나에 미해결 엔트리가 한 줄이라도 남으면 파일 전체가 다음 단계로 못 간다.
- 역행을 허용한다. 검수에서 문제가 발견되면 파일을 앞 단계로 되돌리고 사유를 `notes`에 남긴다.

## 2. 컨텍스트 재구성

전량 추출이 끝난 뒤, 본 번역에 들어가기 전에 추출 텍스트를 기반으로 **번역 컨텍스트 문서**를 만든다. 서사 게임에서는 장면·화자·관계가 핵심이고, UI/퍼즐/액션 중심 게임에서는 시스템 용어, 상태 메시지, 조작 안내, 화면별 길이 제약이 핵심이다. 번역은 문장 단위 치환만으로 닫히지 않으므로, 각 문자열이 어떤 화면·상황·기능에 속하는지 모르면 말투·용어·길이 판단이 추측이 된다. 업계 로컬라이제이션의 로크킷(lockit) 관행을 팬 한글화 규모에 맞게 줄여 적용한다.

컨텍스트 문서의 구성 요소는 게임 유형에 맞춰 고른다.

| 구성 요소 | 내용 |
|---|---|
| 전체 구조 | 서사 게임은 막·장·분기 루트, 비서사 게임은 모드·화면·메뉴 계층 |
| 장면/화면 단위 표 | 장면·층·스테이지·메뉴별 사건, 기능, 등장 인물, 노출 조건 |
| 화자/문체 정보 | 대사 게임은 캐릭터 관계·존비어, UI 중심 게임은 문체 기준과 버튼/상태 표현 |
| 반복 표현 | 반복 개그·말버릇·정형구 또는 시스템 메시지 템플릿 → 용어집과 연동 |
| 핵심 검수 지점 | 서사 클라이맥스, 튜토리얼, 엔딩, 자주 보는 메뉴처럼 품질 체감이 큰 구간 |
| 번역 고려사항 | 말장난·문화 요소·시점 전환·길이 제약·변수 삽입 등 문자열별 주의점 |

### 2.1 소스의 3계보

컨텍스트 문서를 만드는 소스는 세 갈래다. 외부 자료가 있으면 앞쪽부터 활용하고, 없으면 셋째 방법으로 직접 만든다. 어느 길이든 산출물 형식은 같다.

1. **외부 공략·매뉴얼 정리.** 공개된 공략·워크스루·매뉴얼을 수집하되 그대로 쌓아 두지 않는다. 진행 축, 화면 기능, 지명 대응표, 퀘스트 키워드, 조작 용어를 추출해 번역 가이드 형태로 흡수한다.
2. **선행 패치·RE 노트 종합.** 영문 팬패치나 타 언어 패치가 존재하면 그 번역 결과물과 역공학 노트를 컨텍스트 자료로 쓴다. 이때 한계를 문서 머리에 명시한다: **타 언어 번역물은 직역 소스가 아니라 컨텍스트 자료로만 쓴다.** 의역·추가·시점 변경이 흔해서, 직역하면 원문에 없는 내용까지 옮겨 온다.
3. **추출 텍스트 전수 독해로 자체 재구성.** 외부 자료가 없거나 부실하면 추출 텍스트를 처음부터 끝까지 읽고 구조를 직접 복원한다. 비용은 가장 크지만, 추출 데이터 자체에 대한 이해(순서·중복·화자 신호·화면 기능)가 부산물로 따라온다.

### 2.2 살아있는 컨텍스트

컨텍스트 문서는 번역 착수 전에 한 번 쓰고 끝나는 정적 자료가 아니다. 번역 배치가 진행되며 덤프에서 새로 확인된 사실로 계속 정정·갱신되는 살아있는 컨텍스트다. 정정할 때는 옛 서술을 지우지 말고 「무엇을 무엇으로, 어떤 근거로 고쳤는가」를 남겨, 이미 그 서술을 참조해 번역된 분량을 역추적할 수 있게 한다.

### 2.3 서사 순서와 추출 순서

추출 데이터의 엔트리 순서는 **저장 순서이지 플레이 순서가 아니다.** 팬번역 선례에서 흔히 '완전히 무작위 순서의 대사 덤프'라 부르는 고질 문제다. 선례들에서 함께 관찰된 부수 문제도 정리해 둔다: 같은 문자열이 여러 장면에 다용도로 재사용되거나, 미사용·디버그 문자열이 본편 대사 사이에 섞여 있거나, 분기 루트의 거의 같은 대사가 여러 벌 반복되는 경우가 있었다.

실제 사고와 대응 사례:

- 한 사례에서는 씬을 독립 블록으로 가정하고 데이터를 물리적으로 재배치했다가, 씬 경계에 걸쳐 공유되던 데이터의 꼬리가 잘려 대사 대량 유실과 화자 코드 손상이 났다. 저장 순서를 임의로 바꾸는 것은 위험하다.
- 다른 사례에서는 메인 서사의 절반 이상이 단일 region에 몰려 있었다. 저장 순서대로 번역하는 대신 플레이 진행(층) 단위로 작업 순서를 재편성해 배치를 짰다.

권장 절차 — 선례에서 효과가 확인된 방식이지 확립된 표준은 아니다:

1. 엔트리에 **장면 ID를 부여**하거나, 번역 작업용 뷰를 장면순으로 재배열한다. 재배열은 어디까지나 작업 순서·읽기 순서의 논리 재배열이며, `raw/` 기준선이나 ROM 내 물리 배치는 건드리지 않는다(위 물리 재배치 사고가 그 결말이다).
2. 장면순 재배열이 불가능하면 워크스루를 옆에 두고 블록마다 「이 블록은 어느 장면」 주석을 단 채 진행한다.
3. 중복 대사는 고유분만 번역하고 동일 원문에 번역을 전파한다. 단 전파 후에는 루트 간 동일 엔트리의 번역이 서로 달라진 곳을 자동 검출하는 검사를 돌린다(루트별로 따로 번역하다 같은 원문이 다른 번역으로 갈라진 것을 기계 대조로 잡은 사례가 있다). 전파는 기계적 복사이므로, 같은 원문이라도 화자나 장면 톤이 달라 번역이 갈라져야 하는 경우는 전파 대상에서 제외하고 `notes`에 사유를 남긴다.
4. 미사용·디버그로 판정된 문자열은 삭제하지 않는다(라운드트립 불변식 위반). 엔트리에 미사용 표시를 달아 번역 우선순위에서 빼고, 판정 근거(참조하는 포인터 부재, 디버그 메뉴 전용 등)를 적는다. 미사용으로 잘못 판정하면 본편 문자열이 원문 그대로 남으므로, 확신이 없으면 본편 대사와 같이 번역한다.

### 2.4 화자 식별과 존비어 매트릭스

한국어 번역의 품질은 화자 식별에 직결된다. 화자를 모르면 존비어를 정할 수 없고, 존비어가 흔들리면 독자는 즉시 알아챈다. 화자 식별은 한 번에 끝나지 않으며 다음 사이클로 점진 확정한다.

1. **제어코드의 화자 신호를 보존·추출한다.** 화자 ID 워드, 포트레이트(초상화) 지정 코드, 화자별 텍스트 색 지정이 대표적 신호다. 어떤 사례에는 화자 ID 제어코드가, 다른 사례에는 포트레이트 코드가 있었다. 추출 단계에서 이런 코드를 버리지 않고 엔트리에 남겨 두는 것이 화자 식별의 1차 소스다.
2. **신호가 없으면 어조와 스토리 문서로 추정한다.** 원문 언어의 역할어·어조 신호(일본어 사례에서는 1인칭 대명사, 어미, 말버릇)와 §2의 장면 표를 대조해 화자를 추정한다.
3. **추정은 전부 `needs_review`로 마킹한다.** 확정 신호 기반 식별과 추정 식별을 같은 확신도로 다루지 않는다.
4. **배치 진행 중 매핑 표를 정정한다.** 한 사례에서는 배치 1에서 세운 화자 추정이 배치 2의 덤프 대조에서 뒤집혀 매핑 표가 정정됐다. 화자 매핑 표도 §2.2의 살아있는 문서다.

존비어는 **문장 단위로 결정할 수 없다.** 문장마다 따로 정하면 같은 대화 안에서 존대가 오락가락하는 문장 간 불일치가 생긴다는 것이 기계번역 연구에서도 확인된 한국어 특유의 문제다. 따라서 번역 전에 **캐릭터 쌍 단위 스피치 레벨 매트릭스**를 만든다: A가 B에게 말할 때의 등급(해라체/해체/해요체/합쇼체)을 모든 유의미한 캐릭터 쌍에 대해 표로 확정하고, 스토리 진행에 따라 관계가 변하면(적대→우호, 정체 공개 등) 어느 장면부터 등급이 바뀌는지도 명시한다. 형식 예시:

| 화자 → 청자 | 주인공에게 | 라이벌에게 | 연장자 NPC에게 | 비고 |
|---|---|---|---|---|
| 주인공 | — | 해체 | 해요체 | 유아체 어휘 유지 |
| 라이벌 | 해체 | — | 해라체 | 최종장 화해 후 주인공에게 호칭 변화 |
| 연장자 NPC | 하게체 느낌의 해라체 | 해라체 | — | 장중체, 명령형 빈번 |

매트릭스에 없는 쌍의 대화가 번역 중에 나타나면 그 자리에서 임의 결정하지 말고 행을 추가하며 결정 근거를 적는다.

화자 오귀속은 실제 번역 결함으로 직결된다. 한 사례의 선행 영문판 분석에서는 다른 캐릭터의 대사가 주인공 대사로 오귀속되고, 질문-대답으로 주고받는 대화가 한 사람의 독백으로 병합된 결함이 확인됐다. 이처럼 영문 경유 중역에서 생기기 쉬운 손상이며, 화자 매핑 표 없이 번역하면 한국어판도 같은 함정에 빠진다.

### 2.5 산출물 체크리스트

본 번역 착수 전에 다음이 갖춰졌는지 확인한다.

- [ ] 컨텍스트 문서: 전체 구조·장면/화면 표·화자/문체 정보·반복 표현·핵심 검수 지점·번역 고려사항
- [ ] 타 언어 자료를 썼다면 「컨텍스트 전용, 직역 소스 금지」 한계 고지와 미확정 부록 분리
- [ ] 장면 ID 부여 또는 블록별 장면 주석 (불가 시 워크스루 병행 체제)
- [ ] 중복 원문 그룹 식별과 번역 전파·차이 검출 계획
- [ ] 화자 매핑 표 (확정/추정 구분, 추정은 `needs_review`)
- [ ] 캐릭터 쌍 존비어 매트릭스 (관계 변화 시점 포함)
- [ ] 반복 개그·시그니처 표현 목록의 용어집 등재

## 3. 용어집 운영

고유명사·시스템 용어·시그니처 표현의 표기를 번역 시작 전에 확정하고, 번역 전 과정에서 용어집을 단일 진실 원천으로 강제한다. 번역 도중 표기가 흔들리면 나중에 전수 수정해야 하므로, 용어집 구축은 본 번역보다 먼저 끝내는 선행 작업이다.

### 산출물 구성

용어집·말투 가이드는 **3층**으로 분리해 둔다. ① 고유명사(SSOT — 같은 원어 = 같은 역어), ② 캐릭터별 1인칭·어미·감탄사, ③ 상황별(말장난·문화 참조) 처리 메모. 이 3층은 초벌 번역과 검수 **양쪽의 필수 입력**이다. 초벌에만 주고 검수에 안 주면 검수자가 무엇을 기준으로 위반을 판정할지 흔들리고, 검수에만 두면 초벌이 같은 오류를 반복 생산한다.

| 산출물 | 층 | 내용 | 용도 |
|---|---|---|---|
| 고유명사 사전 (TSV/표) | ① | 원문–KO–참조역 대응 + 출처 + 비고. 인명/지명/마법/아이템/시스템 용어 카테고리 분류 | 번역 시 자동 우선 매칭, 검수 시 표기 대조 기준 |
| 감탄사·어미 매핑 | ② | 원문 의성어·감탄사·캐릭터 어미의 KR 대응 | 음역 방지, 캐릭터 말투 일관성 |
| 캐릭터 말투 가이드 | ② | 캐릭터별 1인칭·어조·시그니처 표현·예시 대사 | 화자 식별 후 톤 결정 (§2.4 매트릭스와 연동) |
| 상황별 처리 메모 | ③ | 말장난·문화 참조·번역 불가 항목의 원문 의도와 처리 근거 | 의역의 정당성 추적 (§5 「자동 검출 게이트」와 연동) |
| 출처·충돌 메타 문서 | — | 표기마다 「왜 이 표기인가」의 근거와 충돌 해소 결정 | 리뷰 단계에서 근거 추적 |

### 고유명사 표기 원칙

대원칙은 불변식 하나다: **같은 원어 고유명사는 어디에 나오든 같은 하나의 역어로 옮긴다.** 음역 후보가 여럿 갈리는 것은 예외가 아니라 필연이다. 외부 유저 번역 사례에서는 한 지명이 6종 표기로 갈린 채 배포된 일이 있다. 그래서 표기 확정은 번역 도중이 아니라 번역 전, 선(先)용어집으로 끝낸다.

1. **출처 우선순위를 명문화한다.** 공식 한국어판 표기 > 현행 정식 발매판 계승 표기 > 시리즈 자매 패치 관례 > 자체 신규 추정 순으로 권위를 정하고, 충돌 시 상위 출처를 따른다. 이미 공식·팬덤에서 강하게 정착한 표기는 별도 등급으로 고정해 임의 개명을 막는다.
2. **음역 정확성보다 정착 표기를 우선한다.** 외래어 표기법상 더 정확한 음역이 있어도 팬덤·공식판에 이미 정착한 표기가 있으면 그쪽을 채택한다(카벙클이 정확하지만 카방클 채택).
3. **절대 불변 용어를 따로 표시한다.** 시리즈 정체성에 해당하는 시그니처 표현은 어떤 문맥에서도 변경 금지로 등급을 매겨 두면, LLM이 문맥에 맞춰 임의로 바꾸는 사고를 막을 수 있다.
4. **충돌은 결정 기록과 함께 해소한다.** 후보 표기가 여럿이면 「후보들 → 채택 표기 → 결정 이유」를 표로 남긴다. 이후 같은 논쟁이 재발하지 않는다.
5. **신규 추정 표기는 `needs_review`로 마킹한다.** 선례 없는 캐릭터·아이템의 자체 추정 표기는 사전에 등재하되 검증 필요 플래그를 달아, 사람 검수 단계에서 별도 검토되게 한다.

### 일관성 검증의 층위 분담

불변식을 선언했으면 검증이 따라야 하는데, 순수 기계 검사만으로 닫히지 않는다. 한국어는 용어 뒤에 조사가 붙고 활용·생략이 끼어들어 단순 문자열 매칭이 깨진다. 검증 책임을 세 층으로 나눈다.

```
전 엔트리
  → [기계] 원어 등장 엔트리 추출 + 접미 허용 정규식 매칭
      → 매칭 성공: 통과
      → 매칭 실패: [LLM] 승인 표기 사용 여부 판정 (변형 음역·의역 드리프트)
          → 위반 확정: 해당 줄 재번역
          → 신규 표기·충돌: [사람] 승인 또는 판정
```

- **기계(스크립트) 책임 — 원문 기준 대조.** 검사 방향을 번역문이 아니라 원문에서 잡는다: 용어집의 원어가 등장하는 원문 엔트리를 전부 찾고, 그 엔트리의 번역에 승인 표기가 존재하는지 검사한다. 조사 결합 때문에 승인 표기의 정확 일치가 깨지므로, 1차 매칭은 접미 허용(승인 표기 뒤에 조사 후보 집합이 붙는 것을 허용하는 정규식)으로 한다. 역방향 검사, 즉 「승인 표기가 아닌 변형 음역이 어딘가 존재하는가」는 변형의 모양을 미리 알 수 없으므로 문자열 매칭으로 불가능하다는 점을 검사기 문서에 명시한다.
- **LLM 책임 — 깔때기의 아랫단.** 정규식이 「원어는 있는데 승인 표기 매칭 실패」로 좁혀 준 엔트리에 대해, 이 줄에서 용어가 승인 표기로 쓰였는지를 판정한다. 변형 음역(승인 표기와 닮았지만 다른 표기), 의역 드리프트(용어를 풀어 쓴 경우), 활용·생략이 끼는 대조가 LLM 몫이다. 정규식으로 후보를 좁힌 뒤 LLM이 판정한다. 전수를 LLM에 맡기면 비용과 누락이 같이 커진다.
- **사람 책임 — 결정.** 신규 표기의 승인, 표기 충돌의 최종 판정은 사람만 한다. 기계와 LLM은 위반 후보를 올릴 뿐 사전을 고치지 않는다.

### 지연 치환과 변수 삽입의 조사 문제

표기 일관성을 구조적으로 보장하는 대안으로 **지연 치환** 전략이 있다: 번역 단계에서는 고유명사를 원어(또는 키)로 유지한 채 번역하고, 마지막에 용어집으로 일괄 치환한다. 일관성은 자동으로 보장된다. 다만 대가가 있다. 치환 후 받침 유무에 따라 을/를, 이/가, 은/는이 어긋나므로 **조사 교정 패스가 필수**다. 치환 패스와 조사 교정 패스를 한 묶음으로 설계하지 않으면 일관성은 보장되지만 조사가 깨진다.

변수 삽입 텍스트(`{VAR:..}` 류 — 아이템명·캐릭터명이 런타임에 끼워지는 문장)는 같은 문제의 어려운 버전이다. 레트로 ROM은 런타임에 조사 선택 함수를 끼워 넣기 어려우므로 선택지는 셋이다.

- (a) **조사 회피 문형 재작성 — 기본값.** 「{item}을 얻었다」를 「획득: {item}」처럼 변수 뒤에 조사가 오지 않는 문형으로 고친다.
- (b) **양형 병기.** 「{item}을(를)」. 화면은 덜 깔끔하지만 공정이 단순하다.
- (c) **ASM 후킹으로 조사 분기 — 고급.** 마지막 글자 받침을 보고 조사를 고르는 코드를 훅으로 삽입한다. 비용이 크므로 (a)·(b)로 안 될 때만 간다.

어느 쪽을 고르든 선행 작업은 같다: **그 변수에 들어갈 수 있는 값의 전수 목록을 먼저 덤프한다.** 값 목록 없이 문형을 정하면 빠진 값에서 깨진다. 실측 사례로, 원문이 런타임에 조사를 조립하는 구조였는데 한글 매핑에서 그 조립이 깨져, 문형을 조사 없는 명사구로 재설계해 해소한 경우가 있다.

### 시리즈 공통 용어 상속

같은 시리즈를 여러 플랫폼에서 한글화할 때는 먼저 진행한 프로젝트의 용어집을 후속 프로젝트가 상속한다.

- 후속 프로젝트는 자매 프로젝트들의 사전을 병합해 출발점으로 삼고, 자기 작품에만 나오는 항목만 신규 추가한다. 같은 시리즈의 후속 작업에서는 고유명사 대부분이 상속이고 신규 추정은 소수에 그치는 경우가 많다.
- 자매 간 표기가 갈리면 다수 일치 + 출처 우선순위로 판정하고 결정표에 기록한다.
- 같은 원어가 작품마다 다른 캐릭터를 가리키는 경우가 있으므로, 상속 시 맹목 복사하지 말고 본편 컨텍스트와 대조한다(자매작의 의역명이 본편의 동명 캐릭터와 다른 존재였던 사례가 있다).
- 사전 파일을 그대로 복사해 쓰는 프로젝트가 생기면 원본·사본 관계를 메타 문서에 적어 두어야, 나중에 어느 쪽이 갱신본인지 혼란이 없다.

## 4. 번역 지침

### 제어코드 보존 규칙

추출 스크립트의 텍스트에는 줄바꿈·페이지 넘김·화자 지정·이벤트 트리거 등이 `{br}`, `{end}`, `{page}`, `{delay:1E}` 같은 인라인 마커로 들어 있다(사례 프로젝트 실표기는 대문자형도 있다). 마커 표기 규약의 정본은 `references/strategy/text-extraction.md` §4.4다. 보존 규칙은 다음과 같다.

- **마커는 이름과 개수 모두 원문과 1:1 대응시킨다.** 검증 스크립트가 원문과 번역문의 마커 집합을 자동 대조하게 한다.
- **누락이 곧 소프트락인 코드를 명시한다.** 대화 종료, 컷씬 종료, 이벤트 트리거 코드는 누락 시 게임 진행이 멈춘다. 용어집·지침서에 위험 코드 목록을 명기해 번역자가 우선 인지하게 한다.
- **줄바꿈 마커는 위치 조정이 허용되는 유일한 예외다.** 한국어 어순에 맞게 줄바꿈 위치는 옮길 수 있지만 개수는 화면 줄 수 제약 안에서 관리한다.
- **의미 불명 코드는 임의로 삭제·해석하지 않는다.** 질문 항목으로 분리해 `notes`에 남기고 사람 검수로 넘긴다.

### 길이 예산 의식

번역자는 문장을 쓰는 시점부터 길이 제약을 의식해야 한다. 한글 번역문은 일본어 원문 대비 바이트가 늘어나는 경향이 있다(사례 추정치 1.2~1.5배). 표시 제약은 게임마다 다르므로 프로젝트 지침서에 수치로 명기한다. 사례 값의 예:

- 16×16 고정폭 타일 게임: 대화창 약 14자 × 3줄, 메뉴 8~10자, 아이템명 8자 고정폭 패딩
- 가변폭 폰트 게임: 대화창 최대 4줄 × 가로 약 224px, 한글 1자 약 14~17px

지침의 핵심은 「짧고 명확하게 쓰되, 의미를 깎아 공간 문제를 덮지 않는다」는 점이다. 표시 제약(줄 폭·줄 수)은 번역 단계에서 자연스러운 축약과 줄바꿈 조정으로 먼저 해결한다. 그러나 저장 제약(고정 슬롯·포인터 영역·파일 크기)을 대량으로 넘는다면 번역문을 먼저 깎지 말고 재배치·공간 확장·폰트 폭 조정 같은 엔지니어링 경로를 검토한다. 의미 오해를 부르는 축약(조사 생략으로 다른 뜻이 되는 경우)은 금지하고 QA 검토 대상으로 올린다.

### 말투 일관성

- **번역 우선순위를 선언한다.** 사례에서 쓴 순서는 「원문 충실도 > 캐릭터성 보존 > 한국어 자연스러움 > 영문판 참조」다. 영문판은 컨텍스트 참고용일 뿐이며, 원문과 상충하면 원문을 따른다. 영문판에만 있는 추가 대사를 옮겨 오지 않는다.
- **캐릭터별 말투 규정을 표로 고정한다.** 1인칭, 어조(반말/존대/장중체/사투리/유아체), 시그니처 표현, 예시 대사를 캐릭터마다 정의하고, 캐릭터 쌍 단위 등급은 §2.4의 존비어 매트릭스를 따른다. 「반말 캐릭터의 돌발 존대」, 「3인칭 자기지칭 캐릭터의 1인칭 사용」, 「사투리 캐릭터의 표준어 평준화」가 대표적 훼손 패턴이므로 QA 체크리스트에 넣는다.
- **감탄사는 음역하지 않는다.** 원어 의성어·감탄사는 한국어 정서에 맞는 자연 대체어로 옮기되, 매핑 테이블을 만들어 작품 전체에서 같은 감탄사가 같은 표현으로 번역되게 한다. 단 시리즈 시그니처 감탄사는 예외로 고정 표기를 유지한다.
- **시점을 보존한다.** 원문이 3인칭 나레이션이면 번역도 3인칭을 유지한다. 영문판이 1인칭으로 바꾼 경우가 있어 영문 경유 번역에서 시점 오염이 자주 생긴다.
- **UI·설명문은 명료성을 우선한다.** 대사는 캐릭터성 우선, 메뉴·아이템 설명은 어미 통일(`~한다`, `~할 수 있다`)과 명료성 우선으로 문체 기준을 분리한다.

## 5. LLM·사람 분업

분량이 큰 프로젝트를 사람이 전부 직접 번역하는 것은 부담이 크고, LLM에만 맡기면 표기 흔들림과 오역이 남는다. 3겹 구조로 분업한다.

```
LLM 초벌 번역  →  자동 검증 스크립트  →  LLM 교차 리뷰  →  사람 최종 검수
(in_progress)      (기계 검사)          (needs_review)     (needs_human_review → complete)
```

### 단계별 책임

| 단계 | 담당 | 책임 | 산출 |
|---|---|---|---|
| 초벌 번역 | LLM | 용어집 우선 매칭, 캐릭터 말투 가이드 적용, 마커 보존, `notes`에 번역 근거 기록 | `ko`/`status`/`notes` 채워진 파일 |
| 자동 검증 | 스크립트 | 보호 필드 무결성, 마커 1:1 대응, 번역문 내 원어 문자 잔존, status 유효성, 빈 번역 검출, 인라인 마커 파싱 가능성, 용어 원문 기준 대조(§3) | 실패 항목 자동 플래그 + status 강등 |
| 교차 리뷰 | 초벌과 **다른** LLM 세션/모델 | 용어집 대조 재검, 말투 일관성, 오역·누락, 길이 예산 재확인. 기계적으로 판정 가능한 것만 수정하고 판단 필요 항목은 에스컬레이션 | 수정 반영 + `needs_human_review` 마킹 |
| 사람 최종 | 사람 | 말장난·문화 요소 의역 결정, 신규 표기 승인, 표기 충돌 최종 판정, 실기/에뮬레이터 화면 확인 | `complete/` 승인 |

### 컨텍스트 공급

LLM 초벌의 품질은 모델보다 컨텍스트 설계가 좌우한다. 청크마다 다음 패키지를 구성해 공급한다.

- **컨텍스트 패키지의 구성:** ① 해당 청크 원문에 등장하는 용어만 추린 용어집 발췌, ② 화자 매트릭스에서 이 장면 화자들에 해당하는 행, ③ 스토리 문서(§2)의 해당 장면 절, ④ 직전·직후 엔트리(장면 경계의 연속성 확보).
- **용어집은 동적 주입한다.** 전체 용어집을 통째로 프롬프트에 넣지 않는다. 용어집이 커지면 이 청크와 무관한 용어가 끼어들어 오역을 유발한다. 청크 원문을 스캔해 등장 용어만 골라 주입하는 것이 기본이고, 이는 생성 시점의 제약이다. 사후는 교정이 아니라 **검증**이다. 위반을 검출하면 일괄 치환으로 때우지 말고 해당 줄만 재번역시킨다.
- **확정역을 재주입한다(번역 메모리 원리).** 검수를 통과한 기존 번역에서 유사한 원문 줄, 같은 화자의 확정 대사를 골라 few-shot 예시로 주입한다. 용어와 말투를 동시에 고정하는 가장 싼 수단이다. 단 예시 자격은 검수 통과분에만 준다 — 미검수 번역을 예시로 쓰면 오류가 증식한다.
- **드리프트를 운영으로 막는다.** 청크는 장면 단위로 자르고, 매 청크에 용어집 발췌와 스타일 가이드를 다시 주입한다. 한 번 지시한 것은 세션이 길어지면 잊힌다. 전 배치 완료 후에는 전역 일관성 패스를 1회 돌려 배치 간 드리프트를 잡는다.
- **질문 큐를 운영한다.** 문맥이 불명한 줄은 번역하지 않고 질문 큐에 쌓는다. LLM의 추측 번역은 「모른다」가 아니라 「틀린 확신」으로 귀결되므로, 모르는 것을 모른다고 표시하게 만드는 장치가 출력 규약에 있어야 한다. 질문 큐는 사람 검수 또는 스토리 문서 갱신(§2.2)으로 해소한다.

### 운영 원칙

- **LLM 초벌에는 입력 자료의 적용 순서를 명시한다.** 권장 순서: ① 고유명사 사전 우선 매칭 → ② 감탄사·어미 매핑 적용 → ③ 캐릭터 가이드·존비어 매트릭스 또는 UI 문체 기준으로 톤 결정 → ④ 컨텍스트 문서로 장면/화면 확인 → ⑤ 미해결 표시 항목은 출력에 마킹. 자료를 순서 없이 던지면 사전보다 모델 사전지식이 이기는 사고가 난다.
- **초벌과 리뷰는 컨텍스트를 분리한다.** 같은 세션이 자기 번역을 리뷰하면 같은 오류를 같은 이유로 통과시킨다.
- **사람의 시간은 판단에만 쓴다.** 기계 검사 가능한 것(마커, 표기 일치, 잔존 원어)은 전부 스크립트와 LLM 리뷰가 걸러내고, 사람에게는 「뇌사라는 보스명이 한국어로 너무 직설적인가」 같은 감각 판단과 신규 표기 승인만 도달하게 한다.
- **번역 불가능한 말장난은 의역 + `notes`에 원문 펀치라인 기록**을 규칙으로 한다. 사람 검수자가 원문 의도를 모른 채 의역만 보고 승인하는 일을 막는다.
- **LLM이 절대 만지면 안 되는 필드를 계약으로 못 박는다.** 수정 가능 필드를 3개(`ko`/`status`/`notes`)로 한정하고 검증 스크립트가 git diff로 강제한 사례가 효과적이었다.

### 품질 평가 5축

검수(LLM 교차 리뷰와 사람 최종 검수 공통)의 평가 기준을 5축으로 분리해 적용한다. 축을 섞으면 리뷰가 「전반적으로 어색함」 같은 비실행 지적으로 흐른다.

1. **진행 정보 정확성.** 아이템 이름·획득 조건·퀘스트 지시·힌트의 오역은 플레이어를 실제로 막히게 하므로 최고 심각도로 다룬다. 게임은 보통 한 번만 플레이되며, 잘못된 진행 정보는 그 한 번의 플레이를 실패 경험으로 만들 수 있다. 문학적 결함보다 항상 우선 수정하고, 확신 없는 진행 정보는 사람 검수와 인게임 확인으로 닫는다.
2. **일관성 — 3축으로 분리해서 본다.** ① 용어 표기 일관성(같은 원어 = 같은 표기, §3 불변식), ② 번역 방식 일관성(같은 부류의 용어를 어떤 건 음역하고 어떤 건 완역하는 혼재), ③ 문체·존비어 일관성(같은 화자·같은 관계의 등급 흔들림). 한국 게이머의 번역 평가를 다룬 사례 연구에서 일관성이 최빈 지적 항목이었고, 그중에서도 세 축이 따로 깨진다. 묶어서 보면 원인을 못 잡는다.
3. **캐릭터 보이스.** 존비어 매트릭스·말투 가이드 대비로 검사한다. 이 축은 엔트리 단위 검사로 안 잡히므로, 장면순으로 통독하는 검수 패스를 따로 둔다(§2.3의 장면순 재배열이 여기서도 효자다).
4. **유창성.** 번역투, 조사 오류, 어색한 어순. 의미가 맞아도 읽는 맛을 깎는 축이다.
5. **기술 적합성.** 잘림·줄바꿈 파탄·렌더링 깨짐. 스크립트 검사로 1차를 잡되 최종 판정은 인게임 확인이다(§6, 빌드·검증 문서 연동).

축별 1차 검출 주체와 검사 시점:

| 축 | 1차 검출 | 최종 판정 | 주 검사 시점 |
|---|---|---|---|
| ① 진행 정보 정확성 | LLM 교차 리뷰 | 사람 + 인게임 플레이 | 배치 검수 + QA |
| ② 일관성 3축 | 스크립트(표기) / LLM(방식·문체) | 사람 | 배치 검수 + 전역 일관성 패스 |
| ③ 캐릭터 보이스 | LLM 교차 리뷰 | 사람 장면순 통독 | 메인 서사 배치 검수 |
| ④ 유창성 | LLM 교차 리뷰 | 사람 | 배치 검수 |
| ⑤ 기술 적합성 | 길이·마커 스크립트 | 인게임 확인 | 배치 검수 + 빌드 검증 |

원문에서 이탈했다는 이유만으로 자동 감점하지 않는다. 정보를 더 정확히 전달하는 변형, 말장난의 한국어 재현 같은 **근거 있는 이탈은 플레이어 평가에서 오히려 보상받는다**는 관찰이 사례 연구에 있다. 검수 기준은 「원문과 다른가」가 아니라 「근거가 `notes`에 있는가」다.

### 자동 검출 게이트

「오역·깨짐 = 블로커」는 검수자의 주의력에 맡기면 분량이 커질수록 새어 나간다. 명백한 오역을 사람 검수 **앞에서** 기계가 먼저 걸러 내고, 치명 오역이 남으면 배포 후보로 못 넘어가게 구조로 못 박는다. 사람의 눈은 기계가 못 잡는 판단에만 쓴다.

**원문↔역문 정렬 자동 대조.** §3의 용어 대조가 「표기가 SSOT와 같은가」만 본다면, 정렬 대조는 「원문의 요소가 역문에 대응 없이 사라지거나 변질됐는가」를 본다. 정규식·규칙으로 잡을 수 있는 패턴을 게이트화한다.

- **감탄사·신호어 누락.** 원문에 있는 감탄사·말버릇이 역문에 대응 없이 빠진 줄을 §3 감탄사 매핑(②층)과 대조해 찾는다.
- **1인칭·화자성 상실.** 캐릭터 1인칭이나 시그니처 어미가 가이드에 정의돼 있는데 역문에 없는 줄을 후보로 올린다(최종 판정은 「품질 평가 5축」 ③ 캐릭터 보이스의 장면순 통독).
- **동사 강도·의미 변질.** 「만지다 → 만나다」처럼 닮은 표현으로 의미가 미끄러진 줄, 긍정↔부정·수량·방향이 뒤집힌 줄을 규칙·LLM으로 잡는다.
- **빈 번역·원어 잔존·마커 불일치.** §5 자동 검증 항목과 같은 라인에서 돌린다.

규칙으로 확실히 거를 수 있는 것은 스크립트가 직접 잡고(빈 번역·마커·원어 잔존·감탄사 누락), 모양을 미리 알 수 없는 변질(동사 강도, 화자성 상실)은 정규식으로 후보를 좁힌 뒤 LLM이 판정한다 — §3 깔때기와 같은 분담 원리다.

**오역 심각도 분류.** 검출된 위반을 심각도로 나눠 게이트 통과 기준을 명확히 한다.

| 심각도 | 해당 | 처리 |
|---|---|---|
| 치명 (블로커) | 의미 왜곡, 캐릭터성 상실, 진행 정보 오류(「품질 평가 5축」 ① 진행 정보 정확성) | 0건이어야 배포 후보(`complete/`)로 넘어간다 |
| 중간 | 톤·분위기 손상, 유창성 결함 | 배치 내 해소를 원칙으로, 미해소 시 사유 기록 |
| 경미 | 미세 뉘앙스 차이 | 기록하되 통과 가능 |

치명 0건이 디렉토리 단계 전이(특히 `complete/` 입성)의 게이트 조건이다. 검출기가 치명으로 분류한 항목은 자동으로 `status` 강등 + `notes` 플래그가 붙어, §1 단계 전이 조건에 자연스럽게 걸린다.

**메모 유무를 검수 항목으로 둔다.** 번역 불가(말장난·조사 조립·문화 참조)는 의역하되 `notes`에 원문 의도와 처리 근거를 남긴다(§4 「말투 일관성」, §5 「운영 원칙」). 근거 없는 음역·의역은 검수자에게 오역으로 의심받으므로, **메모가 비어 있는 의역 줄을 검출해 위반 후보로 올린다.** 정렬 대조가 「원문과 다르다」고 표시한 줄에 메모가 없으면, 그것이 근거 있는 이탈인지 그냥 오역인지 기계가 구분할 수 없다 — 이때 메모 부재가 곧 게이트 실패 사유다(「품질 평가 5축」 마지막 문단의 「근거가 `notes`에 있는가」 기준의 기계 강제판).

## 6. 길이 검증 연동

길이 오버플로우를 패치 빌드나 실기 테스트에서 처음 발견하면 수정 비용이 크다. 번역 단계 안에 길이 검증 루프를 심어 조기 검출한다.

### 검증 루프

```
번역 작성 → 길이 검사 스크립트 → 초과 항목 분류
                                      ├─ 표시 제약 → 줄바꿈·문장 조정 → 재검사
                                      └─ 저장 제약 → 엔지니어링 에스컬레이션
                                                   (텍스트 재배치, 포인터 이동, 폰트 폭 조정)
```

- **검사 스크립트를 번역 자산 옆에 둔다.** 대화창 픽셀 폭·줄 수·고정 슬롯 길이를 게임별 수치로 설정한 오버플로우 검사기를 만들어, 배치 완료 시점마다 돌린다. 가변폭 폰트면 글리프 폭 테이블 기반으로 픽셀 단위 계산을 한다.
- **검사 기준은 두 층이다.** ① 표시 제약(줄당 자수/픽셀, 줄 수) — 번역 단계에서 즉시 교정 가능. ② 저장 제약(포인터 슬롯·페이로드 영역 바이트 한도) — 번역만으로 해소하려 들지 말고 텍스트 재배치(relocator) 같은 엔지니어링 가능성을 먼저 판정한다.
- **저장 제약 초과가 대량 발생하면 번역을 멈추지 말고 엔지니어링을 병행 선행시킨다.** 한글 번역이 원문보다 커져 초기 배치부터 페이로드 초과가 예상되면, 빈 공간 풀 할당·포인터 재배치·파일 재배치 같은 수용량 확장 경로를 본격 배치 전에 준비한다. 먼저 공간을 늘리는 경로를 시도하고, 번역 품질을 길이 때문에 깎는 것은 최후 수단으로 남긴다.
- **초과 항목 처리도 상태 기계에 태운다.** 검사 실패 항목은 자동으로 `notes` 플래그 + `status` 강등 처리해, 디렉토리 단계 전이 조건에 자연스럽게 걸리게 한다.
- 짧은 고정 슬롯(이름·메뉴)은 글자 수가 한두 자만 달라도 깨지므로, 주인공 이름처럼 시스템이 직접 그리는 문자열은 폰트 폭과 슬롯 크기를 별도 검증 항목으로 분리한다.

## 7. 배치 운영

스크립트 전체를 한 번에 번역하지 않는다. 추출 단위(region, 뱅크 테이블 등)를 기준으로 배치를 나누고, 쉬운 것부터 무거운 것 순서로 진행한다.

### 배치 분할 원칙

1. **첫 배치는 시범 배치다.** 이름·짧은 메시지처럼 작고 쉬운 파일들로 구성해, 번역 품질이 아니라 **워크플로우 자체**(이동 → 번역 → 검증 → 이동)와 용어집·검증 스크립트의 약점을 검증한다.
2. **난이도 오름차순으로 배열한다.** 권장 순서는 이름/짧은 메시지 → 전투·상태 메시지(말투 일관성 검증) → 시스템/상점/설명문(어미 통일 검증) → 일반 대사 → 메인 다이얼로그(캐릭터 말투 총동원)다. 비서사 게임은 메뉴·상태·튜토리얼처럼 자주 노출되는 UI를 먼저 잡고, 규칙 설명·엔딩·특수 이벤트로 확장한다. 앞 배치에서 확정한 용어·말투 규약을 뒤의 큰 배치가 그대로 쓴다.
3. **메인 서사 배치는 저장 순서가 아니라 장면 순서로 짠다.** §2.3의 재편성을 따른다. 메인 서사가 단일 region에 몰려 있어도 플레이 진행 단위로 잘라 배치하면, 화자·관계 컨텍스트가 연속된 채로 작업할 수 있다.
4. **배치마다 검증 포인트를 명시한다.** 「이 배치에서는 주인공 1인칭 일관성을 본다」, 「이 배치에서는 동사 어미 통일을 본다」처럼 배치별 중점 검사 항목을 계획서에 적어 두면 리뷰가 산만해지지 않는다.
5. **파일 크기를 균등하게 자른다.** 큰 region은 엔트리 수 기준(사례: 32엔트리)으로 분할해 한 파일이 한 작업 세션에 들어가게 한다. 무거운 배치는 파일 단위로 독립 진행한다(병렬 가능하되 한 파일은 한 작업자).

### 진행률 추적

- **배치 계획서를 번역 디렉토리에 둔다.** 파일별 엔트리 수·원문 글자 수·내용 요약·검증 포인트·예상 시간을 표로 만든 계획서가 곧 진행률 대시보드의 원본이 된다.
- **진행률은 디렉토리 집계로 읽는다.** 5단계 구조 덕분에 `ls`와 엔트리 `status` 집계만으로 「전체 N파일 중 complete K, needs_review M」을 스크립트로 뽑을 수 있다. 별도 트래커를 두지 않는다.
- **커밋 메시지에 배치 번호와 단계 전이를 적는다.** `batch 1: translate region_14 (5 entries)`처럼 남기면 git 로그만으로 작업 일지를 따로 쓰지 않아도 된다.
- **배치 진행은 살아있는 문서들의 갱신 주기다.** 배치가 끝날 때마다 스토리 문서(§2.2)·화자 매핑 표(§2.4)의 정정 사항을 반영하고, 질문 큐(§5)를 비운 뒤 다음 배치에 들어간다.
- 글자 수 기반으로 작업량을 추정한다. 시간 견적은 엔트리 수보다 원문 글자 수, 컨텍스트 난도, 검수 필요한 말투/용어 수에 비례한다.

## 8. 검수 대시보드 (도구)

§2.3(장면순 재배열)·§2.4(화자 식별)·§5(불일치 자동검출)를 사람이 한 화면에서 수행하는 내부 IP 도구. `retro-kr-patch-tools` MCP의 `review_dashboard(project_dir, action)`로 **프로젝트별** 기동한다(ROM 무관). 소스 `dashboard/`, 상세 `dashboard/README.md`.

### 원칙
- **편집 원본은 단일 JSON.** 대시보드·빌더가 같은 `translations_json`을 읽고 쓴다. OVERRIDES 같은 2차 소스를 두지 않는다(단일 진실).
- **논리 재배열만.** 대본 뷰의 순서 재정렬은 §2.3 그대로 읽기·작업 순서 재배열이며 ROM 물리 배치·raw 기준선을 건드리지 않는다.
- **자기소유 프로세스만.** `.krpatch/dashboard.pid`로 자기 인스턴스만 관리(광역 kill 금지).
- **코덱 선택.** 프로젝트 코덱(`codec_module`) 있으면 빌드와 100% 동일 로직으로 글리프 미리보기·바이트예산을 계산하고, 없으면 텍스트 전용으로 강등.

### 대본(스토리순) 뷰 — 순서 재정렬은 ROM 아키텍처로 갈린다
- **인라인-바이트코드형**(대사가 스크립트 명령 스트림에 끼워짐): `offset` 순 ≈ 실행순. 별도 포인터 해석 불요.
- **포인터-테이블형**(대사 뱅크 + 코드/테이블 포인터 참조, 다수 JRPG): `pointer` 모드가 ROM 전역을 훑어 각 문자열을 겨냥하는 최소 포인터 위치순으로 정렬(≈ 스크립트 순서). 미참조 문자열은 offset순 뒤로, 커버리지를 표기.
- **한계 명시.** 진짜 분기(전투 분기·조건부 이벤트)까지 따라가려면 ROM별 스크립트 VM 디스어셈블이 필요하며 범용 불가. 두 근사(offset/pointer) + 장면 그룹핑까지가 도구의 책임. 그 이상은 §2.3.2대로 워크스루 병행.
- **화자 묶음.** 이름표 휴리스틱(짧고 부호 없이 개행 종료·가시글자 ≤4자)으로 화자+본문을 대본처럼 묶어, 라인마다 원문을 병기해 문맥·오역을 대조한다(§2.4 화자 식별의 시각 보조).

### 검출·게이트
- 필터: 미번역 / 원문 한자·가나 잔류 / 표기 불일치(같은 원문 → 다른 번역, §5 자동검출) / 재검토 / 수동 / 예산초과.
- 저장 시 코덱으로 인코딩·예산을 즉시 검증하고, 초과·인코딩불가는 강제확인을 요구(§6 길이 검증과 동일 기준).

## 9. 장면-인지 2단계 번역 파이프라인 (도구)

문자열 1개씩 독립 번역(단순 배치)은 **화자·맥락·줄간 일관성**을 잃는다(같은 이름 다르게 표기, 반말/존댓말 혼선, 제어토큰 유실). `retro-kr-patch-tools` MCP `translate_pipeline(project_dir, action)`로 §2.1(문맥 덩어리)·§2.4(화자)·§5(불일치)·§6(길이 루프)을 한 파이프라인에 묶는다(ROM 무관, codec-driven). 소스 `dashboard/translate.py`.

### 3단계
- **① Context Aggregation.** 연속 문자열을 대화 창(장면)으로 묶어 한 프롬프트로. 장면 경계=offset gap 초과 또는 이름표(＿없는 `이름＠`) 경계, 하드캡 ~24줄. 창 = LLM에 주는 문맥 단위.
- **② Stage A (화자/맥락 초벌).** 창 전체를 JSON으로 주고 이름표로 화자 식별 → 화자별 말투 일관 + 자연스러운 초벌. 제어토큰 위치·개수 100% 보존 지시.
- **③ Stage B (자수 최적화 루프).** 줄별로 코덱 `encoded_len`으로 바이트예산·불가문자·제어토큰 멀티셋을 기계검증 → 위반 줄만 피드백 재시도(최대 3회). §6 검증 루프를 LLM 인루프로 구현. 3회 실패 시 baseline 유지(안전, 절대 파손 금지).

### 이름표 고정 (프로젝트별, game-independent)
- **glossary.json = 프로젝트별 이름/용어 SSOT.** `action=glossary`가 웹지식 + **ROM 실제 이름표 grounding**(빈도순 스캔)으로 부트스트랩. 게임마다 독립.
- **이름표는 자유번역 금지.** Stage A 출력을 glossary로 덮어써 화자명을 장면 전체에서 고정(첫등장-일관). 이게 없으면 일관성 도구가 오히려 불일치를 만든다(母親→어미/어멈 실사례).
- **예산-맞춤 별칭.** 이름표 예산은 극히 작다(예: 4바이트=한글 1자). `action=labels`가 예산 초과 이름표를 한국 통용 별칭으로 축약하되, **불확실하면 `label_questions.json`으로 분리해 사용자에게 확인**(임의 축약 금지). 확정 후 glossary 반영.
- **진짜 이름표 판별.** `이름＠` 형태(＠개행) + 다음 줄이 `＿`들여쓴 대사(인접성) + 시스템/색코드 잡음 배제. 카운터·메뉴 텍스트를 이름으로 오인해 고정하는 것을 막는다.

### 순서·게이트
- 권장 순서: glossary → labels →(사용자 확인)→ run. `run scope=all`은 전수(백그라운드), `run --start 0xADDR`는 창 단위(동기·검수용). `apply=True`만 JSON 갱신(백업 `.bak-pre-pipeline`).
- 리포트(`pipeline_report.json`)로 baseline↔pipeline 라인 대조·변경/실패 표기 → §8 대시보드에서 잔여 불일치·실패 줄 사람 검수.
- 적용 후 **제어토큰 무결성 전수 감사**(kr↔jp 멀티셋 일치)로 종단자 유실=스크립트 파손을 차단. 종단자(＄/＊/￥)는 필수, ＿(들여쓰기) 유실은 미관(예산 여유 시 복원).
