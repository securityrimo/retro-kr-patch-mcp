# 프로젝트 컨벤션

한글 패치 프로젝트의 저장소 구조, 도구 언어, CLI 설계, 의존성, 테스트, 원본 자산 취급에 대한 표준 규약이다. 플랫폼이 달라도 이 골격은 동일하게 유지한다. 플랫폼별 기술 세부는 `references/platforms/`의 해당 문서를 본다.

## 1. 저장소 레이아웃

새 프로젝트는 다음 역할 구획으로 시작한다. 매니페스트·패키지 디렉토리명은 언어별 표준으로 바꾸며, 아래 트리의 괄호 안은 Rust를 선택했을 때의 대응 예시다.

```
<프로젝트 루트>/
├─ <manifest>                  # 빌드 매니페스트 (Rust: Cargo.toml)
├─ <lockfile>                  # 의존성 잠금 파일 — 언어 생태계가 제공하면 커밋한다 (Rust: Cargo.lock)
├─ <agent-notes>.md            # 프로젝트 현황·구조·함정 요약 (예: AGENTS.md, CLAUDE.md)
├─ src/                        # 단일 패키지/앱일 때 (소규모 시작 기본형)
│   └─ commands/               # CLI 서브커맨드 구현 단위로 모듈 분리
├─ <재사용 라이브러리>/        # 분리 전환 후: 게임 비특이 재사용 라이브러리 (Rust: crates/)
├─ <게임 특이 앱>/             # 분리 전환 후: 게임 특이 CLI 바이너리 (Rust: apps/)
├─ assets/                     # git 추적 대상 에셋
│   ├─ fonts/                  # 한글 TTF (달무리, 갈무리, 둥근모꼴 등 — 라이선스 확인 후 커밋)
│   ├─ translations/           # 번역 JSON (진척 단계별 하위 디렉토리)
│   └─ translation_guide/      # 고유명사 표기, 용어 통일 가이드
├─ docs/                       # 기술 문서 (포맷 분석, 텍스트 엔진 분석, 계획)
│   └─ archive/                # 완료·폐기된 계획 문서 격리
├─ scripts/                    # 레거시·일회성 분석 스크립트 격리 (아래 참조)
├─ qa_screenshot/              # 에뮬레이터 QA 스크린샷 (이슈 추적용)
├─ roms/                       # 원본·출력 이미지 (.gitignore — §6 참조)
└─ out/                        # 빌드 중간 산출물 (.gitignore)
```

각 디렉토리의 역할과 규칙:

- **`docs/`** — 역공학 결과의 단일 진실 원천. ROM/디스크 구조, 텍스트 인코딩 테이블, 포인터 패턴, 렌더링 엔진 분석을 파일 단위로 분리해 쌓는다. 완료된 계획 문서는 `docs/archive/`로 옮겨 현행 문서와 섞이지 않게 한다. 에이전트 진입점 문서에는 현행 핵심 문서의 목록과 한 줄 요약만 둔다.
- **`assets/translations/`** — 번역 JSON의 본거지. 한 파일 = 한 텍스트 영역(뱅크, 테이블, 스크립트 파일)이 원칙이다. 번역 진척은 디렉토리 위치로 표현한다 — 5단계 구조와 단계 이동 조건은 `references/strategy/translation-workflow.md` §1 참조. 빌드는 `complete/`만 읽는다(미리보기 옵션 허용).
- **`scripts/` (레거시 격리)** — 초기 탐색 단계에서 작성한 Python 원라이너·프로브 스크립트는 여기에 격리한다. 패칭 기능을 전부 주 언어 파이프라인으로 마이그레이션한 뒤에도 분석 스크립트는 삭제하지 않고 이 디렉토리에 남겨 역공학 기록으로 보존한다. 단, **패치 빌드 경로에는 절대 끼워 넣지 않는다** — 빌드는 주 도구의 명령 하나로 끝나야 한다(Rust 사례: `cargo run`).
- **에이전트 진입점 문서** — 프로젝트 상태 표(단계별 완료/진행/미완료), 디렉토리 구조, 빌드·테스트 명령, 알려진 함정을 담는다. 파일명은 사용하는 도구의 관례를 따른다(예: `AGENTS.md`, `CLAUDE.md`). 분석 본문은 docs/로 보내고 여기서는 링크만 건다.

## 2. 도구 언어와 코드 구조

### 2.1 단일 주 파이프라인 원칙

패치 파이프라인(추출·변환·빌드)은 **하나의 주 언어/런타임과 하나의 재현 가능한 빌드 경로**로 묶는다. 모든 보조 도구까지 같은 언어여야 한다는 뜻은 아니다. 이미지 변환기, 압축 검증 도구, 에뮬레이터처럼 외부 프로그램이 더 신뢰할 수 있는 경우에는 버전·입출력·검증 기준을 고정해 주 파이프라인에서 호출한다. 주 언어 요건은 네 가지다.

1. **바이트 단위 정확성** — 패치는 결국 바이트 배열 조작이다. 암묵 변환 없는 바이너리 처리와 명시적 정수 폭으로 오프셋 계산 실수를 실행 전(컴파일·정적 검사) 단계에서 잡을 수 있어야 한다.
2. **빌드를 한 명령으로 재현한다** — 빌드 명령 하나로 전체 패치가 나온다. 외부 어셈블러·이미지 도구를 쓰는 경우에도 버전과 입력 파일을 고정하고, 출력 검증을 빌드 안에 넣는다.
3. **타입과 에러 처리** — 조용한 실패가 없어야 한다. 인코딩 누락·검증 실패는 즉시 에러로 드러난다.
4. **재현 가능한 실행과 프로젝트 간 재사용** — 단일 바이너리, 잠금 파일이 있는 스크립트 환경, 컨테이너 이미지 등 어떤 형태든 다른 환경에서 같은 명령으로 실행되어야 한다. ISA 인코더, 컨테이너 파서는 라이브러리 단위로 분리해 같은 플랫폼의 다음 게임에서 그대로 가져다 쓴다.

Rust는 선행 사례들이 검증한 선택지다. 바이너리 처리, 강한 타입, 단일 실행 파일 배포, property 테스트가 중요하고 프로젝트가 새로 시작한다면 Rust를 우선 검토한다. 이미 성숙한 Python/TypeScript/C# 도구체인이 있거나, 대상 포맷의 검증된 라이브러리가 특정 언어에만 있다면 그 언어를 택해도 된다. 이 문서의 구체 예시는 Rust 기준이므로, 다른 언어를 쓰면 같은 역할 구획을 해당 생태계의 표준 구조로 치환한다. Rust를 쓸 때 새 프로젝트는 에디션 2024를 우선 사용한다(기존 코드 포팅 시 2021 허용).

Python(또는 유사 스크립트 언어)은 탐색 단계의 일회성 분석(헥스 덤프, 빈도 분석, 시각화)에 쓰기 좋다. 주 빌드 경로에 남길 경우에는 임시 스크립트가 아니라 버전 고정, 테스트, 명확한 CLI를 갖춘 정식 도구로 승격한다. 승격하지 않은 스크립트는 `scripts/`에 격리하고 패치 빌드 경로에 끼워 넣지 않는다.

### 2.2 단일 패키지로 시작한다

프로젝트 초기에는 루트에 단일 패키지(Rust 사례: 단일 크레이트) 하나만 둔다. 한 사례는 이 형태로 끝까지 갔다 — `src/commands/`에 서브커맨드별 모듈, `src/font/`, `src/text/` 같은 도메인 모듈로 충분히 버틴다. 처음부터 멀티 패키지로 쪼개면 아직 경계가 불분명한 코드를 억지로 나누게 된다.

### 2.3 라이브러리 분리 시점

핵심은 **재사용 가능한 라이브러리와 게임 특이 앱의 분리**다(Rust 사례: `crates/` + `apps/` workspace). 다음 신호가 보이면 전환한다.

- **ISA 코드가 게임 비특이적으로 굳었을 때** — 디스어셈블러·인코더가 특정 게임 지식 없이 동작하면 ISA 단위 재사용 라이브러리로 뺀다. ISA 디코더·인코더(CPU 명령어)와 플랫폼 코드(ROM 헤더·매퍼·타일 포맷)는 어느 게임에서나 쓰는 라이브러리로, 특정 게임에만 쓰이는 코드(인코딩 테이블·포인터 주소·훅 위치)는 별도 앱으로 분리한다.
- **같은 플랫폼의 후속 게임이 보일 때** — 컨테이너 포맷(디스크 이미지, 압축), 그래픽 변환(planar 비트맵 등)은 다음 프로젝트가 의존성으로 끌어 쓸 수 있게 라이브러리로 분리한다. 다음 게임에서는 게임 특이 앱만 새로 쓰는 것이 목표다.

분리 기준은 한 문장으로: **재사용 라이브러리에는 게임 이름이 등장하지 않는 코드만, 게임 특이 앱에는 게임 특이 로직만** 둔다. 인코딩 테이블, 포인터 주소, 훅 위치는 전부 앱 소속이다.

## 3. CLI 설계

### 3.1 서브커맨드 동사 규약

CLI는 선언적 파서 기반 서브커맨드 구조로 만들고(§4의 "CLI 파싱" 요구 — Rust 사례: `clap` derive), 동사를 다음 계열로 통일한다.

| 계열 | 서브커맨드 예 | 역할 |
|------|--------------|------|
| **info** | `info`, `diff` | 이미지 메타데이터 출력, 두 이미지 비교 |
| **extract** | `extract-text`, `extract-fonts`, `scan-text`, `scan-pointers` | 원본에서 텍스트·글리프·포인터를 끌어내 JSON/PNG로 |
| **disasm** | `disasm`, `analyze` | 코드 영역 디스어셈블, 교차 참조 |
| **check** | `check-glyphs`, `check-overflow` | 번역 텍스트의 글리프 커버리지, 길이 초과 검증 (ROM 없이 dry-run 가능하게) |
| **build** | `build`, `patch` | 번역 JSON + 폰트 TTF + 원본 → 패치 이미지 생성 |

분석용 명령과 빌드용 명령을 한 바이너리에 합칠지 둘로 나눌지는 자유지만(한 사례는 분석 CLI와 패치 CLI를 별도 바이너리로 분리하는 구성을 계획했다), 동사 규약은 유지한다. 같은 동사를 보면 어느 프로젝트에서든 같은 일을 기대할 수 있어야 한다.

빌드 명령의 표준 시그니처는 다음 형태다.

```
patch build \
  --rom <원본 경로> \
  --translations assets/translations \
  --font assets/fonts/<폰트>.ttf \
  --output <출력 경로>
```

원본 경로는 항상 인자로 받는다 — 코드에 하드코딩하지 않는다(§6).

### 3.2 JSON 입출력 원칙

도구 간 데이터 교환은 전부 JSON으로 한다.

- 추출 명령의 출력 = 번역 JSON의 입력. 스키마 정본은 `references/strategy/text-extraction.md` §4.3이다 — 추출기가 채우는 필드(`table_id`/`entry_id`/`ptr_value`/`file_offset`/`raw_hex`/`text`/`flags`)에 번역 단계 필드(`ko`/`status`/`notes` — `references/strategy/translation-workflow.md` 참조)를 더한 구조로 떨군다. 스키마의 Rust 표현 사례:

```rust
#[derive(Serialize, Deserialize)]
struct TranslationEntry {
    entry_id: u32,
    ptr_value: Option<String>,
    file_offset: Option<String>,
    raw_hex: Option<String>,
    text: Option<String>,      // 원문 + 제어코드 표기 (추출기가 채움, 번역 단계 수정 금지)
    ko: String,                // 번역 (추출 시점에는 빈 문자열)
    status: String,
    notes: Option<String>,
    flags: Vec<String>,
}
```

- `diff`, `scan-pointers` 같은 분석 명령도 `--output <파일>.json` 옵션으로 기계가 읽을 수 있는 출력을 지원한다. 사람이 읽는 표 출력과 JSON 출력을 모두 제공하면 후속 자동화(LLM 번역 파이프라인, 검증 스크립트)가 쉬워진다.
- 제어코드는 JSON 안에서 `{br}`, `{end}`, `{delay:1E}` 같은 중괄호 토큰으로 표기해 번역자가 깨뜨리지 않게 한다(표기 규약 정본: `references/strategy/text-extraction.md` §4.4). 빌드 시 원문과 번역문의 제어코드 집합이 일치하는지 검증한다.
- 빌드 파라미터를 JSON으로 외부화할 때도 같은 타입 안전 원칙을 적용한다. 폰트·크기·윤곽선처럼 자주 바꾸는 표현 파라미터를 렌더 타깃이 여럿일 때 설정 파일로 빼는 패턴은 `references/strategy/font-strategy.md` §8을 따른다 — 스키마에 타입·기본값을 박아 로드하고(미지정 필드 무시 금지), 해결 순서는 CLI override → 설정 값 → 기본값으로 둔다. 메모리 주소·뱅크·인코딩 경계 같은 역공학 확정값은 외부화하지 않고 코드에 둔다.

## 4. 의존성 표준

의존성은 언어가 아니라 **기능 요구** 중심으로 정의한다. 새 프로젝트가 갖춰야 할 기능과 요구사항, 그리고 선행 프로젝트들이 검증한 Rust 사례는 다음과 같다. Rust를 쓸 경우 표의 크레이트를 우선 검토하되, 대상 포맷에 더 맞는 검증된 대안이 있으면 선택 근거를 문서화하고 테스트로 고정한다.

| 기능 | 요구사항 | Rust 사례 |
|------|----------|----------|
| CLI 파싱 | 서브커맨드·플래그를 선언적으로 정의. 커스텀 파서를 만들지 않는다 | `clap` (derive) |
| 에러 처리 | 앱 레벨은 컨텍스트를 붙인 전파, 라이브러리 레벨은 구조화 에러 타입 | `anyhow` (앱) / `thiserror` (라이브러리) |
| 직렬화·JSON | 번역 JSON, 분석 결과의 스키마 기반 직렬화 | `serde` + `serde_json` |
| 레거시 인코딩 변환 | Shift-JIS 등 디코딩. 원문 텍스트 식별의 첫 시도 도구 — 후보 영역의 디코드가 에러 없이 통과하면 SJIS 가능성이 높다 | `encoding_rs` |
| TTF 래스터라이즈 | 한글 TTF → 비트맵. 픽셀 크기 지정 후 임계값 처리로 1bpp/4bpp 타일 변환 | `fontdue` |
| PNG 입출력 | 폰트 그리드·글리프·타일맵 시각화 출력 | `png` |
| 체크섬·해시 | 원본 이미지 식별(해시 검증), 디스크 섹터 체크섬 재계산 | `crc32fast` / `md-5` |
| property 테스트 | 라운드트립 property 테스트 (§5) | `proptest` |
| 정규식 (필요 시) | 텍스트 오버플로우 검사 등 제어코드 토큰 파싱 | `regex` |

Rust 프로젝트의 시작 `Cargo.toml` 사례:

```toml
[dependencies]
clap = { version = "4", features = ["derive"] }
anyhow = "1"
thiserror = "2"
```

## 5. 테스트 정책

### 5.1 라운드트립 property 테스트 우선

컨테이너 파서(디스크 이미지, 압축 포맷, 폰트 데이터)의 첫 번째 테스트는 항상 **parse → serialize → 바이트 동일성**이다. 보유한 원본 이미지 전부에 대해 이 성질이 성립해야 다음 단계로 간다. 임의 입력에 대한 성질은 property 테스트 프레임워크로 검증한다(Rust 사례: `proptest`) — 인코딩/디코딩 쌍, 압축/해제 쌍, 주소 변환 쌍(논리↔물리)이 주요 대상이다.

```rust
proptest! {
    #[test]
    fn encode_decode_roundtrip(s in "[가-힣A-Za-z0-9 ]{1,64}") {
        let encoded = encode(&s)?;
        prop_assert_eq!(decode(&encoded)?, s);
    }
}
```

### 5.2 패치 바이트 단위 테스트

패치 적용 로직은 "어느 오프셋에 어떤 바이트가 쓰였는가"를 단위 테스트로 고정한다. 검증 가능한 쓰기 추적 계층을 두는 것이 표준이다 — 모든 ROM 쓰기가 통과하는 단일 추적 타입으로 구현한다(한 사례가 이 방식을 썼다).

- ROM 버퍼에 대한 직접 쓰기를 타입/API 차원에서 금지하고(읽기 전용 접근만 노출 — Rust 사례: `Deref`만 구현하고 `DerefMut`는 구현하지 않음), 모든 쓰기는 오프셋·바이트·의미 라벨을 받는 단일 쓰기 API로만 수행한다.
- 원본 바이트를 아는 위치는 쓰기 전에 기대값을 검증하는 쓰기 API로 쓴다 — 분석이 틀렸으면 패치가 즉시 실패한다.
- 파이프라인 끝에서 쓰기 영역 충돌 검사와 미추적 쓰기 검사를 돌린다.

테스트 파일은 모듈 옆에 분리 배치하는 컨벤션을 따른다(Rust 사례: `*_tests.rs`).

### 5.3 통합 테스트

통합 테스트 디렉토리(Rust 사례: `tests/`)에는 전체 파이프라인 통합 테스트를 둔다. 원본 이미지가 필요한 테스트는 로컬 경로에 파일이 없으면 skip하도록 작성한다(§6 — CI나 다른 환경에는 원본이 없다). 검증 항목:

- 풀 빌드가 에러 없이 완주하고 출력 크기·헤더·체크섬이 기대값과 일치하는가
- 번역 JSON 전체에 대해 글리프 커버리지 검사(`check-glyphs`)가 통과하는가 — 이 검사는 원본 없이도 돌게 만든다
- 텍스트 길이 초과 검사가 통과하는가

에뮬레이터 실행 검증(부팅, 텍스트 표시, 진행)은 자동 테스트로 다 덮을 수 없으므로 `qa_screenshot/`에 스크린샷을 쌓아 이슈 단위로 추적한다. 최종 검증은 정확도 높은 에뮬레이터와 가능하면 실기에서 한다.

## 6. 원본 자산 취급

### 6.1 저작물 비커밋 원칙

원본 ROM·디스크 이미지, 그리고 그것에서 파생된 패치 출력물은 **절대 저장소에 커밋하지 않는다**. 둘 다 저작권 보호 대상이다. `.gitignore`에 처음부터 박는다.

```gitignore
roms/
out/
*.gg
*.sfc
*.bin
*.cue
*.d88
*.iso
```

커밋해도 되는 것과 안 되는 것의 경계:

| 커밋한다 | 커밋하지 않는다 |
|----------|----------------|
| 번역 JSON (원문 텍스트 포함 — 인용 수준) | 원본 ROM/디스크 이미지 |
| 추출한 인코딩 테이블, 포인터 주소 문서 | 패치 적용된 출력 이미지 |
| 라이선스가 허용하는 한글 TTF | 원본에서 통째로 덤프한 바이너리(폰트 영역, 그래픽 영역 원본) |
| 자체 작성 코드·문서·스크린샷 | 메모리 덤프 등 대용량 분석 산출물 |

배포물은 패치된 이미지가 아니라 **BPS/xdelta 등 차분 패치 파일**로 만든다. 차분 패치는 원본 없이는 무의미하므로 배포 가능하다.

### 6.2 로컬 경로 주입 방식

원본 이미지의 위치는 코드에 하드코딩하지 않고 외부에서 주입한다.

- **CLI 인자가 기본** — 모든 명령이 `--rom <경로>` 형태로 원본 경로를 받는다. 문서·CLAUDE.md의 예시 명령에는 `roms/` 하위의 관례적 파일명을 쓴다.
- **관례적 배치** — 원본은 `roms/`(gitignore됨)에 No-Intro/Redump 표준 파일명 그대로 둔다. 파일명을 바꾸지 않아야 해시 검증과 문서의 예시 명령이 맞아떨어진다. 보존 DB 등재본이 없으면 자체 해시를 기준으로 고정한다.
- **해시 검증** — 빌드 도구는 시작 시 입력 이미지의 CRC32/MD5를 알려진 원본 해시와 비교하고, 불일치하면 명확한 에러와 함께 중단한다. 어떤 리비전의 원본을 기준으로 분석했는지가 모든 오프셋의 전제이기 때문이다. 기대 해시값은 코드 상수 또는 커밋되는 매니페스트 JSON으로 관리한다.
- **테스트에서의 주입** — 통합 테스트는 관례 경로(`roms/...`)를 먼저 찾고, 환경 변수로 재정의를 허용하며, 파일이 없으면 해당 테스트를 skip한다. 원본이 없는 환경에서도 테스트 스위트가 실패로 끝나면 안 된다.
