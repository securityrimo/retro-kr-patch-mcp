# 디버깅과 이슈 처리

QA에서 올라온 버그를 재현 고정 → 격리 → 메커니즘 확정 → 수정 → 회귀 편입까지 끌고 가는 방법론이다.
QA 라운드 사이클의 운영(라운드 묶음, 상태 분류, 종료 판정)은 `references/strategy/build-and-verify.md` §6을,
디버거 조작의 플랫폼별 세부(주소 미러링, 레지스터 맵)는 `references/platforms/` 해당 문서를 본다.
본문 사례의 빈도 기반 일반화("흔하다", "전형")는 SKILL.md의 모집단 고지를 따른다. 타 엔진 계열에서는 경향을 가설로 취급한다.

## 1. 디버깅 루프

### 골격 — 과학적 디버깅

디버깅은 Zeller의 과학적 디버깅 루프를 따른다. **관찰**(사실만 적는다. "무엇이 보였나"와 "왜 그런 것 같나"를 분리한다) → **가설**(관찰과 모순 없는 것 하나) → **예측**(가설이 참이면 반드시 관찰될 것을 실험 전에 적는다) → **실험**(예측을 가장 싸게 판정할 최소 실험) → 적중하면 가설을 구체화하고, 빗나가면 기각하고 다음 가설로. 결함을 코드(또는 데이터) 수준에서 지목할 때까지 반복한다.

증상과 원인 사이에는 **결함→감염→실패** 사슬이 있다. 결함(코드·데이터의 잘못)이 실행 중 상태를 감염시키고, 감염이 전파되어 실패(화면의 증상)가 된다. 예: 화면 깨짐(실패) ← VRAM의 잘못된 바이트(감염) ← 그 바이트를 쓴 코드(결함). 디버깅은 실패에서 감염 사슬을 거슬러 **최초 감염 지점**을 찾는 일이다. 증상에서 가까운 곳(렌더러)부터 의심하기 쉽지만, 결함은 보통 훨씬 상류(데이터 파이프라인)에 있다(3절).

전체 공정의 기억술은 Zeller의 TRAFFIC이다: Track(이슈로 기록) → Reproduce(재현) → Automate(재현 자동화·단순화) → Find origins(감염 역추적) → Focus(유력 지점 집중) → Isolate(격리) → Correct(수정). 아래 표준 순서는 이를 한글 패치 작업에 맞게 구체화한 것이다.

### 한 사이클의 표준 순서

```
재현 고정 → 증상 기록 → 가설 수립 → 격리 실험 → 메커니즘 체인 확정 → 수정 → 회귀 검증
```

1. **재현 고정** — 레트로 한글화에서 최소 재현의 실체는 3종 세트다: **버그 직전 세이브스테이트 + 짧은 입력 시퀀스 + ROM/패치 버전 명시**. 재현 없는 수정은 "고쳐졌다"는 판정 자체가 불가능하다. 좋은 재현의 4요건:
   - **최소성** — 불필요한 진행·입력을 제거해 한 사이클을 짧게 만든다. 이후 모든 실험의 단가가 여기서 정해진다.
   - **완전성** — 그 3종 세트만으로 다른 사람·다른 세션에서도 재현된다.
   - **검증성** — 줄이는 도중 같은 증상의 **다른 버그**로 갈아타지 않았는지 확인한다. 갈아타는 것이 최악이다.
   - **결정성** — 매번 같은 결과. 비결정적이면 먼저 결정화(RNG 고정 지점, 입력 무비)부터 한다.
2. **증상 기록** — 스크린샷·메모리 덤프 등 사실만. 추정은 별도 절에 적는다.
3. **가설 수립** — 3절의 증상 사전으로 1차 용의자 순위를 잡는다.
4. **격리 실험** — 2절의 기법으로 가설을 판정한다. 제거 실험이 가능한 가설은 분석보다 제거부터(2절 ⑥).
5. **메커니즘 체인 확정** — 결함→감염→실패를 단계별 인과로 적는다. 예: "공백이 잘못된 타일 코드로 인코딩됨 → 일반 폰트 문자 경로로 처리 → 빈 데이터가 VRAM에 DMA → 배경 타일이 검은색으로 덮임".
6. **수정** — 확정된 결함 지점만 고친다. 한 번에 한 변경.
7. **회귀 검증** — 수정 전 재현 절차로 실패를 한 번 더 확인하고, 수정 후 통과를 확인한 뒤, 그 재현을 회귀 묶음에 영구 편입한다(5절).

### 기록 규율 — 로그북

가설·실험·결과는 머릿속이 아니라 글로 남긴다. 글로 남기지 않으면 같은 가설을 반복 검증하고, 관찰과 추측이 섞인다.

- **추정 원인을 사실처럼 기록하지 않는다.** 모든 원인 서술에 (가설)/(확정) 태그를 붙인다. 확정의 기준은 격리 실험 통과다.
- **아는 것/모르는 것 분리표** — 착수 형식의 한 사례: "확인된 사실 N개"와 "모르는 것 Q1~Qn"을 표로 분리하고, 각 미지수마다 검증 방법을 함께 적는다. 이 표가 실험 계획이 된다.
- **기각된 가설도 기각 증거와 함께 보존한다**(5절 스키마). 재진입을 막고, 진범 발견 시 과거 실패 전체가 한 번에 설명된다.

### 완성 판정

**"이 수정이 버그를 없애는 메커니즘을 한 문장으로 설명할 수 있는가."** 설명하지 못하면 미완성이다. NOP 하나 넣었더니 안 깨지더라 식의 우연 수정은 다른 경로·실기에서 재발한다.

타이밍·현상만 보고 세운 가설은 헛돈다. 증상(타이밍·표면 현상)에 맞춰 패치하지 말고, 원인의 메커니즘을 disasm·실행 추적으로 먼저 확정한 뒤 그 결함을 고친다. 한 사례에서는 의도된 기능인 줄 알았던 타이밍 패딩이 사실 불필요한 것이어서 패딩 제거가 정답이었다 — 현상에 맞춘 보정 패치로는 영영 안 풀렸을 문제다. "낭비처럼 보이는 코드"가 실은 기능일 때(잔상 클리어 등)와 정반대 방향이므로, 어느 쪽이든 메커니즘 확정 전에는 추가도 제거도 하지 않는다.

## 2. 격리 기법 카탈로그

### ① 격리 빌드 (ablation)

의심 변경을 부분 집합으로 나눠 각각만 적용한 빌드를 만들어 비교한다.

- **통과하는 최대 조합 찾기** — CD hang 사례: 폰트만 적용/재압축만 적용/텍스트 패치 적용의 3종 격리 빌드로 앞 둘이 작동하고 텍스트 패치만 hang임을 확인, 가설 6개를 동시에 배제했다. 실패 조합을 좁히기보다 "어디까지 넣어도 통과하는가"를 찾는 쪽이 가설을 한꺼번에 자른다.
- **단일 변수 통제** — 포트레이트 글리치 사례: 20여 개 빌드로 파이프라인 단계별 → 텍스처 내부 → 개별 폰트 파일 → 단일 변수 순으로 좁혀, 특정 폰트 파일의 크기 단독이 범인이고 내용물·포맷·비율은 전부 무관함을 확정했다. 마지막 단계는 반드시 변수 하나만 다른 두 빌드의 비교여야 한다.
- **빌드마다 식별 이름을 부여하고 결과를 표로 누적한다.** 각 ablation 빌드에 어떤 변수를 끈/켠 빌드인지 드러내는 이름을 붙이고, 빌드별 결과(통과/실패·증상)를 한 표에 쌓는다. 위 사례처럼 빌드가 20여 개 규모가 되면, 명명·기록 없이는 어떤 조합을 이미 시험했는지 추적이 무너져 같은 빌드를 반복하거나 결론을 뒤섞는다.
- **함정: ablation 결론은 베이스 구현에 종속된다.** 같은 사례에서 1차 결론(디렉토리 구조 원인)은 베이스 빌드 방식을 바꾸자(이미지 전체 재빌드 구현) 뒤집혔다. 베이스가 바뀌면 기존 ablation 결론을 재검증한다.
- **작동하는 참조 구현의 존재가 강력한 격리 도구다.** 같은 게임의 선행 패치(영문판 등)가 같은 접근으로 정상 동작하면, 접근이 아니라 우리 구현의 버그다. 산출물을 바이트 레벨로 비교해 차이를 격리한다.

### ② 이분 탐색 (델타 디버깅)

동작 상태와 실패 상태를 양 끝으로 놓고 차이를 절반씩 적용해 O(log N)으로 격리한다. 적용 단위는 큰 것부터.

- **자산 단위 이분** — 패치를 구성하는 자산(폰트/인코딩 테이블/텍스트 블록/훅 코드)을 절반씩만 적용한 빌드로 어느 자산이 범인인지 가른다. ①의 격리 빌드를 체계화한 형태다.
- **텍스트 블록 이분** — 범인이 텍스트로 좁혀지면 번역 데이터를 블록 절반씩 원문으로 되돌려 범인 항목 하나까지 좁힌다. 항목이 특정되면 그 항목의 인코딩 바이트를 직접 검사한다.
- **바이너리 디프 이분** — 원본과 패치본의 바이트 차이 영역을 절반씩 적용한다. 의미 단위가 깨지기 쉬운 최후 수단이다.

적용 조건 3가지를 먼저 확인한다.

| 조건 | 의미 | 깨졌을 때 |
|------|------|----------|
| 단조성 | 실패 유발 부분집합을 포함한 상위집합도 실패 | 결과가 널뛰면 이분 결과를 신뢰할 수 없다 |
| 비모호성 | 원인이 하나 | 버그 2개가 공존하면 수렴하지 않는다(6절 증상 겹침) |
| 일관성(판정 가능성) | 중간 구성이 성공/실패로 판정된다 | "부팅조차 안 됨"이 빈발하면 효율이 급락한다 |

전제는 재현이 결정적이고 자동 판정 가능할 것. 함정: 변경 A·B가 **함께 있어야** 실패하는 상호작용이면 단순 이분이 수렴하지 않는다. 한쪽을 고정하고 나머지를 쪼개는 2차원 탐색으로 전환한다. 중간 구성의 빌드 비용도 미리 계산한다(빌드가 느리면 ①의 굵은 격리 빌드가 더 싸다).

### ③ 원본·선행 패치본 비교 (디프 격리)

비교 축은 셋이고, 결함→감염→실패 사슬의 각 층에 대응한다.

- **바이트 디프** — ROM·파일 단위. 증상 영역을 패치가 건드렸는지(패치 버그인지 원본 동작인지) 판정한다. "버그 아님" 확정의 핵심 도구다(5절).
- **상태 디프** — RAM·VRAM·CRAM(팔레트) 덤프 비교. 감염 지점을 특정한다.
- **트레이스 디프** — 실행 로그 비교. 결함 코드를 특정한다(4절).

- **최초 분기점 원칙**: 정상 케이스와 버그 케이스의 디프에서 의미 있는 것은 최초 분기점 하나다. 그 이후의 차이는 전부 파생 결과라 분석 가치가 낮다.
- **노이즈 정규화**: 프레임 카운터·RNG·사운드 등 무관한 차이를 먼저 제거하거나 비교 열에서 빼야 분기점이 보인다.
- 글자 잔상 사례: CRAM(팔레트 RAM) 디프 한 번으로 팔레트 가설을 즉시 제거하고 다음 가설로 넘어갔다. 상태 디프는 가설 기각이 가장 싼 기법이다.

### ④ 레이어 토글

에뮬레이터의 BG/스프라이트/윈도우 레이어 표시 토글로 증상이 어느 레이어에 있는지 즉시 가른다. 비용이 0에 가까우므로 그래픽 증상에서는 항상 먼저 한다. 레이어가 갈리면 3절 결정 트리의 입력이 된다.

### ⑤ 세이브스테이트 이분

감염 시점을 시간축에서 이분한다. 증상 한참 전 스테이트와 직전 스테이트 사이를 절반씩 좁혀 "어느 시점부터 상태가 이미 감염되어 있었는가"를 찾고, 그 직전 구간만 트레이스·브레이크포인트 대상으로 삼는다. 프레임 어드밴스와 결합하면 실험 반복 비용이 0에 가깝다.

### ⑥ 가장 싼 실험 우선

가설마다 판정 비용이 다르다. **제거 실험이 가능한 가설은 분석보다 제거부터 한다.** 한 사례에서는 HITL 관측 5건과 전부 정합하던 VMAIN(VRAM 증가 모드) 가설이, 훅을 뺀 빌드 1개로 즉시 기각됐다. 정황이 아무리 완벽해도 제거 실험을 통과하기 전에는 확정하지 않는다.

## 3. 증상→원인 사전

선행 사례에서 실측된 증상별 1차 용의자다. 공통 원칙: **첫 용의자는 렌더링 코드가 아니라 데이터 파이프라인이다.** 렌더링 버그처럼 보이는 증상의 다수가 번역 데이터·빌드 산출물의 결함이었다.

### 그래픽 깨짐 — 3분류 결정 트리

```
색만 이상 (모양·배치는 정상) ──▶ 팔레트  — 팔레트 RAM 값 자체인지, 타일맵 속성 비트(팔레트 선택)인지 구분
모양 이상 (색·배치는 정상)   ──▶ 타일    — 인코딩 포맷 착오, 전송 위치, 글리프 잘림
배치 이상 (타일 자체는 정상) ──▶ 타일맵  — 인덱스 계산, 주소 계산
```

추가 단서: 깨짐이 **일정 주기로 반복**되면 스트라이드(행 폭·타일 크기·바이트/픽셀) 불일치를 시사한다. 주기를 바이트로 환산하면 어느 변환 단계의 폭이 틀렸는지 좁혀진다.

### 증상별 표

| 증상 | 1차 용의자 | 확인 방법 | 사례 |
|------|-----------|----------|------|

| 특정 루트·분기에서만 발생 | 인덱스 의존 버그(포인터 개수 불일치, 시작 주소 ±1, 인덱스 0 미처리) | 해당 루트가 쓰는 인덱스 경로를 정상 루트와 대조 | 포인터 개수 191 vs 192 + 시작 주소 ±1 + 인덱스 0 미처리의 3중 결함 사례 |
| 크래시(코드 버그처럼 보임) | **데이터 파이프라인 손상이 코드 버그로 위장** | 크래시 주소 분석 전에 입력 데이터·빌드 산출물 무결성 검사 | 격리 빌드로 코드/데이터를 분리하니 번역 인덱스 중복이 파일을 파괴한 것이었던 크래시 사례 — 크래시 주소만 보면 코드 버그 같지만 근본은 빌드 파이프라인 |
| 에뮬레이터에서는 정상, 실기에서 깜빡임·깨짐 | VBlank 예산 초과(전송량이 수직 귀선 시간을 넘김), 미초기화 의존 | 프레임당 VRAM 전송량을 계측해 VBlank 예산과 비교 | — 에뮬레이터는 타이밍에 관대해 초과 전송을 그냥 통과시키는 경우가 흔하다(4절 계층 분리) |

### 한글 패치 특유의 상습 지대

- **VBlank 예산 초과** — VWF·자모 조합 렌더링은 원문 대비 타일 전송량을 늘린다. 원본 엔진이 가정한 프레임당 전송 예산을 넘기면 실기에서만 증상이 난다.
- **동적 타일 영역 공유 충돌** — 한글 글리프를 동적으로 올리는 VRAM/작업 RAM 영역은 게임의 다른 시스템과 공유될 수 있다. "빈 영역"은 관측한 그 시점에 비어 보였을 뿐이다. 전 구간에서 비었는지는 read/write 브레이크포인트로 확정한다.
- **패딩·종료자 처리** — 0x00 계열 패딩이 문자로 해석되는 결함은 서로 다른 두 기종에서 동형 재발했다(위 표). 기종 무관 패턴으로 의심하고, 새 게임에서도 패딩 바이트가 디코더의 어느 경로를 타는지 초기에 확인해 둔다.

## 4. 에뮬레이터 디버깅 전술

필요 요건: read/write/exec 브레이크포인트, 메모리 뷰어, 트레이스 로거, 세이브스테이트+프레임 어드밴스. 갖춘 에뮬레이터를 플랫폼별로 확보한다(사례: Mesen2, GDB 원격 연동 등). 공개 에뮬레이터가 필요 요건을 못 채우면 자체 개조로 메우기도 한다(계측·자동화 인터페이스가 없는 에뮬레이터를 직접 개조해 쓴 사례가 있다).

### 브레이크포인트 레시피

| 목적 | 레시피 |
|------|--------|
| 깨진 화면의 범인 코드 직행 | 깨진 VRAM/타일맵 주소를 특정 → 그 주소에 **write 브레이크포인트** → 마지막으로 쓴 코드가 감염 지점. 콜스택을 거슬러 텍스트 출력 루틴까지 |
| 텍스트 해석 루프 발견 | ROM의 대사 바이트에 **read 브레이크포인트** → 히트한 곳이 해석 루프 |
| 폰트 디코딩 루틴 발견 | ROM의 폰트 데이터에 **read 브레이크포인트** |
| 호출 데이터 확인 | 렌더링 진입점·포인터 룩업에 **exec 브레이크포인트** → 히트 시 레지스터(텍스트 인덱스·포인터 값) 판독 |

주소 표기가 코드의 실제 접근 주소와 같은지 확인한다. 뱅크 미러링으로 브레이크포인트가 안 걸리는 함정이 있다(플랫폼 문서 참조).

보조 도구:

- **메모리 뷰어의 접근 색상 표시** — 어느 영역을 누가 읽고 쓰는지 한눈에 보인다. 영역 공유 충돌 의심 시 첫 관측 수단.
- **코드/데이터 로거(CDL)** — 처음 실행된 명령만 필터해 "이 조작이 어떤 새 코드를 깨우는가"를 본다.
- **라벨/심볼 임포트** — 확정된 루틴에만 이름을 붙여 디버거에 임포트한다. 가설 이름 금지(6절).

### 트레이스 디프

같은 시작점(동일 세이브스테이트)에서 정상 케이스와 버그 케이스를 각각 트레이스 로깅하고 diff로 **최초 분기점**을 찾는다. 분기점 명령에 브레이크포인트를 걸고 재방문해 분기 이유를 판독한다.

- 함정: 프레임 카운터·RNG·사운드가 디프를 노이즈로 채운다. PC 열만 비교하는 등 정규화한다.
- 패치본은 코드 주소가 이동했을 수 있다. 주소가 아니라 호출 시퀀스 수준으로 비교한다.
- 로그 폭발. 브레이크포인트로 구간을 좁힌 뒤에 로깅을 켠다.

### 커스텀 계측

기성 디버거로 부족하면 에뮬레이터를 개조해 원하는 이벤트만 로깅한다. 한 사례에서는 커스텀 에뮬레이터 빌드(VRAM write+캐시 미스+텍스트 진입 로깅)로 일문판/한글판 트레이스 디프를 만들어 WRAM 공유 충돌을 확정했다. Lua 스크립팅·MCP 연동으로 같은 계측이 가능하면 개조보다 먼저 시도한다(`references/strategy/build-and-verify.md` §4).

### 입력 무비 재현

시작 상태+프레임별 입력의 기록·재생으로 재현을 자동화한다. 전제는 결정적 에뮬레이터. 재생 중 임의 프레임에서 정지해 디버거로 전환할 수 있어 재현과 분석이 한 파이프라인이 된다. desync 함정: ROM 체크섬·에뮬레이터 버전·코어 설정에 종속되고, 텍스트 길이 변화만으로 타이밍이 어긋난다. **버그 지점 인근만 짧게** 기록하고, 정확한 타이밍 의존보다 버튼 연타식 기록이 강건하며, 에뮬레이터 버전을 무비와 함께 고정 기록한다.

### HITL 요청서

에이전트가 직접 디버거를 못 돌리는 구간은 사람에게 관측을 요청한다. 왕복을 줄이는 형식: **브레이크포인트 주소 + 확인할 값(레지스터·메모리) + 결과 분기별 다음 행동**을 한 요청서에 담는다. 한 사례에서는 이 형식으로 1회 왕복에 해결됐다. 요청서 디렉토리에는 결과 열(확정/기각/무관)을 둔 인덱스를 유지한다.

### 에뮬레이터 계층 분리

역할별로 에뮬레이터를 분리한다. **디버그용**(트레이스·브레이크포인트 정밀), **자동화용**(headless·스크립팅), **사용자 환경 검증용**(보급률 높은 일반 에뮬레이터). 여러 작품을 한 플랫폼에서 진행하며 이렇게 운영한 사례가 있다. 정밀 에뮬레이터의 에러 메시지는 범위 밖 접근(OOB)을 즉시 노출하지만 실기는 침묵하다 크래시한다. 에뮬레이터는 타이밍·미초기화에 관대하므로(VBlank 초과 전송이 대표) 배포 전 실기 또는 정밀(사이클 정확) 에뮬레이터 검증을 거친다. 정적 검증 통과가 에뮬레이터 동작을 보증하지도 않는다. 빌드 산출물은 반드시 부팅까지 확인한다.

## 5. 이슈 처리 워크플로우

QA 라운드의 사이클 운영(라운드 문서, 상태 4분류, 종료 판정)은 `references/strategy/build-and-verify.md` §6이 정본이다. 여기서는 개별 이슈 하나의 생애를 다룬다.

### 이슈 스키마

이슈 블록 하나에 다음을 기록한다.

- **ID** — 라운드 내 알파벳(A, B, C, …) 또는 프로젝트 전역 ID.
- **증상 스크린샷** — 파일명에 증상을 서술형으로 그대로 적는다("NPC-2번째-대화-색깔만-바뀌어-나옴.png"). 파일명이 이슈 제목이 되고 분석의 입력이 된다.
- **재현 조건** — 1절의 3종 세트(스테이트+입력+버전).
- **추정 원인(가설)과 확정 원인(확정)의 절 분리** — 태그 없이 섞지 않는다.
- **기각된 가설 표** — 가설·기각 증거·기각 일자. 기각 가설 4개를 증거와 함께 표로 보존해 재진입을 막은 사례가 있다.
- **상태** — 수정 완료/재확인 필요/버그 아님/범위 밖(build-and-verify.md §6의 4분류).

### 개별 이슈의 생애

```
보고 → 재현 압축 → 격리 → 메커니즘 확정 → 수정 → 회귀 편입
```

보고(스크린샷+위치+직전 행동)를 받으면 먼저 재현을 3종 세트로 압축하고, 2절 기법으로 격리하고, 1절 기준(메커니즘 한 문장)으로 확정한 뒤 수정한다. 수정된 이슈의 재현 절차는 회귀 묶음에 편입한다.

### "버그 아님" 판정도 비용을 들여 확정한다

증상이 보이면 먼저 원본도 그런지 가른다. 의심 루틴의 바이트를 일문판과 검증된 선행 패치본에서 비교해(JP==EN이면 패치 무관) 원본 동작임을 **증거로** 닫는다. 한 사례에서는 관련 씬 전수 확인과 PPU 설정 바이트의 일문판==영문판 비교로 일부 증상을 "원본 동작"으로 판정했다. 한글 폰트의 획 밀도가 원본의 잠재 특성을 더 드러냈을 뿐이었다. 인상으로 닫은 "버그 아님"은 라운드마다 되살아난다.

### 선제 버그 사냥 — 스트레스 테스트

이슈가 보고되기를 기다리지 않고 의도적으로 한계를 찌른다. 팬번역 QA의 관행이다.

- **텍스트는 도달해야 검증된다.** 전 구간 도달이 가능해진 뒤에 베타를 시작하고, 분기 체크리스트를 만들어 풀 플레이스루로 모든 분기·캐릭터 루트의 텍스트에 실제로 도달한다.
- **오버플로우를 의도적으로 유발한다** — 최장 이름·최장 아이템 조합 등 폭이 최대가 되는 입력을 골라 박스 한계를 찌른다.
- **더미 최대 길이 스트레스** — 모든 항목을 한도 길이의 더미 텍스트로 채운 스트레스 빌드로 전수 한계를 한 번에 확인한다.
- **에뮬레이터와 실기(또는 사이클 정확 에뮬레이터) 양쪽**에서 돌린다(4절 계층 분리).
- **테스터 보고 양식을 미리 합의한다** — 스크린샷·위치·직전 행동·세이브스테이트. 양식이 곧 재현 압축의 입력이다.

### 회귀 묶음

- **빌드 린트가 가장 싼 자동 회귀다** — 포인터 범위·길이 초과·인코딩 가능 여부의 기계 검사를 빌드에 내장한다(build-and-verify.md §1).
- **HITL로 확정한 규칙은 즉시 검사 도구로 옮긴다** — 사람 관측으로 확정한 대사 폭 규칙을 검사 도구로 만들어 전 씬 전수를 회귀 게이트로 돌린 사례가 있다.
- **세이브스테이트 라이브러리** — 수정된 버그마다 재현 스테이트를 보존한다. 함정: 스테이트는 에뮬레이터 버전에 종속된다. 버전을 함께 기록하고, 패치의 RAM 레이아웃이 바뀌면 구 스테이트는 비호환이 된다.
- **입력 무비 라이브러리·체크리스트 회귀** — 무비는 짧게, 체크리스트는 라운드마다 같은 경로로(build-and-verify.md §4).

역할 분담: 자동 검사(빌드 린트·전수 도구)는 빌드가, 플레이 재현·스크린샷 대조는 테스터(사람 또는 에뮬레이터 자동화)가, 격리·확정은 디버깅 담당이 맡는다.

## 6. 함정 모음

- **검증 안 된 이름이 분석을 왜곡한다.** 한 사례에서는 어떤 코드에 붙인 추측성 이름이 분석을 엉뚱한 방향으로 오도했고, 디버거 관측 한 번으로 오분류가 드러났다. 역할이 확정되기 전의 코드·데이터에 가설적 함수명을 붙이지 말고 (가설) 태그로 부른다.
- **정황 일치는 확정이 아니다.** 관측 5건과 전부 정합하던 가설이 제거 실험 1개로 기각된 사례(2절 ⑥)를 기억한다. 확정 조건은 정합의 개수가 아니라 격리 실험 통과다.
- **같은 가설의 변형을 반복하지 않는다.** 렌더 경로를 디스어셈블하지 않은 채 변형 시도만 21회 실패한 사례가 있다. 시도가 두세 번 빗나가면 멈추고 해당 경로를 디스어셈블한다. 실패 기록을 보존한 덕에 원인(렌더러의 ASL 더블링) 발견 시 21회 전체가 한 번에 설명됐다.
- **증상이 겹치면 어떤 수정도 공정한 평가를 받지 못한다.** 한 사례에서는 올바른 수정(ASCII 전각 변환)이 한 번 기각됐다가 재도입됐다. 기각 당시의 실패는 별개 버그(CD 상태 오염 추정)의 증상과 겹쳤던 것으로 재평가됐다. 두 버그가 동시에 살아 있으면 먼저 하나를 격리해 죽인 뒤 평가한다(델타 디버깅의 비모호성 조건 위반이기도 하다).
- **블랙박스 적용 금지.** 선행 패치의 diff를 이해 없이 통째로 적용하면 무관한 변경이 따라 들어온다(영문 패치 diff를 통째로 적용했다가 일문 텍스트가 깨지고 불필요 패치 수백 건이 따라온 사례). 반대 방향의 함정도 있다 — 선행 패치의 코드를 통째로 가져오면 그 코드가 의존하는 암묵적 전역 상태·초기화 순서는 따라오지 않아 깨진다. 명시적 입출력 인터페이스만 가져오고 숨은 상태 의존(전역 변수·선행 초기화·부수 효과)은 전수 검증한다. 주석도 검증 대상이다. "원본은 리틀 엔디언"이라는 틀린 주석이 바이트 순서 버그를 만든 사례가 있다. 문맥만으로 비슷한 한자를 식별하지 않는다(한 사례에서는 글리프를 실측하지 않고 문맥으로 추정한 한자가 오답이어서 닮은 글자를 오식별했다).
- **닮은 코드를 그 함수로 단정하지 않는다.** 비슷하게 생긴 루틴을 라이브러리 함수로 단정해 오도된 사례가 있다. 호출처 17곳의 일관된 사용 패턴이 단일 디스어셈블 인상보다 강한 증거다.
- **계산식·훅 사이트의 용도를 문맥으로 추측하지 않는다.** 좌표·인덱스 계산식의 의미를 주변 코드 인상으로 단정하면 틀린다(한 사례: 텍스처 좌표인 줄 알았으나 화면 좌표였음). 훅 사이트도 마찬가지로 주소만 보고 용도를 추측하면 엉뚱한 동작을 망가뜨린다(폰트 렌더인 줄 알았으나 UI 프레임 DMA였던 사례 — 설치 즉시 되돌렸다). 값의 의미와 코드의 역할은 disasm·메모리 검사·실행 추적으로 확정한 뒤 손댄다. 미검증 이름표를 붙이지 않는 규율(위 첫 항목)의 연장이다.
- **shotgun debugging 금지.** 버퍼 임의 확대, 무관한 초기화 추가 같은 무작위 변경으로 우연한 소멸을 기대하지 않는다. 우연히 사라진 버그는 원인이 살아 있다.
- **동시 다중 변경 금지.** 한 번에 한 변경, 효과 없으면 되돌리고 다음. 두 변경을 함께 넣으면 어느 쪽의 효과인지 알 수 없다.
- **실패한 수정도 기록한다.** 한 사례에서는 "시도한 수정(실패)" 절에 실패 수정 2건(클리어 루틴을 게임 데이터 영역에 배치, VDP 타이밍 부족)을 보존해 같은 길로의 재진입을 막았다. 실패는 기각 가설과 같은 자산이다.
