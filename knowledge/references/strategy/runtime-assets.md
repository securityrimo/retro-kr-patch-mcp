# 런타임 자산 도달성 전략

폰트·스크립트·그래픽·매핑 테이블·압축 블록이 빌드 산출물에 들어갔다는 사실은 게임이 그것을 사용한다는 증거가 아니다. 자산이 **저장 → 탐색 → 적재·변환 → 상주 → 소비** 경계를 실제로 통과하는지 조건부로 판정한다.

자산 형식·배치법·로더 구현·검증 도구는 대상 프로젝트에 맞춰 선택한다. 각 단계의 내부 유효성은 관련 전략으로 검증하고, 여기서는 한 단계의 출력이 다음 단계의 입력으로 이어졌는지 확인한다.

## 1. 발동 조건

다음 변경이 하나라도 있으면 이 게이트를 켠다.

- 원본에 없던 런타임 자산을 추가한다.
- 자산이 기존 슬롯·파일 extent·뱅크·섹터를 넘어 성장하거나 다른 위치로 이동한다.
- 압축·패킹·식별 메타데이터를 바꾸어 로더 입력이나 출력이 달라진다.
- 상주 방식을 바꾸거나 새 버퍼·캐시·DMA·overlay 경로를 사용한다.
- 위치와 크기는 같아도 소비자가 해석할 인코딩·셀·인덱스·포맷 규칙을 바꾼다.

위 조건에 해당하지 않고, 같은 지원 리비전·동일 소비 동작의 기존 증거가 변경 전후 체인이 같음을 직접 보장하는 단순 교체에는 별도 게이트를 만들 필요가 없다. 다른 리비전·로더·상주 방식이나 범위가 더 좁은 과거 증거로 생략하지 않는다.

## 2. 연결 판정

`저장 → 탐색 → 적재·변환 → 상주 → 소비`는 판정을 위한 의존 모델이다. 직접 매핑된 자산처럼 여러 단계가 합쳐지거나 변환 단계가 없는 구조도 있다. 해당하지 않는 단계는 근거와 함께 제외한다. 기존 증거는 같은 지원 리비전·자산 identity·소비 경로·상태 수명 조건에 적용되고 이번 변경이 그 연결을 건드리지 않았을 때만 재사용한다. 하나의 트레이스가 여러 연결을 함께 입증해도 된다.

변경이 닿은 경계에서는 다음을 확인한다.

1. 빌드가 만든 자산의 정체·위치·크기와 런타임 탐색 정보가 같은 대상을 가리키는가?
2. 실제 읽기 경로가 그 대상을 선택하고, 소유 전략에서 검증한 적재·변환 결과를 다음 메모리 경계에 전달하는가?
3. 전달된 자산이 필요한 용량과 수명을 가지며, 관련 상태 전이 뒤에도 소비 시점까지 유효한가?
4. 필요한 소비 경로가 그 자산을 같은 포맷·인코딩·인덱스 규칙으로 읽고, RAM·VRAM 또는 최종 출력에서 결과가 연결되는가?

부팅 성공, 이미지 내부 바이트 일치, 한 화면의 우연한 정상 표시는 각각 변경된 연결 전체의 증거가 아니다. 새 증거의 범위는 변경이 닿는 경계, 소비 경로, 자산 수명을 바꿀 수 있는 상태 전이로 한정한다.

증명 대상이 적재·업로드·초기화·캐시 갱신이면 그 경계 뒤에서 만든 상태는 앞 경계를 통과했다는 증거가 아니다. 상태 저장 자체를 금지하지 않되, 같은 빌드의 경계 전 상태에서 이어가거나 새 실행으로 해당 경계를 다시 통과한다.

## 3. 판정과 다음 행동

- **통과** — 변경이 닿은 연결이 증거로 이어지고 관련 상태 전이 뒤에도 유지된다. 이 판정을 호출한 전략에 결과를 돌려준다.
- **실패** — 처음 끊긴 연결을 소유한 전략으로 돌아간다. 앞 단계의 통과 증거는 유지하고 실패하지 않은 계층까지 다시 설계하지 않는다.
- **불명확** — 마지막으로 확인된 경계와 처음 확인되지 않은 경계를 기록한다. 구현 전 판정이 필요한 위험이면 `references/strategy/poc.md`로 보내 더 작은 프로브를 설계한다.

## 4. 다른 문서와의 경계

- 자산의 저장 공간·포인터·훅은 `references/strategy/reinsertion.md`가 소유한다.
- 압축 식별과 해제·재압축 라운드트립, 변환 결과의 정확성은 `references/strategy/compression.md`가 소유한다.
- 글리프 수요·인코딩·상주 방식은 `references/strategy/font-strategy.md`가 소유한다.
- 베이크드 그래픽의 복원·조판·재인코딩은 `references/strategy/graphics-text.md`가 소유한다.
- 위험을 먼저 증명할 프로브는 `references/strategy/poc.md`, 실패 계층의 격리는 `references/strategy/debugging.md`가 소유한다.
- 최종 이미지 무결성·패치 산출물·회귀 판정은 `references/strategy/build-and-verify.md`가 소유한다.

구체적인 자산 목록, 파일 경로, 매니페스트 필드와 증거 기록 형식은 필요할 때 대상 프로젝트의 문서와 구현이 정한다.
