# 번역 자산 시행 규약

번역 전략이 요구하는 원문 보존, 검토 상태, 가역 제어코드와 빌드 입력 자격을 파일과 데이터에서 아래 의미와 검증으로 보존한다. 프로젝트가 이미 동등한 스키마나 작업 관리 체계를 갖고 있으면 그대로 유지한다. JSON·디렉토리명·필드명 자체는 불변식이 아니다.

## 목차

1. 필수 의미
2. 선택 JSON 프로필
3. 제어코드 토큰
4. 진행 상태 의미
5. 빌드 입력 자격

## 1. 필수 의미

번역 자산 표현은 최소한 다음을 구분해야 한다.

- 원본 이미지와 추출 영역을 식별하는 출처
- 재추출 뒤에도 대조할 수 있는 안정적인 엔트리 식별자
- 원본 바이트와 그 바이트에서 파생한 디코딩 결과
- 번역자가 수정하는 역문
- 미번역·작업 중·검토 필요·사람 판단 필요·빌드 적격을 구분하는 상태
- 의역·표기 결정·미확정 해석의 근거
- 재삽입에 필요한 구조 정보 또는 이를 소유한 분석·빌드 데이터에 대한 안정적인 연결

번역 자산에 포함된 원본 바이트·원문·구조 정보와 연결 식별자는 보호 영역이다. 번역 작업은 역문·검토 상태·판단 근거만 바꾸며, 보호 영역이 달라지면 재추출 또는 명시적인 구조 변경으로 처리한다.

문자 매핑·제어코드 원시 사양·포인터 카탈로그·재삽입 정책의 의미는 `references/conventions/data-formats.md`가 소유한다. 번역 자산은 안정 ID와 원본 식별 정보로 연결하거나 기존 통합 자산 안의 보호 정보로 함께 보존할 수 있다. 어느 경우에도 같은 값을 독립적으로 편집되는 두 사본이 소유하지 않는다.

Agent 번역·리뷰 배치에도 안정 ID, 원문과 raw byte, 가역 token, 추출 기준선과 적용 컨텍스트·용어·문체 SSOT의 스냅샷 식별자를 명시한다. 배치 파일이 이 값을 복사해 보여 주더라도 각 기준선은 한 곳에서만 관리하며, Agent 응답으로 보호 영역을 덮어쓰지 않는다.

병합·재삽입 직전에는 배치에 복사된 source metadata끼리 비교하는 데 그치지 않고, 지원 원본 또는 그 원본에서 다시 만든 현재 추출 기준선과 source identity·raw byte를 대조한다. 현재 기준선과 다르면 stale 결과로 차단하고 재추출 영향 범위를 먼저 판정한다.

### 승인 용어·문체 결정

용어·반복 표현·문체의 승인 SSOT는 최소한 결정의 안정 ID, 원어 대상 또는 상황, 승인 결과, 적용 작품·리비전·장면·화자 관계와 서사 시점의 범위, 근거와 적용 한계, 승인·미확정·충돌 상태를 구분한다. 같은 원어에 범위가 다른 결정을 둘 수 있으므로 원문 문자열만을 전역 유일 키로 쓰지 않는다. 새 근거로 결정이 바뀌면 이전 결정을 조용히 덮지 않고 영향받는 배치가 어느 기준선을 다시 검증해야 하는지 식별할 수 있어야 한다. 파일 수·표 구조·필드명은 프로젝트가 정한다.

## 2. 선택 JSON 프로필

다음 JSON은 필수 의미와 보호 경계를 설명하기 위한 선택 예시다. 기존 구조가 같은 의미와 검증을 제공하면 이 형식으로 바꾸지 않는다.

```json
{
  "table_id": "dialog_group",
  "source": {
    "pointer_table": "0x000000",
    "entry_count": 2,
    "base": "0x000000",
    "method": "pointer_table",
    "terminators": ["FE"]
  },
  "entries": [
    {
      "entry_id": 0,
      "ptr_value": "0x0000",
      "file_offset": "0x000000",
      "raw_hex": "...FE",
      "text": "원문{end}",
      "ko": "역문{end}",
      "status": "in_progress",
      "notes": null,
      "flags": []
    },
    {
      "entry_id": 1,
      "ptr_value": null,
      "file_offset": null,
      "raw_hex": null,
      "text": null,
      "ko": "",
      "status": "untranslated",
      "notes": null,
      "flags": ["null_entry"]
    }
  ]
}
```

- `table_id`와 `entry_id`는 안정적으로 유지한다. 빈 슬롯도 엔트리 순서와 개수에 포함한다.
- `raw_hex`와 `text`는 추출기가 채우는 보호 필드다. `ko`, `status`, `notes`는 번역 단계의 수정 가능 필드다.
- 중복 포인터와 미해독 바이트는 `flags`로 나타내되, 프로젝트가 다른 표현을 쓰면 동일한 의미와 라운드트립을 보존한다.
- 단계 사이에 교환하는 형식은 같은 스키마 정의를 공유하고, 알 수 없는 필드·상태·토큰을 조용히 무시하지 않는다.

## 3. 제어코드 토큰

사람이 편집하는 원문과 역문에는 제어코드를 역파싱 가능한 토큰으로 표현한다. 선택 가능한 문법 예시는 다음과 같다.

- 인자 없음: `{br}`, `{wait}`, `{end}`
- 인자 있음: `{delay:1E}`, `{face:02,05}`
- 미해독 코드: `{op27:0B}`
- 미해독 원시 바이트: `{raw:XX}`

토큰 문법은 바이트열과 전단사여야 한다. 리터럴 중괄호와 충돌하면 이스케이프를 정의한다. 토큰의 보존·이동·재계산·대상 의미 변환·수정 금지 정책은 `references/strategy/text-extraction.md` §4.4가 판정하며, 빌드 입력 검사는 그 정책에 맞춰 집합·순서·파라미터와 변환 결과를 대조한다. 모든 토큰을 원문과 1:1로 강제하지 않는다. 토큰 이름과 구두점은 프로젝트가 바꿀 수 있지만 가역성과 검증 가능성은 유지한다.

## 4. 진행 상태 의미

작업 관리 체계는 최소한 다음 상태 의미를 구분해야 한다. 표의 상태명은 선택 예시다.

| 예시 상태명 | 의미 | 다음 상태의 최소 조건 |
|---|---|---|
| `untranslated` | 추출 기준선, 역문 없음 | 작업 착수 |
| `in_progress` | 번역 또는 수정 중 | 전 엔트리 번역과 자동 검사 통과 |
| `needs_review` | 교차 검토 필요 | 해당하는 독립 교차 검토와 기계 판정 항목 해소 |
| `needs_human_review` | 사람 판단 필요 | 사람의 판단·승인과 근거 기록 |
| `complete` | 자동 검사와 필요한 사람 승인을 통과한 빌드 적격 상태 | 결함 발견 시 앞 상태로 강등 |

상태를 어디에 어떻게 표현할지는 기존 프로젝트 구조가 정한다. 파일 단계와 엔트리 상태를 함께 두면 서로 어긋나지 않는지 검증하고, 가능하면 하나를 다른 하나에서 계산해 중복 소유를 줄인다.

## 5. 빌드 입력 자격

빌드는 개발·PoC와 배포 후보의 입력 정책을 구분한다.

- **개발·PoC 빌드**는 미적격 자산을 명시적으로 선택할 수 있지만 비배포 산출물임을 표시하고, 선택하지 않은 미적격 엔트리는 추출 기준선의 원문을 보존하거나 빌드를 실패시킨다. 미적격 번역을 조용히 섞지 않는다.
- **배포 후보 빌드**는 선언한 현지화 범위에서 적격 판정을 받은 번역만 소비한다. 범위 밖에 의도적으로 남기는 원문·브랜딩·기호는 승인 예외로 기록한다.

배포 대상 엔트리의 적격 판정에는 최소한 다음이 포함된다.

- 보호 필드가 추출 기준선과 일치한다.
- 번역 누락, 승인되지 않은 원문 문자 잔존, 알 수 없는 상태가 없다.
- 제어코드 토큰이 파싱 가능하고 적용된 보존·이동·재계산·대상 의미 변환 정책을 만족한다.
- 글리프 커버리지와 길이·레이아웃 검사가 통과한다.
- 사람 판단이 필요한 항목은 결정과 근거가 남아 있다.

검사 실패나 새 증거가 생기면 적격 상태를 취소하고 필요한 검토 단계로 되돌린다. 상태 이름이나 저장 위치가 아니라 이 판정 결과가 빌드 입력을 결정한다.

병렬·순차 배치를 합칠 때는 각 배치가 사용한 추출·컨텍스트·용어·문체 기준선과 보호 필드가 현재 승인 기준선과 일치하고, 안정 ID의 누락·중복·충돌이 없음을 검증한다. 기준선이 바뀌었으면 영향받는 결과를 다시 검증한다. 같은 엔트리에 서로 다른 역문·상태·근거가 들어오면 입력 순서나 마지막 writer로 고르지 않고 명시적 검토 대상으로 남긴다.
