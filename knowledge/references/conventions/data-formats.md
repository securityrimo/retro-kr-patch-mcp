# 분석·빌드 데이터 시행 규약

텍스트 추출·번역·재삽입·폰트 빌드 사이에서 교환하는 기계 판독 데이터는 전략이 요구한 의미와 통과 조건을 아래 구분과 검증으로 보존한다. 프로젝트에 이미 동등한 형식이 있으면 그대로 유지하며, 직렬화 예시와 필드명은 선택 프로필일 뿐 불변식이 아니다. 텍스트 맵·PoC·HITL·QA처럼 사람이 판정을 추적하는 기록은 `references/conventions/project-records.md`를 따른다.

## 목차

1. 공통 규칙
2. 문자 매핑 테이블
3. 제어코드 사양
4. 포인터 카탈로그
5. 재삽입 정책 데이터
6. 폰트 렌더 프로파일
7. 검증 바인딩

## 1. 공통 규칙

- 같은 산출물을 여러 형식으로 중복 소유하지 않는다. 사람이 보는 표와 도구 입력이 모두 필요하면 한쪽에서 다른 쪽을 생성한다.
- 원본 좌표·원본 바이트·포인터 원시값처럼 재현에 필요한 값은 보호 데이터로 취급한다. 번역문이나 표현 파라미터와 섞어 수작업으로 고치지 않는다.
- 구조화 데이터는 타입과 허용 값을 검증하고, 알 수 없는 필드·상태·열거값을 조용히 무시하지 않는다.
- 오프셋·주소·바이트열은 사람이 검토할 때 진법과 폭을 알아볼 수 있게 직렬화한다. 내부 계산에서는 명시적 정수 폭과 엔디언을 사용한다.
- 엔트리와 영역에는 재추출 뒤에도 대조 가능한 안정적인 키를 둔다. 배열 순번만으로 식별하면 빈 슬롯 추가나 정렬 변경 때 이력이 끊긴다.
- 고정 원본에서 전수 확인한 모집단은 정확한 상수·카탈로그로 표현한다. 휴리스틱 재탐지 결과를 빌드가 조용히 채택하지 않으며, 확정 목록과의 차이는 원본 리비전 또는 사양 재검토가 필요한 오류로 보고한다. 어떤 값을 고정하고 파생할지는 `references/strategy/initial-survey.md` §3.1이 판정한다.

구조화 산출물의 형식 선택과 기계 판독 입출력 원칙은 `references/conventions/project-conventions.md` §3.2를 따른다. 아래 JSON은 필드 의미를 설명하기 위한 선택 예시다. 기존 구조나 다른 직렬화가 같은 의미와 검증을 제공하면 변환하지 않는다. 번역자가 편집하는 원문·역문·상태·토큰·승인 정보는 `references/conventions/translation-artifacts.md`가 소유한다. 분석·빌드 데이터와 번역 자산은 안정 ID와 원본 식별 정보로 연결하며, 번역 자산이 문자 매핑·포인터·재삽입 정책을 다시 정의하지 않는다.

## 2. 문자 매핑 테이블

커스텀 인코딩의 바이트열과 문자·문자열 대응은 프로젝트가 이미 채택한 표현을 따른다. 별도 표현이 없고 단순 대응표가 충분할 때는 다음과 같은 `.tbl` 형식을 선택할 수 있다.

```text
41=あ
42=い
F0=のは
FF00=[END]
```

- 왼쪽은 공백 없는 16진 바이트열이며 바이트 경계를 보존한다.
- 오른쪽은 한 글자뿐 아니라 DTE/MTE가 확장하는 여러 글자일 수 있다.
- 도구가 제어코드 이름을 테이블에 함께 두면 대괄호 표기를 쓸 수 있다. 추출 산출물로 내보낼 때는 `references/conventions/translation-artifacts.md`의 가역 토큰으로 정규화한다.
- 미해독 바이트를 문자로 추정해 채우지 않는다. 조사용 덤프에서는 `<XX>`처럼 원시값이 보이는 표기를 쓰고, 편집 가능한 번역 자산에서는 가역 raw 토큰으로 보존한다.
- 테이블이 바뀌면 기존 번역 자산의 보호 필드를 손으로 고치지 않고 재추출해 차이를 검토한다.

`.tbl`을 쓰지 않는 구현도 디코드와 인코드가 같은 매핑 정의를 공유하고, 다대일·일대다 매핑과 미해독 바이트를 손실 없이 표현해야 한다.

## 3. 제어코드 사양

제어코드 사양은 추출기와 재삽입기가 함께 읽는 단일 데이터로 둔다. 최소 의미는 다음과 같다.

- 코드 바이트 또는 식별 패턴
- 인자 개수와 각 인자의 폭·엔디언·보존 방식
- 문자열이나 메시지를 끝내는지 여부
- 사람이 편집하는 토큰 이름과 의미
- 적용되는 파일·영역·렌더러 범위
- 원본 raw에 포함할 인자와 재배치 때 다시 계산할 인자의 구분

필요한 의미를 설명하면 다음과 같은 구조화 레코드가 된다.

```json
{
  "code": "FF05",
  "name": "delay",
  "arguments": [{ "width": 2, "endian": "big", "policy": "preserve" }],
  "terminates": false,
  "scope": "dialog"
}
```

의미가 아직 불명확해도 코드 경계와 인자 폭은 기록한다. 이때 중립적인 이름을 쓰고 바이트열을 통째로 보존한다. 의미 추정이 바뀌어도 파싱 경계가 흔들리지 않게 하는 것이 우선이다.

## 4. 포인터 카탈로그

고정 원본 리비전에서 포인터 모집단을 전수 확인해 카탈로그 방식을 채택한 경우, 포인터 카탈로그는 원본 좌표계의 포인터를 새 좌표계에서 다시 쓸 수 있게 하는 전수 목록이다. 모집단의 명시 사양 승격 조건은 `references/strategy/initial-survey.md` §3.1이 소유한다. 카탈로그는 최소한 다음을 복원할 수 있어야 한다.

- 포인터 바이트의 원본 저장 위치
- 폭·엔디언과 주소 기준(파일·RAM·뱅크·테이블 상대)
- 원본 raw 값과 해석한 원본 타겟
- splice나 재배치로 저장 위치 자체가 움직이는지 여부
- 보정 뒤 타겟을 검증할 패턴이나 경계 조건
- 문자열 중간 포인터라면 새 번역에서 같은 구조 지점을 찾을 앵커

선택 가능한 JSON 표현 예시는 다음과 같다.

```json
{
  "pointer_id": "dialog/0001/ref-0",
  "storage_offset": "0x000000",
  "width_bits": 16,
  "endian": "little",
  "basis": "ram",
  "base": "0x00000000",
  "raw_value": "0x0000",
  "target_offset": "0x000000",
  "storage_moves": true,
  "target_check": { "kind": "prefix", "bytes": "FF03" },
  "anchor": null
}
```

필드명이나 JSON 사용은 선택 사항이다. 확정 리비전 카탈로그에서는 포인터를 splice 뒤 버퍼에서 다시 읽지 않도록 원본 raw 값과 저장 좌표를 고정하고, 반복 빌드는 승인된 카탈로그를 소비한다. 탐색 스캐너는 생성·갱신 후보만 만들며 새 후보를 조용히 채택하지 않는다.

반대로 여러 리비전, 런타임 생성 데이터, 아직 범위가 확정되지 않은 구조는 고정 카탈로그로 가장하지 않는다. 명시 스키마와 항목 수로 결정적으로 순회할 수 있는 포인터 테이블은 반복 빌드에서 직접 파싱해도 된다. 어느 경로든 휴리스틱 후보를 검토 없이 보정 대상으로 삼거나, 확정된 포인터를 대응 처리 없이 버리면 빌드를 실패시킨다.

## 5. 재삽입 정책 데이터

엔트리별 정책은 번역자가 편집하는 문장과 분리해 빌드 데이터로 관리한다. 번역 자산을 확장한다면 안정적인 엔트리 키로 연결하고, 보호 필드로 취급한다.

표현 형식과 무관하게 다음 의미가 필요하다. 아래 필드명은 예시다.

| 예시 필드명 | 의미 |
|---|---|
| `mode` | `fixed`, `relocate`, `grow` 가운데 선택한 길이·배치 정책 |
| `overflow_policy` | 초과 시 실패, 승인된 번역 조정, 재배치 등 선택한 해소 경로. 자동 부분 절단은 허용하지 않는다 |
| `pad_byte` | 대상 엔진이 무동작으로 처리한다고 확인한 패딩 바이트열 |
| `pad_position` | 종료 시퀀스 전·후 또는 고정 슬롯 잔여부 가운데 확인된 삽입 위치 |
| `terminator_policy` | 종료자 보존·재생성·부재 등 경계 처리 방식 |
| `alignment` | 엔트리나 블록이 유지해야 하는 바이트 정렬 |

패딩·종료자·정렬 값은 전역 기본값을 둘 수 있지만, 경계 모델이나 렌더러가 다른 영역은 별도 프로필로 나눈다. 위험한 값이 누락됐을 때 임의 기본값으로 진행하지 않고 해당 엔트리의 경계 모델을 확정하도록 실패시킨다.

## 6. 폰트 렌더 프로파일

렌더 타깃이 하나면 기존 프로젝트의 빌드 입력으로 표현 파라미터를 전달하면 충분하다. 본문·UI·퍼즐처럼 타깃마다 값이 갈려 어느 설정이 어느 소비 경로를 지배하는지 불명확해질 때만 타입 검증되는 프로파일로 분리한다.

프로파일은 타깃별로 다음 의미 가운데 실제 렌더 경로에 필요한 것만 선언한다.

- 소스 글리프의 프로젝트 상대 식별자와 정확한 버전·라이선스 근거
- 타깃 렌더 경로와 셀·기준선·여백
- 래스터화·이진화·윤곽 처리처럼 결과 픽셀을 바꾸는 파라미터
- 비트 깊이·팔레트·패킹·서브타일 순서처럼 게임 데이터 표현을 정하는 값
- 같은 프로파일로 생성한 비교 결과와 실제 빌드 산출물을 연결하는 식별자

필드명과 직렬화 형식은 프로젝트가 정한다. 서로 다른 렌더러의 필드를 거대한 선택적 스키마 하나에 합치지 말고 타깃 종류별로 검증한다. 값의 덮어쓰기 순서가 필요하면 프로젝트 안에서 한 번만 정의하고, 알 수 없는 타깃·필드·타입은 오류로 처리한다. 검토 결과와 빌드는 같은 해결 결과를 사용해야 한다.

메모리 주소·뱅크·인코딩 경계처럼 역공학으로 확정한 구조값은 이 프로파일에 섞지 않는다. 표현을 자주 바꾸는 값과 확정 구조값을 분리한다.

## 7. 검증 바인딩

데이터 표현은 다음 검사를 기계적으로 판정할 수 있어야 한다.

- 번역 자산의 원문 표기 재인코딩 결과가 보호된 원본 바이트와 같고, 그 바이트가 원본 파일의 기록 위치와 같다.
- 번역 자산의 안정 ID와 추출 기준선이 해당 원본 엔트리 및 해당하는 재삽입 데이터에 모호성 없이 연결된다. 필수 연결의 누락·ID 충돌·적용 범위가 다른 연결은 없고, 공유 엔트리·중복 참조 같은 의도된 관계는 원본 관계로 표현된다.
- 제어코드 사양의 모든 코드가 인자 폭만큼 정확히 소비되며, 미해독·잘림이 0건으로 수렴한다.
- 포인터 raw 값을 기준 주소 규약으로 해석하면 기록한 원본 타겟이 나오고, 역변환하면 같은 raw 값이 나온다.
- 재삽입 뒤 모든 카탈로그 포인터가 유효 범위와 타겟 패턴을 만족하고, 카탈로그 엔트리 수가 예고 없이 변하지 않는다.
- 폰트 프로파일의 모든 타깃이 빌드 산출물과 비교 시트에서 같은 해석 결과를 사용한다.

구체 테스트 코드는 프로젝트 언어와 기존 테스트 구조를 따른다. 필드명을 바꾸면 테스트도 같은 커밋에서 바꾸며, 호환을 위해 알 수 없는 필드를 묵인하는 대신 명시적인 스키마 버전 변환을 둔다.
