# 프로젝트 시행 컨벤션

한글 패치 전략의 불변식과 판정 기준을 프로젝트 구조·인터페이스·검증으로 옮길 때 아래 규약을 지킨다. 먼저 기존 저장소의 구조와 도구체인을 확인하고 동등한 장치가 있으면 유지한다. 디렉토리, 명령, 언어, 라이브러리와 제품은 고정하지 않으며, 플랫폼별 기술 세부는 `references/platforms/`의 해당 문서에서 확인한다.

## 1. 저장소 역할과 단일 소유권

저장소를 만들거나 확장하기 전에 기존 빌드·테스트·문서·자산 흐름을 조사한다. 이미 같은 책임을 맡는 위치와 검증이 있으면 새 구조를 병설하지 않는다. 새 구획이 필요할 때는 이름보다 다음 역할의 소유권을 먼저 분리한다.

- 재현 가능한 빌드에 들어가는 소스 입력
- 원본에서 다시 만들 수 있는 중간·출력 산출물
- 역공학의 현재 결론과 그 근거
- 번역 자산과 검토 상태
- 런타임·QA 증거와 이슈 기록
- 저장소 밖에서 주입하는 원본 이미지와 로컬 캐시

같은 사실이나 상태를 여러 위치가 소유하지 않는다. 사람이 보는 요약과 기계 입력이 모두 필요하면 하나에서 다른 하나를 생성하거나 소유 위치를 명시적으로 가리킨다. 프로젝트 진입 문서는 현재 구조, 재현 진입점, 핵심 문서와 알려진 함정의 포인터만 제공하고 분석 본문이나 진행 상태를 복제하지 않는다.

탐색용 프로브와 일회성 변환은 주 빌드 경로 밖에 둘 수 있다. 반복 빌드에 필요해지는 순간 버전 고정, 명시적 입출력, 오류 전파와 테스트를 갖춘 정식 구성요소로 승격한다. 승격하지 않은 임시 산출물이나 수동 편집 결과를 빌드의 유일한 입력으로 삼지 않는다.

번역 자산의 의미와 검토 상태는 `references/conventions/translation-artifacts.md`, 조사·PoC·QA 기록의 의미와 관계는 `references/conventions/project-records.md`가 소유한다.

## 2. 주 빌드 경로와 코드 경계

### 2.1 단일 주 파이프라인 원칙

추출·변환·재삽입·패치 생성은 **하나의 문서화된 주 경로**에서 원본부터 다시 실행할 수 있어야 한다. 외부 구성요소를 호출해도 되지만 채택 버전, 입력, 출력, 옵션, 실패 조건과 결과 검증을 이 경로가 소유한다. 보조 경로가 같은 산출물을 독립적으로 수정하거나 수동 결과를 이어받아서는 안 된다.

주 경로는 다음 요건을 만족한다.

1. 오프셋·크기·정수 폭·엔디언을 명시해 바이트 단위 결과를 재현한다.
2. 같은 선언 입력에서 같은 의미와 검증 결과를 가진 산출물을 만든다.
3. 인코딩 누락, 손실 변환, 범위 초과, 알 수 없는 상태와 검증 실패를 조용히 건너뛰지 않고 실패로 반환한다.
4. 다른 환경에서도 필요한 버전과 입력을 식별하고 같은 진입점으로 실행할 수 있다.

기존 프로젝트가 여러 구성요소를 묶은 동등한 재현 경로를 이미 갖고 있다면 하나의 언어나 하나의 실행 파일로 합칠 필요는 없다. 중요한 것은 책임과 실패가 한 빌드 그래프에서 추적되는 것이다.

### 2.2 단순한 구조로 시작한다

새 프로젝트는 아직 증명되지 않은 경계를 미리 분리하지 않는다. 한 구성요소 안에서 역할을 모듈로 나누고, 테스트와 사용 증거가 경계를 보여 줄 때만 별도 구성요소로 분리한다. 기존 프로젝트는 규모만을 이유로 합치거나 쪼개지 않는다.

### 2.3 재사용 경계의 승격 조건

게임 특이 지식 없이 동작하고 두 번째 소비처가 확인된 기능은 재사용 단위로 분리할 수 있다. CPU 명령 처리, 컨테이너 파싱, 주소 변환, 그래픽 변환처럼 입력·출력과 불변식을 게임 좌표 없이 설명할 수 있어야 한다. 게임 이름, 원본 리비전, 인코딩 테이블, 포인터 주소, 훅 위치처럼 프로젝트 좌표가 들어가면 게임 특이 구현에 남긴다.

분리 전후에 같은 테스트 벡터와 소비 결과가 유지되어야 한다. “언젠가 재사용할 수 있다”는 기대만으로 공통 계층을 만들지 않는다.

### 2.4 기계어 생성·해석 검산 규칙

패치가 고정된 짧은 명령열을 넘어 기계어를 생성·이동하거나, 제어 흐름·참조의 완전성을 주장한다면 대상 instruction과 mode를 같은 의미 모델로 assemble·disassemble할 수 있어야 한다. 특정 제품이나 라이브러리를 요구하지 않지만 다음 요건은 지킨다.

- 패치가 사용하는 instruction·addressing mode를 단발 opcode helper의 묵시적 fallback 없이 완전하게 표현한다.
- 지원하지 않는 opcode·mode·잘린 명령을 데이터나 성공으로 오인하지 않고 실패시킨다.
- 최종 배치 뒤 생성 영역을 다시 disassemble해 명령 경계, operand, branch target, delay slot·literal pool과 복귀 경로를 검산한다.
- 원본 명령을 이동하면 원래 효과와 live-state 불변식을 새 위치에서도 비교한다.
- 참조·호출 그래프의 완전성을 주장하면 선언한 entry point, bank·overlay·mode 범위와 미해석 영역을 함께 보고한다.

지원 완전성의 분모는 패치가 생성·이동하는 instruction·mode와 분석 대상으로 선언한 code 영역이다. literal·embedded data와 canonical alias를 분리해 보고하고, 분모 밖을 조용히 명령이나 동등한 opcode로 흡수하지 않는다.

고정 지원 리비전에서 확인된 짧은 명령열은 expected bytes와 명령 경계 검증이 붙으면 명시 바이트 사양으로 둘 수 있다. 최종 배치에 따라 달라지는 branch 거리·주소·리터럴은 그 사양에 하드코딩하지 않고 배치 결과에서 파생한다.

## 3. 인터페이스와 교환 데이터

### 3.1 실행 인터페이스 요건

구체 명령 이름과 인터페이스 형태는 기존 프로젝트 관례를 따른다. 새 인터페이스가 필요하면 다음만 고정한다.

- 입력 원본, 설정, 번역 자산과 출력 위치를 명시적으로 전달할 수 있다.
- 호스트별 절대 경로나 숨은 전역 상태를 코드에 박지 않는다.
- 성공·실패가 자동화에서 판정 가능하고, 실패에는 대상 영역과 원인이 포함된다.
- 분석 결과는 후속 단계가 안정적으로 소비할 수 있으며, 사람용 출력만을 파싱하게 만들지 않는다.
- 주 빌드 진입점은 필요한 검증을 우회하지 않고 최종 산출물까지 연결한다.

분석과 빌드를 한 인터페이스에 둘지 나눌지는 프로젝트가 정한다. 기존에 동등한 호출 방식이 있으면 새 명명 규칙이나 래퍼를 덧씌우지 않는다.

### 3.2 기계 판독 가능한 입출력

단계 사이의 교환 데이터는 스키마와 허용 값이 검증되는 기계 판독 형식으로 둔다. 직렬화 형식과 필드명은 프로젝트가 선택하되 다음 의미를 잃지 않는다.

- 원본과 영역을 식별하는 출처, 재추출 뒤에도 유지되는 안정 ID
- 추출기가 보호하는 원시 정보와 사람이 수정하는 정보의 분리
- 정수 폭·엔디언·주소 기준·경계·제어 정보의 명시
- 알 수 없는 필드·상태·토큰과 타입 불일치를 오류로 처리하는 검증
- 스키마 변경을 조용히 흡수하지 않는 명시적 버전 전환

추출 결과와 번역 입력은 같은 보호 정보 기준선을 참조한다. 번역 자산의 의미는 `references/conventions/translation-artifacts.md`, 문자 매핑·제어코드·포인터·재삽입 정책·폰트 프로파일의 의미는 `references/conventions/data-formats.md`가 소유한다. 기존 프로젝트에 동등한 형식이 있으면 변환하지 않고 의미와 검증만 대조한다.

자주 조정하는 표현 파라미터와 역공학으로 확정한 구조값은 서로 다른 소유권으로 둔다. 어느 쪽도 여러 설정 파일과 코드 상수가 동시에 소유하지 않는다.

## 4. 외부 구성요소의 재현 조건

외부 프로그램·라이브러리·서비스가 주 빌드 경로의 결과에 영향을 준다면 다음을 프로젝트에 고정한다.

- 결과를 재현할 수 있는 배포처와 버전 또는 커밋
- 선언 입력·옵션·출력과 실패 전파 방식
- 해당 구성요소가 맡은 능력과 적용 범위
- 대상 원본이나 대표 위험 코퍼스에서의 라운드트립·교차 검증 결과
- 주 빌드 경로에서 결과를 독립적으로 검증하는 방법
- 함께 배포하거나 재배포할 때의 라이선스 조건

어떤 구성요소를 고를지는 프로젝트가 현재 환경과 기존 구현을 조사해 판단한다. 이 문서는 후보를 비교하거나 특정 선택을 권하지 않는다. 이미 동등한 재현·검증 장치가 있으면 새 래퍼나 교체를 요구하지 않는다.

## 5. 테스트 정책

### 5.1 라운드트립 우선

컨테이너, 인코딩, 압축, 주소 변환과 렌더 데이터의 첫 게이트는 왕복 불변식이다. `parse → serialize`처럼 무수정 입력을 왕복시키면 가능한 범위에서 원본 바이트와 동일하게 돌아와야 하고, 여러 직렬화가 허용되는 형식은 게임 소비 결과와 보호해야 할 메타데이터가 동일해야 한다. 대표 표본만 통과한 상태를 전수 형식 지원으로 일반화하지 않으며, 배포 범위의 원본과 위험 경계를 포함한 코퍼스로 확장한다.

인코드↔디코드, 압축↔해제, 논리↔물리 주소처럼 역함수가 있는 쌍은 경계값·빈 값·최대 크기·잘린 입력·잘못된 입력을 자동 생성하거나 체계적으로 열거해 검사한다. 오류 입력이 부분 결과를 성공처럼 내보내지 않는지도 같은 게이트에 포함한다.

### 5.2 Expected Write 규칙

패치가 원본 이미지에 가하는 모든 쓰기는 적용 전에 검증 가능한 Expected Write로 계획한다. 구체 API나 자료형은 프로젝트가 정하지만 각 쓰기는 최소한 소유자·의미, 원본 좌표계와 허용 범위, 기대 원본 바이트 또는 검증 가능한 원본 조건, 최종 바이트나 파생 규칙을 가져야 한다.

빌드는 다음을 실패로 처리한다.

- 기대 원본과 다른 입력에 쓰려는 경우
- 서로 다른 소유자의 쓰기가 같은 범위를 예고 없이 덮는 경우
- 보호·비대상·허용 범위 밖을 수정하는 경우
- 등록한 쓰기 합집합으로 설명되지 않는 최종 diff가 생긴 경우
- 포인터·체크섬·메타데이터 후처리가 추적 계층을 우회한 경우

의도적으로 겹쳐야 하는 변경은 적용 순서에 맡기지 않고 하나의 합성 쓰기나 명시된 의존으로 먼저 해소한다. 최종 출력에서는 각 변경 바이트의 소유자와 이유가 하나로 판정돼야 한다. 기존 프로젝트가 tracked image, 변경 영역 매니페스트, 전후 diff 검증 등 동등한 장치를 갖고 있으면 새 쓰기 API를 강제하지 않는다.

큰 file·bank·sector를 한 단계가 재구성한다는 사실은 그 내부 모든 바이트의 쓰기 허가가 아니다. 실제 writable 하위 범위와 보호 소비자를 구분하고, padding 값이나 상위 container 소유권만으로 허용 범위를 넓히지 않는다.

최종 diff 감사는 원본과 출력의 공통 길이뿐 아니라 길이 변경, 추가된 tail과 잘린 범위까지 포함한다. 기대 최종 크기와 모든 변경 영역이 등록한 쓰기·파생 규칙으로 설명되지 않으면 실패시킨다.

### 5.3 통합 테스트

통합 테스트는 선언한 원본과 번역·설정 입력에서 주 빌드 경로를 실행해 최종 산출물까지 확인한다. 최소한 다음을 다룬다.

- 빌드 완주와 출력 크기·헤더·체크섬·컨테이너 유효성
- 번역 자산의 보호 정보, 글리프 커버리지, 길이·레이아웃과 포인터 범위
- 생성한 차분 산출물을 원본에 적용한 결과와 빌드 산출물의 동일성
- 변경한 런타임 경로에 필요한 정적 게이트와 별도 실행 검증의 연결

원본 이미지가 없는 환경에서는 원본이 필요한 검사를 명시적으로 “실행하지 않음”으로 보고하고, 원본 없이 가능한 스키마·번역·단위 검사는 계속 수행한다. 검사가 빠진 상태를 전체 통과로 표시하지 않는다. 실행 검증의 판정 기준은 `references/strategy/build-and-verify.md`, 증거 기록의 의미는 `references/conventions/project-records.md`를 따른다.

## 6. 원본 자산 취급

### 6.1 저작물 비커밋 원칙

원본 ROM·디스크 이미지와 패치가 적용된 전체 출력 이미지, 원본에서 통째로 추출한 그래픽·폰트·오디오·메모리 덤프는 저장소에 커밋하지 않는다. 저장소의 무시 규칙은 프로젝트가 사용하는 실제 입력·출력 위치와 확장자를 빠짐없이 덮어야 한다.

저장소에는 자체 작성 코드·문서·번역 데이터, 원본을 재현하지 않는 배포 산출물, 재생성 절차와 해시처럼 프로젝트가 배포 권리를 확인한 것만 둔다. 제3자 폰트·도구·기준 자산은 정확한 라이선스와 출처를 확인하고 그 조건이 허용할 때만 추적한다. 특정 산출물이 원본을 포함하지 않는다는 사실만으로 배포 가능하다고 단정하지 않고 프로젝트가 적용되는 권리와 배포 조건을 확인한다.

### 6.2 원본 주입과 식별

원본 이미지 위치는 코드에 하드코딩하지 않고 실행 시점의 입력이나 프로젝트가 문서화한 로컬 설정으로 주입한다. 로컬 기본 위치를 둘 수 있지만 다른 환경에서 재정의할 수 있어야 하며, 호스트별 절대 경로나 개인 파일명은 저장소 문서의 전제가 되지 않는다.

빌드는 입력을 읽기 전에 리비전·리전·크기와 강한 식별 해시를 검증하고, 알려진 원본과 다르면 명확히 실패한다. 기대 식별자는 코드나 검증되는 매니페스트 가운데 한 곳만 소유한다. 여러 원본을 지원하면 원본별 좌표·포맷 프로필을 분리하고, 어느 프로필을 골랐는지 산출물에 남긴다.

팔레트, 타일 베이스, 글리프 원본, 베이크드 그래픽처럼 원본에서 얻을 수 있는 빌드 입력은 가능한 한 선언한 원본에서 다시 추출한다. 선행 패치의 추출물과 수동 작업 파일은 비교·부트스트랩 근거로 사용할 수 있지만 빌드의 유일한 진실 원천으로 삼지 않는다. 재생성할 수 없는 로컬 파생 자산이 불가피하면 그 한계, 식별 해시, 필요한 권리와 검증 절차를 프로젝트 문서에 명시하고 누락 시 빌드를 실패시킨다.
